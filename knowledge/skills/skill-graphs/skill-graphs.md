# Skill Graphs — сети связанных файлов знаний

## Кратко

Skill Graph — архитектурный паттерн управления контекстом: навигация агента по знаниям отвязана от файловой структуры. Файлы находятся по связям (wikilinks в прозе + MOC-навигаторы + YAML descriptions), а не по путям в папках. Агент потребляет минимум контекста, послойно углубляясь только туда, где нужно (progressive disclosure). Физически — сеть маленьких .md-файлов, каждый с одной законченной мыслью.

> **О термине "skill" в skill graph:** Heinrich использует "skill" как синоним "knowledge node / claim": *"each linked file is a standalone methodology claim (= skill)"* и *"okay actually it's about building knowledge bases but that's the same thing…"*. В нашей системе "Skill" = формат SKILL.md (.claude/skills/). Ниже мы используем "файл знаний" / "node" вместо авторского "skill file", чтобы не путать с форматом. Wikilinks в прозе несут семантику ("когда и зачем переходить"), YAML frontmatter позволяет агенту сканировать без чтения полного файла, MOCs (Maps of Content) организуют кластеры. Результат: агент не следует инструкциям, а навигирует предметную область, подтягивая ровно то, что нужно для текущей ситуации. Автор концепции: Heinrich (@arscontexta), февраль 2026.

## Проблема: один скилл — одна глубина

Типичный скилл — один файл с одной способностью: суммаризация, код-ревью, дебаг. Для простых задач этого достаточно.

Но глубокая экспертиза не умещается в один файл. Пример: терапевтический скилл требует знания когнитивно-поведенческих паттернов, теории привязанности, техник активного слушания, фреймворков эмоциональной регуляции. Один файл это не вмещает — теряется либо глубина, либо контекстное окно.

## Решение: Skill Graph

Skill Graph — сеть файлов знаний (.md), связанных wikilinks. Вместо одного большого файла — много маленьких composable-частей, ссылающихся друг на друга.

**Ключевое свойство:** skill graph применяет паттерн навигации (discovery) рекурсивно внутри самого графа. Каждый узел может ссылаться на другие узлы, и граф углубляется настолько, насколько требует предметная область.

## Четыре примитива

Всё строится из стандартных элементов Markdown:

| Примитив | Что делает | Почему важен |
|----------|-----------|-------------|
| **Wikilinks в прозе** | `[[ссылка]]` внутри предложения | Несут смысл: агент понимает *когда и зачем* переходить, а не просто видит reference |
| **YAML frontmatter** | description в заголовке каждого файла | Агент сканирует описания без чтения полного файла |
| **MOCs** (Maps of Content) | Файлы-навигаторы, организующие кластеры | Структурируют граф когда он становится большим |
| **Рекурсивные ссылки** | Node → node → node | Граф идёт так глубоко, как требует домен |

## Progressive Disclosure — пять уровней

Агент потребляет информацию послойно, от самого дешёвого к самому дорогому:

```
Index (точка входа)
  → YAML descriptions (сканирование без чтения)
    → Wikilinks в прозе (навигация по связям)
      → Секции файла (частичное чтение)
        → Полное содержание файла
```

**Большинство решений принимается до чтения хотя бы одного полного файла.** Агент читает index, понимает ландшафт, следует только релевантным путям, игнорирует остальное.

> **Расхождение с реализацией:** Выше — концептуальная модель автора из статьи. В реальном коде arscontexta (верифицировано по GitHub, 2026-03-03) последовательность другая: `file tree + self/ (hook) → YAML descriptions → MOC hierarchy → wiki-link traversal → full content`. Нет файла index.md — роль входа играет tree + self/identity.md. "links" и "sections" у автора не являются отдельными шагами — агент читает MOC целиком, потом при необходимости полный файл. Подробнее: [architecture.md](architecture.md), секция Progressive Disclosure.

## Как устроен Index

Index — не lookup-таблица, а точка входа, направляющая внимание. Агент читает его, понимает ландшафт и следует ссылкам, которые важны для текущего разговора.

Структура index-файла:

```markdown
# domain-name

{1 абзац: философия / зачем эта область}

## Synthesis
Развёрнутые аргументы о том, как части связаны:
- [[argument-1]] — описание в одно предложение
- [[argument-2]] — описание
...

## Topic MOCs
Домен разбивается на подобласти:
- [[topic-1]] — описание
  - [[sub-topic]] — описание
- [[topic-2]] — описание
...

## Cross-Domain Claims
Связи с другими доменами:
- [[claim]] — как связано

## Explorations Needed
Открытые вопросы, пробелы в знаниях
```

## Как устроен Node (отдельный файл)

Каждый файл — одно законченное методологическое утверждение (claim) или техника:

- YAML frontmatter с `description` для быстрого сканирования
- Основной текст — самодостаточный (понятен без чтения других файлов)
- Wikilinks **внутри прозы** — агент видит связь в контексте предложения и понимает *когда* переходить: `"...because [[forced engagement produces weak connections]], the system relies on..."`
- Не ссылки отдельным списком внизу, а часть повествования

## MOCs — Maps of Content

Когда граф растёт, между index и отдельными файлами появляется промежуточный слой — MOCs. Каждый MOC организует кластер связанных файлов знаний (nodes) в навигируемый подтопик.

Пример: в knowledge work skill graph с 249 файлами — 7 MOCs: graph-structure, agent-cognition, discovery-retrieval, processing-workflow и др. Каждый MOC может содержать суб-MOCs (agent-cognition → agent-cognition-hooks, agent-cognition-platforms).

## Примеры доменов

| Домен | Что в графе |
|-------|------------|
| **Терапия** | Когнитивно-поведенческие паттерны, теория привязанности, активное слушание, эмоциональная регуляция — всё связано |
| **Трейдинг** | Риск-менеджмент, психология рынка, размер позиции, технический анализ — контекст перетекает между частями |
| **Юриспруденция** | Паттерны контрактов, compliance, юрисдикции, цепочки прецедентов — всё traversable от одной точки входа |
| **Компания** | Оргструктура, продуктовые знания, процессы, онбординг, культура, конкурентный ландшафт |

Ни один из этих доменов не умещается в один файл, но все работают как графы.

## Папки для глаз, связи для агента

arscontexta использует плоскую структуру (249 файлов в одной notes/). Но когда у базы два потребителя — человек и агент — нужны оба интерфейса: папки для человеческой навигации, wikilinks + MOC для агентской. Оба работают с одними файлами через разные механизмы.

**Проблема "куда класть файл" снимается графом.** Файл физически в одной папке, но через topics в YAML и wikilinks связан с несколькими контекстами. Пример для CEO: нод "Ищем CTO с опытом в AI" входит одновременно в Найм (Команда), Продуктовую стратегию (Продукт), Бюджетирование (Финансы). В папках — один родитель. В графе — три точки доступа.

## Skill Graphs vs одиночные Skills

| | Одиночный Skill | Skill Graph |
|---|----------------|-------------|
| **Объём** | Один файл, до 500 строк | Десятки-сотни файлов |
| **Глубина** | Одна задача / процесс | Целая предметная область |
| **Контекст** | Загружается целиком | Агент навигирует, берёт нужное |
| **Связи** | Cross-references в секции Related | Wikilinks в прозе — семантически богатые |
| **Масштабирование** | Ограничен размером файла / контекстного окна | Растёт с доменом, ограничен только временем навигации |
| **Результат** | Агент следует инструкциям | Агент понимает предметную область |

## Как построить

**Простой путь:** установить плагин arscontexta, выбрать research preset, указать тему. Плагин разворачивает структуру и наполняет через /learn и /reduce. → [cases/arscontexta.md](../../cases/arscontexta.md).

**Ручной путь:** skill graph не обязан жить в `.claude/skills/`. Ключевое — index-файл, который говорит агенту что существует и как навигировать. Формат index описан выше. Далее: создать файлы-узлы, связать wikilinks в прозе, организовать MOCs по мере роста.

## Связь с принципами из !skills.md

Skill graph — не замена Skills (SKILL.md), а дополнительный слой: граф знаний, по которому агент навигирует. Концепция развивает три принципа из [!skills.md](../!skills.md):

- **"Много маленьких скиллов лучше одного большого"** → skill graph доводит это до архитектуры: не просто "разбей на части", а "свяжи в навигируемый граф"
- **"Cross-references"** → wikilinks в прозе — более мощная версия секции "Related Skills"
- **"Progressive Disclosure"** → skill graph расширяет 3 уровня (metadata → body → resources) до 5 уровней (index → descriptions → links → sections → content)

## Значение

Skills (SKILL.md) — context engineering: курированное знание, инжектируемое по триггеру. Skill graph — дополнительный слой: агент навигирует структуру файлов знаний, подтягивая то, что требует ситуация. Skills задают процесс и правила, skill graph даёт предметную глубину. Вместе — разница между агентом, который следует инструкциям, и агентом, который понимает предметную область.
