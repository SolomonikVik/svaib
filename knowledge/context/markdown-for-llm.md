# Анатомия идеального Markdown-файла для человека + LLM + RAG

## Кратко

Консолидированное исследование (3 модели: Claude Opus 4.6, Gemini 3.0, GPT 5.2) по оптимизации Markdown-файлов для одновременного использования человеком, LLM и RAG-системами. Ключевые находки: `description` в YAML — самое ценное поле; "summary first" критичен из-за Lost in the Middle; каждая секция должна быть автономным чанком; структурный чанкинг по заголовкам превосходит семантический; YAML обрабатывается LLM как текст, не как структурированные данные.

---

## Executive Summary

1. **`description` в YAML — самое ценное поле.** 2-3 предложения о содержании файла. Используется для discovery агентами (Claude SKILL.md, Cursor rules) и для reranking в RAG.

2. **Заголовки (H1-H6) — самый важный структурный элемент.** Одновременно навигация для человека, семантический сигнал для LLM и граница чанка для RAG. Заголовки должны быть самоописывающими: "Бюджет проекта Alpha Q1", не просто "Бюджет".

3. **Summary first — всегда.** Начало файла имеет наивысший attention weight. Эффект "Lost in the Middle" (Stanford/Meta, 2024): информация в середине контекста теряет до 20%+ точности.

4. **Каждая секция — автономный чанк.** Не "Он решил...", а "Иванов решил...". Повторять субъект, контекст и ключевые термины в каждой H2-секции. Anthropic Contextual Retrieval: -35% ошибок поиска, -67% при комбинации с BM25+reranking.

5. **Плоский YAML, <300 строк, таблицы <20 строк, вложенность списков <=2 уровня.** Структурная простота бьёт сложное форматирование по всем бенчмаркам.

6. **Структурный чанкинг по заголовкам превосходит семантический** для хорошо структурированных файлов: 70.5% hit rate vs 63.8% (HuggingFace, nuclear engineering).

7. **YAML обрабатывается LLM как текст, не как структурированные данные.** Claude Projects, Custom GPTs, Cursor — все читают YAML как обычный текст. Пишите YAML-поля так, чтобы они читались как natural language.

## 1. Метаданные файла (YAML Frontmatter)

### 1.1 Какие поля реально работают

**Обязательные:**

| Поле | Зачем | Примечание |
|:-----|:------|:-----------|
| `title` | Идентификация для человека и LLM | Совместимо со всеми экосистемами (Obsidian, Jekyll, Hugo, Docusaurus) |
| `type` | Маршрутизация агента | `decision`, `reference`, `procedure`, `person`, `project` |
| `status` | Решает проблему №1: поиск устаревшего контента | `current`, `draft`, `archived`, `superseded` |
| `updated` | Сигнал свежести | Дата последнего смыслового обновления |
| `tags` | Retrieval-сигналы + навигация | Совместимо с Obsidian, Jekyll, Docusaurus |
| `description` | **Самое ценное поле для LLM.** Используется для discovery и reranking | 2-3 предложения: о чём файл + когда его использовать |

**Полезные:** `context_priority` (high/medium/low — приоритет загрузки в контекст), `related`/`depends_on`/`feeds_into` (типизированные связи), `scope` (personal/team/company), `aliases` (синонимы для retrieval, совместимо с Obsidian).

### 1.2 Куда класть инструкции для AI

Коротко в YAML (`description`, `context_priority`) + отдельная секция в теле файла для сложных инструкций. Это самый надёжный подход — разные пайплайны по-разному обрабатывают YAML.

### 1.3 Версионирование

`updated` в YAML — минимально необходимое. Опционально: `version` + мини-changelog в теле файла.

### 1.4 Антипаттерны в метаданных

- **Глубоко вложенный YAML** — LLM путаются, Obsidian не поддерживает
- **Больше ~12-15 полей** — бремя поддержки превышает пользу
- **Технические поля SSG** (`layout`, `permalink`, `draft`) — тратят токены без пользы для AI
- **Полная инструкция агенту только в YAML** — пайплайны могут выкинуть YAML
- **Коллизии имён:** Hugo резервирует `type` — учитывать при миграции между экосистемами

### 1.5 Как инструменты реально парсят YAML

| Инструмент | Как обрабатывает YAML |
|:-----------|:---------------------|
| Claude Projects | Как текст. Не парсит структурно. Семантический поиск по всему файлу включая YAML |
| Cursor | Парсит ровно 3 поля: `description`, `alwaysApply`, `globs`. Остальное — текст |
| Custom GPTs | Игнорирует YAML-структуру полностью. Только семантический + BM25 поиск |
| Стандартные RAG (LangChain, LlamaIndex) | Парсит YAML как metadata для фильтрации. Тело файла — для embedding |

**Вывод:** YAML — это прежде всего **человекочитаемый, LLM-видимый структурированный текст**. Пишите поля так, чтобы они хорошо читались как natural language.

## 2. Структура тела файла

### 2.1 Порядок секций (оптимальный)

```
YAML frontmatter
# Заголовок (= title)               <- Позиция с наивысшим вниманием
Summary / Кратко: 2-3 предложения   <- Якорный чанк для RAG
[Инструкции для агента]              <- Опционально, ближе к началу
## Основной контент — секция 1
## Основной контент — секция 2
## Связанные файлы                   <- Позиция "конца" — тоже хорошее внимание
```

**Почему этот порядок:** "Lost in the Middle" (Liu et al., 2024) — начало и конец контекста вспоминаются лучше всего, середина теряет 20%+ точности. Повторить критический факт и в summary, и в релевантной секции тела — страхует от позиционного bias.

### 2.2 Заголовки — самый важный элемент

- **Самоописывающие:** "Ключевые риски запуска проекта Alpha", не "Важно" и не "Риски"
- **Контекст в заголовке:** Включать субъект (проект, компанию, период), чтобы чанк был понятен автономно
- **H2 = основная единица чанкинга.** LangChain `MarkdownHeaderTextSplitter` и LlamaIndex `MarkdownNodeParser` оба режут по заголовкам и переносят путь заголовков в metadata чанка
- **`strip_headers=False`** — важно сохранять заголовки в тексте чанка для семантического поиска

### 2.3 Как LLM видит разные элементы Markdown

| Элемент | Понимание LLM | Практический вывод |
|:--------|:-------------|:-------------------|
| Заголовки H1-H6 | ★★★★★ | Основной структурный сигнал |
| Маркированные/нумерованные списки | ★★★★★ | Парсятся как дискретные элементы |
| Блоки кода | ★★★★★ | Чётко отделены от прозы |
| XML-теги инлайн | ★★★★★ | Anthropic рекомендует; Claude тренирован на XML |
| **Жирный** / *курсив* | ★★★★ | Жирный повышает вес текста |
| Простые таблицы (<20 строк) | ★★★★ | Хорошо понимаются |
| YAML frontmatter | ★★★★ | Распознаётся между `---` |
| Вложенные списки (3+ уровня) | ★★★ | Связи между уровнями теряются |
| Сложные таблицы (100+ строк) | ★★ | Заголовки "забываются" |

**Бенчмарк:** Markdown key-value формат = **60.7% accuracy** vs таблицы 51.9% vs CSV 44.3% (1000 записей, GPT-4.1-nano). Когда имена полей повторяются для каждой записи, LLM работают значительно лучше.

### 2.4 Размер файла

<300 строк, sweet spot 2000-8000 токенов. Секции проектировать на ~1-3 экрана. Если файл больше — разбивать на несколько с чёткими связями.

### 2.5 Разделение инструкций и контента

**Три паттерна:** YAML frontmatter (структурированные metadata), HTML-комментарий `<!-- AI: ... -->` (краткая инструкция без визуального шума), отдельная секция с trigger/instruction парами (сложные правила работы с файлом).

**Пример trigger-пары:** "Trigger: агент обновляет факты → Instruction: обнови только секции с [SOURCE]"

### 2.6 Данные внутри файла

Таблицы — только когда выигрывают у списков, максимум ~20 строк, с текстовой интерпретацией рядом. Данные лучше в читабельном markdown; если нужна машинная структура — порционно и с объяснением.

## 3. RAG-оптимизация единицы файла

### 3.1 Что индексируется и как

| Что | Как попадает в поиск | Практический вывод |
|:----|:--------------------|:-------------------|
| YAML frontmatter | Metadata для фильтрации, не для embedding (в стандартных RAG) | Tags/dates → фильтры. Description → должна также быть в теле |
| Имя файла | Metadata, не embedding | "Filename = поисковый запрос" — дублируйте термины в H1 и summary |
| Заголовки | Граница чанка + metadata на дочерних чанках | Самоописывающие заголовки = встроенный retrieval index |
| Тело текста | Embedding (основной вектор поиска) | Ключевые термины должны быть в тексте, не только в YAML |

### 3.2 Чанкинг: структурный vs семантический

Структурный чанкинг по заголовкам превосходит семантический для хорошо структурированных markdown: 70.5% hit rate vs 63.8% (HuggingFace, nuclear engineering). Семантический чанкинг — для неструктурированного текста.

**Рекомендуемый пайплайн:**

```
1. MarkdownHeaderTextSplitter (по H1/H2/H3, strip_headers=False)
2. RecursiveCharacterTextSplitter (для слишком длинных секций: ~512-1024 токенов)
3. Инъекция файлового контекста: "File: strategy.md | Section: Q1 Goals > Revenue"
```

### 3.3 Как сделать чанк самодостаточным

| Приём | Пример |
|:------|:-------|
| Повторять субъект | Не "Он решил...", а "Совет директоров решил..." |
| Контекст в первой строке секции | "Контекст: бюджет проекта Alpha на Q1 2026" |
| Ключевые термины в каждой секции | Те же слова, по которым будут искать |
| Заголовок = тема чанка | "Ценовая стратегия — Q1 Board Meeting", не "Ценообразование" |
| Не начинать с "Как выше" / "Как ниже" | Без отсылок к другим частям файла |

Anthropic Contextual Retrieval (2024): инъекция контекста в чанк перед embedding даёт -35% ошибок, -67% с BM25+reranking.

### 3.4 Дополнительные RAG-приёмы

- **Секция "На какие вопросы отвечает этот документ?"** (FAQ-стиль) — повышает semantic similarity с запросами
- `---` **(горизонтальная черта)** — RAG-системы воспринимают как границу темы и разрезают текст в этом месте. Использовать только при жёсткой смене темы
- **Не дробить слишком мелко** — микрофайлы по 5 строк создают шум в поисковой выдаче. Один связный документ на тему лучше 10 заметок

## 4. Связи файла с другими файлами

### 4.1 Как LLM видит ссылки

LLM не может переходить по ссылкам (`[[wikilinks]]` или `[markdown](file.md)`). Видит их только как семантические сигналы: "этот файл связан с X". Для перехода нужен tool use.

### 4.2 Типизированные связи

| Тип связи | Семантика |
|:-----------|:----------|
| `parent` / `child` | Иерархия |
| `depends_on` | Этот файл зависит от... |
| `feeds_into` | Этот файл питает... |
| `supersedes` / `superseded_by` | Версионирование / замена |
| `implements` | Связь решения с исполнением |
| `source` | Каноничный первоисточник факта |
| `related` | Общая связь |

Хранить в секции "Связанные файлы" в теле с описанием зачем связано. Связи без описания — шум для LLM.

### 4.3 Source of Truth: маркеры

`[SOURCE]` — факт живёт здесь, агент может обновлять. `[REF: [[file#section]]]` — факт живёт в другом файле, здесь только ссылка, агент обязан обновлять в первоисточнике.

### 4.4 Transclusion

`![[file]]` (Obsidian embed) невидим для LLM — видит литеральную строку, не содержимое. Для AI-пайплайнов нужно материализовать/flatten.

### 4.5 Embed vs Link

Факт критичен и краток → вставить копию. Факт справочный → ссылка с `[REF:]`.

## Чеклист при создании/ревью файла

- [ ] `description` в YAML заполнен (2-3 предложения)
- [ ] `status` в YAML актуален
- [ ] H1 = title из YAML
- [ ] Секция "Кратко" есть и содержит ключевые факты
- [ ] Каждый H2-заголовок самоописывающий (включает субъект)
- [ ] Каждая секция начинается с контекста (понятна автономно)
- [ ] Субъект повторяется, не "он/она/это"
- [ ] Таблицы < 20 строк
- [ ] Списки <= 2 уровня вложенности
- [ ] Файл < 300 строк
- [ ] [SOURCE] / [REF:] маркеры расставлены
- [ ] Секция "Связанные файлы" с описанием связей
- [ ] Нет "слепых" ссылок без контекста
