# LLM Wiki — персональная база знаний, поддерживаемая LLM

## Кратко

Паттерн организации персональных знаний, где LLM инкрементально компилирует и поддерживает структурированную markdown-вики. Вместо классического RAG (каждый раз ищем по сырым источникам) — LLM обрабатывает каждый новый источник, обновляет десятки страниц вики, поддерживает кросс-ссылки и консистентность. Автор — Андрей Карпатый (бывший директор AI в Tesla, основатель Eureka Labs). Гист вызвал волну реализаций: llm-wiki-kit, sage-wiki, memex, atomic-knowledge и др.

---

## Архитектура: три слоя

```
Raw Sources  →  The Wiki  →  Schema
(сырьё)        (знание)      (структура)
```

**Raw Sources** — любые входные материалы: статьи, встречи, документы, решения, ссылки. Сохраняются как source of truth — к ним можно вернуться при сомнениях в wiki.

**The Wiki** — структурированный markdown. Не свалка заметок, а компилированное знание: страницы по сущностям, саммари по темам, кросс-ссылки. LLM обновляет при каждом ingest.

**Schema** — мета-уровень: какие типы страниц существуют, какие связи допустимы, какие конвенции именования. По сути — онтология вики.

## Три операции

### Ingest (загрузка нового источника)

LLM обрабатывает один источник за раз. Извлекает ключевое, пишет саммари, обновляет страницы сущностей, поддерживает кросс-ссылки. На один источник обновляется 10-15 файлов вики.

Ключевое: это не "добавь заметку" — это компиляция. LLM решает что из нового источника важно, как оно связано с уже существующим знанием, какие страницы нужно обновить.

### Query (запрос к вики)

Вопрос к базе знаний. Отличие от обычного чата: **хороший ответ сам становится страницей вики**. Знание не исчезает в истории чата — оно компаундится. Каждый вопрос потенциально обогащает базу.

### Lint (проверка целостности)

Периодический аудит вики. Чеклист:
- Противоречия между страницами
- Устаревшие факты (по датам источников)
- Осиротевшие страницы (нет входящих ссылок)
- Пропущенные кросс-ссылки (тема упоминается, но ссылки нет)
- "Дыры" — темы, которые часто упоминаются, но не имеют своей страницы

## Инсайт: почему LLM подходит для этой работы

> "Тяжёлая часть базы знаний — не чтение и не мышление, а бухгалтерия."

Человек бросает поддержку вики, потому что maintenance burden растёт экспоненциально: каждый новый файл создаёт новые связи, которые нужно обновить. LLM не устаёт от бухгалтерии: обновить 15 файлов за раз, проверить все кросс-ссылки, найти противоречия — это именно то, что модели делают хорошо.

## Реализация (минимальная)

Три файла инфраструктуры:
- **index.md** — каталог страниц по категориям (навигация)
- **log.md** — хронологический журнал всех операций (аудируемость)
- **CLAUDE.md** (или аналог) — конфигурация: структура вики, конвенции, workflow

Всё остальное — страницы вики. Markdown в git. Портабельно, аудируемо, принадлежит пользователю.

## Комьюнити-реализации

Гист вызвал волну реализаций с разными фокусами:
- **llm-wiki-kit** — MCP-сервер, интеграция с Claude Code
- **sage-wiki** — multi-format extraction (PDF, видео, аудио)
- **memex** — filesystem-sandboxed daemon (работает в фоне)
- **atomic-knowledge** — markdown-first протокол (ближе к Zettelkasten)
- Специализированные: для fiction writing, enterprise service delivery, semiconductor research

Паттерн намеренно абстрактный — seed, а не prescription. Каждая реализация адаптирует под свой контекст.

## Критика и ограничения

**LLM-саммари могут подменять мышление.** Когда LLM пишет саммари за человека — человек не выстраивает собственную структуру понимания. Контраргумент: LLM берёт на себя бухгалтерию, а не мышление. Но граница размытая.

**Ошибки компаундятся.** Неверный факт, записанный при ingest, влияет на все будущие обновления. Mitigation: raw sources сохраняются как source of truth, к ним можно вернуться. Но на практике это требует дисциплины.

**Без lint вики дрейфует.** Особенно в быстро меняющихся доменах. Lint — не опция, а необходимость. Вопрос: кто запускает и как часто.

**Нет модели стоимости контекста.** Карпатый не ставит вопрос "сколько контекста загружать при query". Wiki растёт — query дорожает. Progressive disclosure не описан.

## Связь с другими подходами

**arscontexta (6Rs Pipeline)** — Reweave и Verify из 6Rs = более детализированная версия Lint. arscontexta добавляет Reflect (поиск неочевидных связей) и Rethink (рефакторинг структуры). Карпатый проще, arscontexta глубже. → [skill-graphs/architecture.md](skill-graphs/architecture.md)

**ICM (Van Clief, 2026)** — "configure the factory, not the product" = близко к подходу Карпатого: правь schema/конфигурацию, а не отдельные страницы. ICM добавляет разделение reference (стабильные правила) vs working (per-run артефакты), которого у Карпатого нет.

**File-based Memory** — LLM Wiki работает на уровнях 0-1 спектра памяти (статические файлы + авто-поддержка, без embeddings, без графовых БД). Осознанный выбор: портабельность и аудируемость важнее скорости поиска. → [ai-system-files.md](ai-system-files.md)

**Наша knowledge/** — по сути мы уже делаем вариант LLM Wiki: Виктор кидает источник → AI исследует → записывает в структурированный файл → обновляет сводный !файл и README. Ingest работает. Query-as-accumulation и Lint — пока нет.
