# AI-Ready Architecture

## Кратко

Принципы структуры кода, при которой AI-агенты могут безопасно вносить изменения. Ключевая проблема: AI не удерживает глубокие цепочки сайд-эффектов — видит 1-2 уровня, дальше слепнет. Решение: sinks вместо pipes (каждое действие самодостаточно), глубокие модули с честными интерфейсами, progressive disclosure через структуру директорий. Классические принципы (low coupling, information hiding, Parnas 1972) — не новые, но критичные именно для AI: каждая скрытая зависимость = потенциальная ошибка агента.

---

## Проблема

AI-агенты начинают каждую сессию с нуля — без памяти о системе, без интуиции, без "я просто знаю что эта функция ещё пишет в лог". Человек-разработчик постепенно выстраивает ментальную модель скрытых связей. AI этого не может.

Следствие: всё неявное для AI **не существует**. Если обновление в БД неявно запускает нотификацию → обновление метрики → блокировку аккаунта — AI увидит только обновление в БД. Остальное — вне его контекстного окна и понимания.

---

## Принципы

### 1. Sinks вместо Pipes — убирай каскады

**Pipe** — компонент, который при вызове запускает цепочку неявных действий в других частях системы. Изменение в одном модуле "протекает" через трубы в непредсказуемые места.

**Sink** — компонент, эффект которого замкнут внутри его границ. Blast radius любого изменения ограничен модулем.

**Практически:** если нужно "обновить БД + отправить нотификацию + пересчитать лимиты + проверить блокировку" — не делай триггерную цепочку (обновление БД автоматически вызывает следующее звено). Вместо этого — плоская оркестрация: одна функция вызывает все шаги последовательно и явно.

```
# Pipe (плохо для AI): каскад скрытых сайд-эффектов
update_record()  # → trigger: send_notification()
                 #   → trigger: update_metrics()
                 #     → trigger: check_limits()

# Sink (хорошо для AI): плоская оркестрация
def process_update():
    update_record()
    send_notification()
    update_metrics()
    check_limits()
```

В классической терминологии это **low coupling** — минимум скрытых связей между модулями.

### 2. Глубокие модули с честными интерфейсами

Из философии John Ousterhout ("A Philosophy of Software Design"): модуль должен быть существенным по функциональности, но простым по интерфейсу. Сложность скрыта внутри, наружу — минимальный контракт.

**Honest interface** — если модуль меняет стейт, пишет в базу, шлёт запрос или имеет другие побочные эффекты — это **видно** в его интерфейсе. Нет скрытого поведения.

> "If a module's behavior depends on state or side effects that aren't visible in its interface, your codebase is not AI-ready." — Ian Bull

Принцип information hiding (Parnas, 1972) + принцип honest interfaces = модуль можно понять не читая его внутренности. Для AI это критично — он работает с интерфейсами, а не с ментальной моделью системы.

### 3. Progressive Disclosure через структуру проекта

Кодовая база должна быть навигируемой на возрастающих уровнях детализации:
1. Директории верхнего уровня → подсистемы (понятно ЧТО система делает)
2. Типы и контракты → интерфейсы (понятно КАК части взаимодействуют)
3. Реализация → конкретный код (понятно КАК именно работает)

**File System as Architecture** — структура директорий должна отражать концептуальную архитектуру. AI (и человек) может понять систему, просмотрев только дерево файлов.

Аналогия: наш knowledge/ уже так устроен — README в каждой папке, границы категорий, навигация от общего к частному. Тот же принцип для кода.

### 4. C4-подобная структуризация

Вместо "паутины мелких модулей" — структурировать по иерархии: подсистемы → компоненты → модули. Каждый уровень понятен без спуска в детали нижнего. Все зависимости явно в интерфейсах.
