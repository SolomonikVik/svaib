# Онтология планирования проекта v0.5
## Файловая система для управления целями, планами и прогрессом

> **Версия:** v0.5 (10 февраля 2026)
> **Контекст:** File-first подход. Markdown + Git. Используются человеком и LLM-агентом.
> **Scope:** Один проект. Масштабируется на несколько проектов и вложенные подпроекты.
> **Автор:** Виктор Соломоник (svaib)
> **Источники:** Ревью Claude Opus, GPT-5.2, Gemini 3.0, Advanced OKR (GO2OKR), концепция «Бизнес по-человечески»

---

## Принципы

1. **LLM + People centered.** Система одинаково удобна для человека и для LLM-агента. Читаемо глазами, парсится моделью без потери смысла.
2. **Разделение по частоте изменений.** Файлы с разной частотой обновления живут отдельно.
3. **Компактность.** Каждый файл — ровно столько, сколько нужно. Лишнее = шум = потраченные токены.
4. **Один файл — одна роль.** Каждый файл отвечает на один вопрос.
5. **Сверху вниз.** Vision → Goal → Plan → Weekly Progress → Timeline. Каждый уровень питается предыдущим.
6. **Минимум дублирования.** У каждого факта одно каноническое место.
7. **Модульность.** Полная структура на максимуме, но разворачивается по запросу. Не все файлы обязательны.
8. **ISO 8601.** Даты везде в формате YYYY-MM-DD.

Эта онтология — **блок регулярного менеджмента** внутри более широкой системы управления проектом/компанией.


- **Продукт** — техническая документация, спецификации. Отдельная папка.
- **Процессы** — бизнес-процессы, регламенты, протоколы встреч. Отдельная папка.
- **Knowledge** — персональная/профессиональная база знаний. Ссылки, заметки, инсайты, инструменты. Не привязана к планированию — это накопленные знания, которые помогают принимать решения. У каждого клиента своя: CEO хранит управленческие практики, разработчик — технические находки, предприниматель — идеи и контакты.
- **Inbox** — общая входящая. Не привязана к проекту. Периодически разгребается.
- **Архив** — устаревшие файлы, старые версии, закрытые подпроекты.
- **README + SKILL.md** — в корне проекта. Точка входа для LLM и людей.

## Файловая структура

```
{project}/
  README.md                         ← сводка проекта (точка входа)
  SKILL.md                          ← инструкция для LLM-агента
  management/
    01_vision.md                    ← кто мы, зачем, миссия
    02_goal.md                      ← куда идём: цели + стратегия + KR
    03_plan.md                      ← что делаем: квартал → неделя → день
    04_weekly_progress_2026.md      ← недельные итоги (детальный лог)
    05_timeline.md                  ← история проекта: кварталы, годы, вехи
    ideas.md                      ← идеи, гипотезы, задачи на потом
    decisions.md                    ← принятые решения и обоснования
    reflections.md                  ← лог размышлений и переживаний
    metrics.md                      ← описание системы метрик (опционально)
    team.md                         ← команда, партнёры, роли (опционально)
```

Первые 5 файлов пронумерованы — задают логику чтения сверху вниз. Остальные — по алфавиту.

```

```

| Планирование квартала | ideas, metrics |

| Анализ состояния, энергии, паттернов | reflections |

## Файл 1: `01_vision.md`

### Назначение
Идентичность проекта. Кто мы, зачем существуем, какое будущее видим.

По Advanced OKR — содержит **Aim** (State Goal, 3+ лет).

### Содержит
- Суть проекта (1-3 предложения)
- Миссия / большая цель
- Для кого (целевая аудитория)
- Личная цель основателя в проекте
- Ценности и принципы (опционально)
- Описание продукта / бизнес-модели (если есть)

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | Раз в 6-12 месяцев |
| Размер | Компактный (до 1-2 страниц) |
| LLM читает | Всегда (базовый контекст) |
| Статус | Ядро |

### Когда обновлять
- Пивот или существенное изменение видения
- Раз в полгода — перечитать и проверить актуальность
- Конец года — в рамках годовой ретроспективы

## Файл 2: `02_goal.md`

### Назначение
Целеполагание и стратегия. Куда идём, как планируем, как измерим.

Показывает **актуальное состояние** целей. Открыл — видишь текущее. Историю и итоги — в `05_timeline.md`.

### Содержит

#### Долгосрочная цель (1-3 года)
Вытекает из vision. По Advanced OKR — **Strategy Goal** (State Goal).

#### Стратегия / Roadmap
Как планируем достичь цели. Крупные вехи, ключевые ставки.
> При разрастании → отдельный файл `strategy.md`

#### Цель года
По Advanced OKR — **Strategic Objective** (Change Goal).
Формула: *глагол + существительное + новое свойство*

#### Цель текущего квартала + Key Results
По Advanced OKR — **Objective** (Change Goal) + **KR** (3-5 штук).

Типы KR:
- **Outcome** (влияние): конверсия, NPS, retention
- **Output** (действия): количество клиентов, фич
- **Milestone** (веха): да/нет

> Если метрик больше, чем помещается здесь → см. `metrics.md`

### Правило каноничности KR
- **В goal** KR существуют только как **план** на текущий квартал (целевые значения).
- **Факт (план vs факт)** фиксируется только в `05_timeline.md` по итогам квартала.
- В goal факт не дописывается "по ходу". Goal всегда показывает "куда идём", не "где были".

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | Раз в квартал (цель квартала). Раз в год (цель года, стратегия) |
| Размер | Компактный. НЕ растёт |
| LLM читает | Часто (контекст «куда идём») |
| Статус | Ядро |

### Ритуал
- **Начало квартала:** сформировать цель квартала + KR
- **Начало года:** пересмотреть цель года, обновить стратегию
- **Конец года:** пересмотреть долгосрочную цель

```markdown
- [ ] задача 2

- [ ] задача 1

W07 (2026-02-10 — 2026-02-16)
Если направлений/подроектов нет - то единым списком
### направление/подпроект A
- [ ] задача 1
### направление/подпроект B
- [ ] задача 1
### направление/подпроект C
- [ ] задача 1

## План квартала (Q1 2026)
Цель/Фокус квартала: [одно предложение]
Задачи в рамках направлений/подпроектов на квартал (вытекают из goal)
Если направлений нет - то единым списком

### направление/подпроект A
- [ ] задача 1
- [ ] задача 2

### направление/подпроект B
- [ ] задача 1
- [ ] задача 2

### направление/подпроект C
- [ ] задача 1
- [ ] задача 2
```

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | Ежедневно (день), еженедельно (неделя), раз в квартал (квартал) |
| Размер | Компактный и стабильный — не растёт, обновляется |
| LLM читает | Часто (оперативный контекст) |
| Статус | Ядро |

### Ритуалы

**Каждое утро:** план дня из плана недели.

**Каждый понедельник:** итоги прошлой недели → weekly progress. Новый план недели.

**Начало квартала:** из goal взять цель и KR → план квартала. Просмотреть ideas.

### Характеристики
| Параметр | Значение |

| Размер | Растёт медленно (4-5 записей в год). Один файл на весь проект |

## Файл 6: `ideas.md`

### Назначение
Структурированная парковка для идей, гипотез и отложенных задач.

### Структура

```markdown
# Ideas

## Идеи
(мысли, которые могут пригодиться — не оформлены как задачи)
- 2026-02-10: описание идеи

## Гипотезы
(нужно проверить, прежде чем делать)
- 2026-02-10: гипотеза + как проверить

## Задачи на потом
(понятно что делать, но не сейчас)
- 2026-02-10: описание задачи
```

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | По мере появления |
| Размер | Растёт, периодически чистится |
| LLM читает | При планировании квартала. НЕ по умолчанию |
| Статус | Ядро |

### Ритуал
- **Планирование квартала:** просмотреть ideas → вытащить релевантное в plan
- **Раз в квартал:** почистить — удалить неактуальное
- **В любой момент:** набросать запись, не думая о структуре

### Типы записей (для LLM)
- **Идея** → нужно оформить и обдумать
- **Гипотеза** → нужно проверить/валидировать
- **Задача на потом** → можно брать в plan напрямую

## Файл 7: `decisions.md`

### Назначение
Реестр принятых решений. Выборы, которые определяют траекторию проекта.

Без этого файла LLM будет предлагать уже отвергнутые варианты, а человек забудет *почему* принял решение.

### Структура

```markdown
# Decisions

## 2026-02-10: [Название решения]
**Статус:** Актуально
**Решение:** Что именно решили
**Контекст:** Почему вопрос встал
**Альтернативы:** Что рассматривали и почему отвергли
**Следствия:** Что вытекает для плана/целей

## 2026-01-15: [Название решения]
**Статус:** Пересмотрено (см. решение от 2026-02-10)
...
```

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | По мере принятия значимых решений |
| Размер | Растёт медленно |
| LLM читает | При стратегических обсуждениях, пересмотре целей |
| Статус | Ядро |

### Что является «решением»
**Да:** выбор, закрывающий альтернативы — «Не идём в B2C», «Выбрали Dify вместо Langflow».
**Нет:** операционный выбор внутри недели — «Сегодня делаю A, не B».

### Архивирование
При накоплении >30-40 решений — архивные (со статусом «Пересмотрено» старше года) переносятся в `archive/decisions_YYYY.md`.

## Файл 8: `reflections.md`

### Назначение
Лог размышлений и переживаний. Неструктурированное пространство для мыслей, чувств, интуиций.

### Отличие от других файлов
- **Ideas** — идеи, которые *могут стать задачами*
- **Decisions** — выборы, которые *уже сделаны*
- **Reflections** — мысли и чувства, которые *пока ни то ни другое*

Примеры: «Чувствую, что стратегия Z не работает, но не могу сформулировать почему...», «Упадок сил последние две недели, нужно пересмотреть нагрузку», «Заметил паттерн: клиенты спрашивают про X, но реально нуждаются в Y...»

### Структура

```markdown
# Reflections

## 2026-02-10
(текст размышления / переживания)
Теги: #стратегия #энергия #клиенты
Signal: что это меняет? (ничего / проверить гипотезу / обсудить решение / снизить нагрузку)

## 2026-02-03
...
```

Дата, теги и signal ставятся автоматически (LLM-агентом при записи).

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | Свободная |
| Размер | Растёт |
| LLM читает | По запросу |
| Статус | Ядро |

### Архивирование
Раз в год старые записи переносятся в `archive/reflections_YYYY.md`.

## Файл 9 (опциональный): `metrics.md`

### Назначение
Описание системы метрик проекта. Не сами данные, а карта: какие метрики, где считаются, как часто анализируются, кто отвечает.

Сами данные живут в Google Sheets, BI-системах, CRM и т.д.

### Когда создавать
- Метрик больше, чем помещается в KR в goal
- Есть внешние источники данных
- Нужна прозрачность: кто за какую метрику отвечает

### Структура

```markdown
# Metrics

## [Название метрики]
**Что измеряет:** ...
**Целевое значение:** ... (связь с KR в goal)
**Где считается:** Google Sheets / BigQuery / CRM / ...
**Как часто анализируется:** еженедельно / ежемесячно
**Кто отвечает:** ...
**Ссылка:** [ссылка на дашборд/таблицу]
```

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | При изменении системы метрик |
| Размер | Компактный, стабильный |
| LLM читает | При подведении итогов, планировании |
| Статус | Опционально |

## Файл 10 (опциональный): `team.md`

### Назначение
Описание команды, партнёров, ключевых людей вокруг проекта.

### Когда создавать
- Появился хотя бы один партнёр, подрядчик или сотрудник
- LLM нужно знать контекст о людях без объяснений каждый раз

### Содержит

## Мета: `README.md`

### Назначение
Сводка проекта для быстрого контекста. LLM начинает с этого файла.

Живёт в корне проекта, не внутри management/.

### Содержит
- Что за проект (1-2 предложения)
- Текущая стадия
- Ключевые цели (кратко, ссылка на goal)
- Текущий фокус (кратко, ссылка на plan)
- Команда (кратко)
- Карта файлов: что где лежит

### Характеристики
| Параметр | Значение |
|---|---|
| Частота обновления | При значимых изменениях |
| Размер | Максимально компактный (1 страница) |
| LLM читает | Всегда первым |

### Назначение
Инструкция для LLM-агента в формате skill: как работать с файловой системой проекта.

Живёт в корне проекта, не внутри management/.

### Содержит

### Для SVAIB-продукта
Ключевой deliverable консалтинга. Скелет стандартный, наполнение — под клиента.

## Вложенные проекты

### Два типа

**Полный проект** — имеет собственный vision и цели:
- Ядро: 01_vision, 02_goal, 03_plan, 04_weekly_progress, 05_timeline, ideas, decisions, reflections
- Опциональные: metrics, team
- Мета: README, SKILL.md

**Подпроект** — работает на цели родителя:
- Наследует vision, goal, timeline, decisions от родителя
- Имеет собственные: plan, weekly_progress
- Может иметь: ideas, metrics
- Должен иметь: одну строку "зачем существует и когда закрывается" (в начале plan или в отдельном README)

### Критерии
- **Когда создавать подпроект:** один plan перестаёт вмещать оперативку
- **Когда подпроект → полный проект:** появилась собственная цель, отличная от родительской

### Структура папок (пример)

```
svaib/
  README.md
  SKILL.md
  management/
    01_vision.md
    02_goal.md
    03_plan.md
    04_weekly_progress_2026.md
    05_timeline.md
    ideas.md
    decisions.md
    reflections.md
    metrics.md                    # опционально
    team.md                       # опционально
    projects/
      lab/
        plan.md
        weekly_progress_2026.md
      dev/
        plan.md
        weekly_progress_2026.md
        ideas.md                # если нужен
        metrics.md                # если нужны свои
```

## Масштабирование для клиентов SVAIB

Система проектировалась для двух контекстов:

**Соло-предприниматель / малая команда:**
- Развёртывается ядро (8 файлов) + мета (2 файла)
- Опциональные файлы — по мере роста
- Один человек ведёт всё, LLM помогает

**CEO с командой (до 100-200 человек):**
- Та же структура, но для личного пространства CEO (не для всей компании)
- Team разворачивается в папку с профайлами
- Metrics расширяется (CEO отслеживает больше метрик)
- Strategy может выноситься в отдельный файл
- Подпроекты для ключевых направлений
- Протоколы встреч — в большой рамке (Процессы), не в management/

Принцип: система не усложняется для клиента. CEO не должен вести больше файлов, чем соло-предприниматель. Разница — в глубине наполнения, не в количестве.

## Ритуалы (сводка)

| Когда | Что делаем | Файлы |
|---|---|---|
| Каждое утро | План дня из плана недели | 03_plan |
| Каждый понедельник | Итоги недели → weekly progress. Новый план недели | 03_plan → 04_weekly_progress |
| Начало квартала | Цель квартала + KR. План квартала. Просмотр ideas | 02_goal, 03_plan, ideas |
| Конец квартала | Итоги квартала (KR план vs факт) → timeline | 04_weekly_progress → 05_timeline |
| Конец года | Итоги года → timeline. Пересмотр vision/strategy. Новые цели | 05_timeline, 02_goal, 01_vision |
| По мере | Решения → decisions. Мысли → reflections. Идеи → ideas | decisions, reflections, ideas |
