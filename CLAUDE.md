## Проект

**svaib** — экспериментальная AI-мастерская Виктора. Цель: создать международную AI-компанию с классным продуктом и клиентами по всему миру.

**Три направления:**
- **Лаборатория** — внутренняя AI-мастерская, ускоряющая всё остальное. Прототип: мастерская Железного человека с Джарвисом — R&D продукта, автоматизация через AI-помощников
- **Продукт** — Second AI Brain — персональная AI-инфраструктура для руководителя. Пять частей: онтология (какие сущности бывают) → методология (как с ними работать) → scaffold (каркас папок и файлов) → skills (мастерская промптов и навыков по доменам) → plugin (собранный пакет клиенту). Данные у клиента, методология у нас — плагин не хранит данные, это интеллектуальный слой поверх
- **Публикации** — Telegram, семинары, YouTube — открыто делимся инсайтами и движением

**Миссия:** делать предпринимателей по всему миру сильнее через AI. Не просто облегчать управление, а помогать строить бизнес по-человечески — где уважают себя, сотрудников и клиентов.

**Стадия:** "100 недель кринжа" — контракт Виктора с собой (старт сентябрь 2025). Первая половина — построить рабочий продукт и фреймворк. Вторая — доказать бизнес-состоятельность. Сейчас ранний MVP, неопределённость высокая.

**Стратегия: три ставки:**
1. Лаборатория как акселератор — Виктор строит AI-помощников, каждый из которых усиливает всё остальное: контент, работу с клиентами, сам продукт. Роль — архитектор, не исполнитель
2. Сайт как воронка — весь маркетинг ведёт на svaib.com
3. Ранние клиенты как R&D — 2-3 early adopter клиента = оплачиваемый полигон для продукта

## Правила

1. **Подход к задачам:** Для комплексных задач — сначала обсудить план крупными мазками, потом детализировать. Для простых — сразу к делу. Если не уверен — лучше обсудить. Задавай правильные вопросы, а не просто давай ответ.

2. **Не плоди сущности без необходимости**
Новый элемент (категория, поле, слой, шаг, тип) — только если его нельзя выразить через существующие И без него теряется что-то важное. Лучше 5 чётких, чем 10 размытых. 

3. **Фиксация знаний:** Важные данные и находки из сессии должны быть записаны в соответствующий файл проекта, не оставаться только в чате. Куда — зависит от типа: команда, README папки, файл контекста. Если не очевидно куда — обсудить с Виктором. Данные записывать точно, без обобщений.

4. **Помни о cutoff date.** Технологии, версии, API, модели — всё, что могло измениться за последние 6-12 месяцев, не выдавать как факт. Если предполагаешь обновление — сказать "по данным на [cutoff], но рекомендую проверить" или предложить веб-поиск.

5. **Стиль файлов:** Пиши для AI и человека одновременно — кратко, по существу, экономя токены, но не теряя ничего важного. Лучше сохранить, чем потерять.

6. **Memory — только с разрешения Виктора.** НЕ записывать в auto memory без явного запроса. Знания по теме → в соответствующий файл проекта, не в memory.

7. **Разбор ошибок:** Если Виктор спрашивает "почему так произошло" — это запрос на инженерный анализ. Запустить субагента `rescue` для диагностики. НЕ использовать антропоморфные объяснения ("не вдумался", "поторопился") — только: где сбой, почему, что починить.

8. **Не делай выводов о том, чего не читал.** Если предмет анализа — конкретные файлы или артефакты проекта, прочитай их прежде чем оценивать. Инструкция по созданию чего-то — не замена чтению самих артефактов. Не изобретай термины, которых нет в системе. При неполной картине — спроси или исследуй, но не выдавай частичное знание за анализ.

9. **Содержательный текст = только текст, без tool calls.** В Claude Code текст и tool calls уходят в одном ходе — Виктор видит текст, но действия уже выполняются. Поэтому: если в сообщении есть вопрос к Виктору ИЛИ содержательный ответ на его вопрос (объяснение, обоснование, варианты) — отправляй только текст. Никаких tool calls в том же ходе. Дождись реакции, потом действуй. Исключение: краткий статус перед действием ("Делаю X", "Готово, обновляю файл") — можно совмещать с tool calls.
