# Ideas

## Кратко

Структурированная парковка для идей, гипотез, отложенных задач и инсайтов. Четыре типа записей: идеи (нужно оформить и обдумать), гипотезы (нужно проверить/валидировать), задачи на потом (можно брать в plan напрямую), инсайты (наблюдения для встраивания в фреймворк/продукт). Каждая запись с датой. При планировании квартала — просматривать и вытаскивать релевантное в plan; инсайты — встраивать в `framework/methodology/`, `framework/architecture.md`, `framework/scaffold/` по мере зрелости.

**Работа с файлом:** дописывать по мере появления, в любой момент. LLM читает при планировании квартала, НЕ по умолчанию. Раз в квартал — чистить: удалить неактуальное; разобранные инсайты помечать `→ разобрано в [файл]` или удалять запись. Растёт, периодически чистится. Набросать запись можно не думая о структуре — LLM распределит по секциям.

## Связанные файлы

- 03_plan.md — задачи из ideas берутся в план при планировании квартала (feeds_into)
- 02_goal.md — цели определяют, какие идеи релевантны
- decisions.md — решение может закрыть идею или превратить гипотезу в задачу
- reflections.md — размышления могут порождать идеи и гипотезы
- ../../framework/methodology/ — куда оседают принципы и модели из инсайтов
- ../../framework/architecture.md — куда оседают позиционирующие инсайты
- ../../framework/scaffold/ — куда оседают изменения каркаса из инсайтов

### 2026-02-27 — Scaffold как каркас мышления (не хранения)

**Контекст:** Анализ архитектуры CoMan OS (Максим Опилкин) + собственный опыт заполнения scaffold.

**Инсайт:** Scaffold — не просто файлы для хранения данных. Это каркас мышления: конкретные точки, в которые можно думать. Weekly progress, timeline, reflections — каждый файл задаёт вопрос, на который руководитель отвечает. Структура подсказывает, о чём думать.

**Два полюса снижения барьера:**
- "Garbage in, order out" (CoMan OS) — бросай хаос, система разберёт → снижает барьер входа
- "Вот каркас, внутри него ты свободен" (Second AI Brain) — структура подсказывает → снижает барьер мышления

**Для фреймворка:** Это позиционное отличие. Каркас масштабируется через копирование (scaffold), автоструктурирование — через технологию (дороже, сложнее). Для нашей аудитории (руководитель + малая команда) каркас ценнее.

**Клиентское подтверждение:** Клиенты платили именно за понятную чёткую структуру — "делай вот это, и всё работает".

→ Куда: framework/methodology/ (принцип), возможно framework/architecture.md (позиционирование)

### 2026-02-27 — Дух файла: стиль текста влияет на поведение AI

**Контекст:** Обсуждение с Claude о том, как стиль написания файлов влияет на качество ответов.

**Инсайт:** LLM считывает стиль, тон, энергию текста и подстраивается (in-context learning на уровне стиля). Vision, написанный с душой ("камень за камнем, как дворец Шеваля"), буквально делает помощников лучше — задаёт рамку: терпение, кумулятивность. Vision "для галочки" — даёт сухие формальные ответы.

**Для фреймворка:** Это не эзотерика, а механика контекстного окна. Значит scaffold-шаблоны должны не только задавать структуру, но и задавать тон. Примеры в шаблонах важны не меньше полей.

**Практический вывод:** В methodology/ стоит добавить принцип: "пиши файлы так, как хочешь чтобы AI с тобой разговаривал".

→ Куда: framework/methodology/ (принцип), framework/scaffold/ (примеры в шаблонах)

### 2026-02-28 — Anthropic knowledge-work-plugins как референс для блока "помощники"

**Контекст:** Anthropic выложили open source репо [knowledge-work-plugins](https://github.com/anthropics/knowledge-work-plugins) — 15 first-party + 4 partner-built плагинов для knowledge work ролей: productivity, sales, customer-support, product-management, marketing, legal, finance, data, enterprise-search, bio-research, design, engineering, HR, operations + partner (apollo, brand-voice, common-room, slack). Каждый плагин = набор skills (команды) + .mcp.json (коннекторы к внешним сервисам). Работают в Cowork и Claude Code. Пост в Telegram (ProductsAndStartups): автор попробовал data-плагин на статистике Telegram-канала — 6 команд (/explore-data, /analyze, /write-query, /create-viz, /build-dashboard, /validate), за 2 минуты получил интерактивный дашборд.

**Инсайт 1 — Паттерн "одна роль = один набор скиллов + коннекторы":**
Anthropic фактически стандартизировали паттерн ролевого помощника: роль определяется через набор domain-specific команд + подключения к данным через MCP. Это тот же паттерн, который Second AI Brain закладывает в слой 3 "Помощники" (architecture.md): skills по методологии + коннекторы к данным клиента. Разница: у Anthropic роли generic (для любого маркетолога), у нас — роли привязаны к scaffold-структуре и методологии конкретного руководителя.

**Инсайт 2 — Идея помощника "Data Scientist для аналитики":**
Data-плагин показывает реальный use case: сырые данные (CSV, SQL, API) → профилирование → вопросы → визуализация → дашборд. Для клиентов Second AI Brain это может быть помощник "по запросу", который анализирует бизнес-метрики: SEO, маркетинговая воронка, engagement, unit economics. Не нужен штатный аналитик — руководитель загружает данные, помощник строит картину. Добавить в каталог будущих помощников plugin/.

**Инсайт 3 — Каталог ролей как ориентир:**
15 ролей Anthropic — готовый чеклист: какие помощники бывают в knowledge work. Не все релевантны для Second AI Brain (bio-research — нет), но несколько прямо ложатся: productivity (задачи, ритуалы), product-management (roadmap, specs), operations (процессы). Полезно как input при проектировании каталога помощников в plugin/.

→ Куда: framework/architecture.md (валидация паттерна "помощники"), framework/plugin/ (каталог ролей, идея data-помощника)

### 2026-02-28 — Три модели структурирования знаний (CoMan OS / Abdullin / Second AI Brain)

**Контекст:** Анализ архитектуры Рината Абдуллина (abdullin.com) + сравнение с CoMan OS (Максим Опилкин). Детали: `framework/_inbox/scaffold/ARCHITECTURE_rinat.md`, `framework/_inbox/scaffold/ARCHITECTURE_maxim.md`.

**Инсайт — три модели, и наша = гибрид:**
- **CoMan OS (Максим):** хаос → система структурирует сверху вниз через OKR-дерево. Connecting tree задаёт единую причинную иерархию.
- **Codex (Ринат):** хаос → AI дистиллирует + автоматически строит связи снизу вверх. Pipeline: capture → distill → reflect.
- **Second AI Brain (мы):** скелет задаёт рамку → AI наращивает внутри. Не "человек заполняет каркас" и не "AI строит из хаоса", а двусторонняя система: scaffold = скелет (форма, рамка мышления), AI = мясо (связи, грани, enrichment).

**Метафора:** Скелет определяет форму организма. Но мясо, связки, нервы — наращиваются уже внутри, в том числе с помощью AI.

**Что можно взять:**
- **Дистилляция** (от Рината) — расширить pipeline за пределы встреч: ссылки, голосовые заметки, PDF → карточка знания. Естественное развитие `_inbox/`.
- **Автоматические связи** (от Рината) — агент при добавлении файла проверяет, с чем он связан. Сейчас у нас связи вручную в YAML.
- **AGENT_EDITABLE блоки** (от Рината) — разметка зон в scaffold, где AI может писать. Разделение "человеческих" и "агентских" секций.
- **SOUL.MD** (от Рината) — подтверждение инсайта "дух файла". Ценности и тон в файле формируют поведение AI.

→ Куда: framework/architecture.md (модель "скелет + AI enrichment"), framework/methodology/ (принцип двусторонней работы), framework/plugin/agents/ (auto-linking agent)

### 2026-03-01 — Chunk-based Architecture: файл = чанк, контекст = сборка

**Контекст:** Собственная идея Виктора (из задачника) + пост Дмитрия Матскевича (@whyaimatskevich, t.me/whyaimatskevich/183, /197) про second brain как инфраструктуру слоёв: raw → alignment → UFC (Unified Field Context) → agent.

**Исходная формулировка (Виктор):** "Если файл больше чанка — это плохой файл. Есть файлы/пазлы/детали, есть сборки (детерминированные и агентские — агент знает какие есть файлы и на первом шаге понимает какая ему нужна сборка, а на следующем шаге происходит сборка, которая уже подгружается в контекст)."

**Инсайт — три слоя архитектуры:**

1. **Чанки** — атомарные файлы (~50 строк), каждый выражает одну сущность/мысль. Не vision целиком, а отдельно: видение, ценности, долгосрочная цель, стиль лидерства. У чанков — теги и связи (граф). Файл больше чанка = плохой файл (сигнал, что в нём смешаны разные сущности).

2. **Предустановленные сборки** — файлы-рецепты без собственного контента. Описывают какие чанки подтянуть. Пример: "vision = видение + долгосрочная цель + ценности". Аналог текущих файлов (vision, goal, plan), но декларативные, не монолитные.

3. **Динамические сборки** — двухходовка агента: (1) определить какие чанки нужны для текущей задачи, (2) собрать их в контекстный файл → передать координатору. Агент сам решает какая сборка нужна — не по шаблону, а по задаче.

**Аналогия:** человеческая память — не готовые документы, а фрагменты с тегами и связями (граф), которые мозг пересобирает под задачу каждый раз заново.

**Связь с моделью Матскевича:**
- Его raw (слой реальности) ≈ наши чанки (атомарные факты)
- Его alignment/goals (слой целей) ≈ наши предустановленные сборки (фильтр через цели)
- Его UFC = f(raw, goal) ≈ наши динамические сборки (контекст под задачу)
- Его "second brain = не агент, а инфраструктура" ≈ наша архитектура: данные + память + помощники
- Наша модель конкретнее: определяет формат хранения (чанк как файл) и два механизма сборки (предустановленный + динамический)

**Что меняет в текущем фреймворке:**
- Принцип онтологии "один файл — одна роль" усиливается до "один файл — один чанк"
- Текущие файлы (vision, goal, plan) становятся либо чанками, либо сборками — но не гибридами
- Протокол чтения для LLM (methodology.md) заменяется на механизм сборок
- Сборки (architecture.md, "будущее развитие") из будущего становятся центральным слоем
- file_spec.md уже содержит принцип "каждая секция — автономный чанк" — теперь это поднимается на уровень файла

**Открытые вопросы:**
- Гранулярность: как определить границу чанка? "50 строк" — жёсткое правило или ориентир?
- Текущие файлы типа plan (ежедневно обновляется, содержит день + неделю + период) — как разбить на чанки без потери удобства?
- Формат рецепта сборки: YAML-список файлов? Промпт? Декларативное описание?
- Хранение графа связей: в YAML каждого чанка? Отдельный индекс?

→ Куда: framework/architecture.md (принцип "чанк как единица хранения, сборка как единица контекста"), framework/ontology/ (правила чанкования), framework/methodology/ (протокол сборок вместо протокола чтения)

### 2026-03-10 — Позиционная модель: скелет + динамическое разворачивание

**Контекст:** Формирование product vision. Виктор фиксирует ключевое позиционное отличие Second AI Brain от двух полюсов на рынке.

**Инсайт — три модели, наша = середина:**

| Полюс | Пример | Принцип |
|-------|--------|---------|
| Автоструктурирование | ArsContexta, CoMan OS (Опилкин) | Хаос на входе → AI наводит порядок |
| Жёсткий каркас | Классические шаблоны, ноушен-сетапы | Заполни N файлов по инструкции |
| **Середина (мы)** | Second AI Brain | Скелет на уровне сущностей + динамическое разворачивание |

**Суть:** Мы работаем на уровне сущностей (vision, goal, plan, decisions, meetings) — это не файлы, а типы мышления. Скелет задаёт форму и подсказывает о чём думать, но какие сущности нужны, в каком объёме, как связаны — определяется контекстом конкретного руководителя. Scaffold — не жёсткая структура, а динамически разворачиваемый каркас.

**Связь с предыдущими инсайтами:**
- "Scaffold как каркас мышления" (2026-02-27) — подтверждает: scaffold подсказывает, а не диктует
- "Три модели структурирования" (2026-02-28) — уточняет: наша модель "скелет + AI enrichment" — это именно середина, не один из полюсов
- "Chunk-based architecture" (2026-03-01) — технический механизм этой середины: сущности = чанки, scaffold = предустановленные сборки

**Для фреймворка:** Это не техническая деталь, а позиционное решение. Должно быть зафиксировано в architecture.md и отражено в методологии онбординга (scaffold разворачивается под клиента, а не клиент подстраивается под scaffold).

→ Куда: framework/architecture.md (позиционирование), framework/methodology/ (принцип динамического разворачивания), framework/00_product.md (зафиксировано)

### 2026-03-18 — Google Docs как альтернативный формат данных клиента

**Контекст:** Тестирование Google Drive коннектора в Claude Project. Проверка Drive Search, Drive Fetch, RAG, scope-контроля.

**Инсайт:** Клиенту не обязателен Markdown-scaffold. Google Docs — нативный формат с нулевым порогом входа. Клиент работает в привычном инструменте, Claude читает через Drive Fetch. Это потенциально меняет слой "Данные" в архитектуре.

**Факты (верифицированы):**
- Drive Fetch читает полный текст Google Doc по ID, актуальная версия
- Drive Search фильтрует по folder ID, но только один уровень вглубь (не рекурсивно)
- RAG в Project Knowledge отстаёт на 2+ часа — нельзя полагаться на свежесть
- Только Google Docs — Sheets, Slides, Markdown не читаются через Drive API
- semantic_query игнорирует фильтр по папке — scope ненадёжен на уровне API
- Scope ограничивается системной инструкцией (folder ID + правила), не API-фильтрами
- Форматирование частично сохраняется: H2 → ##

**Что затрагивает в фреймворке:**
- **architecture.md** — слой "Данные" сейчас = Markdown. Google Docs = альтернативная реализация того же слоя. Нужно обобщить
- **scaffold/** — два формата доставки: Markdown (Claude Code / техн. клиенты) и Google Docs (Claude Project / руководители). Содержание одно, упаковка разная
- **00_product.md** — "данные — открытый Markdown" → обобщить до "открытые данные у клиента"
- **methodology/onboarding.md** — радикально упрощается для Google Docs клиентов
- **Ключевые файлы** — дублировать в Project Knowledge для быстрого доступа, но актуальные данные брать через Drive Fetch

**Архитектурные выводы для клиентов:**

Клиент работает в Google Docs — ноль порог входа, привычный инструмент
Markdown-scaffold для клиентов не нужен — Google Docs нативнее и проще
Scope ограничивается системной инструкцией (folder ID + правила "где что искать"), а не API-фильтрами
Актуальные данные — через Drive Fetch по ID, не через RAG
Ключевые файлы — дублировать в Project Knowledge для быстрого доступа, но не полагаться на их свежесть

→ Куда: framework/architecture.md (обобщить слой "Данные"), framework/scaffold/ (два формата), framework/00_product.md (уточнить формулировку), framework/methodology/onboarding.md (упрощение)
