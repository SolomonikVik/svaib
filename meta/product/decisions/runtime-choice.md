# Выбор runtime для Second AI Brain

## Кратко

Решение на текущем этапе: **не писать свой runtime с нуля**, а использовать **готовый lightweight runtime как delivery-слой**. Текущий основной кандидат — `HKUDS/nanobot`.

Важно: это **не окончательная ставка на конкретный инструмент навсегда**, а продуктовое решение о границах продукта. Наш продукт — не runtime, а **управленческий дизайн, методология и plugin-слой** поверх клиентских данных.

## Статус

**Accepted.** Решение принято по направлению, но конкретный runtime ещё проходит пилотную проверку.

**Update 28.03.2026:** Обнаружен новый кандидат — Claude Code Scheduled Tasks (cloud-режим). По итогам первого теста выглядит сильнее nanobot для нашего кейса, т.к. это нативный Claude без посредников. Подробнее — в секции «Альтернативы».

## Решение

На текущем этапе **не пишем свой runtime с нуля**, а используем **готовый lightweight runtime как delivery-слой**.

Граница продукта фиксируется так:

**Second AI Brain = данные клиента + наша методология + plugin-слой.**

Runtime, интерфейс, orchestration engine и конкретный AI-host — **заменяемые детали**, а не ядро продукта.

Текущий основной кандидат для пилотного delivery-слоя:

- **`HKUDS/nanobot`**.

Это решение означает:

- **решение принято по направлению** — не писать свой runtime сейчас;
- **конкретный runtime ещё проходит проверку** — `nanobot` выбран как основной кандидат для короткого пилота.

## Контекст

Весной 2026 у нас возник вопрос: что делать с delivery-слоем продукта.

С одной стороны, есть соблазн собрать свой минимальный runtime из зрелых кирпичей. С другой — это уводит внимание в инфраструктурную разработку, тогда как ценность svaib лежит в другом месте.

Из [product_vision.md](../product_vision.md), ранних клиентских разговоров и последних экспериментов видно:

- клиентам нужен не "ещё один AI-фреймворк", а работающие управленческие циклы;
- барьер входа критичен: сложные инженерные инструменты снижают adoption;
- выигрыш даёт не код рантайма, а качество ритуалов, контекста, навыков и автоматизации;
- наш delivery должен быть простым, быстрым и заменяемым.

Отдельный фактор: недавний опыт с meeting analysis показал, что избыточная самодельная сложность легко съедает десятки часов без пропорционального прироста качества. Это усиливает тезис: **не строить лишнюю инфраструктуру там, где она не является нашим core value**.

Почему `nanobot` сейчас выглядит лучшим кандидатом:

- зрелость для новой волны agent runtimes: около **34K stars** на момент исследования;
- формат `SKILL.md` совпадает с нашей логикой skills и снижает стоимость адаптации;
- `HEARTBEAT.md` хорошо ложится на наш паттерн ритуалов и proactive-циклов;
- есть `Telegram`, `MCP` и scheduled-механики из коробки;
- multi-instance запуск через отдельные конфиги лучше совпадает с нашей клиентской изоляцией;
- model-agnostic и lightweight, то есть не тянет нас в vendor lock-in и не раздувает delivery-слой;
- ближе к нашей логике `skills / automation / workspace`, чем тяжёлые экосистемы вроде OpenClaw;
- снижает time-to-demo по сравнению с самописным рантаймом.

Но это ещё не означает, что он идеально совпадает с нашим plugin-слоем без адаптации.

## Альтернативы

### Claude Code Scheduled Tasks (discovery 28.03.2026)

Cloud-режим Claude Code: задачи крутятся на серверах Anthropic по расписанию (cron). Комп не нужен. UI: claude.ai/code/scheduled.

Почему это сильный кандидат:

- **Нативный Claude** — наши скиллы и промпты работают без адаптерного слоя, это тот же Claude Code;
- **Cloud без инфраструктуры** — не нужен VPS, Docker, self-hosting;
- **Коннекторы из коробки** — Google Drive, Google Calendar, GitHub;
- **Bash (curl)** — протестирована отправка в Telegram через Bot API, работает;
- **Клиент не видит сложности** — для клиента это Telegram + Google Drive, без GitHub/терминала/CLI.

Потенциальная продуктовая схема для клиента:
- Наша сторона: скилл (методология + промпт) + Telegram-бот (@svaib_bot, один на всех клиентов)
- Клиентская сторона: Google Drive с файлами (scaffold) + Telegram
- Связка: cloud scheduled task + коннектор Google Drive + коннектор Google Calendar + curl в Telegram Bot API

Закрывает все три критерия проверки из этого документа:
1. Приём запроса через простой интерфейс — Telegram
2. Proactive сценарий по расписанию — cron
3. Наш методологический слой без переписывания — скиллы работают нативно

Риски:
- Research preview — возможности, лимиты и pricing могут измениться;
- Vendor lock-in на Anthropic (но мы и так на Claude);
- Неизвестны ограничения по количеству задач, длительности выполнения, объёму данных.

**Статус:** протестировано базово (28.03.2026), требуется полноценный пилот на реальном клиентском сценарии.

### OpenClaw

Не берём как базовый runtime сейчас:

- слишком тяжёлый и сложный для нашей стадии;
- security-профиль хуже подходит для CEO-кейса с чувствительными данными;
- тащит за собой большой объём лишней платформенной сложности.

### Свой runtime

Не берём сейчас:

- это уводит нас в инфраструктурную разработку вместо методологии;
- у нас уже есть негативный опыт, когда самодельная сложность съедала непропорционально много времени;
- это не усиливает наш главный продуктовый рычаг.

### NanoClaw

Не берём сейчас:

- слишком сильная привязка к конкретному вендорному стеку;
- хуже совпадает с нашим требованием к model-agnostic delivery-слою;
- для нас риск lock-in сейчас важнее, чем выигрыш от минимализма.

### ZeroClaw

Пока не берём, но держим в watchlist:

- выглядит серьёзно по безопасности и инженерной дисциплине;
- потенциально силён как future runtime;
- но на текущем этапе проигрывает по скорости кастомизации и time-to-demo.

## Следствия

### Для продукта

- усиливаем позицию "методология важнее инструмента";
- снижаем риск уйти в инфраструктурное R&D;
- можем быстрее показать клиенту работающий сценарий.

### Для framework

- нужно чётче зафиксировать, что такое наш plugin-контракт независимо от конкретного runtime;
- нужно отделить сущность "наш skill/agent/hook" от специфики Claude Code / Cowork;
- нужно описать минимальный адаптерный слой между нашим plugin и внешним runtime.

### Для dev

- не начинаем писать собственный runtime как отдельный продукт;
- проверяем `nanobot` на реальном сценарии;
- оцениваем стоимость адаптации и риски vendor-shape.

## Что этим решением НЕ утверждается

- Мы **не утверждаем**, что `nanobot` — наш постоянный runtime на годы.
- Мы **не утверждаем**, что текущий формат `plugin/` уже полностью runtime-agnostic.
- Мы **не утверждаем**, что нужно перестраивать продукт под экосистему `nanobot`.
- Мы **не утверждаем**, что свой runtime никогда не понадобится.

Это решение только о ближайшем фокусе: **не тратить силы на создание delivery-инфраструктуры раньше времени**.

## Критерии проверки решения

Решение считается подтверждённым, если в коротком пилоте внешний runtime позволяет без существенной самописной инфраструктуры реализовать 3 вещи:

1. Приём запроса через простой интерфейс для клиента.
2. Запуск одного проактивного сценария по расписанию.
3. Подключение нашего методологического слоя без переписывания логики продукта под конкретный runtime.

Минимальный сценарий проверки:

1. Обработка встречи.
2. Подготовка к ближайшей встрече.
3. Короткий proactive brief в Telegram.

Если для этого потребуется писать заметный объём glue-кода, обходить ограничения платформы или менять структуру нашего plugin под runtime, решение нужно пересмотреть.
