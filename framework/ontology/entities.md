# Сущности v2 (DRAFT)

> **DRAFT** — черновая версия, будет дорабатываться. Не опираться как на факт. Требует валидации на реальных клиентских кейсах и стабилизации границ между сущностями.

## Контекст: откуда и зачем

Мы строили модель анализа транскриптов встреч (синтез из трёх источников: наша практика, Опилкин/Go2OKR, обзор исследований). В процессе пришли к инсайту: сущности типа "решение", "задача", "идея" — не изобретение для встреч. Они уже неявно существовали в файловой онтологии Second AI Brain (decisions.md хранит решения, ideas.md — идеи, plan.md — задачи). Но нигде не были описаны как самостоятельные единицы.

**Ключевой инсайт:** онтология должна строиться от сущностей, а не от файлов. Файлы — контейнеры. Сущности — то, что в них живёт. Встреча, планирование, рефлексия — события-источники, которые порождают одни и те же сущности.

**Зачем нужен этот каталог:**
- Встреча — не отдельный мир со своей классификацией. Это событие, из которого извлекаются универсальные бизнес-сущности и раскладываются по своим местам
- Скилл анализа транскриптов не изобретает свои категории, а маппит речь на существующие сущности (речевые маркеры → тип сущности → целевой файл)
- Единый словарь сущностей для всего фреймворка: scaffold, methodology, plugin говорят на одном языке

**Как собирались (v1 → v2):**
- v1: 13 сущностей плоским списком. 10 вытащены из существующих файлов онтологии, 3 (проблема/блокер, открытый вопрос, факт) выявлены при анализе транскриптов
- v2: переработка после исследования 40+ фреймворков и валидации в 3 LLM. Введена типология (атомарные / коллекции / композиции). Vision и Стратегия стали композициями. Добавлены Принцип/Ценность и Риск. Убрана Веха (поглощена Фактом с датой)
- v3: добавлена атомарная сущность #11 Запрос/Потребность — выраженное ожидание от любого актора (себя, клиента, партнёра). Выявлена при тестировании на реальном транскрипте встречи

## Архитектура модели

### Два слоя

```
Слой 1: СУЩНОСТИ          →  Что существует и меняется
        (атомарные,            Единый каталог: люди, задачи, решения,
         коллекции,            проекты, идеи — всё здесь
         композиции)
        ↕
Слой 2: РИТУАЛЫ            →  В процессе чего создаются/меняются сущности
        И ИСТОЧНИКИ            Встреча, планирование, рефлексия, стратсессия...
```

### Три типа сущностей

| Тип | Определение | Аналогия |
|-----|-------------|----------|
| **Атомарная** | Неделимая единица. Извлекается из потока, имеет одно чёткое определение | `string`, `number` — примитив |
| **Коллекция** | Упорядоченный набор атомарных одного типа. Может быть пустой или неполной | `List<Task>` — список однотипных |
| **Композиция** | Сборка из разных сущностей. Части связаны — целое теряет смысл без ключевых частей. Состав мягкий: варьируется от контекста | `Object { partA, partB, partC }` — структура из разнотипных частей |

### Атомарные (11)

| # | Сущность | Определение | Ключевые атрибуты | Контейнер |
|---|----------|-------------|-------------------|-----------|
| 1 | **Цель** | Измеримый результат к сроку | Горизонт (день/неделя/период/год/долгосрочная), метрики успеха, статус | 02_goal.md |
| 2 | **Решение** | Выбор, закрывающий альтернативы | Аргумент/обоснование, альтернативы, статус (в силе/пересмотрено) | decisions.md |
| 3 | **Задача** | Конкретное действие с owner | Owner, срок, статус | 03_plan.md |
| 4 | **Идея** | Предложение без обязательства | Область, источник | ideas.md |
| 5 | **Инсайт** | Новое понимание реальности, меняющее ментальную модель. Не содержит действия (≠ Идея), не проверяется одной метрикой (≠ Факт), не закрывает альтернативы (≠ Решение) | Область, поддерживающие факты, импликации | reflections.md |
| 6 | **Факт** | Объективное, проверяемое утверждение | Источник, дата, область | profile.md, company.md, knowledge/ |
| 7 | **Принцип / Ценность** | Устойчивое правило или убеждение, которым руководствуемся | Область (личный/командный/продуктовый) | 01_vision.md, my_profile.md |
| 8 | **Метрика** | Измеримый показатель с числовым выражением | Название, период, план (целевое), факт (фактическое) | metrics.md, 02_goal.md |
| 9 | **Открытый вопрос** | Поднято, но не решено; требует возврата | Контекст, срочность | 03_plan.md, ideas.md |
| 10 | **Риск** | Потенциальная негативная ситуация, которая ещё не случилась | Вероятность, влияние, митигация | 03_plan.md, ideas.md |
| 11 | **Запрос / Потребность** | Выраженное ожидание или потребность от любого актора (клиент, партнёр, сам руководитель). Ещё не Цель и не Задача — входящий сигнал | Источник (кто), суть (что хочет), статус (выявлен → оценён → принят/отклонён) | ideas.md, profile.md |

#### Речевые маркеры (для AI-классификации)

| Сущность | Маркеры в речи |
|----------|---------------|
| Цель | "хотим достичь", "к концу квартала", число + срок |
| Решение | "решили", "выбираем A, отказываемся от B", "договорились" |
| Задача | имя + действие + срок: "Вася подготовит к пятнице" |
| Идея | "а что если", "можно попробовать", "хорошо бы" |
| Инсайт | "оказывается", "я понял что", "на самом деле", "проблема не в X, а в Y" |
| Факт | число, ссылка на источник: "у нас 200 сотрудников", "churn = 15%" |
| Принцип | "мы всегда", "мы никогда", "для нас важно" |
| Метрика | "будем мерить", "показатель", "KPI", "план X — факт Y" |
| Открытый вопрос | "нужно разобраться", "непонятно", "вернёмся к этому" |
| Риск | "если не..., то...", "есть опасность", "может случиться" |
| Запрос / Потребность | "хочу", "мне нужно", "нам нужен", "было бы здорово", "можно ли", "ожидаю", "не хватает" |

#### Правила разграничения

| Пара | Как различать |
|------|--------------|
| Решение vs Задача | Решение = выбор ("делаем пилот"). Задача = действие ("Вася подготовит к пятнице") |
| Решение vs Идея | Решение = зафиксировано, альтернативы закрыты. Идея = предложение, можно не делать |
| Идея vs Задача | Идея = "хорошо бы". Задача = конкретный owner и действие |
| Инсайт vs Факт | Факт = объективное, проверяемое ("churn = 15%"). Инсайт = интерпретация ("churn из-за онбординга, а не цены") |
| Инсайт vs Идея | Инсайт = понимание "почему" (описывает реальность). Идея = предложение "что делать" (описывает возможность) |
| Инсайт vs Решение | Инсайт открывает пространство альтернатив. Решение закрывает альтернативы |
| Риск vs Факт | Риск = ещё не случилось (P < 100%). Факт = уже есть (P = 100%). Реализовавшийся Риск → Факт |
| Принцип vs Решение | Принцип = устойчивое правило ("мы не работаем с госсектором"). Решение = выбор в ситуации ("уходим от клиента X") |
| Метрика vs Факт | Метрика = числовой показатель, отслеживается в динамике (план/факт). Факт = единичное утверждение |
| Запрос vs Факт | Факт = реальность ("у нас 200 человек"). Запрос = ожидание изменения ("хочу архитектуру хранения") |
| Запрос vs Идея | Запрос = "мне/ему нужно X" (потребность актора). Идея = "а что если попробовать Y" (предложение решения). Запрос описывает **что хочется**, Идея — **как можно сделать** |
| Запрос vs Задача | Запрос = входящий сигнал, может быть отклонён. Задача = взята в работу, есть owner |
| Запрос vs Цель | Запрос = выраженная потребность любого актора. Цель = наш измеримый результат к сроку. Принятый Запрос → Цель или Задача |

## Принципы модели

1. **Бритва Оккама** — новая сущность только если нельзя выразить через существующие И без неё теряется информация
2. **Один факт = одно место** — не дублировать между сущностями
3. **Каждая сущность отличима** — AI и человек могут надёжно отличить от соседних (таблица разграничений)
4. **Сущности определяются независимо от контейнеров** — файлы-контейнеры описаны в ontology.md
5. **Ключевые атрибуты, не обязательные** — набор атрибутов варьируется от проекта к проекту
6. **Композиции мягкие** — состав меняется от клиента к клиенту, ключевые части фиксированы
