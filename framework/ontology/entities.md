# Сущности (DRAFT)

> **DRAFT** — черновая версия, будет дорабатываться. Не опираться как на факт. Требует валидации на реальных клиентских кейсах и стабилизации границ между сущностями.

## Кратко

Каталог типов бизнес-информации Second AI Brain. Определяет какие сущности существуют в управленческом домене (решение, задача, цель, идея...), как их отличать друг от друга, и по каким правилам классифицировать. 

## Связанные файлы

- ontology.md — файловая структура (контейнеры для сущностей)
- rituals.md — каталог ритуалов (когда какие сущности создаются/обновляются)
- ../methodology/methodology.md — протоколы, decision frames, масштабирование
- ../methodology/meeting_analysis.md — методология анализа транскриптов: маппинг речи → сущности (DRAFT)
- architecture.md — как слои данных, памяти и помощников работают вместе

## Архитектура модели

### Два слоя

```
Слой 1: СУЩНОСТИ          →  Что существует и меняется
        (атомарные,            Единый каталог: люди, задачи, решения,
         коллекции,            проекты, идеи — всё здесь
         композиции)
        ↕
Слой 2: РИТУАЛЫ            →  В процессе чего создаются/меняются сущности
        И ИСТОЧНИКИ            Встреча, планирование, рефлексия, стратсессия...
```

### Три типа сущностей

| Тип | Определение | Аналогия |
|-----|-------------|----------|
| **Атомарная** | Неделимая единица. Извлекается из потока, имеет одно чёткое определение | `string`, `number` — примитив |
| **Коллекция** | Упорядоченный набор атомарных одного типа. Может быть пустой или неполной | `List<Task>` — список однотипных |
| **Композиция** | Сборка из разных сущностей. Части связаны — целое теряет смысл без ключевых частей. Состав мягкий: варьируется от контекста | `Object { partA, partB, partC }` — структура из разнотипных частей |

## Слой 1: Сущности

### Атомарные (11)

| Сущность | Определение | Ключевые атрибуты |
|----------|-------------|-------------------|
| **Цель** | Желаемое состояние или результат к сроку (outcome, не action) | Горизонт (день/неделя/период/год/долгосрочная), метрики успеха, статус |
| **Решение** | Выбор, закрывающий альтернативы | Аргумент/обоснование, альтернативы, статус (в силе/пересмотрено) |
| **Задача** | Конкретное действие с owner | Owner, срок, статус |
| **Идея** | Предложение без обязательства. Может быть просто предложением ("а что если X"), а может — гипотезой в проверяемой форме ("если X, то Y") | Область, источник |
| **Инсайт** | Новое понимание реальности, меняющее ментальную модель. Не содержит действия (≠ Идея), не проверяется одной метрикой (≠ Факт), не закрывает альтернативы (≠ Решение) | Область, поддерживающие факты, импликации |
| **Факт** | Объективное, проверяемое утверждение | Источник, дата, область |
| **Принцип / Ценность** | Устойчивое правило или убеждение, которым руководствуемся | Область (личный/командный/продуктовый) |
| **Метрика** | Измеримый показатель с числовым выражением | Название, период, план (целевое), факт (фактическое) |
| **Открытый вопрос** | Поднято, но не решено; требует возврата | Контекст, срочность |
| **Риск** | Потенциальная негативная ситуация, которая ещё не случилась | Вероятность, влияние, митигация |
| **Запрос / Потребность** | Выраженное ожидание или потребность от любого актора (клиент, партнёр, сам руководитель). Ещё не Цель и не Задача — входящий сигнал | Источник (кто), суть (что хочет), статус (выявлен → оценён → принят/отклонён) |

#### Речевые маркеры (для AI-классификации)

| Сущность | Маркеры в речи |
|----------|---------------|
| Цель | "хотим достичь", "к концу квартала", число + срок, "хотим выйти на", "довести до", "чтобы стало" |
| Решение | "решили", "выбираем A, отказываемся от B", "договорились" |
| Задача | имя + действие + срок: "Вася подготовит к пятнице", "сделай", "подготовь", "напиши" (императив + owner) |
| Идея | "а что если", "можно попробовать", "хорошо бы", "предположим что", "если мы сделаем X, то Y" |
| Инсайт | "оказывается", "я понял что", "на самом деле", "проблема не в X, а в Y" |
| Факт | число, ссылка на источник: "у нас 200 сотрудников", "churn = 15%" |
| Принцип | "мы всегда", "мы никогда", "для нас важно" |
| Метрика | "будем мерить", "показатель", "KPI", "план X — факт Y" |
| Открытый вопрос | "нужно разобраться", "непонятно", "вернёмся к этому" |
| Риск | "если не..., то...", "есть опасность", "может случиться" |
| Запрос / Потребность | "хочу", "мне нужно", "нам нужен", "было бы здорово", "можно ли", "ожидаю", "не хватает" |

#### Правила разграничения

| Пара | Как различать |
|------|--------------|
| Решение vs Задача | Решение = выбор ("делаем пилот"). Задача = действие ("Вася подготовит к пятнице") |
| Решение vs Идея | Решение = зафиксировано, альтернативы закрыты. Идея = предложение, можно не делать |
| Идея vs Задача | Идея = "хорошо бы". Задача = конкретный owner и действие |
| Инсайт vs Факт | Факт = объективное, проверяемое ("churn = 15%"). Инсайт = интерпретация ("churn из-за онбординга, а не цены") |
| Инсайт vs Идея | Инсайт = понимание "почему" (описывает реальность). Идея = предложение "что делать" (описывает возможность) |
| Инсайт vs Решение | Инсайт открывает пространство альтернатив. Решение закрывает альтернативы |
| Риск vs Факт | Риск = ещё не случилось (P < 100%). Факт = уже есть (P = 100%). Реализовавшийся Риск → Факт |
| Принцип vs Решение | Принцип = устойчивое правило ("мы не работаем с госсектором"). Решение = выбор в ситуации ("уходим от клиента X") |
| Метрика vs Факт | Метрика = числовой показатель, отслеживается в динамике (план/факт). Факт = единичное утверждение |
| Запрос vs Факт | Факт = реальность ("у нас 200 человек"). Запрос = ожидание изменения ("хочу архитектуру хранения") |
| Запрос vs Идея | Запрос = "мне/ему нужно X" (потребность актора). Идея = "а что если попробовать Y" (предложение решения). Запрос описывает **что хочется**, Идея — **как можно сделать** |
| Запрос vs Задача | Запрос = входящий сигнал, может быть отклонён. Задача = взята в работу, есть owner |
| Запрос vs Цель | Запрос = выраженная потребность любого актора. Цель = наш измеримый результат к сроку. Принятый Запрос → Цель или Задача |
| Цель vs Задача | Цель = желаемое состояние, результат (outcome). Задача = конкретное действие (output). Цель отвечает на "что должно измениться?", Задача — на "кто что делает?". Одна Цель может порождать несколько Задач |
| Метрика vs Цель | Метрика = шкала, показатель (конверсия, churn, выручка). Цель = точка на шкале + срок ("конверсия 5% к Q2"). Метрика существует без цели. Цель ссылается на Метрику как критерий успеха |
| Открытый вопрос vs Риск | Открытый вопрос = не знаем ответ (проблема знания сейчас). Риск = знаем что может пойти плохо (угроза в будущем). "Непонятно как масштабировать" = вопрос. "Если не масштабируем — потеряем клиента" = риск. Вопрос может породить Риск после разбора |
| Инсайт vs Принцип | Инсайт = новое понимание реальности, описательный ("клиенты уходят из-за скорости ответа"). Принцип = правило действия, предписательный ("отвечаем в течение 15 минут"). Инсайт может затвердеть в Принцип, но не каждый Инсайт становится Принципом |
| Запрос vs Открытый вопрос | Запрос = есть актор с потребностью ("клиенту нужно X"). Открытый вопрос = есть неопределённость без заказчика ("непонятно, какой подход выбрать"). Тест: есть субъект-источник ожидания → Запрос. Есть неясность, требующая разбора → Открытый вопрос |

## Проверки целостности

Правила, которые AI применяет при анализе. Выражают ключевые связи между сущностями без формального графа отношений.

| Проверка | Что ловит |
|----------|-----------|
| Задача без Цели | Действие без понятного "зачем". AI спрашивает: "Ради какой цели эта задача?" |
| Решение без основания | Выбор без опоры. AI фиксирует: что Решение закрывает (Открытый вопрос, Запрос, Идею) и на чём стоит (Факт, Инсайт) |
| Идея не зафиксирована как Решение | Обсудили, всем понравилось, но статус не изменился. AI различает: это *решили* или *предложили*? |
| Метрика без привязки к Цели | Число отслеживается, но непонятно зачем. Метрика — линейка, Цель — зачем мерим |
| Решение противоречит Принципу | Цель хорошая, метрика растёт, но решение нарушает ценности. AI ловит конфликт |

## Принципы модели

1. **Бритва Оккама** — новая сущность только если нельзя выразить через существующие И без неё теряется информация
2. **Один факт = одно место** — не дублировать между сущностями
3. **Каждая сущность отличима** — AI и человек могут надёжно отличить от соседних (таблица разграничений)
4. **Сущности определяются независимо от контейнеров** — файлы-контейнеры описаны в ontology.md
5. **Ключевые атрибуты, не обязательные** — набор атрибутов варьируется от проекта к проекту
6. **Композиции мягкие** — состав меняется от клиента к клиенту, ключевые части фиксированы
