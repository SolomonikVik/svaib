# Протокол встречи — schema

Общий язык протокола как выходного артефакта: каноническое дерево секций, маппинг сущностей → секции, правила оформления. Каждый формат (полный, Telegram, email и др.) сам определяет свой набор секций, но использует этот маппинг и правила.

**Иерархия:**
- `entities.md` → что извлекаем (11 типов атомарных сущностей)
- **этот файл** → как из сущностей собирается протокол (маппинг, секции, правила)
- `L2-prompt-protocol-full.md`, `L2-prompt-protocol-telegram.md` → как рендерим конкретный формат

## Связанные файлы

- [entities.md](entities.md) — канон атомарных сущностей (определения, маркеры, правила разграничения)
- [../methodology/meeting_overlays.md](../methodology/meeting_overlays.md) — каталог L2-надстроек (протокол = одна из них)
- [../skills/meeting-analysis/L2-prompt-protocol-full.md](../skills/meeting-analysis/L2-prompt-protocol-full.md) — реализация: полный протокол

## Каноническое дерево секций

```
КОНТЕКСТ ВСТРЕЧИ
├── Тема / проект
├── Дата
├── Участники
└── Цель встречи (если озвучена)

ИТОГ ВСТРЕЧИ
└── 2-3 предложения: что произошло, прояснилось, изменилось

РЕШЕНИЯ И ДОГОВОРЁННОСТИ

ЗАДАЧИ
├── По исполнителям (кто → что → срок)
└── Исполнитель не назначен

К СЛЕДУЮЩЕЙ ВСТРЕЧЕ
└── Задачи, вопросы и ожидания, привязанные к следующей встрече

ВАЖНАЯ ИНФОРМАЦИЯ
├── Проблемная зона (риски, открытые вопросы)
├── Запросы и потребности
├── Идеи и гипотезы
├── Инсайты и наблюдения
└── Справочные материалы

ТРЕБУЕТ УТОЧНЕНИЯ
```

**Правило:** нет информации — нет секции. Пустые секции не пишем.

**Форматирование:** секции — H2 (`## Решения и договорённости`), подсекции — H3 (`### Исполнитель`). Если платформа не поддерживает Markdown (Telegram) — CAPS LOCK + bold.

## Маппинг: сущности → секции протокола

Канонический маппинг. Тип из выжимки (entities.md) сохраняет смысл — секция протокола задаёт читательскую упаковку, не переклассификацию.

| Тип в выжимке | Секция протокола | Примечание |
|---------------|-----------------|------------|
| Решение | Решения и договорённости | |
| Задача | Задачи | По исполнителям, без owner → "Не назначен" |
| Цель | Решения и договорённости или не включать | "Решили выйти на X к Q2" → решение. "Хотелось бы" → не включать |
| Запрос | Запросы и потребности | Входящий сигнал — что хочет актор |
| Идея | Идеи и гипотезы | |
| Инсайт | Инсайты и наблюдения | |
| Факт | Справочные материалы или не включать | Только если ценно участникам |
| Принцип | Решения и договорённости или Справочные | Принят на встрече → Решения. Озвучен как факт → Справочные |
| Метрика | Справочные материалы | |
| Открытый вопрос | Проблемная зона | |
| Риск | Проблемная зона | |
| Требует уточнения | Требует уточнения | |

## Общие правила (для любого формата)

**Один факт — одно место.** Не дублировать между секциями. Исключение: решение + задача сосуществуют ("решили делать пилот" в Решениях + "Вася подготовит пилот к пятнице" в Задачах — разные слои, не дублирование).

**Протокол = только факты этой встречи.** Додумывать то, чего нет в выжимке — запрещено.

**Досье = только пометки.** Если подано досье — использовать только для пометок ("подтвердили: ...", "уточнили: ..."). НЕ удалять пункты на основании досье, НЕ добавлять информацию из досье.

**Цитаты-опоры убираются.** Они — служебная информация выжимки, не для протокола.

**Живой язык.** "Решили", "договорились", "упёрлись", "застряли". Запрещено: "в рамках обсуждения", "было принято решение", "участники пришли к выводу".

## Форматы-реализации

| Формат | Файл | Особенности |
|--------|------|-------------|
| Полный | `L2-prompt-protocol-full.md` | Все секции, без ограничений длины, Markdown |
| Telegram | `L2-prompt-protocol-telegram.md` | Action-only: итог, задачи, решения и договорённости, к следующей встрече. Лимит ~4000 символов, эмодзи |

Каждый формат — отдельный самостоятельный промпт (архитектурное решение, зафиксировано в `meeting_overlays.md`). Маппинг и правила из этого файла — общий контракт. Каждый формат сам определяет свой набор секций.
