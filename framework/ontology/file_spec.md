# Спецификация файла Second AI Brain — формат, структура, правила

## Кратко

Стандарт файла в системе Second AI Brain: как должна выглядеть любая единица контента, чтобы она одинаково хорошо работала для человека, LLM и RAG. Определяет YAML-заголовок, порядок секций, правила заголовков, связи между файлами, ограничения размера. Онтология описывает КАКИЕ файлы бывают — эта спецификация описывает КАК они устроены.

**Работа с файлом:** reference — обновлять при изменении стандарта. Все scaffold-шаблоны должны соответствовать этой спецификации. При расхождении scaffold и spec — spec приоритетнее.

## Связанные файлы

- ontology.md — типы файлов и их контентная часть (vision, goal, plan...). Spec дополняет: онтология = что внутри, spec = как оформлено
- ../scaffold/principles.md — принципы проектирования структуры (миссия файла, развёртывание, маршрутизация). file_spec = КАК устроен файл, principles = КОГДА создавать и КАК маршрутизировать
- ../methodology/methodology.md — ритуалы работы с файлами (когда обновлять, как переносить)
- ../scaffold/ — шаблоны файлов, должны соответствовать этой спецификации
- ../plugin/agents/ — агенты создают и валидируют файлы по этой спецификации
- ../../knowledge/context/markdown-for-llm.md — исследование-первоисточник (3 модели, научные данные)

## Принципы

1. **LLM + People centered.** Читаемо глазами, парсится моделью без потери смысла
2. **Summary first.** Критические факты — в начале файла (наивысший attention weight)
3. **Каждая секция — автономный чанк.** Понятна без чтения остальных секций
4. **Один файл — одна роль.** Не смешивать план и итоги, цели и решения
5. **Компактность.** Лишнее = шум = потраченные токены
6. **Filename = поисковый запрос.** Имя содержит слова, по которым будут искать
7. **Минимум дублирования.** У каждого факта одно каноническое место

## Именование файлов

- **Формат:** `snake_case.md` — строчные буквы, подчёркивания, без транслита
- **Префиксы:** опционально, конвенция в разработке
- **Принцип:** filename = поисковый запрос. AI-инструменты ищут по именам файлов
- **filename** = сжатая суть (стабильная ссылка), **title** в YAML = развёрнутое описание
- **Management-ядро:** нумерация задаёт порядок чтения (`01_vision.md`, `02_goal.md`...)

### Датированные файлы (meetings/, decisions/)

Файлы, привязанные к конкретной дате (протоколы встреч, транскрипты), используют паттерн:

```
YYYY-MM-DD_{тема}_{тип}.md
```

| Элемент | Правило | Пример |
|---------|---------|--------|
| Дата | ISO 8601, всегда первая (сортировка) | `2026-03-27` |
| Тема | snake_case, английский, содержит кто/что | `one_on_one_vasily`, `strategy_session`, `marketing_review` |
| Тип | Суффикс, определяет формат содержимого | `summary`, `transcript` |

**Типы:**
- `_summary` — обработанная выжимка (структурированный .md)
- `_transcript` — сырая транскрибация (формат как есть: .md или .txt)

**Примеры:**
- `2026-03-27_marketing_review_summary.md` — выжимка встречи по маркетингу
- `2026-03-27_marketing_review_transcript.md` — транскрипт той же встречи
- `2026-03-27_one_on_one_vasily_summary.md` — выжимка 1-on-1 с Василием

**Размещение:** выжимки — в `meetings/`, транскрипты — в `meetings/archive/`.

## Анатомия файла

Точный порядок секций. Обоснование — в скобках.

```
---
YAML frontmatter                     (метаданные для агента)
---
# Заголовок                          (= title; наивысший attention weight)

## Кратко                            (якорный чанк, ключевые слова для поиска)
2-4 предложения, ≤100 слов.

**Работа с файлом:**                 (опционально; инструкция агенту)
Частота, правила, ограничения.

## Связанные файлы                   (граф связей с описанием)
- file.md — почему связан

---

## Контентная секция 1               (каждая — автономный чанк)

## Контентная секция 2

---
```

**Верхний блок** (YAML → Кратко → Работа с файлом → Связи) — агент читает первым и получает полную картину: что за файл, как с ним работать, что рядом. Только потом — контент.

## YAML-заголовок

### Обязательные поля

| Поле | Тип | Назначение | Пример |
|:-----|:----|:-----------|:-------|
| `title` | string | Развёрнутое описание содержания | "Цели и стратегия проекта Alpha на 2026" |
| `updated` | date | Дата последнего смыслового обновления (ISO 8601) | 2026-02-18 |
| `version` | integer | Номер версии. Новый файл = 1, обновление = N+1 | 3 |

### Опциональные поля

| Поле | Когда нужно | Значения |
|:-----|:-----------|:---------|
| `type` | Когда поведение агента неочевидно из контекста | `reference`, `plan`, `log`, `decision`, `procedure` (значения — ниже) |
| `scope` | management/ | Когда файл релевантен: `product_core`, `planning`, `always`... |
| `priority` | management/ | `critical`, `high`, `medium`, `low` |
| `status` | Файлы, которые могут устареть | Зависит от контекста (см. ниже) |
| `source` | knowledge/ | URL первоисточника |
| `source_type` | knowledge/ | `article`, `repo`, `video`, `thread`, `docs`, `talk` |
| `tags` | knowledge/ | Ключевые слова для retrieval |
| `review_by` | knowledge/ | Дата проверки актуальности (+3 месяца) |
| `publish` | knowledge/ | `true` = готово для публикации на сайт |

### Значения type (открытый вопрос)

Маппинг type на онтологические сущности (vision, goal, plan...) в проработке. Пока type — подсказка агенту, не строгое правило. Поведение агента определяется блоком "Работа с файлом".

| type | Семантика | Поведение агента |
|:-----|:----------|:----------------|
| `reference` | Справочная информация | Читать. Менять только по команде |
| `plan` | Оперативный план | Обновлять по ритуалу (ежедневно/еженедельно) |
| `log` | Хронологический лог | Только дописывать. Не менять старые записи |
| `decision` | Принятое решение | Не менять. Учитывать при советах и планировании |
| `procedure` | Инструкция / процедура | Следовать. Не менять без согласования |

### Когда status нужен

Management-ядро (vision, goal, plan, weekly_progress, timeline) — **status не нужен**. Эти файлы всегда актуальны по определению. Их обновляют, не архивируют.

| Контекст | Значения status | Пример |
|:---------|:---------------|:-------|
| decisions | `current` → `superseded` | Решение пересмотрено |
| knowledge/ | `raw` → `processed` → `integrated` → `archived` | Workflow обработки |
| Проектные документы | `draft` → `current` → `archived` | Жизненный цикл документа |

### Правила YAML

- **Плоский.** Без вложенных объектов. LLM путаются в глубокой вложенности.
- **<=12 полей.** Больше — бремя поддержки превышает пользу.
- **Natural language.** Поля должны читаться как текст. LLM обрабатывают YAML как текст, не как структурированные данные.
- **Без технических полей SSG.** `layout`, `permalink`, `draft` — тратят токены без пользы для AI.

## Тело файла

### Секция "Кратко" (обязательна)

2-4 предложения, ≤100 слов. Executive summary + ключевые слова для поиска.

**Что включать:**
- Что это за файл (одно предложение)
- Ключевые термины (слова, по которым будут искать)
- Связь с контекстом (опционально)

**Пример:** *"Цели и стратегия проекта Alpha на 2026. Долгосрочная цель (3 года), стратегия, цель года и периода с метриками успеха."*

### Блок "Работа с файлом" (опционально)

3-5 строк после "Кратко". Инструкция агенту: как работать с этим конкретным файлом.

**Когда нужен:** management-ядро, scaffold-шаблоны, процедуры.
**Когда не нужен:** knowledge-статьи, разовые документы.

**Пример:**
```markdown
**Работа с файлом:** обновлять в начале каждого квартала.
Менять: цель периода, метрики. Не менять: долгосрочную цель (только при пивоте).
При конфликте с plan → приоритет у goal.
```

### Секция "Связанные файлы"

Граф связей **с описанием**. Связь без описания бесполезна — агент не знает, зачем переходить.

**Формат:**
```markdown
## Связанные файлы

- vision.md — миссия и идентичность, goal конкретизирует видение в цели
- plan.md — оперативный план, вытекает из целей этого файла (feeds_into)
- timeline.md — итоги периодов, метрики план vs факт фиксируются там, не здесь
```

Типизация связей (feeds_into, depends_on, supersedes, implements) — указывается в описании, не в отдельном поле.

### Заголовки

Самый важный структурный элемент файла. Одновременно навигация для человека, семантический сигнал для LLM и граница чанка для RAG.

- **Самоописывающие:** "Бюджет проекта Alpha Q1 2026", не "Бюджет"
- **H2 = единица чанкинга.** RAG-системы режут по H2 и переносят путь заголовков в metadata чанка
- **Контекст в заголовке:** включать субъект (проект, период, компанию)

### Автономность секций

Каждая H2-секция Second AI Brain должна быть понятна автономно — без чтения остальных секций файла.

| Правило | Плохо | Хорошо |
|:--------|:------|:-------|
| Повторять субъект | "Он решил..." | "CEO Alpha решил..." |
| Контекст в начале | (сразу к делу) | "В рамках ребрендинга Q1 2026..." |
| Конкретные термины | "Сделали это" | "Запустили A/B тест лендинга" |
| Без отсылок | "Как выше...", "См. раздел 2" | Повторить факт |

## Связи между файлами

### Навигационные vs контентные ссылки

Два типа ссылок с разной стоимостью обслуживания:

| Тип | Где живёт | Пример | Хрупкость |
|:----|:----------|:-------|:----------|
| **Навигационная** | README, секция "Связанные файлы" | `- plan.md — оперативный план` | Низкая: одно место, легко обновить |
| **Контентная** | Внутри текста файла | `См. methodology/meeting_analysis.md` | Высокая: рассыпана по тексту, каскадные правки при перемещении |

**Правило:** минимизировать контентные ссылки на файлы, которые могут переехать. Навигационные ссылки из README — норма. Чем меньше вложенность папок и короче пути — тем меньше ломается при реорганизации.

### Маркеры [SOURCE] / [REF:]

Каждый изменяемый факт имеет ОДНО каноническое место. Маркеры ставит агент, не руководитель.

**Синтаксис:** маркер ставится в конце строки, после факта.

```markdown
// В goal.md (источник):
**Цель Q1:** 5 пилотных клиентов [SOURCE]

// В plan.md (ссылка):
Цель квартала: 5 пилотных клиентов [REF: goal.md]
```

**Правила:**
1. Каждый изменяемый факт — один файл с `[SOURCE]`
2. Все остальные упоминания — `[REF: файл.md]` (ссылка на файл, не на секцию)
3. При изменении SOURCE → агент находит все REF → проверяет актуальность

**Когда маркировать:** ключевые метрики, цели, решения — факты, которые часто меняются и цитируются в нескольких файлах. Стабильные факты — можно не маркировать.

### Embed vs Link

**Факт критичен и краток** → вставить копию с [REF:]. Агент видит факт сразу, не нужно открывать другой файл.

**Факт справочный или объёмный** → ссылка в "Связанные файлы". Агент переходит по необходимости.

## Ограничения

| Параметр | Лимит | Почему |
|:---------|:------|:-------|
| Размер файла | <300 строк | Lost in the Middle. Sweet spot: 2000-8000 токенов |
| Таблицы | <20 строк | Заголовки "забываются" у LLM на длинных таблицах |
| Вложенность списков | <=2 уровня | Связи между уровнями теряются |
| Поля YAML | <=12 | Бремя поддержки > польза |
| H2-секция | 512-1024 токена | Оптимальный размер чанка для RAG |

**Если файл >300 строк:** разбить на несколько с чёткими связями. Добавить `## Contents` (буллет-лист секций) в начало, если файл близок к лимиту.

## Уровни соответствия

| Уровень | Где применяется | Обязательно | Опционально |
|:--------|:---------------|:------------|:------------|
| **Полный** | management/ ядро | YAML (title, updated, version) + Кратко + Работа с файлом + Связанные файлы + автономные секции + [SOURCE]/[REF:] | type, scope, priority |
| **Стандартный** | knowledge/, проектные файлы | YAML (title, updated, version) + Кратко + Связанные файлы | type, tags, source, status, review_by |
| **Мета** | README.md, CLAUDE.md | Свой формат (будет описан позже) | — |

## Чеклист для агента

### Блокирующие (файл невалиден без них)

- [ ] `title` заполнен (развёрнутое описание)
- [ ] `updated` = дата смыслового обновления
- [ ] `version` = целое число (новый = 1, обновление = N+1)
- [ ] H1 = title из YAML
- [ ] Секция "Кратко" есть (2-4 предложения, ≤100 слов)
- [ ] Файл <300 строк

### Качество (проверять при ревью)

- [ ] "Связанные файлы" с описанием (не голые ссылки)
- [ ] "Работа с файлом" есть (для management/scaffold)
- [ ] H2-заголовки самоописывающие, секции автономны
- [ ] Таблицы <20 строк, списки ≤2 уровня вложенности
- [ ] `[SOURCE]` / `[REF:]` для ключевых изменяемых фактов
- [ ] YAML плоский, ≤12 полей

## Антипаттерны

| Антипаттерн | Почему плохо | Что делать |
|:------------|:-------------|:-----------|
| description в YAML + "Кратко" | Дублирование → рассинхронизация | Только "Кратко" |
| Глубоко вложенный YAML | LLM путаются, Obsidian не поддерживает | Плоский YAML |
| Связи в YAML без описания | `related: goal.md` — агент не знает зачем | Связи в теле с описанием |
| Заголовок "Важно" или "Примечание" | Неинформативен, чанк без контекста | Самоописывающий заголовок |
| "Как было сказано выше..." | Секция теряет автономность | Повторить факт |
| Таблица на 50+ строк | Заголовки "забываются" | Разбить или заменить списком |
| Микрофайлы по 5 строк | Шум в поисковой выдаче | Один связный документ на тему |
| Два [SOURCE] для одного факта | Непонятно где правда | Один [SOURCE], остальные [REF:] |
| Transclusion `![[file]]` | Невидима для LLM — только Obsidian рендерит | Явная ссылка или копия с [REF:] |
| `---` как декорация | RAG-системы режут по `---` | Только как семантическая граница (смена темы) |

## Открытые вопросы

Спецификация — рабочий документ. Ниже — вопросы, которые осознанно оставлены на проработку.

- **type → онтология.** Как 5 значений type маппятся на 8 сущностей онтологии (vision, goal, ideas...)? Возможно нужен другой подход
- **status.** Какие значения, кому нужен, кому нет. Нужен ли status management-ядру? Сейчас — "не нужен", но decisions точно нуждается
- **Формат мета-файлов.** README.md и CLAUDE.md — "свой формат", но нигде не описан
- **[SOURCE]/[REF:] и reflections.** "Полный" уровень требует маркеры для всего ядра, но reflections — неструктурированный лог, маркировка бессмысленна
- **Порог для Contents.** Когда файл "близок к лимиту"? >250 строк? Не определено
