## Кратко

Справочник сущностей Second AI Brain: связи между файлами, правила размещения, принципы проектирования и правила вложенных проектов. Отвечает на вопросы: какие файлы бывают, как связаны, что куда класть. Детальная спецификация каждой сущности (назначение, структура, правила работы) -- в scaffold-шаблонах: `../scaffold/management/`, `../scaffold/direction/`, `../scaffold/team/`.

## Связанные файлы

- entities.md — атомарные сущности (решение, задача, идея, инсайт...). Онтология описывает ФАЙЛЫ-контейнеры, entities — ЧТО в них живёт
- behavioral_patterns.md — поведенческие паттерны на встречах (сопротивление, увод темы, дисбаланс и др.). Аналог entities для поведенческого домена: не "что решили", а "как решали"
- rituals.md — каталог ритуалов (повторяющиеся управленческие процессы, в которых создаются/меняются сущности)
- file_spec.md — формат файлов: YAML, секции, связи, ограничения. Онтология описывает КАКИЕ файлы бывают, file_spec — КАК они устроены
- ../scaffold/principles.md — принципы проектирования структуры: миссия файла, развёртывание, маршрутизация. Онтология описывает ЧТО существует, principles — КАК решать что создавать
- ../methodology/methodology.md — протоколы, decision frames, масштабирование
- ../scaffold/ — готовый каркас с шаблонами всех сущностей (включает спецификацию каждой сущности)
- ../README.md — обзор фреймворка

## Принципы

1. **LLM + People centered.** Система одинаково удобна для человека и для LLM-агента. Читаемо глазами, парсится моделью без потери смысла.
2. **Разделение по частоте изменений.** Файлы с разной частотой обновления живут отдельно.
3. **Компактность.** Каждый файл — ровно столько, сколько нужно. Лишнее = шум = потраченные токены.
4. **Один файл — одна роль.** Каждый файл отвечает на один вопрос.
5. **Сверху вниз.** Vision → Goal → Plan → Weekly Progress → Timeline. Каждый уровень питается предыдущим.
6. **Минимум дублирования.** У каждого факта одно каноническое место.
7. **Модульность.** Полная структура на максимуме, но разворачивается по запросу. Не все файлы обязательны.
8. **ISO 8601.** Даты везде в формате YYYY-MM-DD.

## Большая рамка

Онтология описывает **блок регулярного менеджмента** внутри более широкой структуры проекта/компании.

```
┌──────────────────────────────────────────────────────┐
│                   ПРОЕКТ / КОМПАНИЯ                   │
│                                                      │
│  ┌──────────┐  ┌───────────┐  ┌────────────────────┐│
│  │ Продукт  │  │ Процессы  │  │    Регулярный      ││
│  │          │  │           │  │    менеджмент       ││
│  │ • vision │  │ • бизнес- │  │   (ЭТА ОНТОЛОГИЯ)  ││
│  │ • описа- │  │   процессы│  │                    ││
│  │   ния    │  │ • регла-  │  │  management/       ││
│  │ • прайс  │  │   менты   │  │   meetings/        ││
│  │ • specs  │  │ • SOP     │  │                    ││
│  └──────────┘  └───────────┘  └────────────────────┘│
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │Knowledge │  │  Inbox   │  │  Архив   │           │
│  │          │  │ (общий)  │  │          │           │
│  │ • база   │  └──────────┘  └──────────┘           │
│  │   знаний │                                       │
│  │ • ссылки │  ┌─────────────┐                      │
│  │ • заметки│  │ README +    │                      │
│  │          │  │ MyProfile + │                      │
│  │          │  │ SKILL.md    │                      │
│  └──────────┘  └─────────────┘                      │
└──────────────────────────────────────────────────────┘
```

| Блок | Что внутри |
|------|-----------|
| **Management** | Регулярный менеджмент (включая meetings/ — выжимки встреч по meeting_analysis.md) |
| **Продукт** | Всё о продукте: стратегия (vision, позиционирование), описания, коммерция (прайс, офферы), техническое (specs, API, docs) |
| **Процессы** | Бизнес-процессы, регламенты, SOP |
| **Knowledge** | Персональная база знаний (ссылки, заметки, инструменты). Не привязана к планированию |
| **Inbox** | Общая входящая, не привязана к проекту |
| **Архив** | Устаревшие файлы, старые версии |
| **README + MyProfile + SKILL.md** | Точка входа для LLM и людей (в корне проекта). MyProfile — персональный профиль руководителя |

## Файловая структура

Формат каждого файла (YAML-заголовок, обязательные секции, ограничения) — в [file_spec.md](file_spec.md). Детальная спецификация каждой сущности (назначение, связи, правила) — в scaffold-шаблоне.

```
{project}/
  README.md                         ← сводка проекта (точка входа)
  my_profile.md                     ← персональный профиль руководителя
  SKILL.md                          ← инструкция для LLM-агента
  team/                               ← профайлы людей + оргструктура (развёрнутая форма management/team.md)
    README.md                         ← оргструктура, маппинг направлений
    {person}.md                       ← профайл: стиль, сильные стороны, коучинг-лог
  management/
    01_vision.md                    ← кто мы, зачем, миссия
    02_goal.md                      ← куда идём: цели + стратегия + метрики
    03_plan.md                      ← что делаем: период → неделя → день
    04_weekly_progress_2026.md      ← недельные итоги (детальный лог)
    05_timeline.md                  ← история проекта: кварталы, годы, вехи
    ideas.md                        ← идеи, гипотезы, задачи на потом
    decisions.md                    ← принятые решения и обоснования
    reflections.md                  ← лог размышлений и переживаний
    metrics.md                      ← описание системы метрик (опционально)
    team.md                         ← микропрофили ключевых людей (свёрнутая форма → team/ при росте)
    meetings/                       ← стратсессии + кросс-функциональные встречи
  {direction}/                        ← зона ответственности (N штук: marketing/, sales/...)
    01_overview.md                    ← миссия, scope, лидер, цели, KPI
    02_active.md                      ← проблемы, задачи, вопросы, риски
    03_progress.md                    ← хроника по встречам
    meetings/                         ← протоколы встреч по направлению
  product/                            ← продукт: стратегия, описания, прайс, specs
  processes/                          ← бизнес-процессы, регламенты, SOP
```

Первые 5 файлов пронумерованы — задают логику чтения сверху вниз. Остальные — по алфавиту.

## Связи между файлами

```
my_profile ────→ 01_vision (личное видение и фокус → проектный vision)
my_profile ────→ 02_goal (фокус года → цели проекта)
my_profile ────→ decisions (ценности/границы → фильтр для решений)
                  │
01_vision ────→ 02_goal (миссия → стратегия → цели года → цели периода)
                  │
02_goal ────────→ 03_plan (цель периода → план периода → план недели → план дня)
                  │
ideas ────────→ 03_plan (задачи/гипотезы → работа на период)
                  │
03_plan ────────→ 04_weekly_progress (итоги недели → детальный лог)
                  │
04_weekly_progress → 05_timeline (итоги кварталов/лет → сводная история)
                  │
decisions ──────→ 02_goal, 03_plan (решения влияют на стратегию и задачи)
                  │
metrics ────────→ 02_goal (метрики подтверждают движение к целям)
                  │
meetings ──────→ 04_weekly_progress (результаты встреч → недельные итоги)
```

## Правила роста сущностей

Некоторые секции при разрастании выносятся в отдельный файл. Известные случаи:

| Секция | В каком файле | Выносится в |
|--------|--------------|-------------|
| Стратегия | 02_goal.md | strategy.md |
| Команда | team.md | папка team/ с профайлами (триггер: 3+ человек с регулярными 1-on-1) |

Общий принцип: если секция обновляется с другой частотой, чем основной файл, и перестаёт помещаться — выноси.

## Вложенные проекты

### Два типа

**Полный проект** — имеет собственный vision и цели:
- Ядро: 01_vision, 02_goal, 03_plan, 04_weekly_progress, 05_timeline, ideas, decisions, reflections
- Опциональные: metrics, team
- Мета: README, my_profile, SKILL.md

*my_profile — один на всё дерево проектов (в корневом проекте). В подпроектах не дублируется.*

**Подпроект** — работает на цели родителя:
- Наследует vision, goal, timeline, decisions от родителя
- Имеет собственные: plan, weekly_progress
- Может иметь: ideas, metrics
- Должен иметь: одну строку "зачем существует и когда закрывается" (в начале plan или в отдельном README)

### Критерии

- **Когда создавать подпроект:** один plan перестаёт вмещать оперативку
- **Когда подпроект → полный проект:** появилась собственная цель, отличная от родительской

### Структура папок (пример)

```
project/
  README.md
  my_profile.md                      # один на всё дерево
  SKILL.md
  management/
    01_vision.md
    02_goal.md
    03_plan.md
    04_weekly_progress_2026.md
    05_timeline.md
    ideas.md
    decisions.md
    reflections.md
    metrics.md                    # опционально
    team.md                       # опционально
    meetings/                     # протоколы встреч
    projects/
      lab/
        plan.md
        weekly_progress_2026.md
      dev/
        plan.md
        weekly_progress_2026.md
        ideas.md                  # если нужен
        metrics.md                # если нужны свои
  product/                          # стратегия, описания, прайс, specs
  processes/                        # бизнес-процессы, регламенты, SOP
```
