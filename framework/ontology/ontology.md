## Кратко

Справочник сущностей Second AI Brain: связи между файлами, правила размещения, принципы проектирования и правила вложенных проектов. Отвечает на вопросы: какие файлы бывают, как связаны, что куда класть. Детальная спецификация каждой сущности (назначение, структура, правила работы) -- в scaffold-шаблонах: `../scaffold/01_ceo/`, `../scaffold/02_strategy/`, `../scaffold/03_team/`, `../scaffold/04_management/`.

## Связанные файлы

- entities.md — атомарные сущности (решение, задача, идея, инсайт...). Онтология описывает ФАЙЛЫ-контейнеры, entities — ЧТО в них живёт
- behavioral_patterns.md — поведенческие паттерны на встречах (сопротивление, увод темы, дисбаланс и др.). Аналог entities для поведенческого домена: не "что решили", а "как решали"
- protocol_format.md — schema семейства протоколов: маппинг сущностей → секции, приоритеты, общие правила. Entities определяет ЧТО извлекаем, protocol_format — КАК собираем в протокол
- rituals.md — каталог ритуалов (повторяющиеся управленческие процессы, в которых создаются/меняются сущности)
- file_spec.md — формат файлов: YAML, секции, связи, ограничения. Онтология описывает КАКИЕ файлы бывают, file_spec — КАК они устроены
- ../scaffold/principles.md — принципы проектирования структуры: миссия файла, развёртывание, маршрутизация. Онтология описывает ЧТО существует, principles — КАК решать что создавать
- ../methodology/methodology.md — протоколы, decision frames, масштабирование
- ../scaffold/ — готовый каркас с шаблонами всех сущностей (включает спецификацию каждой сущности)
- ../README.md — обзор фреймворка

## Принципы

1. **LLM + People centered.** Система одинаково удобна для человека и для LLM-агента. Читаемо глазами, парсится моделью без потери смысла.
2. **Разделение по частоте изменений.** Файлы с разной частотой обновления живут отдельно.
3. **Компактность.** Каждый файл — ровно столько, сколько нужно. Лишнее = шум = потраченные токены.
4. **Один файл — одна роль.** Каждый файл отвечает на один вопрос.
5. **Сверху вниз.** Vision → Goal → Plan → Weekly Progress → Timeline. Каждый уровень питается предыдущим.
6. **Минимум дублирования.** У каждого факта одно каноническое место.
7. **Модульность.** Полная структура на максимуме, но разворачивается по запросу. Не все файлы обязательны.
8. **ISO 8601.** Даты везде в формате YYYY-MM-DD.

## Файловая структура

Формат каждого файла (YAML-заголовок, обязательные секции, ограничения) — в [file_spec.md](file_spec.md). Детальная спецификация каждой сущности (назначение, связи, правила) — в scaffold-шаблоне.

```
{project}/
  README.md                         ← сводка проекта (точка входа)
  CLAUDE.md                         ← инструкция для AI-агента (runtime-specific)
  .claude/skills/                   ← исполняемые навыки (runtime-specific)
  01_ceo/
    01_my_profile.md                ← персональный профиль руководителя
    02_active.md                    ← личные задачи и открытые вопросы CEO
    03_thoughts.md                  ← идеи, рефлексии, гипотезы
    04_decisions.md                 ← личные решения CEO
  02_strategy/
    01_vision.md                    ← кто мы, зачем, миссия
    02_goal.md                      ← куда идём: цели + стратегия + метрики-план
    03_plan.md                      ← что делаем: сборка плана периода/недели
    04_weekly_progress.md           ← недельные итоги
    05_timeline.md                  ← история проекта: кварталы, годы, вехи
    06_decisions.md                 ← стратегические решения компании
  03_team/
    README.md                       ← обзор, маппинг людей ↔ подразделений
    org_structure.md                ← оргструктура (имя → роль / путь в 04_management/)
    {person}.md                     ← профайл: стиль, сильные стороны, коучинг-лог
  04_management/
    README.md                       ← правила дерева подразделений/направлений
    glossary.md                     ← глоссарий: варианты имён, ошибки транскрибации, жаргон
    meetings/                       ← стратсессии + кросс-функциональные встречи
    {org_node}/                     ← подразделение/направление (единица или группировка)
      01_overview.md                ← миссия, scope, лидер, KPI
      02_active.md                  ← задачи, вопросы, риски
      03_progress.md                ← хроника по встречам
      04_decisions.md               ← решения уровня подразделения/направления
      meetings/                     ← протоколы встреч по узлу
      {sub_node}/                   ← дочернее подразделение/направление
  product/                          ← продукт: стратегия, описания, прайс, specs
  processes/                        ← бизнес-процессы, регламенты, SOP
  knowledge/                        ← база знаний
  clients/                          ← досье именованных клиентов
  _inbox/                           ← входящее на разбор
  archive/                          ← архив
```

Корневой канон и правила развёртывания задаются `../scaffold/MODEL.md`. Эта секция — онтологическая сводка, а не альтернативная спецификация.

## Связи между файлами

```
01_ceo/01_my_profile ─→ 02_strategy/01_vision (личное видение и фокус → vision компании)
01_ceo/01_my_profile ─→ 02_strategy/02_goal (фокус года → цели компании)
01_ceo/01_my_profile ─→ 01_ceo/04_decisions, 02_strategy/06_decisions (ценности/границы → фильтр решений)
                         │
02_strategy/01_vision ─→ 02_strategy/02_goal (миссия → стратегия → цели года → цели периода)
                         │
02_strategy/02_goal ───→ 02_strategy/03_plan (цель периода → план периода → план недели)
                         │
04_management/*/02_active ─→ 02_strategy/03_plan (актуальная работа подразделений → сборка плана компании)
                         │
02_strategy/03_plan ───→ 02_strategy/04_weekly_progress (итоги недели → детальный лог)
                         │
02_strategy/04_weekly_progress ─→ 02_strategy/05_timeline (итоги периода → сводная история)
                         │
04_management/*/04_decisions, 01_ceo/04_decisions ─→ 02_strategy/06_decisions (когда решение поднимается на уровень курса)
                         │
meetings/ ─────────────→ 04_management/*, 03_team/*, 01_ceo/*, 02_strategy/* (обновляют канонические файлы по миссии)
```

## Правила роста сущностей

Некоторые секции при разрастании выносятся в отдельный файл. Известные случаи:

| Секция | В каком файле | Выносится в |
|--------|--------------|-------------|
| Стратегия | 02_goal.md | strategy.md |
| Команда | — | 03_team/ с профайлами (один человек — один файл, с первого дня) |

Общий принцип: если секция обновляется с другой частотой, чем основной файл, и перестаёт помещаться — выноси.

## Вложенные проекты

Для CEO-scaffold эта зона пока не является зафиксированным каноном. До отдельной фиксации:

- не изобретать старое дерево `management/projects/`;
- использовать текущий корневой канон из `../scaffold/MODEL.md`;
- отдельные домены раскладывать через `product/`, `processes/`, `clients/`, `knowledge/` или через дерево `04_management/`, если речь о реальной управленческой единице/группировке.
