# Онтология

## Кратко

Справочник сущностей Second AI Brain: типы файлов, их поля, связи между ними, правила размещения. Отвечает на вопросы: какие файлы бывают, что в каждом, как связаны, что куда класть. Принципы проектирования и правила вложенных проектов.

## Связанные файлы

- ../methodology/methodology.md — как работать с этими сущностями (ритуалы, протоколы, decision frames)
- ../scaffold/ — готовый каркас с шаблонами всех сущностей
- ../README.md — обзор фреймворка

---

## Принципы

1. **LLM + People centered.** Система одинаково удобна для человека и для LLM-агента. Читаемо глазами, парсится моделью без потери смысла.
2. **Разделение по частоте изменений.** Файлы с разной частотой обновления живут отдельно.
3. **Компактность.** Каждый файл — ровно столько, сколько нужно. Лишнее = шум = потраченные токены.
4. **Один файл — одна роль.** Каждый файл отвечает на один вопрос.
5. **Сверху вниз.** Vision → Goal → Plan → Weekly Progress → Timeline. Каждый уровень питается предыдущим.
6. **Минимум дублирования.** У каждого факта одно каноническое место.
7. **Модульность.** Полная структура на максимуме, но разворачивается по запросу. Не все файлы обязательны.
8. **ISO 8601.** Даты везде в формате YYYY-MM-DD.

## Большая рамка

Онтология описывает **блок регулярного менеджмента** внутри более широкой структуры проекта/компании.

```
┌──────────────────────────────────────────────────────┐
│                   ПРОЕКТ / КОМПАНИЯ                   │
│                                                      │
│  ┌──────────┐  ┌───────────┐  ┌────────────────────┐│
│  │ Продукт  │  │ Процессы  │  │    Регулярный      ││
│  │          │  │           │  │    менеджмент       ││
│  │ • specs  │  │ • бизнес- │  │   (ЭТА ОНТОЛОГИЯ)  ││
│  │ • docs   │  │   процессы│  │                    ││
│  │ • API    │  │ • встречи │  │  management/       ││
│  │          │  │ • логи    │  │                    ││
│  └──────────┘  └───────────┘  └────────────────────┘│
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │Knowledge │  │  Inbox   │  │  Архив   │           │
│  │          │  │ (общий)  │  │          │           │
│  │ • база   │  └──────────┘  └──────────┘           │
│  │   знаний │                                       │
│  │ • ссылки │  ┌──────────┐                         │
│  │ • заметки│  │ README + │                         │
│  │          │  │ SKILL.md │                         │
│  └──────────┘  └──────────┘                         │
└──────────────────────────────────────────────────────┘
```

| Блок | Что внутри |
|------|-----------|
| **Management** | Регулярный менеджмент (эта онтология) |
| **Продукт** | Техническая документация, спецификации, API |
| **Процессы** | Бизнес-процессы, регламенты, протоколы встреч |
| **Knowledge** | Персональная база знаний (ссылки, заметки, инструменты). Не привязана к планированию |
| **Inbox** | Общая входящая, не привязана к проекту |
| **Архив** | Устаревшие файлы, старые версии |
| **README + SKILL.md** | Точка входа для LLM и людей (в корне проекта) |

## Файловая структура

```
{project}/
  README.md                         ← сводка проекта (точка входа)
  SKILL.md                          ← инструкция для LLM-агента
  management/
    01_vision.md                    ← кто мы, зачем, миссия
    02_goal.md                      ← куда идём: цели + стратегия + KR
    03_plan.md                      ← что делаем: квартал → неделя → день
    04_weekly_progress_2026.md      ← недельные итоги (детальный лог)
    05_timeline.md                  ← история проекта: кварталы, годы, вехи
    ideas.md                        ← идеи, гипотезы, задачи на потом
    decisions.md                    ← принятые решения и обоснования
    reflections.md                  ← лог размышлений и переживаний
    metrics.md                      ← описание системы метрик (опционально)
    team.md                         ← команда, партнёры, роли (опционально)
```

Первые 5 файлов пронумерованы — задают логику чтения сверху вниз. Остальные — по алфавиту.

## Связи между файлами

```
01_vision ────→ 02_goal (миссия → стратегия → цели года → цели квартала)
                  │
02_goal ────────→ 03_plan (цель квартала → план квартала → план недели → план дня)
                  │
ideas ────────→ 03_plan (задачи/гипотезы → работа на квартал)
                  │
03_plan ────────→ 04_weekly_progress (итоги недели → детальный лог)
                  │
04_weekly_progress → 05_timeline (итоги кварталов/лет → сводная история)
                  │
decisions ──────→ 02_goal, 03_plan (решения влияют на стратегию и задачи)
                  │
metrics ────────→ 02_goal (метрики подтверждают движение к KR)
```

## Сущности

### 1. `01_vision.md` — Кто мы, зачем?

**Назначение:** Идентичность проекта. Кто мы, зачем существуем, какое будущее видим. По Advanced OKR — содержит **Aim** (State Goal, 3+ лет).

**Содержит:**
- Суть проекта (1-3 предложения)
- Миссия / большая цель
- Для кого (целевая аудитория)
- Личная цель основателя в проекте
- Ценности и принципы (опционально)
- Описание продукта / бизнес-модели (если есть)

| Параметр | Значение |
|---|---|
| Частота обновления | Раз в 6-12 месяцев |
| Размер | Компактный (до 1-2 страниц) |
| LLM читает | Всегда (базовый контекст) |
| Статус | Ядро |

**Когда обновлять:**
- Пивот или существенное изменение видения
- Раз в полгода — перечитать и проверить актуальность
- Конец года — в рамках годовой ретроспективы

---

### 2. `02_goal.md` — Куда идём?

**Назначение:** Целеполагание и стратегия. Показывает **актуальное состояние** целей. Открыл — видишь текущее. Историю и итоги — в `05_timeline.md`.

**Содержит:**

**Долгосрочная цель (1-3 года):** Вытекает из vision. По Advanced OKR — **Strategy Goal** (State Goal).

**Стратегия / Roadmap:** Как планируем достичь цели. Крупные вехи, ключевые ставки.
> При разрастании → отдельный файл `strategy.md`

**Цель года:** По Advanced OKR — **Strategic Objective** (Change Goal). Формула: *глагол + существительное + новое свойство*

**Цель текущего квартала + Key Results:** По Advanced OKR — **Objective** (Change Goal) + **KR** (3-5 штук).

Типы KR:
- **Outcome** (влияние): конверсия, NPS, retention
- **Output** (действия): количество клиентов, фич
- **Milestone** (веха): да/нет

> Если метрик больше, чем помещается здесь → см. `metrics.md`

**Правило каноничности KR:**
- В goal KR существуют только как **план** на текущий квартал (целевые значения).
- **Факт (план vs факт)** фиксируется только в `05_timeline.md` по итогам квартала.
- В goal факт не дописывается "по ходу". Goal всегда показывает "куда идём", не "где были".

| Параметр | Значение |
|---|---|
| Частота обновления | Раз в квартал (цель квартала). Раз в год (цель года, стратегия) |
| Размер | Компактный. НЕ растёт |
| LLM читает | Часто (контекст «куда идём») |
| Статус | Ядро |

---

### 3. `03_plan.md` — Что делаем?

**Назначение:** Оперативное управление. Что конкретно делаем прямо сейчас. Содержит ТОЛЬКО будущее и настоящее.

**Структура:**

```markdown
# Plan

## План дня
Дата: YYYY-MM-DD
### направление/подпроект A
- [ ] задача 1
- [ ] задача 2

### направление/подпроект B
- [ ] задача 1

## План недели
WXX (YYYY-MM-DD — YYYY-MM-DD)
### направление/подпроект A
- [ ] задача 1
### направление/подпроект B
- [ ] задача 1

## План квартала (QX YYYY)
Цель/Фокус квартала: [одно предложение]
### направление/подпроект A
- [ ] задача 1
- [ ] задача 2
```

Если направлений/подпроектов нет — единым списком.

| Параметр | Значение |
|---|---|
| Частота обновления | Ежедневно (день), еженедельно (неделя), раз в квартал (квартал) |
| Размер | Компактный и стабильный — не растёт, обновляется |
| LLM читает | Часто (оперативный контекст) |
| Статус | Ядро |

---

### 4. `04_weekly_progress_YYYY.md` — Что было (детально)?

**Назначение:** Детальный лог. Что реально происходило неделя за неделей. «Сырые данные» проекта. Из них собираются квартальные и годовые итоги в timeline.

**Структура:**

```markdown
# Weekly Progress YYYY

## Q1

### W07 (YYYY-MM-DD — YYYY-MM-DD)
Фокус: ...
Сделано: ...
Не сделано: ...
Заметки: ...

### W06 (YYYY-MM-DD — YYYY-MM-DD)
...
```

**Именование по годам:** `04_weekly_progress_2026.md`, `04_weekly_progress_2027.md`

| Параметр | Значение |
|---|---|
| Частота обновления | Еженедельно |
| Размер | Растёт (~50 записей в год). Новый файл каждый год |
| LLM читает | По запросу (НЕ по умолчанию) |
| Статус | Ядро |

---

### 5. `05_timeline.md` — История проекта

**Назначение:** Сводная история проекта. Открыл — увидел всю траекторию за все годы. Не детальный лог (это weekly progress), а сжатая хроника: цели, результаты, пивоты, ключевые события. Один файл на весь проект, сквозной.

**Структура:**

```markdown
# Timeline

## 2026

### Цель года
...

### Q1 (январь — март)
Цель квартала: ...
KR: план vs факт
Результат: ...
Ключевые события: ...

### Q2 (апрель — июнь)
...

## 2025

### Цель года
...
```

**Правила:**
- **KR план vs факт** фиксируется только здесь (не в goal).
- Timeline — единственное каноническое место для итогов кварталов и лет.

| Параметр | Значение |
|---|---|
| Частота обновления | Раз в квартал (итоги квартала). Раз в год (итоги года) |
| Размер | Растёт медленно (4-5 записей в год). Один файл на весь проект |
| LLM читает | При стратегических обсуждениях, пересмотре целей |
| Статус | Ядро |

---

### 6. `ideas.md` — Какие есть идеи?

**Назначение:** Структурированная парковка для идей, гипотез и отложенных задач.

**Структура:**

```markdown
# Ideas

## Идеи
(мысли, которые могут пригодиться — не оформлены как задачи)
- YYYY-MM-DD: описание идеи

## Гипотезы
(нужно проверить, прежде чем делать)
- YYYY-MM-DD: гипотеза + как проверить

## Задачи на потом
(понятно что делать, но не сейчас)
- YYYY-MM-DD: описание задачи
```

**Типы записей (для LLM):**
- **Идея** → нужно оформить и обдумать
- **Гипотеза** → нужно проверить/валидировать
- **Задача на потом** → можно брать в plan напрямую

| Параметр | Значение |
|---|---|
| Частота обновления | По мере появления |
| Размер | Растёт, периодически чистится |
| LLM читает | При планировании квартала. НЕ по умолчанию |
| Статус | Ядро |

---

### 7. `decisions.md` — Что решили?

**Назначение:** Реестр принятых решений. Выборы, которые определяют траекторию проекта. Без этого файла LLM будет предлагать уже отвергнутые варианты, а человек забудет *почему* принял решение.

**Структура:**

```markdown
# Decisions

## YYYY-MM-DD: [Название решения]
**Статус:** Актуально
**Решение:** Что именно решили
**Контекст:** Почему вопрос встал
**Альтернативы:** Что рассматривали и почему отвергли
**Следствия:** Что вытекает для плана/целей
```

**Что является «решением»:**
- **Да:** выбор, закрывающий альтернативы — «Не идём в B2C», «Выбрали Dify вместо Langflow».
- **Нет:** операционный выбор внутри недели — «Сегодня делаю A, не B».

**Архивирование:** При накоплении >30-40 решений — архивные (со статусом «Пересмотрено» старше года) переносятся в `archive/decisions_YYYY.md`.

| Параметр | Значение |
|---|---|
| Частота обновления | По мере принятия значимых решений |
| Размер | Растёт медленно |
| LLM читает | При стратегических обсуждениях, пересмотре целей |
| Статус | Ядро |

---

### 8. `reflections.md` — О чём думаю/чувствую?

**Назначение:** Лог размышлений и переживаний. Неструктурированное пространство для мыслей, чувств, интуиций.

**Отличие от других файлов:**
- **Ideas** — идеи, которые *могут стать задачами*
- **Decisions** — выборы, которые *уже сделаны*
- **Reflections** — мысли и чувства, которые *пока ни то ни другое*

Примеры: «Чувствую, что стратегия Z не работает...», «Упадок сил последние две недели...», «Заметил паттерн: клиенты спрашивают про X, но нуждаются в Y...»

**Структура:**

```markdown
# Reflections

## YYYY-MM-DD
(текст размышления / переживания)
Теги: #стратегия #энергия #клиенты
Signal: что это меняет? (ничего / проверить гипотезу / обсудить решение / снизить нагрузку)
```

Дата, теги и signal ставятся автоматически (LLM-агентом при записи).

| Параметр | Значение |
|---|---|
| Частота обновления | Свободная |
| Размер | Растёт |
| LLM читает | По запросу |
| Статус | Ядро |

**Архивирование:** Раз в год старые записи переносятся в `archive/reflections_YYYY.md`.

### 9. `metrics.md` — Какие метрики, где? (опционально)

**Назначение:** Описание системы метрик проекта. Не сами данные, а карта: какие метрики, где считаются, как часто анализируются, кто отвечает. Сами данные живут в Google Sheets, BI-системах, CRM.

**Когда создавать:**
- Метрик больше, чем помещается в KR в goal
- Есть внешние источники данных
- Нужна прозрачность: кто за какую метрику отвечает

**Структура:**

```markdown
# Metrics

## [Название метрики]
**Что измеряет:** ...
**Целевое значение:** ... (связь с KR в goal)
**Где считается:** Google Sheets / BigQuery / CRM / ...
**Как часто анализируется:** еженедельно / ежемесячно
**Кто отвечает:** ...
**Ссылка:** [ссылка на дашборд/таблицу]
```

| Параметр | Значение |
|---|---|
| Частота обновления | При изменении системы метрик |
| Размер | Компактный, стабильный |
| LLM читает | При подведении итогов, планировании |
| Статус | Опционально |

### 10. `team.md` — Кто в команде? (опционально)

**Назначение:** Описание команды, партнёров, ключевых людей вокруг проекта.

**Когда создавать:**
- Появился хотя бы один партнёр, подрядчик или сотрудник
- LLM нужно знать контекст о людях без объяснений каждый раз

**Содержит:** Ключевые люди: роль, зона ответственности, контекст.

> При разрастании → папка `team/` с отдельными профайлами.

| Параметр | Значение |
|---|---|
| Частота обновления | При изменениях в команде |
| Размер | Компактный |
| LLM читает | При обсуждении задач, связанных с людьми |
| Статус | Опционально |

## Мета-файлы

### `README.md`

Сводка проекта для быстрого контекста. LLM начинает с этого файла. Живёт в корне проекта, не внутри management/.

**Содержит:** что за проект (1-2 предложения), текущая стадия, ключевые цели (кратко, ссылка на goal), текущий фокус (кратко, ссылка на plan), команда (кратко), карта файлов.

| Параметр | Значение |
|---|---|
| Частота обновления | При значимых изменениях |
| Размер | Максимально компактный (1 страница) |
| LLM читает | Всегда первым |

### `SKILL.md`

Инструкция для LLM-агента в формате skill: как работать с файловой системой проекта. Живёт в корне проекта, не внутри management/.

**Содержит:** архитектура файлов и их роли, протокол чтения, ритуалы и их реализация, специфика проекта.

Ключевой deliverable консалтинга. Скелет стандартный, наполнение — под клиента.

## Вложенные проекты

### Два типа

**Полный проект** — имеет собственный vision и цели:
- Ядро: 01_vision, 02_goal, 03_plan, 04_weekly_progress, 05_timeline, ideas, decisions, reflections
- Опциональные: metrics, team
- Мета: README, SKILL.md

**Подпроект** — работает на цели родителя:
- Наследует vision, goal, timeline, decisions от родителя
- Имеет собственные: plan, weekly_progress
- Может иметь: ideas, metrics
- Должен иметь: одну строку "зачем существует и когда закрывается" (в начале plan или в отдельном README)

### Критерии

- **Когда создавать подпроект:** один plan перестаёт вмещать оперативку
- **Когда подпроект → полный проект:** появилась собственная цель, отличная от родительской

### Структура папок (пример)

```
project/
  README.md
  SKILL.md
  management/
    01_vision.md
    02_goal.md
    03_plan.md
    04_weekly_progress_2026.md
    05_timeline.md
    ideas.md
    decisions.md
    reflections.md
    metrics.md                    # опционально
    team.md                       # опционально
    projects/
      lab/
        plan.md
        weekly_progress_2026.md
      dev/
        plan.md
        weekly_progress_2026.md
        ideas.md                  # если нужен
        metrics.md                # если нужны свои
```
