## 1. Проблематика

Текущий scaffold моделирует «как CEO управляет» через одну парадигму — line management через подразделения/направления. На живых клиентах модель ломается.

### Узел [1] — `04_management/` смешивает три слоя в одной папке

- Оргструктура (кто где работает) — статика
- Управление (задачи / встречи / решения) — динамика
- Кросс-сессии / стратсессии / glossary — служебное

Плюс `04_management/` — единственная обязательная папка под управление, поэтому первичная ось управления по умолчанию = подразделения. Это навязывает один стиль.

### Узел [2] — Нет места для оргструктуры компании

Оргструктура закопана внутри `04_management/`. «Карта компании» ≠ «как CEO управляет», но в текущей модели они совмещены.

### Узел [3] — Нет масштабирования вниз

Сейчас scaffold = инструмент CEO. Запрос Клиента: *«канонический контекст компании, не CEO, просто сейчас с этим контекстом работает CEO»*. Пространство должно работать у CEO → C-1 → команды.

### Узел [4] — Природа темы у разных CEO разная (диагноз для [1])

Управленческая тема (например, маркетинг) бывает:

- **Подразделением** — есть отдел, лидер, KPI (Клиент2)
- **Процессом** — нет отдела, есть тема (CEO без маркетинг-директора)
- **Проектом** — кросс-функциональная инициатива с горизонтом и завершением (Клиент1: 3 OKR ≈ 3 проекта)

Scaffold по умолчанию кладёт всё в «подразделение». Это и объясняет, почему [1] стала свалкой.

### Эмпирика по клиентам

| | Стиль управления | Реакция на текущую модель |
|---|---|---|
| Клиент 1 | OKR / кросс-функционально | «Подразделения = княжества, мыслим OKR» — запрос на пересборку 04_management ~01.05 |
| Клиент 2 | Структурно через подразделения | По факту в scaffold плоская `management/` без узлов — Iron Law не дал развернуть |
| Project-driven (гипотеза) | Через проекты | Не покрыто |

## 3. Как хотим (fern-модель)

### Идея

Каждая сущность scaffold разворачивается по принципу папоротника (fern). Метафора задаёт три свойства модели:

- **Самоподобие** — часть устроена так же, как целое: маркетинг как файл и компания как папка имеют одну форму, отличаются плотностью наполнения.
- **Разворачивание из «fiddlehead»** — закрученная точка раскрывается в полный лист по мере роста живой работы. У ботаников это юная скрученная вайя папоротника.
- **Переменная глубина у соседних узлов** — одна вайя раскрылась полностью, соседняя ещё в крутке. В отличие от строгого фрактала fern допускает разную зрелость у соседних узлов одного уровня.

«Прогрессивное разворачивание» (progressive disclosure) — общий паттерн. Fern уточняет его: разворот идёт не линейно по предписанному сценарию, а по живости каждой ветви независимо.

### Стадии разворачивания (фрактальная единица)

```
Стадия 0 — УПОМИНАНИЕ
  отсутствует в scaffold; живёт в текстах других файлов
  (статус под вопросом — см. ξ в открытых)

Стадия 1 — ФАЙЛ
  marketing.md
  одна страница: что это, кто отвечает, состояние
  по сути сжатый overview (5 блоков, может быть в 1 строке каждый)

Стадия 2 — ПАПКА С OVERVIEW
  marketing/
  ├── README.md         карта папки
  └── 01_overview.md    5 блоков канона, каждый раскрыт

Стадия 3 — ПАПКА С ПАСПОРТОМ И ОПЕРАЦИОНКОЙ
  marketing/
  ├── README.md
  ├── profile.md        ↑ если ядро не лезет в overview
  ├── 01_overview.md
  ├── 02_active.md      операционный хаб запустился
  ├── 03_progress.md
  ├── 04_decisions.md
  └── meetings/

Стадия 4 — ПОЛНЫЙ КОНТУР
  + 02_backlog.md
  + _inbox/
  + architecture.md (опц., если есть устройство)

Стадия 5 — ВЕТВЛЕНИЕ
  marketing/
  ├── ... (всё из стадии 4)
  └── {sub_node}/        каждый подузел — снова с Стадии 1
       (content / seo / events — каждое со своей стадией)
```

### Пример: маркетинг от Стадии 1 до Стадии 4

Один и тот же маркетинг у CEO, который растёт от темы до направления.

**Стадия 1 — `04_company/marketing.md`** (тема есть, отдела нет)

```markdown
---
title: "Маркетинг"
updated: 2026-01-15
type: reference
---

# Маркетинг

## Суть
Тема существует, отдельного отдела нет. CEO лично запускает рекламу
через подрядчиков.

## Цели и КР
2026 — выйти на 50 заявок/месяц, CAC ≤ 5000 ₽.

## Команда
CEO + 1 подрядчик (контент).

## Фокус
Узкое место — нет канала с предсказуемой стоимостью лида.
Сейчас: тестируем 3 канала параллельно.

## Дорожная карта
Q1 — выбор основного канала. Q2 — нанять маркетолога. Q3 — масштабирование.
```

**Стадия 2 — `04_company/marketing/`** (нашли канал, нужно описать как работаем)

```
marketing/
├── README.md         — карта папки
└── 01_overview.md    — те же 5 блоков, но раскрытые
```

**Стадия 3** — нанят маркетолог, появилась операционка

```
marketing/
├── README.md
├── 01_overview.md
├── 02_active.md      — задачи маркетолога, риски, повестка следующей встречи
├── 03_progress.md    — хроника еженедельных встреч
├── 04_decisions.md   — решения уровня направления
└── meetings/         — протоколы 1-on-1 с маркетологом
```

**Стадия 4** — направление выросло, появилось стабильное ядро (бренд-tone, позиционирование) и пайплайн будущей работы

```
marketing/
├── README.md
├── profile.md        — стабильное ядро: позиционирование, бренд-tone, табу
├── 01_overview.md
├── 02_active.md
├── 02_backlog.md     — будущая работа: Q2-Q3 инициативы
├── 03_progress.md
├── 04_decisions.md
├── _inbox/           — входящее на разбор от маркетолога
└── meetings/
```

Каждый переход — естественный рост, не миграция. То же относится к `projects/marketing_launch/` (если запустили проект) и к `processes/lead_pipeline.md` (если оформили процесс).

### Универсальный механизм — работает везде в scaffold

В fern-модели обязательную папку `04_management/` предлагается переименовать в `04_company/` с уточнённой миссией: вместо «как CEO управляет» → «структура компании». Управленческая семантика (встречи, операционка, решения) живёт у каждой сущности внутри company по fern-логике, не на уровне самой папки.

```
04_company/              корневая «компания» сама разворачивается
  ├── marketing.md       Стадия 1: один файл (нет отдела, есть тема)
  ├── sales/             Стадия 3: папка с overview + операционкой
  └── product/           Стадия 5: подразделение с под-узлами

01_ceo/                  CEO как сущность тоже разворачивается
                         (или частный случай человека в 03_team/, см. λ ниже)

projects/                каждый проект — Стадия 4-5
clients/                 каждый клиент — Стадия 3-4 со специфической раскладкой
metrics/                 от Стадии 1 (один файл) до Стадии 5 (полная вертикаль)
knowledge/               от Стадии 1 (knowledge.md) до Стадии 5 (распределённая wiki)
```

### Что закрывает

| Проблема | Решение |
|---|---|
| **[1]** свалка в 04_management | Растворяется. Внутри `04_company/` — узлы на разных стадиях. Никто не «прибит» к одному формату. |
| **[2]** нет места оргструктуры | `04_company/` = сама оргструктура. От плоского списка файлов до глубокого дерева. |
| **[3]** масштабирование вниз | Открывает дорогу к масштабированию: каждый человек — fern-сущность со своей стадией. Внутри — та же модель. Ссылается на общее `04_company/`. **Конкретный механизм — в открытом вопросе λ.** |
| **[4]** природа темы | Маркетинг **как направление** → `04_company/marketing/`. Маркетинг **как проект** → `projects/marketing_launch/`. Маркетинг **как процесс** → `processes/marketing_pipeline.md`. Природа = выбор папки, а не раскладки. Раскладка единая, разворачивается по необходимости. |

### Сильные стороны

1. **Iron Law — физика модели.** Пустая сущность = несуществующая (нет даже Стадии 1). Не правило «не плоди пустоты» снаружи, а свойство самой формы.
2. **Симметрия subject ↔ object.** CEO, C-1, маркетинг-директор — все имеют своё «корневое пространство», устроенное одинаково. «Канонический контекст компании» (запрос Клиент 1) становится естественным следствием.
3. **Один паттерн — для всего.** Подразделение, проект, продукт, клиент, человек — все fern-сущности. Раскладка одна, разворот разный. Файл превращается в папку без перепиливания, папка в дерево — каждое движение естественный рост, не миграция.

#### μ. Память: две оси — структурированная и неструктурированная

Память в scaffold делится по способу управления.

**Структурированная** — canonical scaffold с fern-разворотом (миссия каждого файла, цифровые префиксы, 5 блоков overview). Жёсткая дисциплина места и формы. Внутренний протокол работы агента с этой памятью — [memory/01_context_memory.md#L57-L91](../memory/01_context_memory.md#L57-L91) (раздел 1: та же базовая развилка структурированная vs неструктурированная).

**Неструктурированная** — две разные сущности с разной функцией:

- **`docs/`** — *хранилище.* Свалка для документов клиента, которые не нужно разруливать: pdf-ы, скриншоты, шаблоны договоров, «всякие разные вещи». Без LLM-разбора. Может появляться как подузел любой сущности по необходимости (`projects/{name}/docs/`, `04_company/{node}/docs/`) или как самостоятельная папка верхнего уровня.
- **`knowledge/`** — *разруливаемая база.* LLM Wiki в стиле Карпатого: 3 слоя (sources / wiki / schema), 3 операции (ingest / query / lint). Клиент кидает источник — LLM компилирует страницы, поддерживает связи. У svaib уже работает эта схема для собственной [knowledge/](../../knowledge/) — живой пример паттерна, на который опираемся. Описание паттерна — [knowledge/context/llm-wiki.md](../../knowledge/context/llm-wiki.md).

##### Открытые подвопросы внутри μ

- **μ.1 — Сосуществуют или альтернативы?** У одного клиента есть `docs/` и `knowledge/` одновременно (две оси неструктурированной памяти) или одна из них покрывает обе потребности?
- **μ.2 — Граница «хранить vs разруливать» — где она проходит?** Один документ может пойти и в `docs/` (как файл), и в `knowledge/` (как источник для wiki). Дублирование или маршрутизация по функции?
- **μ.3 — `knowledge` по подразделениям.** Может ли knowledge разворачиваться по подразделениям (каждый отдел ведёт свою часть + общая компанийная)? Это fern на knowledge или плоская wiki?
- **μ.4 — `processes/` — это knowledge или своя сущность?** Регламенты — структурированная память (с миссией процесса) или неструктурированная (часть wiki)?
