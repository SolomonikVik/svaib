# Принципы проектирования файловой структуры

## Кратко

Принципы, по которым проектируется и разворачивается файловая структура Second AI Brain. Не описывают КАК устроен один файл (→ file_spec.md) и не описывают КАКИЕ файлы бывают (→ ontology.md). Описывают КАК принимать решения о структуре: какие файлы создавать, когда разделять, как маршрутизировать информацию. Выведены из клиентской практики и рыночного анализа (март 2026).

**Работа с файлом:** reference — обновлять при новых инсайтах о структуре. Консультировать при проектировании scaffold для нового домена или клиента.

## Принцип 1: Миссия файла как критерий решения

Каждый файл имеет миссию — ответ на один вопрос (КТО? ГДЕ? ЧТО ДЕЛАЕМ? ПОЧЕМУ? ЧТО ДЕЛАТЬ?). Миссия — это не описание содержимого, а **критерий принятия решения** для агента: добавлять информацию в этот файл или нет.

Это не механическая маршрутизация "тип сущности → файл". Одна и та же сущность (например, Решение) может попасть в разные файлы в зависимости от содержания. "Перешёл на cowork" → setup (конфигурация). "Структуру делать по зонам" → decisions (стратегическое решение). Тип сущности подсказывает, но миссия файла решает.

**Реализация:** блок "Работа с файлом" в каждом файле (file_spec.md). Содержит: что сюда, что НЕ сюда, тест-вопрос. "Кратко" объясняет назначение файла, "Работа с файлом" — правила для агента.

**Процесс обновления:** агент не берёт сущность и ищет файл для неё. Агент проходит по файлам проекта и для каждого спрашивает: "нужно ли обновить этот файл по его миссии с учётом новой информации?" Это принципиально другой алгоритм — файл-центричный, не сущность-центричный. Сам процесс обновления описывается в methodology/.

## Принцип 2: Permanent vs Current

Информация, которая живёт дольше одного цикла (встречи/недели), хранится отдельно от операционной. Разная частота обновления = разные файлы.

| Тип | Примеры | Поведение |
|-----|---------|-----------|
| **Permanent** | Профиль человека, факты о бизнесе, стратегические решения | reference — обновлять при новых данных |
| **Current** | Задачи, открытые вопросы, риски, повестка | plan — перезаписывать, не растёт |
| **Log** | Хроника встреч, история решений | log — только дописывать |

Смешивание permanent и current в одном файле — главный источник "файлов-свалок". Признак: файл одновременно содержит "Кто клиент" и "Что делать на этой неделе".

## Принцип 3: State-файл vs Log-файл

Два режима работы с информацией:

**State-файл** (active.md, plan.md): отражает текущее состояние. Перезаписывается. Целевой размер фиксирован (40-80 строк). Выполненное подтверждается на встрече, затем удаляется. История — в источнике (протоколы встреч, git).

**Log-файл** (progress.md, decisions.md, weekly_progress.md): хроника, только дописывается. Старые записи не меняются. Растёт, но каждая запись компактна.

**Формат записей в log-файлах:** повторяющийся структурированный блок с одинаковыми полями (не все обязательны). Не свободный текст и не жёсткая таблица. Пример для decisions.md:

```markdown
## YYYY-MM-DD: Название решения
**Статус:** Актуально
**Решение:** Что решили
**Почему:** Контекст
**Альтернатива:** Что отвергли
```

Без фиксированного формата записей log-файл через 10 встреч становится непарсируемым — ни для человека, ни для AI.

Ошибка: сделать tracking.md одновременно state и log. Результат — растущая портянка, где невозможно найти актуальное.

## Принцип 4: Развёртывание (секция → файл → папка)

Два аспекта: когда разделять один файл на несколько и какие папки/файлы создавать у клиента.

### Триггеры разделения (секция → файл)

| Триггер | Что происходит | Пример |
|---------|---------------|--------|
| **Частота** | Секция обновляется чаще/реже остального файла | Задачи (ежедневно) в project (ежемесячно) → отдельный active.md |
| **Объём** | Секция >50 строк или >15 записей | Решения в project.md → отдельный decisions.md |
| **Множественность** | Появляется N экземпляров одного типа | 1 человек → 1 файл 03_team/{person}.md |
| **Домен** | Содержание относится к другой предметной области | Техническая настройка в project → отдельный setup.md |
| **Функция чтения** | Разные блоки читаются разными агентами для разных задач | active.md читается ежедневно при планировании, progress.md — раз в месяц при стратегическом обзоре |

### Обратная свёртка

Если файл <20 строк и обновляется одновременно с родителем → вернуть секцией. Не плодить микрофайлы.

### Iron Law: разворачивай только живое

> **У клиента создаются только те папки и файлы, за которыми стоит реальная работа: встреча, задача, решение, человек. Пустые ветки «на всякий случай» не создаются.**

«Живое» = регулярная работа CEO, не факт существования сущности в компании. У клиента может быть 50 отделов — в `04_management/` лежат только те, с которыми CEO работает напрямую.

Применение к слоям scaffold:

- **4 обязательные папки** (`01_ceo/`, `02_strategy/`, `03_team/`, `04_management/`) — создаются при старте со всеми каноническими файлами. Пустые файлы допустимы.
- **Подразделение/направление `04_management/{org_node}/`** — разворачивается при живой работе. Внутри сразу 4 файла + `meetings/`.
- **Опциональные папки** (`product/`, `clients/`, `knowledge/`, `processes/`) — по живому триггеру в домене.
- **`03_team/{person}.md`** — при перечислении C-1, первой встрече, первом упоминании с явной ролью.

Удаление, слияние, перенос — только вручную с подтверждением CEO. Риск потерять историю работы.

Конкретные правила раскладки и применения Iron Law к scaffold Second AI Brain — в [MODEL.md](MODEL.md).

## Принцип 5: Один аспект — одно место

Каноническое место информации определяется не "фактом", а **аспектом** — миссионным вопросом, на который файл отвечает. Одно событие может законно обновить несколько файлов, если затрагивает разные аспекты.

"Перешёл на cowork" обновляет:
- setup.md (аспект: конфигурация системы — что установлено и как работает)
- profile.md (аспект: опыт человека с инструментами — как он работает)

Это не дублирование — разные файлы отвечают на разные вопросы об одном событии. Дублирование — когда один и тот же аспект записан в двух местах (например, текущий тариф и в setup, и в project).

Связи между файлами — через ссылки [REF:]. При изменении источника — проверить ссылки.

## Принцип 6: Deal ≠ Activities ≠ Tasks

Три сущности, которые никогда не живут в одном файле:
- **Deal/Project** = что мы делаем (паспорт, scope, условия) → project.md
- **Activities** = что произошло (встречи, звонки) → meetings/, progress.md
- **Active** = что актуально сейчас (задачи, вопросы, риски, повестка) → active.md

Источник: CRM-системы (HubSpot, Salesforce), PRINCE2, PMBoK. Подтверждено на практике: смешение в одном файле → "свалка".

## Принцип 7: Catch-all как страховка

Если агент не может однозначно маршрутизировать информацию — она остаётся в источнике (протокол встречи, inbox). Не теряется, разбирается позже. Лучше не разложить, чем разложить неправильно.

## Принцип 8: "Scaffold" — внутренний термин

В промптах, оркестраторах и любых runtime-инструкциях для агента — "файлы проекта", "папка клиента", "досье". "Scaffold" остаётся как термин уровня фреймворка (где мы проектируем структуру). Агенту, который идёт обновлять файлы, лишний слой абстракции не нужен — он создаёт место, где модель может свернуть не туда.

**Источник:** S16 (2026-03-30). Модель читает "обнови scaffold" → сначала резолвит абстракцию, потом ищет файлы. Читает "обнови файлы проекта в clients/..." → сразу действие.
