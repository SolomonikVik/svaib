# Clients — шаблоны для клиентских проектов

Scaffold для ведения проектов с внешними контрагентами: клиенты, партнёры, подрядчики. Универсальная модель — подходит для консалтинга, IT-услуг, дизайна, любого сервисного бизнеса.

Готовый каркас для нового проекта. Скопировать папку, заменить плейсхолдеры `{...}`, начать заполнять.

## Структура

```
{project_name}/
  profile.md        — кто контрагент (человек, предпочтения, контекст)
  company.md        — его бизнес (продукт, цифры, люди, стратегия)
  project.md        — что делаем (сделка, модель работы, метрики успеха,
                      карта решений/продуктов, программа)
  decisions.md      — почему так решили (стратегические решения с контекстом)
  setup.md          — техническая конфигурация (инструменты, подписки, структура)
  tasks.md          — что делать сейчас (задачи, вопросы, идеи, риски,
                      повестка следующей встречи)
  progress.md       — как двигались (хроника по встречам)
  meetings/         — протоколы встреч
    archive/        — полные транскрибации (сырые)
  deliverables/     — материалы для контрагента
  docs/             — юридические документы
```

## Принцип: миссия файла как фильтр

Каждый файл имеет блок **"Кратко"** (миссия) и **"Работа с файлом"** (инструкция агенту). Агент перед размещением информации проверяет: **соответствует ли это миссии файла?** Не соответствует — ищет другой файл или оставляет в протоколе встречи.

Это не механическая маршрутизация "тип сущности → файл", а осмысленная: агент соотносит содержание с назначением документа. Одна и та же сущность (например, Решение) может попасть в разные файлы в зависимости от содержания. "Перешёл на новый инструмент" → setup (конфигурация). "Структуру делать по зонам" → decisions (стратегическое решение).

Подробнее: [../principles.md](../principles.md)

## Маршрутизация по миссиям

| Информация | Тест | Файл |
|------------|------|------|
| Факт о человеке | Про личность, предпочтения, опыт? | profile.md |
| Факт о бизнесе | Останется при смене руководителя? | company.md |
| Стратегическое решение | Через 3 мес спросят "почему так"? | decisions.md |
| Запрос на новое решение/продукт | Контрагент хочет новую возможность/автоматизацию? | project.md → Карта решений |
| Задача | Кто-то должен что-то сделать? | tasks.md |
| Открытый вопрос | Не решено, требует возврата? | tasks.md |
| Идея | Предложение без обязательства? | tasks.md |
| Риск | Может пойти плохо? | tasks.md |
| Мелкое/контекстное | Не подходит ни под одну миссию? | остаётся в meetings/ |

## Разграничение файлов

| Файл | Миссия (вопрос) | Что хранить | Что НЕ класть |
|------|-----------------|------------|---------------|
| profile.md | **КТО** этот человек? | Личность, боли, позиция по теме проекта, предпочтения, опыт, принципы, язык, личное | Бизнес → company. Запросы на решения → project. Задачи → tasks |
| company.md | **ГДЕ** он работает? | Компании, продукты, цифры, оргструктура, ключевые люди, стратегия, технологии | Личные цели → profile. Наши решения → decisions |
| project.md | **ЧТО** мы делаем? | Сделка, модель работы, метрики успеха, карта решений/продуктов, программа/этапы | Задачи → tasks. Решения с контекстом → decisions. Динамика → progress |
| decisions.md | **ПОЧЕМУ** так решили? | Стратегические решения с контекстом (почему, альтернативы, статус) | Мелкие решения → meetings/. Задачи → tasks |
| setup.md | **КАК** настроено? | Инструменты, подписки, структура папок, конфигурация | Задачи по настройке → tasks. Опыт контрагента → profile |
| tasks.md | **ЧТО ДЕЛАТЬ** сейчас? | Задачи (по owner), открытые вопросы, идеи, риски, повестка след. встречи | Завершённые → удалить. Решения → decisions. Факты → profile/company |
| progress.md | **КАК ДВИГАЛИСЬ?** | Хроника по встречам (2-4 строки на встречу) | Полные протоколы → meetings/. Задачи → tasks |

### Правила сортировки (при обработке встреч)

**Принцип "вход → выход":**
- profile.md "Боли" = **сырой вход** — контекст человека, с чем пришёл
- project.md "Карта решений" = **обработанный выход** — что мы строим, с метриками

```
Встреча → сырые боли → profile.md "Боли" (контекстные, краткие, 5-8 пунктов)
                              ↓
           анализ → project.md "Карта решений" (боль → решение → метрика)
```

**Тест: куда записать?**

| Фраза контрагента | Куда | Почему |
|-------------------|------|--------|
| "У меня файлы копятся, нет системы" | profile → Боли | Контекстная проблема, описывает ситуацию |
| "Хочу чтобы процесс X был автоматизирован" | project → Карта решений | Конкретный запрос на решение |
| "Стиль — прямой, на ты, должен спорить" | profile → Предпочтения | Личностная характеристика |
| "Давай на следующей встрече разберём Y" | tasks → Повестка | Операционка |
| "Договорились — 100К за блок" | project → Статус | Условия сделки |
| "Я 13 лет в отрасли, команда 100 человек" | company → Общее/Цифры | Факт о бизнесе |
| "У нас 2000 клиентов" | company → Цифры | Факт о бизнесе |
| "Игорь — прям двигает тему" | company → Ключевые люди | Человек в компании |
| "Хотим MRR 2.7M к 2028" | company → Стратегия | Цель компании, не человека |
| "Хочу выйти из операционки" | profile → Личная стратегия | Личная траектория |
| "Решили делать структуру по зонам" | decisions → новая запись | Стратегическое решение |
| "Бросить инструкцию по X" | tasks → {Владелец} | Конкретная задача |
| "Перешёл на новый инструмент" | setup (конфигурация) + profile (опыт) | Состояние системы + характеристика человека |
| "Если не масштабируем — потеряем клиента" | tasks → Риски | Потенциальная проблема |
| "А что если через GitHub?" | tasks → Идеи | Предложение без обязательства |
| "Нужно разобраться с доступом" | tasks → Открытые вопросы | Не решено, требует возврата |

**Красный флаг:** Если в profile.md "Боли" появляется **"хочет [конкретное решение]"** — это запрос → project.md "Карта решений".

**Граничное правило profile vs company:** Факт останется верным при смене руководителя → company.md. Факт про этого конкретного человека → profile.md.

## Развёртывание

Структура не жёсткая — разворачивается по мере роста данных:
- **Новый проект** → 3 файла (profile + project + tasks), остальные создаются по мере появления данных
- **company.md** → появляется когда есть факты о бизнесе (обычно после встречи 1)
- **decisions.md** → появляется когда есть первое стратегическое решение
- **setup.md** → появляется когда начинается техническая настройка
- **progress.md** → появляется после встречи 2 (есть динамика)

Подробнее: [../principles.md](../principles.md) → Принцип 4.

## Соответствие file_spec.md

Все шаблоны следуют спецификации файла Second AI Brain:
- YAML frontmatter (title, updated, version)
- Секция "Кратко" (<=100 слов)
- Блок "Работа с файлом"
- Секция "Связанные файлы"
- Автономные секции, <300 строк
