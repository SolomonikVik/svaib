# Обработка встречи с контрагентом

> **DRAFT** — черновая процедура, требует проверки на практике.

Инструкция для координатора (основной чат). Субагент делает L1-экстракцию, координатор читает выжимку, делает дельты сам и обсуждает с исполнителем.

> **Известное ограничение:** процедура использует только entity extractor (бизнес-сущности). Канонический пайплайн (skills/meeting-analysis/README.md) предполагает два параллельных экстрактора (entity + behavior) с конкатенацией выходов. Behavior extractor пока не стабилизирован — когда будет готов, Шаг 1 нужно расширить на два параллельных субагента.

## Когда использовать

Исполнитель просит обработать встречу / транскрипт лежит в `{project_dir}/meetings/_transcript.md`.

## Переменные

Перед стартом определи:
- `{project_dir}` — папка проекта с контрагентом (содержит profile.md, meetings/ и т.д.)
- `{output_dir}` — рабочая папка для временных файлов субагента (например, `_inbox/subagents/`)
- `{name}` — идентификатор проекта/контрагента (используется в именах файлов)

## Критические правила

1. **НЕ читай транскрипт.** Не подгружай в свой контекст — только передавай путь субагенту. Транскрипт большой и забьёт контекст.
2. **Выжимку — ЧИТАЙ.** После работы субагента прочитай выжимку — она ~100 строк, тебе нужен контекст встречи для дальнейшей работы.
3. **Субагент — foreground.** НЕ используй `run_in_background: true`.

## Последовательность

### Шаг 0: Подготовка

1. Убедись что транскрипт на месте: `{project_dir}/meetings/_transcript.md`
2. Создай рабочую папку если нет (для временных файлов субагента)

### Шаг 1: L1-экстракция (субагент)

Запусти субагент через Agent tool:

```
Прочитай промпт из файла framework/skills/meeting-analysis/L1-prompt-entity-extractor.md (секция "Промпт", содержимое внутри блока ```).

Прочитай профайл контрагента для контекста: {project_dir}/profile.md

Прочитай транскрипт: {project_dir}/meetings/_transcript.md

Определи участников встречи по контексту транскрипта и профайла.

Примени промпт L1 к транскрипту. Профайл используй как контекст (кто есть кто, термины, ключевые люди).

Результат ЗАПИШИ в файл: {output_dir}/{name}_extract.md

Use only simple, single bash commands - no &&, no ||, no pipes, no ;, no redirects.

Выводи только результат, без комментариев.
```

### Шаг 2: Читай выжимку + досье, делай дельты

1. Прочитай выжимку: `{output_dir}/{name}_extract.md`
2. Прочитай файлы досье **которые существуют**: profile.md, company.md, project.md, tasks.md, setup.md, decisions.md, progress.md. На ранних этапах проекта часть файлов (company, setup, decisions, progress) может ещё не быть создана — это нормально, см. README.md → Развёртывание
3. **Файл-центричный проход:** пройди по каждому файлу и спроси по его миссии: "нужно ли обновить этот файл с учётом новой информации?" НЕ раскидывай сущности из выжимки — проверяй файлы.

**Чеклист по файлам:**

| Файл | Миссия | Тест-вопрос |
|------|--------|-------------|
| profile.md | КТО этот человек | Узнали новое о человеке (стратегия, предпочтения, боли, принципы, язык)? |
| company.md | ГДЕ он работает | Новые факты о бизнесе (цифры, люди, оргструктура, стратегия компании)? Тест: останется при смене руководителя? |
| project.md | ЧТО мы делаем | Новые запросы на решения? Изменилась модель работы, метрики успеха, программа? |
| decisions.md | ПОЧЕМУ так решили | Были стратегические решения? Тест: через 3 мес спросят "почему так?" — это ответ? |
| tasks.md | ЧТО делать сейчас | Новые задачи (по owner)? Открытые вопросы? Идеи? Риски? Повестка следующей встречи? |
| setup.md | КАК настроено | Изменилась конфигурация (инструмент, структура, настройки)? |
| progress.md | КАК двигались | Дописать 2-4 строки о встрече (что сделано, ключевой сдвиг, обратная связь) |

4. Покажи дельты исполнителю по каждому файлу, применяй после ОК

**Ключевые правила маршрутизации:**

**Один аспект — одно место.** Одно событие может законно обновить несколько файлов, если затрагивает разные аспекты. "Перешёл на новый инструмент" → setup (конфигурация) + profile (опыт человека). Это НЕ дублирование — разные файлы отвечают на разные вопросы. Дублирование — когда один аспект в двух местах.

**Catch-all.** Если не можешь однозначно маршрутизировать — оставь в протоколе встречи (meetings/). Лучше не разложить, чем разложить неправильно.

**Антипаттерны (что НЕ класть):**
- Конфигурацию (инструмент, настройка) в profile → setup
- Прогресс ("разобрал 50 файлов") в profile/company → progress или протокол
- Задачи в project → tasks
- Решения без контекста в project → decisions (с why + альтернативы)
- Конкретные события/даты в "Кратко" → progress

**Формат записей в log-файлах:**

progress.md — дописать блок:
```
## Встреча N (YYYY-MM-DD) — тема
{2-4 строки: что сделано, ключевой сдвиг, обратная связь}
```

decisions.md — дописать блок (только для стратегических решений):
```
## YYYY-MM-DD: Название решения
**Статус:** Актуально
**Решение:** Что решили
**Почему:** Контекст
**Альтернатива:** Что рассматривали и отвергли
```

**Даты-источники.** Volatile-данные (цифры, статусы, решения, люди) — ставь inline-дату `(YYYY-MM-DD)`. Стабильные факты — без дат.

### Шаг 3: Финализация

1. Переименуй выжимку в протокол: `mv {output_dir}/{name}_extract.md {project_dir}/meetings/YYYY-MM-DD_{тема}.md`
2. Архивируй транскрипт: `mv {project_dir}/meetings/_transcript.md {project_dir}/meetings/archive/YYYY-MM-DD_{тема}_transcript.md`
3. Создай чистый `_transcript.md` для следующей встречи
