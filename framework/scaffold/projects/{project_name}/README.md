# README: управление проектом

## Что это за папка

Универсальная структура для управления любым проектом. Семь файлов ядра — каждый отвечает на один вопрос:

| Файл | Вопрос | Природа |
|------|--------|---------|
| `_inbox/01_inbox.md` | Что прилетело и пока не разобрано? | Буфер (разгребается) |
| `01_overview.md` | Зачем проект и куда идём? | Конституция (меняется редко) |
| `02_backlog.md` | Что будем делать? | План (намерение) |
| `02_active.md` | Что делаем сейчас? | Работа (обязательство) |
| `03_progress.md` | Что сделали и как двигаются метрики? | История (append-only) |
| `04_decisions.md` | Почему так решили? | Архив решений (append-only) |
| `meetings/` | Что обсуждали и о чём договорились? | Протоколы встреч |

## Что лежит в папке

```
{project_name}/
├── README.md             — этот файл
├── _inbox/
│   └── 01_inbox.md       — входящее на разбор
├── 01_overview.md        — суть, цели и КР, границы, команда, дорожная карта
├── 02_active.md          — повестка встречи, задачи, вопросы, проблемы и риски
├── 02_backlog.md         — план по вехам + идеи и гипотезы
├── 03_progress.md        — динамика метрик + хроника по встречам
├── 04_decisions.md       — стратегические решения с контекстом
└── meetings/             — протоколы встреч проекта
    ├── README.md
    └── archive/          — сырые транскрипты
```

## Ключевой принцип

**Не плоди сущности без необходимости.** Новый файл / раздел / категория — только если без него теряется что-то важное и это нельзя выразить через существующее.

## Порядок работы при старте проекта

1. **Пишем `01_overview.md`** — цель, КР, границы, вехи дорожной карты
2. **Декомпозируем вехи в `02_backlog.md`** — план по вехам + зона для идей
3. **Берём ближайшие задачи в `02_active.md`** — назначаем ответственного и срок
4. **По мере выполнения — в `03_progress.md`** — фиксируем факт с датой
5. **В конце периода — обновляем метрики** в overview и progress
6. **Значимые развилки — в `04_decisions.md`** — с контекстом и альтернативами

# Задачи

## Откуда приходят задачи

Два паттерна:

**Плановое (top-down):** декомпозиция вехи из дорожной карты. Веха → разбиваем на задачи → в backlog под веху.

**Хаотичное (bottom-up):** идея прилетела извне — со встречи, от клиента, в голову пришла, из другого файла.

В зависимости от зрелости:
- **Сырое** (обрывок, транскрипт, вложение) → `_inbox/01_inbox.md`
- **Идея/гипотеза без разбора** → `02_backlog.md` → секция «Идеи и гипотезы»
- **Понятная задача под веху** → `02_backlog.md` → «План по вехам» под нужную веху
- **Срочная, делаем сейчас** → `02_active.md` (редкий случай — для горящего)

Дальше работает общий жизненный цикл.

## Жизненный цикл

```
overview (веха с целью)
    ↓ декомпозируем
backlog (задача под вехой — намерение)
    ↓ взяли в работу
active (задача с ответственным и сроком — обязательство)
    ↓ выполнили + подтвердили
progress (строка с датой — факт)
```

## Инварианты

1. **Задача живёт в одном операционном файле за раз** (backlog / active / progress). Не копируется — переносится.
2. **Движение вперёд или откат назад** (active → backlog при блокере), но не в двух местах одновременно.
3. **Overview и decisions не содержат задач.** Это другой уровень абстракции — цели и решения.

## Правила переноса

**Идеи → План по вехам (разбор идей внутри backlog):**
- На разборе: идея ясна и привязывается к вехе → переезжает в «План по вехам» под веху
- Идея отклонена → удалить (или зафиксировать отказ как решение, если важно)
- Идея принципиальная, с альтернативами → в `04_decisions.md` как решение

**backlog → active:**
- Задача получает ответственного и срок
- Формулировка уточняется (из «настроить CRM» в «настроить CRM: воронка, импорт, обучение — до 30.04, Петя»)
- Из backlog удаляется

**active → progress:**
- После подтверждения на встрече (не просто «я сделал», а «обсудили и закрыли»)
- В progress уходит финальная формулировка — что реально сделали
- Из active удаляется

**active → backlog (откат):**
- Если задача заблокирована или отложена
- Если оказалась слишком крупной — декомпозируется и возвращается подзадачами

**active → удаление:**
- Если передумали делать — просто удалить
- Если решение значимое — зафиксировать в `04_decisions.md`

## Гигиена — диагностические маркеры

- **Задача висит в active > 2–3 недель** → слишком крупная, заблокирована или не делается. Разобрать.
- **Backlog > 50 строк** → давно не было разбора. Почистить.
- **Overview > 2 экранов** → распухло. Вынести детали в соседние файлы.
- **В active > 3 задач на человека** → лимит задач в работе сломан, реально не делается ничего.

# Метрики

## Жизненный цикл

```
overview (КР с целью и текущим значением — снимок «где мы»)
    ↕ связаны
progress (таблица по периодам — источник правды, история движения)
```

Метрики, в отличие от задач, **не движутся** между файлами. Они живут в двух местах одновременно: снимок текущего значения в overview, полная история в progress.

## Инварианты

1. **Source of truth — таблица «Динамика метрик» в `03_progress.md`.** Снимок в overview = копия последнего значения из неё.
2. **Метрики в таблице = метрики из КР в overview.** Не больше. Операционные метрики (звонки в день, скорость ответа, NPS) — в отдельный файл или дашборд, не в progress.
3. **Частота обновления** (месяц / неделя / квартал) — задаётся на старте проекта и фиксируется в overview.

## Правила обновления

**Когда:** в конце каждого периода (месяц / неделя — в зависимости от проекта).

**Что делаем:**
1. Добавляем новую колонку в таблицу «Динамика метрик» в `03_progress.md`
2. Переписываем «Сейчас» и «Обновлено» в таблице КР в `01_overview.md`

Это минимальный контролируемый дубль: одно число в двух местах. Синхронизируется за одно действие в конце периода.

# Общие практики

## Разбор backlog

Раз в 2–4 недели разбираем backlog:
- Идеи и гипотезы → в план под веху, удалить или в решения
- План по вехам → пересматриваем приоритеты
- Поднимаем готовое → в active

## Опциональные расширения ядра

По мере необходимости:
- `docs/` — материалы, документы, артефакты
- {другие папки по природе проекта}
