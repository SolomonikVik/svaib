# Semantic model: <domain>

Шаблон domain-файла. Копировать как `<domain>.md` (`finance.md`, `sales.md`, `operations.md`, `people.md`). Заполнять под бизнес CEO.

Семантическое описание домена для AI-агента: какие источники, какие метрики, какие маршруты под типовые вопросы. Числа не хранятся здесь — только семантика. Расчёты идут через Python (xlsx-skill) поверх файлов в `source/`.

## datasets

Источники данных домена. Один dataset = один лист в xlsx (или один CSV). Если на одном листе **несколько таблиц с разными системами координат** — фиксировать координаты каждой через `regions` (см. ниже).

```
- name: <dataset_id>           # стабильный ID, snake_case
  source: source/<file>.xlsx#<sheet>
  primary_key: <column>
  description: <что это>
  synonyms: [<как CEO называет в речи>]
  antonyms: [<омонимы — звучат похоже, означают другое; из речи клиента>]
  regions:                     # опционально, если на листе несколько таблиц
    - name: <region_id>
      rows: [<start>, <end>]   # 0-indexed
      header_row: <row>
      description: <что эта подтаблица>
  fields:
    - name: <field_id>
      expression: <column_name>          # имя колонки в xlsx (или координата `B5` для нерегулярных листов)
      description: <что это, единица измерения>
      synonyms: [<как CEO называет>]
      antonyms: [<омонимы>]
      is_time: <true|false>              # для полей-дат
```

## relationships

Связи между датасетами (если есть). Опциональная секция.

```
- name: <relationship_id>
  from: <dataset_id>
  to: <dataset_id>
  from_columns: [<column>]
  to_columns: [<column>]
```

## metrics

Бизнес-определения метрик. Каждая метрика = один паспорт.

```
- name: metric_<id>            # стабильный ID, не привязан к колонкам
  version: 1                   # инкремент при изменении формулы
  owner: <кто отвечает за определение>
  direction: <increase|decrease>  # хорошо когда факт ↑ или ↓?
  expression: <SQL или Python через датасеты и поля>
  formula_human: <объяснение для CEO без кода>
  description: <бизнес-смысл, известные нюансы>
  synonyms: [<как CEO называет в речи>]
  antonyms: [<омонимы — звучит похоже, но не эта метрика>]
  plan_required: <true|false>  # есть ли план в источнике, или метрика только-факт
```

**`direction` — обязательное поле.** Без него `narrative` не различает «факт ниже плана» как победу или провал.

- `increase` — хотим больше: выручка, MRR, retention, конверсии, продуктивность, capacity, NPS.
- `decrease` — хотим меньше: churn, отток, расходы, бэклог, время отклика, ошибки, аварии.

**Если синонима в речи нет** — пометка `synonyms: ["в речи не зафиксировано"]`. Не выдумывать. Если CEO произнесёт новый синоним — добавить в следующей итерации.

## routes

Готовые решения типовых вопросов CEO. Триггер → query → freshness → формат ответа.

```
- name: route_<id>
  trigger: [<фраза CEO 1>, <фраза 2>]
  extractor: <путь к per-client extractor + параметры вызова>
  freshness: <snapshot|live|hybrid>
  response_format: <число | таблица | narrative + цифра>
```

**`extractor`** — ссылка на per-client Python-скрипт (см. `orchestrator-metrics.md` Шаг 2, Режим А). Скрипт принимает `metric_id` (или список), пишет результат в файл, агент читает. **Не подставлять Python-код прямо в `query`** — теряется детерминированность, каждый запуск LLM генерирует разный код.

## known_issues

Известные косяки источника. **Обязательная секция** — без неё агент тихо отвечает неточно.

```
- type: <div0|wrong_label|empty_period|merged_cells|comment_rows|plan_missing|...>
  location: <координата или диапазон в xlsx>
  description: <что не так и как обходить>
  workaround: <что делает extractor: skip | mark | substitute | error>
```

**Типичные категории:**

- **`div0`** — `#DIV/0!` в формулах источника (план = 0). Extractor пропускает или маркирует `status: "div0"`.
- **`wrong_label`** — заголовок колонки врёт (например, «01.2026 Цель» в месте, где факт). Зафиксировать реальное значение.
- **`empty_period`** — период пустой, хотя ожидался заполненным. Маркировать `status: "missing"`, не считать «нулём».
- **`merged_cells`** — объединённые ячейки в заголовках. Extractor читает через openpyxl, не пытается распарсить через `read_excel(header=N)`.
- **`comment_rows`** — строки-комментарии/гипотезы вперемешку с метриками. Extractor пропускает по списку известных строк.
- **`plan_missing`** — у метрики нет плана в источнике. `plan_required: false` в паспорте. Запросы про «разрыв план/факт» исключают эту метрику.
- **`multi_table_sheet`** — на одном листе несколько таблиц с разными системами координат. См. `regions` в `datasets`.

## Правила заполнения

**Стабильные ID.** Маршруты ссылаются на `metric_<id>`, не на физические колонки. Если колонка переименуется — поменяется только `expression` метрики, маршруты не сломаются.

**Версия метрики.** При изменении формулы — `version` инкрементируется. По trace всегда видно, на какой версии паспорта собран ответ.

**Синонимы — из речи клиента.** Не выдумывать. Аудит синонимов по transcripts и decisions — обязателен **до** первого тест-промпта. Где синонима в речи нет — пометка «в речи не зафиксировано».

**Антисинонимы — отдельная защита.** Если в речи клиента есть омонимы (одно слово = разные сущности — например, «контракт с сотрудником» и «контракт с клиентом») — фиксировать в `antonyms`, чтобы извлекатель не привязал данные не к той метрике.

**`known_issues` — обязательная секция.** Каждый клиентский xlsx — это «подарок»: DIV/0, врущие метки, пустые периоды, объединённые ячейки. Без явной фиксации косяков агент тихо отвечает неточно.

**Iron Law.** Маршрут добавляется, когда вопрос задавался ≥2 раз и есть проверенный ответ. Один разовый удачный ответ маршрутом не становится.
