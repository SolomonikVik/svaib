# metrics/ — система метрик бизнеса

Опциональная папка. Семантика метрик бизнеса CEO для AI-агента: какие источники, какие метрики, какие маршруты под типовые вопросы. Числа не дублируются здесь — они живут в `source/` (xlsx-файлы клиента). Расчёты выполняются Python-ом (xlsx-skill / Pandas) поверх источников. AI читает семантику, дёргает код, возвращает воспроизводимый ответ.

**Стартовая точка у нового клиента — первый слой:** `business-metrics.md` + служебная колонка с каноническими именами метрик в xlsx + AI-аналитик target metrics CEO. Без паспортов L2, без extractor-а, без narrative. Спецификация формата metrics-файлов — [`../../methodology/metrics/metrics-spec.md`](../../methodology/metrics/metrics-spec.md). Остальные файлы каркаса (`01_metrics.md`, `template-domain.md`, `extractors/`) — второй слой, разворачиваются по живому триггеру.

## Что лежит

При разворачивании папки создаётся только каркас:

- `README.md` — этот файл
- `business-metrics.md` — базовый файл metrics-вертикали (L1): target metrics CEO одним списком с пятью полями метрики. Точка входа для каждого клиента
- `01_metrics.md` — витрина L2 (target metrics CEO, ссылки на паспорта в доменах); появляется, когда вводятся domain-файлы
- `template-domain.md` — шаблон функционального domain-файла L2
- `source/` — исходные xlsx-файлы клиента (read-only для AI)
- `extractors/` — per-client Python-скрипты L2; правила в `extractors/README.md`

**Функциональные domain-файлы создаются по живому триггеру** — когда у CEO появилась реальная работа в домене (не «на всякий случай»). Минимальный канон:

> **Источник правды по канону имён domain-файлов** — [`../../methodology/metrics/architecture.md`](../../methodology/metrics/architecture.md), раздел «Канонические имена domain-файлов». Здесь — копия для удобства; при разворачивании у клиента эта копия едет вместе со scaffold.

| Файл | Что покрывает |
|------|---------------|
| `finance-metrics.md` | прибыль, маржа, расходы, cash, P&L |
| `sales-metrics.md` | выручка, pipeline, конверсии, средний чек |
| `operations-metrics.md` | delivery, качество, сроки, capacity, SLA |
| `people-metrics.md` | команда, найм, удержание, производительность |

Расширение за пределы минимального канона — тоже по живому триггеру:

| Файл | Триггер появления |
|---|---|
| `customer-metrics.md` | retention/churn/LTV стали отдельным контуром |
| `marketing-metrics.md` | отдельная машина привлечения, CAC по каналам |
| `product-metrics.md` | цифровой продукт с метриками использования |
| `strategy-metrics.md` | связь долгосрочных целей с метриками |

## Правила уровня папки

**Семантика отдельно от данных.** Числа — только в `source/`. Markdown-файлы здесь описывают, не считают.

**Цифры в одном месте.** Значения нигде не дублируются. Верхние файлы (`01_metrics.md`, `../02_strategy/02_goal.md`, `../04_company/{node}/01_overview.md`) ссылаются на паспорт метрики, а не пересказывают её.

**LLM не считает в голове.** Любой расчёт — через детерминированный код (xlsx-skill / Pandas). Соблазн «прикинуть на маленькой таблице» — главный источник тихих ошибок.

**ID метрики = её каноническое имя.** Имя, под которым метрика записана заголовком в metrics-файле — её идентификатор. Никаких отдельных `metric_<id>` полей. Ссылки между файлами — по каноническому имени (`{domain}-metrics.md#{имя}`).

**Синонимы — из речи CEO.** Если на L2 добавляется поле `synonyms` в датасетах и метриках — оно заполняется из транскриптов и решений, не выдумывается.

**Один паспорт — один файл.** Метрика живёт в одном metrics-файле. В других — ссылки `{domain}-metrics.md#{имя}`.

**Iron Law.** Domain-файл создаётся, когда у CEO живая работа в домене. Не «на всякий случай».
