# metrics/ — система метрик бизнеса

Опциональная папка. Семантика метрик бизнеса CEO для AI-агента: какие источники, какие метрики, какие маршруты под типовые вопросы. Числа не дублируются здесь — они живут в `source/` (xlsx-файлы клиента). Расчёты выполняются Python-ом (xlsx-skill / Pandas) поверх источников. AI читает семантику, дёргает код, возвращает воспроизводимый ответ.

**Стартовая точка у нового клиента — первый слой:** `canonical_metrics.md` + колонка с каноническими названиями в xlsx + помощник по колонке. Без паспортов, без extractor-а, без narrative. Рамка — [`../../methodology/metrics/first-layer.md`](../../methodology/metrics/first-layer.md). Остальные файлы каркаса (`01_metrics.md`, `template-domain.md`, `extractors/`) — второй слой, разворачиваются по живому триггеру.

## Что лежит

При разворачивании папки создаётся только каркас:

- `README.md` — этот файл
- `canonical_metrics.md` — первый слой: каноническое имя + бизнес-смысл + как считается. Базовый артефакт у каждого клиента
- `01_metrics.md` — витрина второго слоя (target metrics CEO, ссылки на паспорта в доменах); появляется, когда вводятся паспорта
- `template-domain.md` — шаблон domain-файла второго слоя
- `source/` — исходные xlsx-файлы клиента (read-only для AI)
- `extractors/` — per-client Python-скрипты второго слоя; правила в `extractors/README.md`

**Domain-файлы создаются по живому триггеру** — когда у CEO появилась реальная работа в домене (не «на всякий случай»). Минимальный канон:

> **Источник правды по канону имён domain-файлов** — [`../../methodology/metrics/architecture.md`](../../methodology/metrics/architecture.md), раздел «Канонические имена domain-файлов». Здесь — копия для удобства; при разворачивании у клиента эта копия едет вместе со scaffold.

| Файл | Что покрывает |
|------|---------------|
| `finance.md` | прибыль, маржа, расходы, cash, P&L |
| `sales.md` | выручка, pipeline, конверсии, средний чек |
| `operations.md` | delivery, качество, сроки, capacity, SLA |
| `people.md` | команда, найм, удержание, производительность |

Расширение за пределы канона — тоже по живому триггеру:

| Файл | Триггер появления |
|---|---|
| `customer.md` | retention/churn/LTV стали отдельным контуром |
| `marketing.md` | отдельная машина привлечения, CAC по каналам |
| `product.md` | цифровой продукт с метриками использования |
| `strategy.md` | связь долгосрочных целей с метриками |

## Правила уровня папки

**Семантика отдельно от данных.** Числа — только в `source/`. Markdown-файлы здесь описывают, не считают.

**Цифры в одном месте.** Значения нигде не дублируются. Верхние файлы (`01_metrics.md`, `../02_strategy/02_goal.md`, `../04_company/{node}/01_overview.md`) ссылаются на паспорт метрики, а не пересказывают её.

**LLM не считает в голове.** Любой расчёт — через детерминированный код (xlsx-skill / Pandas). Соблазн «прикинуть на маленькой таблице» — главный источник тихих ошибок.

**Синонимы — из речи CEO.** Поле `synonyms` в датасетах и метриках заполняется из транскриптов и решений, не выдумывается.

**Один паспорт — один файл.** Метрика живёт в одном domain-файле. В других — ссылки `domain.md#metric_<id>`.

**Iron Law.** Domain-файл создаётся, когда у CEO живая работа в домене. Не «на всякий случай».
