# Карта метрик

Витрина target metrics CEO + карта, какой домен закрывает какую часть бизнеса. AI-агент при любом вопросе про метрики читает этот файл первым: определяет домен, идёт в нужный domain-файл за паспортом и маршрутом.

**Не дублирует значения и формулы.** Только карточки-ссылки на паспорта в доменах.

## Target metrics CEO

Метрики, которые CEO держит в голове постоянно. Каждая — карточка-ссылка на паспорт.

| Метрика | Что измеряет | Паспорт |
|---------|--------------|---------|
| _выручка_ | <одной фразой> | `sales.md#metric_revenue` |
| _прибыль_ | <одной фразой> | `finance.md#metric_profit` |
| _маржа_ | <одной фразой> | `finance.md#metric_margin` |
| _cash_ | <одной фразой> | `finance.md#metric_cash` |
| _<метрика>_ | <одной фразой> | `<domain>.md#metric_<id>` |

> Заполняется при онбординге: 5–10 метрик, которые CEO называет в речи.

## Глоссарий CEO

Опциональная секция. Слова и обороты CEO, которые не являются метриками сами по себе, но регулярно встречаются в его вопросах и должны однозначно интерпретироваться оркестратором. Заполняется по аудиту transcripts.

| Оборот CEO | Что значит | На какую процедуру маппится |
|------------|------------|-----------------------------|
| _«что у меня горит»_ | топ-N метрик с худшим разрывом план/факт в плохую сторону | compound route, набор = все метрики с `plan_required: true`, ранжирование по \|Δ%\|, фильтр по `direction`-aware bad-side |
| _«готов меня к встрече с X»_ | бриф по метрикам куратора X + его ДЗ из прошлой встречи + риски | compound route, набор = метрики юнита X + transcripts последней встречи |
| _«мы тут на встрече решили»_ | смотри `04_decisions.md` соответствующего юнита | вопрос вне metrics — переход в meeting/decisions cycle |

> Без глоссария оркестратор интерпретирует обороты по контексту, что снижает воспроизводимость. Слово-кандидат добавляется в глоссарий, когда CEO произнёс его ≥2 раз в разных вопросах.

## По OKR

Опциональная секция. Заполняется, если у CEO структура OKR и он мыслит метрики через цели. Срез поверх паспортов: какие метрики работают на какую OKR. Сами OKR живут в `../02_strategy/02_goal.md` — здесь только навигация на паспорта.

```
### OKR{N} — {название на языке CEO}
- {метрика как называет CEO} → `<domain>.md#metric_<id>`
- {метрика как называет CEO} → `<domain>.md#metric_<id>`
```

> Не дублирует определения метрик и не хранит OKR. Только маршрут от клиентского ракурса («что с OKR3?») к паспортам в доменах.
