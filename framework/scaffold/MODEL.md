# Scaffold Model

> **Этот файл не переносится клиенту.** MODEL.md — методологический документ framework. Он живёт рядом со scaffold как источник правил, но при развёртывании клиентского пространства не копируется. Файлы внутри `scaffold/` (обязательные и опциональные папки, включая их README) спроектированы самодостаточными: не содержат ссылок на эту модель, `principles.md`, `memory/file_spec.md` и другие framework-only документы — чтобы у клиента не было «висящих» зависимостей.

Прескриптивная модель scaffold в Second AI Brain. Какие папки существуют, какие миссии у них, какие файлы внутри, как информация маршрутизируется, когда что разворачивается у клиента. Source of truth для всего, что касается структуры scaffold.

**Работа с файлом:** reference — читается первым при любой работе со scaffold. Обновляется по решению после клиентского опыта, не по ходу работы.

**Scope этого файла:** модель scaffold. Не содержит реализацию скилла развёртывания (→ [../skills/scaffold/_draft_spec.md](../skills/scaffold/_draft_spec.md)).

## 1. Базовые принципы

1. **Scaffold как замысел.** Scaffold — полная модель всех возможных сущностей и правил их развёртывания. Не набор файлов для копирования, а модель, которая разворачивается избирательно под контекст клиента.
2. **Разворачивай только живое.** У клиента создаются только те подразделения/направления, папки и файлы, за которыми стоит реальная работа: встреча, задача, решение, человек. Пустые ветки «на всякий случай» не создаются.
3. **Source of truth — во framework.** Каноны scaffold живут в `framework/scaffold/`. У клиента — либо развёртывание через скилл (будущее), либо ручное по правилам развёртывания (текущее).
4. **Управленческий контур CEO ≠ HR-оргсхема.** Scaffold моделирует то, с чем CEO реально работает: с кем встречается, кому делегирует, чью работу контролирует. Если CEO работает с «маркетинг+продажи» как с одним блоком — это блок в scaffold, независимо от того, как эти отделы расположены в формальной HR-структуре.
5. **Миссия папки как фильтр.** Каждая обязательная папка отвечает на один вопрос. Агент перед размещением информации проверяет: «соответствует ли миссии этой папки?». Не соответствует — ищет другую.
6. **Всё опционально внутри подразделения/направления.** Лидер, KPI, логика группировки, миссия — могут быть пустыми. Клиент принимается с любой стадией структурности.
7. **Конвенция важнее конфигурации.** Тип подразделения/направления, уровень, логика группировки — выводятся из структуры папок и свободного описания, не из атрибутов в YAML.

## 2. Онтология подразделений

Две концептуальные сущности внутри `04_management/`:

- **Единица** — неделимая в контексте управления. Концептуально имеет лидера и KPI. Клиентские синонимы: отдел, подразделение, команда, служба, группа, департамент. EN: department, team, unit, function.
- **Группировка** — объединение нескольких единиц в один управляемый вектор. Лидер опционален. Может иметь собственные цели/OKR. Клиентские синонимы: направление, блок, вектор, дирекция, кластер, департамент. EN: division, area, group, cluster, domain.

**Тип выводится из структуры, не хранится атрибутом.** Нет дочерних организационных подразделений/направлений (папок с `01_overview.md`) → единица. Есть дочерние подразделения/направления → группировка. Служебные подпапки (`meetings/`, `archive/`) не считаются. Эволюция (единица → группировка при появлении дочернего подразделения/направления) происходит сама по мере создания папок.

**Логика группировки** — почему единицы объединены в одну группировку. Свободное описание в `01_overview.md` группировки, не атрибут в YAML. Типовые логики:

- Функциональная — коммерция = продажи + маркетинг
- По OKR — снижение churn = product + support + implementation + account management
- Операционная — бэкофис = финансы + HR + legal
- Продуктовая — линейка A = разработка A + маркетинг A + поддержка A
- Географическая — хаб Алматы = продажи + операции + склад

**Уровни от CEO вниз.** Level 1 — ближайший к CEO (C-1). Level 2 — следующий вниз. Level 3+ — глубже. Нумерация от точки зрения CEO, не от абсолютной иерархии компании. На одном уровне могут сосуществовать и единицы, и группировки.

**Клиентский словарь — в `04_management/glossary.md`.** Клиент говорит на своём языке («департамент», «направление», «блок»), агент читает словарь и отвечает на клиентском языке. Внутренне оперирует онтологией единица/группировка.

**За рамками онтологии:** модель людей и ролей (один человек в нескольких единицах, совмещение, матричное подчинение) — живёт в профилях `03_team/`, не в структуре единиц.

## 3. Структура корня scaffold

```
scaffold/
├── 01_ceo/          ← обязательная
├── 02_strategy/     ← обязательная
├── 03_team/         ← обязательная
├── 04_management/   ← обязательная
│
├── clients/         ← опциональная
├── knowledge/       ← опциональная
├── processes/       ← опциональная
├── product/         ← опциональная
│
├── _inbox/
└── archive/
```

**Обязательные папки (01–04)** пронумерованы по смыслу: CEO → стратегия → люди → структура управления. Нумерация отражает последовательность чтения агентом при старте и смысловую зависимость блоков.

**Опциональные папки** — без префиксов, по алфавиту, без приоритета. Разворачиваются у клиента по триггеру живой работы (раздел 8).

### Миссии обязательных папок

| Папка              | Миссия (вопрос)                    | Что сюда                                                                                             | Что НЕ сюда                                                                                          |
| ------------------ | ---------------------------------- | ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| **01_ceo/**        | Кто CEO как человек?               | Личность, ценности, стиль работы. Личные задачи и открытые вопросы. Личные мысли, идеи, рефлексии. Личные твёрдые решения CEO. | Артефакты компании → 02_strategy. Задачи подразделений → 04_management. Профили сотрудников → 03_team. |

## 4. Раскладка файлов

### 4.1 `01_ceo/`

```
01_ceo/
├── 01_my_profile.md      ← кто я, ценности, стиль работы
├── 02_active.md          ← личные задачи и открытые вопросы CEO
├── 03_thoughts.md        ← мысли впрок: идеи, рефлексии, тревоги, гипотезы
└── 04_decisions.md       ← личные твёрдые решения CEO
```

Порядок: стабильный контекст (01) → текущая работа (02) → мягкое (03) → твёрдое (04).

**`01_my_profile.md`** — личность CEO как входные данные для бизнес-решений: ценности, стиль мышления, компетенции, границы. Reference-файл. Обновляется раз в 6-12 месяцев.

**`02_active.md`** — личные задачи и открытые вопросы CEO. Две секции, без секций по исполнителям (исполнитель один — сам CEO):

```markdown
## Задачи

- [ ] {что сделать (срок | контекст)}

## Открытые вопросы

- [ ] {что решить (контекст)}
```

State-файл: перезаписывается, не растёт. Содержит **только прямые личные задачи и вопросы CEO** — то, что нигде больше не живёт. Задачи по подразделениям — в `04_management/{node}/02_active.md`. Сборка «всё, что на CEO» — агент собирает по запросу (раздел 7).

Задача = делать (действие). Открытый вопрос = решить (думать до развилки). Дозревший открытый вопрос переезжает в `04_decisions.md` (личное), в `02_strategy/06_decisions.md` (курс), в `04_management/{node}/04_decisions.md` (подразделение/направление), или превращается в задачу.

**`03_thoughts.md`** — мысли впрок: идеи, рефлексии, тревоги, гипотезы. Один сборник на всё — низкий порог записи. Внутри — по датам, опционально теги. Разбор — отдельным ритуалом, не на входе записи. Один файл вместо ideas+reflections: классификация на входе («идея или рефлексия?») убивает запись.

**`04_decisions.md`** — личные твёрдые решения CEO. Отличаются от мыслей по жёсткому признаку: **на мысль агент не опирается, на решение опирается как на факт**. Смешивать с thoughts нельзя — агент не понимает, что считать руководством к действию.

Примеры решений, которым некуда деться кроме `01_ceo/`:

- «расстаюсь с Васей» (готовится, в оргструктуру рано)
- «пока не запускаем направление X» (нет курса компании, нет подразделения)

Работает как **буфер** (зафиксировано лично → потом вынесено в `02_strategy/` или `04_management/`) или **конечная точка** (наружу не пойдёт принципиально).

### 4.2 `02_strategy/`

```
02_strategy/
├── 01_vision.md           ← миссия, идентичность, видение будущего
├── 02_goal.md             ← долгосрочная цель, стратегия, цели периода, метрики (план)
├── 03_plan.md             ← план периода → неделя → день
├── 04_weekly_progress.md  ← хроника по неделям
├── 05_timeline.md         ← сводная история проекта, план vs факт
└── 06_decisions.md        ← реестр стратегических решений компании
```

Порядок: идентичность (01) → куда (02) → что делаем сейчас (03) → хроника (04) → история (05) → почему так (06).

Все файлы — артефакты компании. Видит внешний человек (инвестор, преемник, новый сотрудник) как «кто они, куда идут, что делают, почему так».

**`03_plan.md`** — не чистый state-файл, а сборка (раздел 7). Показывает срез текущей работы компании: пересечение активити по подразделениям/направлениям и стратегических целей.

### 4.3 `03_team/`

```
03_team/
├── README.md               ← обзор: как устроена папка
├── org_structure.md        ← маппинг людей на подразделения/направления в 04_management/
└── {person}.md             ← профайл сотрудника
```

`org_structure.md` маппит каждого человека на путь в `04_management/{node}`. Профайлы сотрудников (`{person}.md`) — стиль, сильные стороны, развитие, коучинг-лог.

### 4.4 `04_management/`

```
04_management/
├── README.md               ← модель подразделения/направления, правила маршрутизации
├── meetings/               ← стратсессии CEO, кросс-встречи верхнего уровня
├── glossary.md             ← клиентский словарь, жаргон, ошибки транскрибации
│
├── {org_node}/             ← подразделение/направление (единица или группировка)
│   ├── 01_overview.md      ← миссия, лидер (опц.), KPI (опц.), логика группировки (опц.)
│   ├── 02_active.md        ← задачи, вопросы, риски (секции по исполнителям)
│   ├── 03_progress.md      ← хроника по встречам
│   ├── 04_decisions.md     ← решения уровня подразделения/направления
│   ├── meetings/           ← встречи по этому подразделению/направлению
│   │   └── archive/        ← сырые транскрипты
│   └── {sub_node}/         ← дочернее подразделение/направление
│       └── ...
└── ...
```

На верхнем уровне `04_management/` лежат только: `README.md`, `meetings/`, `glossary.md`, дерево подразделений/направлений. Больше ничего.

#### Шаблон подразделения/направления `{org_node}/`

Одинаков для единицы и группировки. Тип определяется наличием дочерних подразделений/направлений.

- **`01_overview.md`** — миссия подразделения/направления, лидер (опционально), KPI (опционально), логика группировки (для группировок, опционально). Reference-файл, меняется редко.
- **`02_active.md`** — state-файл: задачи, вопросы, риски, повестка следующей встречи. Секции по исполнителям (`### {CEO}`, `### {Лидер}`, `### Не назначен`). Перезаписывается, не растёт.
- **`03_progress.md`** — log-файл: хроника по встречам, 2-4 строки на встречу.
- **`04_decisions.md`** — решения уровня этого подразделения/направления с обоснованием.
- **`meetings/`** — протоколы встреч по этому подразделению/направлению. `meetings/archive/` — сырые транскрипты.

**Всё внутри подразделения/направления опционально.** Лидер, KPI, логика группировки, миссия — могут быть пустыми. Клиент принимается с любой стадией структурности.

**Произвольная вложенность.** Level = глубина от `04_management/`. На одном уровне могут сосуществовать единицы и группировки.

#### `glossary.md`

Единый словарь клиентского пространства. Содержит:

- **Клиентские синонимы подразделений/направлений** — как клиент называет свои единицы и группировки («у нас это служба», «у нас дирекция»). Агент маппит на онтологию единица/группировка.
- **Бизнес-жаргон клиента** — специфическая терминология отрасли и компании.
- **Ошибки транскрибации** — типичные ошибки транскрибатора на именах сотрудников, клиентов, терминах. Правильные написания.

Обязательно читается L2-пайплайном meeting-analysis перед обработкой встреч.

#### `04_management/meetings/` (верхний уровень)

Содержит:

- Стратсессии CEO — встречи всей верхушки компании
- Кросс-функциональные встречи, охватывающие несколько верхнеуровневых подразделений/направлений

Маршрутизация встреч — раздел 5.

## 5. Правило общей родительской папки (встречи)

Протокол встречи ложится в `meetings/` общей родительской папки, охватывающей всех участников встречи.

| Состав встречи                                                         | Куда протокол                      |
| ---------------------------------------------------------------------- | ---------------------------------- |
| Внутри одного подразделения/направления                                | `04_management/{node}/meetings/`   |
| Между дочерними подразделениями/направлениями одной группировки        | `04_management/{parent}/meetings/` |
| Между верхнеуровневыми подразделениями/направлениями / стратсессии CEO | `04_management/meetings/`          |

Примеры:

- 1-on-1 с маркетинг-директором → `commercial/marketing/meetings/`
- Планёрка коммерции (sales + marketing) → `commercial/meetings/`
- Стратсессия «все C-1» → `04_management/meetings/`

В `meetings/` — только обработанные выжимки (`_summary.md`). В `meetings/archive/` — сырые транскрипты (`_transcript.md`).

## 6. Маршрутизация решений

В модели есть три уровня файлов с решениями — личный, компанейский, по подразделению/направлению. Граница:

| Критерий                                                                                             | Куда                                                                  |
| ---------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| Решение влияет на курс компании, цели, стратегию, vision. Через 3 месяца спросят «почему мы двигаемся туда?». | `02_strategy/06_decisions.md`                                         |
| Решение касается только одного подразделения/направления и не меняет курс. Через 3 месяца спросят «почему у commercial такой подход?». | `04_management/{node}/04_decisions.md`                                |
| Решение охватывает несколько подразделений/направлений, но не меняет курс (например, как перераспределить нагрузку между sales и marketing). | `04_management/{parent}/04_decisions.md` (общая родительская папка)   |
| Личное решение CEO — ещё не вынесено наружу, или принципиально не пойдёт наружу.                     | `01_ceo/04_decisions.md`                                              |
| Сомнение «курс или одно подразделение/направление?»                                                  | **Выше**, в `02_strategy/`. Компания важнее.                          |

Мелкие операционные выборы («не делать сегодня X», «перенести звонок») — не решения, не маршрутизируются, остаются в протоколах встреч или `02_active.md`.

## 7. Модель сборок задач

### Источник истины — там, где задача живёт по природе

- Задача по подразделению → `04_management/{node}/02_active.md`, секция `### {CEO}` или `### {Лидер}`
- Личная задача CEO (нигде больше не живёт) → `01_ceo/02_active.md`

**Дубля нет.** Задача живёт в одном месте — в том, к которому относится по природе. Развилка «куда писать первоначально» очевидна: либо задача привязана к подразделению/направлению, либо нет.

### Формат секций в подразделениях/направлениях

`04_management/{node}/02_active.md` использует секции по исполнителям:

```markdown
## Задачи

### {CEO}

- [ ] {Задача (срок | из встречи YYYY-MM-DD)}

### {Лидер}

- [ ] {Задача (срок | из встречи YYYY-MM-DD)}
- [x] {Выполненная — перенести в повестку встречи}

### Не назначен

- [ ] {Задача (YYYY-MM-DD)}
```

Секция = исполнитель. Явный маркер «кто делает», без тегов вроде `@ceo`.

### Сборки не материализуются

**«Всё, что на CEO»** — агент собирает по запросу из `01_ceo/02_active.md` + секций `### {CEO}` во всех `04_management/{...}/02_active.md`. Файл не материализуется. Нет проблемы синхронизации, нет вопроса «куда писать первоначально».

**`02_strategy/03_plan.md`** — сборка плана периода/недели компании. Собирается из активити по подразделениям/направлениям + стратегических целей из `02_goal.md`. Назначение: увидеть, верно ли двигаемся — соответствуют ли активити стратегическим целям. План = пересечение «что делаем» и «куда идём».

**Механика сборки `03_plan.md`:** гибрид. Файл существует, видим и читаем. Обновляется понедельничным ритуалом планирования (скилл `weekly-planning`).

### Темпы

| Файл                                | Темп обновления                                   |
| ----------------------------------- | ------------------------------------------------- |
| `04_management/{node}/02_active.md` | Ежедневно / после каждой встречи                  |
| `01_ceo/02_active.md`               | Ежедневно                                         |
| `02_strategy/03_plan.md`            | Еженедельно (понедельник, ритуал weekly-planning) |
| `02_strategy/02_goal.md`            | Начало периода (месяц / квартал)                  |

## 8. Правила развёртывания

Описывают, что и когда создаётся у клиента. Текущий этап — развёртывание ручное по этим правилам. Автоматизация в виде скилла — [../skills/scaffold/_draft_spec.md](../skills/scaffold/_draft_spec.md), откладывается до 2-3 успешных ручных развёртываний.

### Iron Law

> **Разворачивай только живое. Пустые ветки не создавай. «Живое» = появилась реальная работа (встреча, задача, человек), а не «заполнены поля».**

### Что создаётся при старте клиента

1. **`01_ceo/`** — все 4 файла создаются сразу (пустые допустимы). Низкий порог записи, нужны с первого дня.
2. **`02_strategy/`** — все 6 файлов создаются сразу (пустые допустимы).
3. **`03_team/`** — `README.md`, `org_structure.md` + по одному `{person}.md` на каждого C-1 (перечисляет CEO при онбординге). Профайлы нужны с первого дня для обработки встреч и коучинга.
4. **`04_management/`** — `README.md`, `meetings/`, `glossary.md`. Дерево подразделений/направлений не разворачивается сразу — только по триггеру (ниже).
5. **Опциональные папки** — НЕ создаются при старте. Разворачиваются по триггеру.

### Триггер разворачивания подразделения/направления `04_management/{node}/`

Подразделение/направление разворачивается: клиент явно работает с ним как существующим («у меня есть отдел маркетинга, мы встречаемся, ставим цели, я контролирую»).

**Подразделение/направление НЕ разворачивается**, если упомянуто в разговоре без артефактов работы — держим в `02_active.md` как контекст.

При развёртывании подразделения/направления создаются все 4 файла (`01_overview.md`, `02_active.md`, `03_progress.md`, `04_decisions.md`) + `meetings/`. Пустые допустимы.

### Триггер разворачивания `03_team/{person}.md`

Новый файл сотрудника создаётся при:

- Явном перечислении C-1 при онбординге
- Первой встрече с участием этого человека
- Первом упоминании с явной ролью

### Триггеры опциональных папок

Разворачиваются, когда у клиента появляется реальная работа в домене, не «на всякий случай».

| Папка          | Триггер                                                                                              |
| -------------- | ---------------------------------------------------------------------------------------------------- |
| **clients/**   | CEO лично работает с конкретными именованными клиентами, это важная часть управления (аккаунт-менеджмент, B2B-продажи, консалтинг, агентский бизнес). |
| **knowledge/** | У CEO есть база знаний, которую он хочет структурировать — живая работа с материалом, а не свалка ссылок. |
| **processes/** | Есть формализованные бизнес-процессы или регламенты, которые нужно держать в системе.                |
| **product/**   | У компании есть продукт (то что поставляется клиентам — SaaS, услуги, консалтинг, B2B-сервис, физический товар) и есть специфика для документирования: позиционирование, внутреннее устройство, стадия реализации, продуктовые решения. В подавляющем большинстве случаев разворачивается. Отличие от `projects/{...}`: продукт — постоянный контур, проект — временная активность с завершением. При развёртывании создаётся раскладка из 7 файлов (`README.md`, `00_product.md`, `01_overview.md`, `02_active.md`, `02_backlog.md`, `03_progress.md`, `04_decisions.md`), пустые файлы допустимы. `architecture.md` — опциональный внутри папки, разворачивается при наличии внутреннего устройства продукта (SaaS, tech, физический продукт со стадиями производства); для услуг/консалтинга не разворачивается. Подпапки (`docs/`, `specs/`, `pricing/` и т.п.) — только по живому триггеру, канон жёстко не фиксируется. Встречи по продукту — в `04_management/{node}/meetings/` по правилу общей родительской папки (раздел 5), не здесь. Канон шаблона — [product/README.md](product/README.md). 02_active **без** Session Handoff (это для svaib направлений). |
| **projects/**  | У CEO есть именованные проекты с горизонтом и завершением (миграция CRM, запуск продукта, стройка склада). Отличие от `04_management/{org_node}/`: проект — временная активность, направление — постоянная. При развёртывании создаётся вся полная раскладка проекта (`_inbox/01_inbox.md`, `01_overview.md`, `02_active.md`, `02_backlog.md`, `03_progress.md`, `04_decisions.md`, `meetings/`), пустые файлы допустимы. Канон шаблона — [projects/README.md](projects/README.md). 02_active **без** Session Handoff (это для svaib направлений). |

Когда опциональная папка разворачивается — её файлы создаются сразу внутри (по тому же правилу, что для обязательных): пустые допустимы, плейсхолдеры не оставляем.

### Удаление, слияние, перенос подразделений/направлений

**Только вручную.** Агент при запросе на удаление, слияние или перенос подразделения/направления — останавливается и просит подтверждения CEO или Виктора. Автоматика не применяется: риск потерять историю работы.

### Чего не создаёт ни один шаг развёртывания

- Пустые ветки «на всякий случай»
- Файлы под плейсхолдеры, если подразделения/направления ещё нет
- Копии файлов для «будущих подразделений»
