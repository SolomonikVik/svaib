# product/ — продуктовый контур

Рабочее пространство продукта компании: что за продукт, как он устроен, где мы в его стройке, какие задачи/решения/хроника. Читается агентом, когда CEO работает над продуктом (позиционированием, roadmap, устройством, продуктовыми решениями).

## Природа

**Продукт — то, что компания поставляет клиентам.** SaaS, услуги, консалтинг, B2B-сервис, физический товар — всё это продукт. Меняется наполнение, но схема контура одна.

**Продукт — постоянный контур.** Этим отличается от `../projects/{...}` (временная активность с горизонтом завершения) и от `../04_management/{org_node}/` (подразделение/направление, которое может быть не про продукт).

**Опциональная папка.** Разворачивается, когда у компании есть специфика продукта для документирования: позиционирование, устройство, стадия реализации, продуктовые решения. Не разворачивается «на всякий случай» — пустые ветки не создаются.

## Структура

```
product/
├── README.md          — этот файл
├── 00_product.md         — ЧТО за продукт и зачем (проблема, ICP, решение, принципы, границы, бизнес-модель, стадия)
├── architecture.md    — КАК устроен продукт внутри (компоненты, связи, стек, интеграции) — разворачивается по триггеру
├── 01_overview.md     — состояние реализации: стадия и roadmap периода
├── 02_active.md       — операционный хаб (повестка, задачи, вопросы, риски)
├── 02_backlog.md      — полный список задач и идей
├── 03_progress.md     — хроника встреч + динамика метрик (блок метрик опционален)
└── 04_decisions.md    — продуктовые решения с контекстом
```

## Правила развёртывания

**Триггер разворачивания папки.** У компании есть продукт, который поставляется клиентам, и есть что документировать: позиционирование, устройство, стадия, продуктовые решения. В подавляющем большинстве случаев папка нужна.

**Раскладка при триггере.** Создаются `README.md` + 6 файлов из 7 шаблонов: `00_product.md`, `01_overview.md`, `02_active.md`, `02_backlog.md`, `03_progress.md`, `04_decisions.md`. Пустые файлы и секции допустимы — каркас ставится с первого дня.

**`architecture.md` — опциональный внутри папки.** Разворачивается, когда у продукта есть внутреннее устройство, которое осмысленно документировать: SaaS, tech-продукт, физический продукт со стадиями производства. Для услуг / консалтинга / простой дистрибуции — не разворачивается; «из чего состоит решение» ложится в `00_product.md` (секция «Решение»).

**Подпапки — по живому триггеру.** Типовые: `docs/` (документация продукта), `specs/` (технические спеки), `pricing/` (прайсы и тарифы отдельным документом). Канон подпапок жёстко не фиксируется — природа продукта у разных клиентов отличается. Разворачиваются при появлении реального материала, не заранее.

**Встречи по продукту — не здесь.** Протоколы встреч ложатся в `meetings/` ближайшей общей родительской папки, охватывающей участников: продуктовая встреча внутри продуктовой команды → `../04_management/{product-node}/meetings/`; кросс-функциональная встреча между несколькими направлениями → выше в дереве `04_management/`. В `03_progress.md` — только краткие записи по встречам (2-4 строки), не полные протоколы.

## Связи с `../02_strategy/`

Продукт компании — часть её стратегии. Три связи между `product/` и `../02_strategy/`:

| № | Откуда | Куда | Частота |
|---|--------|------|---------|
| ① | `../02_strategy/01_vision.md` (блок «Продукт») | [00_product.md](00_product.md) | Стабильная, меняется при пивотах |
| ② | `../02_strategy/02_goal.md` (фокус «Продукт») | [01_overview.md](01_overview.md) | Каждый период (декада / квартал) |
| ③ | [03_progress.md](03_progress.md) | `../02_strategy/04_weekly_progress.md` | Еженедельно |

Если продукт у компании один и он же — ядро бизнеса, эти связи — основной маршрут между стратегическим и продуктовым слоями.
