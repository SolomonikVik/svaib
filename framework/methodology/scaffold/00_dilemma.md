# Scaffold root choice — дилемма, варианты, решение

Фиксация архитектурного спора о главной оси scaffold и принятого выбора. Временный файл — потом войдёт в [01_architecture.md](01_architecture.md) или останется как ADR. Подробности и канон распределения — позже отдельной работой.

> **Важно для читателя.** На момент написания канонические файлы методологии scaffold ([01_architecture.md](01_architecture.md), [02_folder-spec.md](02_folder-spec.md), [03_contours.md](03_contours.md), [03_node-files.md](03_node-files.md), [deployment.md](deployment.md)) описывают **старую модель Аспект > Домен** (`01_ceo / 02_strategy / 03_team / 04_company / 05_metrics`). Этот файл — новое решение (**Вариант 3 — Матрица**), ещё не интегрированное в канон. План трансформации канона — отдельной правкой ниже.

## Дилемма

Бизнес многомерен — домены × аспекты × проекты × люди × клиенты × время. Папочное дерево одномерно — файл живёт в одной точке. Любая иерархия = выбор **одной главной оси**, остальные измерения становятся вложенностью, ссылками или метаданными. Универсального ответа нет; это design problem с осознанными компромиссами под наш ICP (CEO 5-300 человек, работающий с AI).

В сущностях scaffold мы смешали две разные категории, не назвав их явно:

- **Домены** — что компания делает (маркетинг, продажи, продукт, финансы, операции, клиенты, IT, PR…).
- **Аспекты** — как компания это делает (стратегия, метрики, люди, процессы, проекты, встречи, решения, документы, знания).

Ключевое — **аспекты разворачиваются на разных уровнях**: на уровне компании (vision, оргструктура, target metrics, кросс-функциональные процессы и проекты, стратсессии) и на уровне домена (стратегия маркетинга, команда маркетинга, метрики маркетинга, процессы маркетинга, проекты маркетинга).

## Ограничения и не-ограничения

**Реальное ограничение одно** — файл лежит в одной точке дерева.

**Не ограничения** — реализуемы в любой модели и не различают варианты:
- метаданные через YAML;
- дашборды и навигационные сборки в README/overview;
- ссылки и связи между файлами;
- generated views (`_views/`), пересобираемые helper-скиллом;
- slash-команды для on-demand агрегации;
- веб-морда поверх git в будущем.

Поэтому навесы не различают принципиальные варианты — только улучшают выбранный.

## Варианты которые рассматривали

**Вариант 1 — Аспект > Домен.** Корень = типы сущностей (`ceo / strategy / team / company / metrics / projects / clients / …`). Домены (маркетинг, продажи, финансы) — узлы внутри `company/`. У каждой сущности один дом по её типу. Это текущая модель scaffold v1.

**Вариант 2 — Домен > Аспект.** Корень = функции (`marketing / sales / product / finance / operations / …`). Аспекты разворачиваются внутри домена (стратегия маркетинга, команда маркетинга, метрики маркетинга). Cross-функция требует отдельного решения — дубли, индексные регистры или YAML+views.

**Вариант 3 — Матрица.** Аспекты разворачиваются и на уровне компании (на корне), и внутри каждого домена — на разных уровнях по правилу. На корне живёт компанийное (target metrics, оргструктура, стратсессии, кросс-проекты, vision). Внутри домена — доменное (метрики маркетинга, команда маркетинга, проекты маркетинга). Один факт — один дом, но дом выбирается по уровню (компания или домен), а не по типу.

## Выбор

**Вариант 3 — Матрица.**

Обоснование: первый и второй — это выбор «или функции, или типы». Матрица говорит «и то, и то, на разных уровнях», что честнее отражает устройство бизнеса. Vision и target metrics — это про компанию целиком, они принадлежат уровню компании. Метрики маркетинга и команда маркетинга — про конкретный домен, они принадлежат уровню домена. Эти вещи не одного уровня, и насильно класть их рядом (как в варианте 1) или насильно разносить дублями (как в варианте 2) — компромиссы, которых матрица не делает.

Из трёх требований к scaffold ([01_architecture.md § Что такое scaffold](01_architecture.md)) — карта управления для CEO, навигационная среда для AI-агента, поддерживаемая память для команды — матрица сейчас лучше других совмещает все три на уровне архитектурного принципа.

Выбор фиксирует принцип, а не финальное дерево папок. Конкретная раскладка корня и доменных узлов определяется отдельно.

## Что уже прояснили после выбора

Матрица читается как одна и та же логика управления на разных уровнях: компания, домены, поддомены — на каждом уровне разворачивается только то, для чего есть живая работа. Терминология (домен / узел / контур / поддомен / аспект) — открытый вопрос, фиксируется при интеграции в канон. Раскладки ниже.

### Корневая раскладка матрицы

На уровне scaffold выделяется отдельная папка компании как компанийный уровень матрицы. Это сделано для читаемости дерева и удобства UI-навигации: компанийное не растворяется среди доменов, а домены остаются корневыми рабочими областями.

```text
scaffold/
├── 00_ceo/                  ← личный контекст CEO
├── 01_company/              ← компанийный уровень матрицы
│
├── clients/                 ← домен, если есть как отдельная область управления
├── finance/                 ← домен, если есть
├── knowledge/               ← общая база знаний компании, опц.
├── marketing/               ← домен, если есть
├── operations/              ← домен, если есть
├── product/                 ← домен, если есть
├── sales/                   ← домен, если есть
├── ...                      ← support, hr, it, pr-gr и др. по живой модели клиента
│
├── _inbox/                  ← служебная папка верхнего уровня
└── zz_archive/              ← общий архив
```

Домены не нумеруются: их состав не является фиксированным каноном и зависит от реальной управленческой модели клиента.

### `01_company/` — компанийный уровень матрицы

`01_company/` не натягивается на старую универсальную анатомию узла. Уровень всей компании собирается через компанийные аспекты:

```text
01_company/
├── _inbox/
│
├── 01_strategy/
│   ├── 01_vision.md
│   ├── 02_goal.md
│   ├── 03_strategy.md
│   ├── 04_progress.md
│   ├── 05_decisions.md
│   └── README.md
│
├── 02_team/
│   ├── org-structure.md
│   ├── {person}.md
│   └── README.md
│
├── 03_metrics/
│   ├── business-metrics.md
│   ├── extractors/          ← опц.
│   ├── source/
│   └── README.md
│
├── docs/                    ← опц.
├── glossary.md              ← общекомпанийный словарь для meeting-analysis
├── meetings/                ← встречи уровня всей компании
├── processes/               ← кросс-функциональные процессы, опц.
├── projects/                ← кросс-функциональные проекты, опц.
├── zz_archive/
└── README.md
```

Отдельные `overview / backlog / progress / decisions` для компании не вводятся автоматически:

- смысловой вход компании дают `01_strategy/01_vision.md`, `02_goal.md`, `03_strategy.md`;
- компанийный backlog собирается из доменов и кросс-функциональных проектов;
- компанийный progress и стратегические decisions уже живут в `01_strategy/04_progress.md` и `05_decisions.md`.

### Домен как саморазворачивающаяся область

Домен разворачивается по тому же принципу, что раньше универсальный узел, но может включать дополнительные аспектные папки:

```text
marketing/
├── _inbox/
├── docs/                    ← опц.
├── knowledge/               ← опц.
├── meetings/
├── metrics/                 ← опц.
├── processes/               ← опц.
├── projects/                ← опц.
├── team/                    ← опц.
├── zz_archive/
│
├── 01_overview.md
├── 02_active.md
├── 02_backlog.md
├── 03_progress.md
├── 04_decisions.md
├── profile.md               ← опц.
├── strategy.md              ← опц.
└── README.md
```

`team/`, `metrics/`, `projects/`, `processes/`, `knowledge/`, `strategy.md`, `profile.md` не создаются «на всякий случай». Они появляются, если на уровне конкретного домена возникла собственная живая работа и отдельный слой памяти действительно нужен.

### Домен может содержать поддомены

Если клиент мыслит не отдельными `marketing/` и `sales/`, а общим коммерческим блоком, структура может ветвиться:

```text
commercial/
├── README.md
├── 01_overview.md
│
├── marketing/
├── sales/
└── partnerships.md
```

Родительский домен не обязан сразу разворачивать всю инфраструктуру. На его уровне появляются только те элементы, которые реально относятся ко всему родительскому домену: общая стратегия, общие встречи, общие проекты, общие процессы, общие метрики, общие решения.

## Что считается доменом

**Домен** — устойчивая область управления бизнесом, у которой есть собственная ответственность и которая может быть предметом регулярного управленческого внимания.

Признаки домена:

- CEO или руководитель реально мыслит его как отдельную область;
- у него есть свой фокус, задачи, решения и динамика;
- вокруг него могут разворачиваться команда, метрики, процессы, проекты;
- он не сводится к разовой инициативе или отдельному артефакту.

Примеры доменов:

- `marketing/`;
- `sales/`;
- `product/`;
- `finance/`;
- `operations/`;
- `clients/` / `customer-success/`;
- `support/`, `hr/`, `it/`, `pr-gr/` — если это реальные управленческие области у клиента.

Не являются доменом сами по себе:

- разовая кампания → проект;
- регулярная процедура → процесс;
- конкретный сотрудник → человек;
- отдельный KPI → метрика.

## Распределительный принцип

После выбора матрицы главный вопрос сводится к одному:

> что живёт на родителе, что на ребёнке, что на потомке?

Канонический ответ — два связанных правила:

> 1. **Если есть операционный владелец — сущность живёт у него.**
> 2. **Если владельца нет — сущность живёт на наименьшем общем предке всех узлов, которым она нужна.**

**Приоритет: владелец перевешивает общего предка.** Если head of marketing ежедневно ведёт CAC, а смотрят её sales и CEO — CAC живёт в `marketing/metrics/`, не в `01_company/03_metrics/`. Остальные узлы получают ссылку или представление. Логика — класть туда, где **управляют**, а не туда, где **видят**; иначе сущность становится «ничейной» на верхнем уровне.

Этот подход — стандарт инженерных дисциплин: **bounded context** в DDD, **lift state up** в React, **нормализация** в реляционных БД, **каскад** в CSS и config-системах. Везде один и тот же ответ: факт живёт на наименьшем общем предке тех, кому он нужен; если есть владелец — у владельца.

Это также обобщение уже существующего в каноне **правила общей родительской папки**: раньше оно применялось к встречам, задачам и решениям; в матрице тот же принцип распространяется на `strategy`, `team`, `metrics`, `projects`, `processes` и другие аспектные слои.

### Порядок выбора уровня хранения

1. **Есть один явный операционный владелец?**  
   Сущность живёт у него.
2. **Нет одного владельца, но все участники находятся внутри одного родительского домена?**  
   Сущность живёт у этого родителя.
3. **Сущность затрагивает несколько корневых доменов?**  
   Сущность живёт на уровне `01_company/`.

Примеры:

- метрика только маркетинга → `marketing/metrics/`;
- общая метрика sales + marketing → родительский `commercial/metrics/`;
- target metric компании → `01_company/03_metrics/`;
- маркетинговая стратегия → `marketing/strategy.md`;
- стратегия всей компании → `01_company/01_strategy/`;
- кросс-функциональный проект нескольких корневых доменов → `01_company/projects/`;
- проект только одного домена → `{domain}/projects/`.

### `team` и `metrics` — не отдельная архитектурная развилка

После фиксации распределительного принципа `team` и `metrics` раскладываются тем же правилом:

**Люди:**
- участник домена (маркетолог, sales-менеджер) → `{domain}/team/`;
- **руководитель домена** (head of marketing, head of sales) → `{domain}/team/` — он сам себе operational owner внутри своего домена;
- руководитель родительского домена (head of commercial над marketing + sales) → `{parent-domain}/team/`;
- **C-level / руководитель уровня компании** (CFO, COO, CTO, подчинённые напрямую CEO) → `01_company/02_team/`.

**Метрики:**
- метрика только домена → `{domain}/metrics/`;
- метрика общего родительского домена → `{parent-domain}/metrics/`;
- target metric компании → `01_company/03_metrics/`.

Оставшийся вопрос по `team` и `metrics` уже не архитектурный, а навигационно-технический: нужны ли агрегирующие views, YAML-метаданные и какие именно ссылки поддерживать между уровнями.

## Плюсы выбора

- **Один факт — один дом.** Нет «второй правды»: уровень определяется природой факта.
- **Согласуется с собственной онтологией framework.** Управленческие циклы (`ontology/management_cycles.md`) получают физическое отражение в слое scaffold.
- **Самоподобная структура.** Каждый уровень матрицы — узел с триггерным разворотом аспектов. Компанийный узел адаптирует анатомию через свои аспекты (`01_strategy/`, `02_team/`, `03_metrics/`), доменный — через универсальные файлы + опциональные аспектные слоты. Принцип «разворачивается только живое» один и тот же на всех уровнях.

## Минусы и оговорки

- **Cross-domain navigation.** Остаётся задача навигационного механизма для сущностей, которые физически живут в одном узле, но должны быть видимы из нескольких (один человек в двух доменах, метрика двух доменов, проект нескольких доменов). Варианты — YAML-теги, ссылки из узла-«потребителя», generated views; решается отдельно.
- **Эмпирика не проверена.** Выбор сделан архитектурно, на ICP не валидирован.

## Что дальше

Открытые вопросы следующей очереди:

- **Терминология:** `узел`, `домен`, `поддомен`, `контур`, `аспект` — что становится каноном.
- **Cross-domain navigation:** как оформляются ссылки, YAML-метаданные и потенциальные generated views между уровнями.
- **Корневой `knowledge/` vs доменный `knowledge/`** — оба возможны, нужно правило.

Задачи трансформации канона и миграция клиентского `framework/scaffold/` — в § План трансформации канона ниже.
