# Scaffold README spec — канон карты папки

Канон `README.md` как карты папки в scaffold для AI-агента: структура, обязательные блоки, миссия папки + миссии файлов внутри неё. Каркас, аннотация, доставка через триплет с [AGENTS.md](#доставка-readme-агенту), отличия от [01_overview.md](#readme-vs-01_overviewmd).

## Миссия README

README — карта папки. Контейнер описывает себя через README; сущность папки — через `01_overview.md`. Эти роли не пересекаются.

Главная функция README — **маршрутизация AI-агента**: что лежит, куда идти, что прочитать дальше. Всё, что не помогает агенту принять решение, режется.

## Принципы README

1. **Ландшафт, не каталог.** Показывает устройство и направляет внимание; не перечисляет все файлы.
2. **Конкретика > обобщения.** «Все роуты регистрируются в `src/routes/index.ts`» работает; «следуй best practices» — мёртвый текст.
3. **Pointer-driven.** Тяжёлый контент по ссылке. README не объясняет — отправляет на SOT: канон файла → [02_file-spec.md](02_file-spec.md), решения → `04_decisions.md`, состояние сущности → `01_overview.md`.
4. **Эволюция от ошибок.** Новое правило в README = записанный прецедент агентской ошибки, а не превентивная гипотеза.
5. **Иерархия: ближайший побеждает.** Локальный README папки перекрывает корневой при совпадении правил. В корне держим только универсальное.

## Главный читатель — LLM-агент

Агент видит репозиторий как текстовые строки: путь файла, результат `Grep`, фрагмент содержимого. README — один из ключевых сигналов «копать глубже или хватит».

**Что важно из этого следует:**

- **Имя папки и путь работают как фрейминг до README.** `04_company/sales/meetings/` уже сообщает область, сущность и тип материала — README должен дополнять, не повторять.
- **Заголовки H2 в README — основные навигационные якоря.** Агент часто получает структуру через `Grep '^## '`, не читая содержимое. H2 должны быть самоописывающими и уникальными.
- **Ссылки внутри предложения — с контекстом перехода.** Голая ссылка без объяснения, зачем туда идти, бесполезна.

**Место в памяти.** README — **warm on-demand слой**: загружается агентом при заходе в папку или домен. Не оплачивается токенами всегда (в отличие от `MEMORY.md`/`CLAUDE.md`). Симптом нарушения — рост до сотен строк: значит, просочилось то, что нужно раз в месяц; место такого контента в cold-файле или `01_overview.md`.

Источник логики — [`memory/01_context_memory.md`](../../memory/01_context_memory.md).

## Доставка README агенту

**Проблема.** README — warm on-demand: агент подтянет его при заходе в папку, но не гарантированно в момент работы. У каждого агента свой нативный файл инструкций (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`), и нужна схема, которая работает со всеми и не плодит дублирование.

**Решение.** В каждой значимой папке размещаем **триплет**:

| Файл | Содержание | Зачем |
|---|---|---|
| `README.md` | Единственный источник: миссия папки, карта, правила | Source of truth |
| `AGENTS.md` | `@README.md` (pointer-only) | Стандарт индустрии: Codex, Gemini CLI, Cursor читают нативно |
| `CLAUDE.md` | `@README.md` (pointer-only) | Временный костыль: Claude Code пока не поддерживает AGENTS.md нативно |

**Принципы:**

- `AGENTS.md` и `CLAUDE.md` ссылаются **на README напрямую**, не друг на друга. При удалении CLAUDE.md ничего не ломается.
- `AGENTS.md` и `CLAUDE.md` — **строго pointer-only**. Никаких локальных правил «чуть-чуть тут» — иначе появится вторая правда. Все правила живут в README.
- Триплет ставится **только в значимых папках** (контуры, узлы, методологические папки, домены).
- **В обычных подпапках триплет не ставим.** Если локального README нет, агент опирается на ближайший README выше по дереву.

**План отката костыля.** Когда Claude Code добавит нативную поддержку AGENTS.md — удаляем все CLAUDE.md (`find . -name "CLAUDE.md" -delete`). Остаётся пара `README.md` + `AGENTS.md`.

## Когда README обязателен

- ✅ В корне scaffold (`scaffold/README.md`) — корневая карта.
- ✅ В каждом каноническом контуре (`01_ceo/`, `02_strategy/`, `03_team/`, `04_company/`, `05_metrics/`).
- ✅ В каждом узле, развёрнутом до полной формы (`{node}/README.md` рядом с `01_overview.md`).
- ⚠️ В опциональных контурах (`projects/`, `clients/`, `processes/`, `product/`, `knowledge/`) — обязателен, как только контур разворачивается.

**Когда может отсутствовать:**

- В служебных папках без управленческой миссии (`docs/`, `meetings/`, `_inbox/`) — README **опционален**, появляется только когда содержимого много и нужна навигация.
- В маленьких папках (1-2 файла) README опционален — создаётся, только если помогает направлять внимание агента.

## YAML README

### Минимум

```yaml
---
title: "..."
description: "..."
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### Поля

- **`title`** — короткая вывеска README. Совпадает с H1 (см. [02_file-spec.md § Title и H1](02_file-spec.md)).
- **`description`** — **дешёвая карточка релевантности**, ≤140 символов. Машинная подсказка для `rg '^description:'`, по которой агент решает, заходить в эту папку или нет. **Не** «миссия папки», **не** пересказ содержимого.
- **`created`** — когда README создан.
- **`updated`** — дата последнего смыслового обновления.
- **`version`** для README не нужен.

### `description` ≠ аннотация

| Артефакт | Где | Длина | Функция |
|---|---|---|---|
| `description` (YAML) | YAML | ≤140 символов, одна строка | Машинная карточка релевантности для `rg '^description:'` |
| Аннотация (под H1) | Тело README, без заголовка | до 25-30 слов | Миссия папки + быстрая навигация по соседям через inline-ссылки |

Разные функции, разные читатели, разные форматы.

## Каркас README

```markdown
---
title: "..."
description: "..."
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# {Имя папки} — {one-line purpose}

{Аннотация: миссия папки + inline-ссылки на ключевых соседей. До 25-30 слов. Без заголовка секции.}

## Что лежит
| Файл / папка | Миссия (зачем существует) | Когда читать |
|---|---|---|

## Маршруты (опц.)
| Триггер задачи | Что читать |
|---|---|

## Связи
- {соседняя папка/файл} — что там и зачем переходить
```

### Состав каркаса

| Блок | Обязательность | Что внутри |
|---|---|---|
| Аннотация (под H1, без заголовка) | обязательная | Миссия папки + inline-ссылки на ключевых соседей. До 25-30 слов. Канон — [02_file-spec.md § Аннотация](02_file-spec.md) |
| `## Что лежит` | обязательный | Таблица содержимого. Колонка «Миссия» = одна строка зачем существует каждый файл |
| `## Маршруты` | опциональный | Таблица «триггер задачи → что читать» |
| `## Связи` | обязательный | Граф связей с соседними папками/доменами с описанием перехода |

**Колонка «Миссия» — единственное место**, где живёт миссия файлов внутри папки. Сами файлы свою миссию не дублируют (у них есть title/H1 + аннотация). Применяется к README любой папки: контура, узла, методологической, домена.

**Пример аннотации README папки:**

```markdown
# 04_company — управление структурой компании

Место, где CEO ведёт подразделения, направления, встречи, решения. Содержание — в таблице ниже; протокол развёртывания узлов — в [deployment.md](../methodology/scaffold/deployment.md).
```

### Связи в README — обязательны

В обычных файлах scaffold связи живут inline в тексте (см. [02_file-spec.md § Связи между файлами](02_file-spec.md)). В README блок `## Связи` обязателен — README по своей функции **задаёт граф связей** папки с другими папками и доменами.

## README vs `01_overview.md`

| Артефакт | Описывает | Главная роль | Длина |
|---|---|---|---|
| `README.md` | Папку как **контейнер** | Маршрутизация AI-агента | 30-150 строк |
| `01_overview.md` | Сущность папки **как объект** | Понять «что это, в каком состоянии, куда дальше» | До 300 строк |

**В README не пишем:** метрики, миссию сущности, состояние стройки, лидера, KPI → это в `01_overview.md` или `profile.md`.

**В `01_overview.md` не пишем:** голый листинг файлов («файл X лежит здесь, файл Y там»). Смысловые ссылки в канве текста допустимы — «принципы и бизнес-модель — в [profile.md](profile.md)».

`01_overview.md` **не является** картой связей сущности. Карта папки и связи живут в `README.md`.

## Ограничения

- Папочный README — 30-150 строк (warm-слой не должен раздуваться).
- Корневой README scaffold — до 200 строк (точка входа, оплачивается часто).
- Общие правила формата — [02_file-spec.md](02_file-spec.md).

Если README разрастается — частую механику выносим в отдельный файл (`MAINTENANCE.md`, `ROUTES.md`), README остаётся короткой картой со ссылкой.
