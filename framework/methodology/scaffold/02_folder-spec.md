# Scaffold folder spec

## Миссия файла

Спецификация папки scaffold: что такое папка/узел, как она называется, как формулируется миссия, что считается дочерним узлом, какие служебные подпапки допустимы, какой README нужен, и какие канонические папки существуют в scaffold.

## Черновик

Файл временно совмещает два уровня:

1. **Общая спецификация папки** — универсальные правила для любой папки/узла scaffold.
2. **Канонические папки scaffold** — состав и миссии конкретных корневых зон и типовых папок.

Если второй раздел разрастётся, вынести его в отдельный `02_folder-templates.md`.

## Общая спецификация папки

Заполнить:

- папка как контейнер и как узел;
- корень, родительский узел, дочерний узел, листовой узел;
- что считается дочерним узлом, а что служебной подпапкой;
- имя папки;
- миссия папки;
- README как карта папки;
- когда папку не создавать.

## Канонические папки scaffold

### 🟡 `01_ceo/`

```
01_ceo/
├── 01_my_profile.md      ← кто я, ценности, стиль работы
├── 02_active.md          ← личные задачи и открытые вопросы CEO
├── 03_thoughts.md        ← мысли впрок: идеи, рефлексии, тревоги, гипотезы
└── 04_decisions.md       ← личные твёрдые решения CEO
```

Порядок: стабильный контекст (01) → текущая работа (02) → мягкое (03) → твёрдое (04).

**`01_my_profile.md`** — личность CEO как входные данные для бизнес-решений: ценности, стиль мышления, компетенции, границы. Reference-файл. Обновляется раз в 6-12 месяцев.

**`02_active.md`** — личные задачи и открытые вопросы CEO. Содержит **только прямые личные задачи и открытые вопросы CEO** — то, что нигде больше не живёт. 

**`03_thoughts.md`** — мысли впрок: идеи, рефлексии, тревоги, гипотезы. Один сборник на всё — низкий порог записи. Внутри — по датам, опционально теги. Разбор — отдельным ритуалом

**`04_decisions.md`** — личные твёрдые решения CEO. Отличаются от мыслей по жёсткому признаку: **на мысль агент не опирается, на решение опирается как на факт**. Работает как **буфер** (зафиксировано лично → потом вынесено в `02_strategy/` или `04_management/`) или **конечная точка** (наружу не пойдёт принципиально).

### 🟡  `02_strategy/`

```
02_strategy/
├── 01_vision.md           ← миссия, идентичность, видение будущего
├── 02_goal.md             ← долгосрочная цель, стратегия, цели периода, метрики (план)
├── 03_plan.md             ← план периода → неделя → день
├── 04_weekly_progress.md  ← хроника по неделям
├── 05_timeline.md         ← сводная история проекта, план vs факт
└── 06_decisions.md        ← реестр стратегических решений компании
```

Порядок: идентичность (01) → куда (02) → что делаем сейчас (03) → хроника (04) → история (05) → почему так (06).
Папка еще дорабатывается. Открытый вопрос, связанный со стратегией, в том числе со стратегией конкурентной, с маркетинговой стратегией, с продуктовой стратегией. 

### 🟡 `03_team/`

```
03_team/
├── README.md               ← обзор: как устроена папка
├── org_structure.md        ← маппинг людей на подразделения/направления в 04_company/
└── {person}.md             ← профайл сотрудника
```

`org_structure.md` маппит каждого человека на путь в `04_company/{node}`. Профайлы сотрудников (`{person}.md`) — стиль, сильные стороны, развитие, коучинг-лог.
⁉️ **Открытый вопрос: орг-структура в команде или в компании**

### `04_company/`

```
04_company/
├── README.md               ← модель подразделения/направления, правила маршрутизации
├── meetings/               ← стратсессии CEO, кросс-встречи верхнего уровня
├── glossary.md             ← клиентский словарь, жаргон, ошибки транскрибации
│
├── {org_node}/             ← подразделение/направление (единица или группировка)
│   ├── 01_overview.md      ← миссия, лидер (опц.), KPI (опц.), логика группировки (опц.)
│   ├── 02_active.md        ← задачи, вопросы, риски (секции по исполнителям)
│   ├── 03_progress.md      ← хроника по встречам
│   ├── ...                 ← см. полную форму узла в 01_architecture.md
│   ├── meetings/           ← встречи по этому подразделению/направлению
│   └── {sub_node}/         ← дочернее подразделение/направление
│       └── ...
└── ...
```

На верхнем уровне `04_management/` лежат только: `README.md`, `meetings/`, `glossary.md`, дерево подразделений/направлений. Больше ничего.

⁉️ **Открытый вопрос: `glossary.md` в команде или в компании или еще где?**
#### Шаблон подразделения/направления `{org_node}/`

Одинаков и для родительского узла (где есть свои подразделения) и для листового (единичного) 
Полный шаблон лежит в [01_architecture](01_architecture.md)

**Всё внутри подразделения/направления опционально.** Лидер, KPI, логика группировки, миссия — могут быть пустыми. На одном уровне могут сосуществовать и родительские узлы и листовые. Клиент принимается с любой стадией структурности.

#### Правило общей родительской папки

Информация фиксируется в общей родительской папке охватывающей всех задействованных участников.
##### Встречи
* ***`04_company/meetings/` (верхний уровень)** содержит: стратсессии CEO — встречи всей верхушки компании, кросс-функциональные встречи, охватывающие несколько верхнеуровневых подразделений/направлений
* Протокол встречи ложится в `meetings/` общей родительской папки, охватывающей всех участников встречи.

| Состав встречи                                                         | Куда протокол                      |
| ---------------------------------------------------------------------- | ---------------------------------- |
| Внутри одного подразделения/направления                                | `04_management/{node}/meetings/`   |
| Между дочерними подразделениями/направлениями одной группировки        | `04_management/{parent}/meetings/` |
| Между верхнеуровневыми подразделениями/направлениями / стратсессии CEO | `04_management/meetings/`          |
##### Решения



- `05_metrics/`;
- `projects/`;
- `clients/`;
- `knowledge/`;
- `processes/`;
- `product/`;
- `_inbox/`;
- `archive/`.

Для каждой папки: миссия, базовый состав, порядок чтения, что сюда, что не сюда, особенности структуры.
