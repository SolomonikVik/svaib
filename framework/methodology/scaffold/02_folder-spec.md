# Scaffold folder spec — канон папки каркаса

Спецификация папки scaffold как абстрактной единицы: типы (контур / узел / служебная), имя, миссия, полная форма узла, правила жизни. Анатомия конкретных контуров — в [02_folder-templates.md](02_folder-templates.md), карта папки — в [02_readme-spec.md](02_readme-spec.md), миссии файлов узла — в [02_file-templates.md](02_file-templates.md).

## 🔵 Папка как сущность

Папка в scaffold — контейнер управленческого контекста. Она существует не «на всякий случай», а под понятную миссию, и описывает себя через `README.md` (см. § README как карта папки). Для LLM путь и имя — часть контекста: имя задаёт фрейминг, вложенность — уровень погружения, полный путь читается как описание места файла (`04_company/sales/meetings/` уже сообщает область, сущность и тип материала). Каждый уровень вложенности должен добавлять смысл; техническая вложенность без смысловой роли удлиняет путь и ухудшает навигацию.

## 🔵 Типы папок

По функциональной роли папка scaffold относится к одному из трёх типов: контур, узел или служебная папка.

### Контур

Корневая папка scaffold: `01_ceo/`, `02_strategy/`, `03_team/`, `04_company/`, `05_metrics/`, `projects/`, `product/`, `clients/`, `processes/`, `knowledge/`.

Состав и анатомия каждого контура — в [02_folder-templates.md](02_folder-templates.md).

### Узел

Управляемая сущность внутри контура: подразделение, направление, проект, продукт, именованный клиент, процесс или другая управляемая тема. Узел отвечает на вопрос «какую сущность мы здесь ведём». У него есть собственное состояние, задачи, решения и жизненный цикл.

**Инвариант: сущность ↔ файл, узел ↔ папка.** Управляемая сущность может жить одним файлом (стадия 1 развёртывания). Узлом она становится при развороте в папку с README и каноническими файлами (стадия 2). До этого момента «узел» — корректный термин только в смысле «будущий узел». Стадии — в [01_architecture.md § Стадии развёртывания](01_architecture.md).

Узлы образуют дерево внутри контура. Родительский узел содержит дочерние узлы; листовой узел дочерних узлов не содержит. Служебные подпапки (`_inbox/`, `docs/`, `meetings/`, `zz_archive/`) не считаются дочерними узлами.

```text
04_company/
├── commercial/        ← родительский узел
│   ├── sales/         ← листовой узел
│   └── marketing/     ← листовой узел
└── operations/        ← листовой узел
```

В большинстве случаев контур — контейнер для узлов внутри (`04_company/`, `projects/`, `clients/`). Некоторые контуры одновременно ведутся как управляемая сущность по полной форме узла, например `product/` с `01_overview.md` и `04_decisions.md`.

### Служебная папка

Технический контейнер для материалов или процедур: `_inbox/`, `docs/`, `meetings/`, `zz_archive/`, `source/`, `extractors/`. Не имеет самостоятельной управленческой миссии.

Может жить как на корне scaffold (`_inbox/`, `zz_archive/`), так и внутри контура или узла. Вложенный случай при необходимости называется «служебная подпапка» — это форма, не отдельный тип. Подробно — § Служебные папки.

## 🔵 Миссия папки

У каждой значимой папки (контур, узел) есть миссия — зачем это место существует в управленческой модели. Миссия отвечает не на «что сюда можно положить», а на «зачем это место есть». При решении «нужен ли ещё один файл или папка» — оцениваем миссию папки и как она соотносится с миссиями соседей.

**Где живёт:** в аннотации под H1 в `README.md` папки. Канон — [02_readme-spec.md § Структура README](02_readme-spec.md).

**Хорошая миссия:**
- описывает роль папки в системе, а не перечисляет возможные файлы;
- помогает решить, класть ли сюда новую информацию;
- отличает папку от соседних контуров;
- выдерживает разные масштабы клиента.

## 🔵 Имя папки

Имя папки должно быть коротким, стабильным и понятным агенту без дополнительного объяснения.

**Формула:**

```text
{optional-prefix}_{semantic-name}
```

- **`semantic-name`** — английский в нижнем регистре, `kebab-case`. Слова внутри семантической части — через дефис: `weekly-progress`, `org-structure`. Одно слово — без дефиса: `metrics`, `projects`, `clients`. Канон именования согласован с [02_file-spec.md § Имя файла](02_file-spec.md).
- **`optional-prefix`** — двузначное число + `_`, опционально. Используется для обязательных контуров (`01_ceo`, `02_strategy`, `03_team`, `04_company`, `05_metrics`) и для папок, унаследовавших имя у канонического файла (`04_decisions/`).
- **Опциональные контуры** — без числового префикса: `projects/`, `product/`, `clients/`, `processes/`, `knowledge/`.
- **Служебные папки с принципиально иным режимом** — с префиксом подчёркивания: `_inbox/`. Префикс сигнализирует «не часть основной модели». Архив прижимается вниз: `zz_archive/`.

**Связь имени папки с именем файла.** Когда папка появляется как разворот ранее существовавшего файла, её имя точно повторяет имя файла без расширения: `marketing.md` → `marketing/`, `04_decisions.md` → `04_decisions/`. Инвариант: ссылки на старый файл чинятся механически.

## 🔵 Универсальная анатомия узла

Универсальная анатомия узла — максимальная раскладка управляемой сущности: подразделения, направления, проекта, продукта, именованного клиента, процесса или другой темы.

```text
{node}/
├── _inbox/            ← входящее на разбор
├── docs/              ← документы и артефакты узла (опц.)
├── meetings/          ← протоколы встреч
│
├── README.md          ← карта папки
├── 01_overview.md     ← смысловой вход
├── 02_active.md       ← что в работе сейчас
├── 02_backlog.md      ← будущая работа
├── 03_progress.md     ← хроника
├── 04_decisions.md    ← журнал решений
└── profile.md         ← паспорт сущности (опц.)
```

Полная форма не означает, что каждый узел создаётся сразу в полном составе. Узел может существовать в любой степени развёрнутости — от одного файла до полной папки-узла с дочерними узлами. Стадии развёртывания — в [01_architecture.md](01_architecture.md). Миссии файлов узла — в [02_file-templates.md](02_file-templates.md).

**Применимость по контурам:**
- **Применяется** — там, где есть задачи, backlog, прогресс, решения и встречи: `04_company/`, `projects/`, `product/`, `clients/`.
- **Не применяется** — у контуров с фиксированной анатомией: `01_ceo/`, `02_strategy/`, `03_team/`, `05_metrics/`.
- **Открытый канон** — `processes/`, `knowledge/`.

Подробные раскладки конкретных контуров — в [02_folder-templates.md](02_folder-templates.md).

## 🔵 Служебные папки

Канон служебных папок и контуров их применения. Определение типа — в § Типы папок.

| Папка | Роль | Где появляется |
|---|---|---|
| `_inbox/` | Входящее на разбор | Где есть поток неразобранного материала |
| `docs/` | Документы и артефакты клиента | В любом контуре или узле |
| `meetings/` | Протоколы встреч | Где есть регулярные встречи |
| `zz_archive/` | Завершённый или холодный контекст | В любой папке, где нужен архив |
| `source/` | Исходные данные, не редактируются агентом | `05_metrics/` |
| `extractors/` | Скрипты или процедуры для работы с источниками | `05_metrics/` |

## 🔵 README как карта папки

Каждая значимая папка scaffold имеет `README.md`. README — **карта папки**; главная функция — маршрутизация AI-агента: что лежит, куда идти, что читать дальше. Канон — [02_readme-spec.md](02_readme-spec.md).

Для folder-spec важно: README делает папку читаемой как контейнер. Он фиксирует миссию папки, объясняет содержимое и задаёт переходы к связанным контекстам.

Папка задаёт территорию поиска, но агент не обязан обходить её как дерево. Нормальная навигация — через README, ссылки, поиск по именам и путям, релевантные переходы между контекстами.

## 🔵 Правила жизни папки

**Когда папка появляется и расширяется:**
- появилась регулярная операционная работа, требующая `02_active.md` / `03_progress.md` / `04_decisions.md`;
- исходный файл (например `04_decisions.md`, `02_backlog.md`, `marketing.md`) перерос одну страницу — становится папкой со своим README. Имя — по § Имя папки.

**Соседи независимы.** Узлы внутри одной родительской папки могут быть на разной глубине; выравнивать соседей не требуется.

**Структура растёт от живого.** `scaffold/` не должен опережать реальную работу. Пустая или почти пустая папка — антипаттерн: шумит в навигации, заставляет агента читать впустую.

**Папки не обязаны идеально классифицировать все знания.** Если тема пересекает несколько контуров, использовать README, ссылки, навигаторы, профильные файлы или доменные представления — а не ломать структуру ради единственной директории.
