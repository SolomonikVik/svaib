# Scaffold file spec — канон md-файла каркаса

Канон md-файла scaffold: имя, YAML, шапка (H1 + аннотация), тело (H2-секции, текст внутри), правила файла/блока (опц.), связи. **Канон README папки — отдельно: [02_readme-spec](02_readme-spec.md)

## 🔵 Принципы работы с файлом

1. **LLM + People centered.** Читаемо глазами, парсится моделью без потери смысла.
2. **Один файл — одна роль.** Не смешивать план и итоги, current и log.
3. **Filename = поисковый запрос.** Имя содержит слова, по которым будут искать
4. **Минимум обвязки.** Не дублировать то, что уже выводится из path + filename + README + H1.
5. **Каждая H2-секция — автономный чанк.** Понятна без чтения соседних (RAG режет по H2).
6. **Компактность.** Лишнее = шум = потраченные токены. 
7. **Лимит файла** — <300 строк; больше — разбивать на несколько с явными связями, близко к лимиту — `## Обзор` в начало.
8. **Минимум дублирования.** У каждого факта одно каноническое место

## 🔵 Миссия файла

У каждого канонического файла scaffold должна быть миссия: зачем этот файл существует, какую работу делает в узле или контуре. При поиске или маршрутизации новой информации (задачи, решения, мысли, факта) — в первую очередь сверяемся с миссией файла

**Хорошая миссия файла:**
- описывает работу файла в системе
- подсказывает в каком случае файл стоит читать
- помогает решить, ляжет ли сюда новый факт;
- отличает файл от соседей в той же папке (`02_active` vs `02_backlog` vs `03_progress`);

**Где живёт миссия файла:** 
* универсальная миссия канонического типа (что вообще делает `02_active.md`, `04_decisions.md` и т.д.) — в [02_file-templates.md](02_file-templates.md). 
* адаптация миссии под конкретный узел — в колонке «Миссия» таблицы `## Что лежит` в README папки (см. [02_readme-spec.md](02_readme-spec.md)). 
* внутри самого файла отдельная секция «Миссия файла» **не пишется**: миссия считывается через **путь файла + filename + title/H1 + обязательной аннотации под H1** (см. § Аннотация).

## 🔵 Имя файла

**Принципы:** 
* `filename` = поисковый запрос. AI-инструменты ищут по именам файлов. `filename` = передает миссию, сжатая суть и стабильная ссылка
* **filename = id файла.** Rename → точечная правка ссылок.

**Общая формула:**
- `{optional-prefix}_{semantic-name}_{optional-type}.md`.

**Semantic-name:**
* `kebab-case.md` — строчные, дефисы между словами, английский, без транслита.
* слова внутри смысловой части имени разделяются дефисами: `system-principles.md`, `weekly-progress.md`, `file-spec.md`.

**Optional-prefix:**
- **Числовой** двузначный + `_` — задаёт порядок чтения и сортировку: `01_overview.md`, `02_active.md`, `04_decisions.md`.
- **датированные файлы** (например meetings/, decisions/): ISO 8601, всегда первая `YYYY-MM-DD_{semantic-name}.md` 

**Optional-type:**
* **для обработки встреч:** 
	* `_summary` — обработанная выжимка; 
	* `_transcript` — сырая транскрибация. 

## 🔵 YAML

**YAML** — стандартизированные метаданные для поиска: description, status, type, tags. Агент не парсит YAML как базу данных — ищет текстовым поиском (`rg '^status: raw$'`). YAML работает потому что это конвенция: одинаковый формат, предсказуемое место, индустриальный стандарт.
### Поля

- **`title`** — зеркало H1 для YAML-scan и карточки файла. Подробнее — см. § Шапка файла.
- **`created`** (`YYYY-MM-DD`) — когда файл создан. Помогает понимать возраст.
- **`updated`** (`YYYY-MM-DD`) — дата последнего смыслового обновления.
- **`version`** — только для SOT, спек, templates, skills, где важна версия канона.
- **`status`** (`raw` / `draft` / `review` / `final`) — стадия готовности. `final` = редактирование агентом запрещено без спроса.

**Опциональные поля** `domain`, `tags`, `source` — добавляются при появлении живой потребности (поиск по доменам, knowledge-теги, и т.д.).

**Лимит — ≤8 полей.** Больше — бремя поддержки превышает пользу.
### Минимум для файла

```yaml
---
title: "..."
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```


## 🔵 Шапка файла

После YAML и до первой H2 тела идёт **шапка** — фиксированная последовательность элементов, передающая миссию и суть файла без чтения тела.

Шапка работает в двух режимах. **H1** виден человеку при открытии файла (Obsidian, IDE). **`title` в YAML** — машине при YAML-scan и сборке карточки файла. Делают одну работу, по умолчанию совпадают (`title` = зеркало H1, см. § YAML).

**Что обязательно:**
- H1 — всегда.
- Одно из двух: **аннотация** (дефолт) или **`## Обзор`** (для длинных reference-файлов).

### H1

**Формула:**

```
{домен/сущность} — {что внутри / зачем читать}
```

**Требования:**

1. **Информативность.** Отвечает на «что это и для чего», не метка. `Overview` — плохо. `Marketing — операционный контур` — нормально.
2. **Лимит:** ≤120 символов, одна строка.
3. **`title` (YAML) = H1 по умолчанию.** Расхождение допустимо только если `title` нужен под отдельную задачу (редкий случай).

**Можно упростить,** если path + filename самодостаточны: `04_company/marketing/02_active.md` → `Marketing — активные задачи` хватает.

### Аннотация

**Дефолт для всех файлов scaffold.** Абзац сразу под H1.

> **Без `##`-заголовка.** Это абзац под H1, а не H2-секция.

**Что внутри:**
- Описание сути файла — что внутри и зачем.
- Inline-ссылки на соседние/связанные файлы по контексту — задаёт быструю навигацию (особенно удобно в Obsidian).

**Лимит:** до 25-30 слов.

**Пример:**
> Спецификация папки scaffold как абстрактной единицы: типы, имя, миссия, полная форма узла, правила жизни. Анатомия конкретных контуров — в [02_folder-templates.md](02_folder-templates.md), карта папки — в [02_readme-spec.md](02_readme-spec.md), миссии файлов узла — в [02_file-templates.md](02_file-templates.md).

### Обзор (опц.)

Расширенная сводка для длинных reference-файлов, где аннотации мало. **Заменяет аннотацию** — одно из двух.

**Когда нужно:**
- Файл большой (>300 строк), через 25-30 слов суть не передать.
- Полезна и человеку, и LLM (исследование, длинная методология, аналитика).

**Когда не нужно:** operational-файлы (`02_active`, `02_backlog`), короткие reference, всё, что покрывается аннотацией.

**Формат:** секция `## Обзор`, 2-4 предложения, ≤100 слов.

## 🔵 Тело файла: H2-секции

H2 задают структуру файла — человеку даёт навигацию и понимание, LLM — карту файла без чтения тела (поиск `^##` ) и семантический сигнал.
H2 в т.ч. единица чанкинга для RAG (250-500 слов)

- **Самоописывающие.** «Бюджет проекта Alpha Q1 2026», не «Бюджет».
- **Контекст в заголовке** — субъект, период, проект, если не очевидно.
- **Уникальные** в файле.
- **Автономные** — каждая секция понятна без чтения соседних.
  - Повторять субъект, не «Он решил…».
  - Контекст в начале секции.
  - Конкретные термины, не «сделали это».
- **H3 — внутренняя детализация**, не главный навигационный якорь.

## 🔵 Текст внутри секций

Как пишется сам текст в теле — параграф, список, таблица, выделения, маркеры. Цель: читается одинаково хорошо человеком и LLM, без лишних токенов.

### Когда параграф, когда список, когда таблица

- **Параграф** — для прозы, рассуждения, объяснения. Один абзац — одна мысль.
- **Маркированный список** — для дискретных элементов (правила, шаги, перечисление). Самая надёжная форма для LLM. Вложенность ≤2 уровня — глубже связи теряются.
- **Key-value список** (`**поле** — описание`) — для набора однотипных параметров с пояснением: поля YAML, атрибуты сущности, параметры. Лучший формат для извлечения LLM.
- **Простая таблица** — когда сравниваешь несколько сущностей по 2-4 одинаковым осям и все ячейки заполнены. Лимит — ≤20 строк; больше — заголовки «забываются».
- **Сложная таблица** (>20 строк, длинные пояснения, пустые ячейки) — нет. Перевести в key-value список или разбить.

### Выделения

- **Жирный** — повышает вес термина для LLM и человека. Для ключевых слов, имён сущностей, важных ярлыков.
- *Курсив* — лёгкий акцент, цитата, термин в первом упоминании.
- Если жирно всё — не выделено ничего. Норма — несколько ключевых слов на абзац.

### Эмодзи-маркеры

Для человека - реальное UX-преимущество, а LLM это не путает
- **Только частотные эмодзи** — кружки/квадраты/галочки/стрелки (🔵 ✅ → 📁), не экзотика
- **Последовательно** — один эмодзи = один тип секции по всему scaffold. Тогда это мини-онтология, а не украшением

**Канон в проработке** — см. [open-questions.md § Канонический набор эмодзи-маркеров](open-questions.md). Эмодзи в подзаголовках H2 — на усмотрение автора, жёсткого запрета нет.

#### Текущий рабочий набор:

| Маркер | Значение                             |
| ------ | ------------------------------------ |
| 🔵     | Выверено человеком, можно положиться |
| 🤖     | Правил агент, не выверено вручную    |
| ⁉️     | Открытый вопрос                      |
| ✅      | Закрыто                              |

#### Светофор G/Y/R

Частный случай эмодзи-маркера для статуса. Один символ + одна строка комментария.

| Символ | Значение |
|---|---|
| 🟢 | Идёт по плану |
| 🟡 | Риск / задержка / требует внимания |
| 🔴 | Блокер / срыв / эскалация |

**Где появляется (по контексту, опционально):** `01_overview`, `02_active`, `02_backlog`, метрики, KR, проекты.

Аббревиатура `G/Y/R` понятна нейросетям; для человека всегда подписывается «(светофор)».

## 🔵 Правило работы с файлом (секцией) (опц.)

Локальная инструкция агенту, которую нельзя вывести из имени, README или canonical templates.

**Когда нужно:**
- Локальная специфика этого файла.
- Локальное исключение из общего канона.
- Предупреждение о типичной ошибке.

**Когда не нужно:** в каноническом файле узла, если режим понятен из имени.

**Синтаксис:**

```markdown
> **Правило файла:** {одно конкретное проверяемое правило, действующее на весь файл}

H2-секция

> **Правило блока:** {одно правило, действующее только на этот блок}
```

**Требования к правилу:**

- Конкретное и проверяемое: «новые записи в начало; старые не редактировать», не «будь аккуратен».
- Одно-два правила. Если больше — выносить в README, canonical templates, procedure или skill.

**Enforcement:** хук Rule Injection (тип B) перед Edit подтягивает правило файла + ближайшее правило блока. Спека — [`memory/01_context_memory.md` § 5.5](../../memory/01_context_memory.md). Жёсткий запрет нарушения — отдельный validator, не Rule Injection.


## 🔵 Связи между файлами

**По умолчанию — inline в тексте** через markdown-ссылки с контекстом перехода: `подробнее — [folder-spec.md](02_folder-spec.md)`.

**Голая ссылка без объяснения, зачем переходить, бесполезна** — агент не знает приоритета.

**Навигационный граф папки живёт в README, не в файле.**

**Стабильность ссылки.** Если возможно ссылку давать — на папку или README (стабильны при rename файла внутри).
