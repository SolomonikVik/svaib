# Scaffold file spec — канон md-файла scaffold

Канон md-файла scaffold: имя, YAML, шапка (H1 + аннотация), тело (H2-секции, текст внутри), правила файла/блока (опц.), связи. Канон README папки — в [02_readme-spec.md](02_readme-spec.md).

## 🔵 Принципы работы с файлом

1. **LLM + People centered.** Читаемо глазами, парсится моделью без потери смысла.
2. **Один файл — одна роль.** Не смешивать план и итоги, текущую работу и хронику.
3. **Компактность.** Лишнее = шум = потраченные токены.
4. **Лимит файла** — <300 строк; больше — разбивать на несколько с явными связями, близко к лимиту — `## Обзор` в начало.
5. **Минимум дублирования.** У каждого факта одно каноническое место.

## 🔵 Миссия файла

У каждого канонического файла scaffold есть миссия: зачем этот файл существует, какую работу делает в контуре или узле. При поиске, чтении или маршрутизации новой информации (задачи, решения, мысли, факта) — в первую очередь сверяемся с миссией файла.

**Хорошая миссия файла:**
- описывает работу файла в системе;
- подсказывает, в каком случае файл стоит читать;
- помогает решить, ляжет ли сюда новый факт;
- отличает файл от соседей в той же папке (`02_active` vs `02_backlog` vs `03_progress`).

**Где живёт миссия файла:**
- универсальная миссия канонического типа (что вообще делает `02_active.md`, `04_decisions.md` и т.д.) — в [03_node-files.md](03_node-files.md);
- адаптация миссии под конкретный узел — в колонке «Миссия» таблицы содержимого в `README.md` папки. Канон — [02_readme-spec.md](02_readme-spec.md);
- внутри самого файла отдельная секция `Миссия файла` не пишется: миссия считывается через путь, имя файла, `title` / H1 и аннотацию под H1.

## 🔵 Имя файла

**Принципы:**
- `filename` = поисковый запрос. Агенты ищут по именам файлов; имя передаёт миссию и сжатую суть.
- `filename` = id файла. Rename → точечная правка ссылок.

**Общая формула:**

```text
{optional-prefix}_{semantic-name}_{optional-type}.md
```

**Semantic-name:**
- `kebab-case` — строчные английские слова через дефисы, без транслита: `system-principles.md`, `weekly-progress.md`, `file-spec.md`.

**Optional-prefix:**
- **Числовой** двузначный + `_` — задаёт порядок чтения и сортировку: `01_overview.md`, `02_active.md`, `04_decisions.md`.
- **Датированный.** ISO 8601, всегда первый: `YYYY-MM-DD_{semantic-name}.md`. Используется в `meetings/`, `04_decisions/`.

**Optional-type:**
- Для обработки встреч:
  - `_summary` — обработанная выжимка;
  - `_transcript` — сырая транскрибация.


## 🔵 YAML

**YAML** — стандартизированные метаданные для поиска. Агент не парсит YAML как базу данных — ищет текстовым поиском (`rg '^status: raw$'`). YAML работает потому что это конвенция: одинаковый формат, предсказуемое место, индустриальный стандарт.

### Поля

- **`title`** — зеркало H1 для YAML-scan и карточки файла. Подробнее — § Шапка файла.
- **`created`** (`YYYY-MM-DD`) — дата создания.
- **`updated`** (`YYYY-MM-DD`) — дата последнего смыслового обновления.
- **`version`** — только для SOT, spec, templates, skills, где важна версия канона.
- **`status`** (`raw` / `draft` / `review` / `final`) — стадия готовности. `final` = редактирование агентом запрещено без согласования.

**Опциональные поля** `domain`, `tags`, `source` — добавляются при появлении живой потребности (поиск по доменам, knowledge-теги).

**Лимит — ≤8 полей.** Больше — бремя поддержки превышает пользу.

### Минимум для файла

```yaml
---
title: "..."
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```
## 🔵 Шапка файла

После YAML и до первой H2 тела идёт **шапка** — фиксированная последовательность элементов, передающая миссию и суть файла без чтения тела.

H1 виден человеку при открытии файла (Obsidian, IDE), `title` в YAML — машине при YAML-scan. По умолчанию `title` = зеркало H1 (см. § YAML).

**Что обязательно:**
- H1 — всегда.
- Одно из двух: **аннотация** (дефолт) или **`## Обзор`** (для длинных справочных файлов).

### H1

**Формула:**

```text
{домен/сущность} — {что внутри / зачем читать}
```

**Требования:**

1. **Информативность.** Отвечает на «что это и для чего», не метка. `Overview` — плохо. `Marketing — операционный контур` — нормально.
2. **Лимит:** ≤120 символов, одна строка.
3. **`title` (YAML) = H1 по умолчанию.** Расхождение допустимо только если `title` нужен под отдельную задачу (редкий случай).

**Можно упростить,** если path + filename самодостаточны: `04_company/marketing/02_active.md` → `Marketing — активные задачи` хватает.

### Аннотация

**Дефолт для всех файлов scaffold.** Абзац сразу под H1, без `##`-заголовка.

**Что внутри:**
- Описание сути файла — что внутри и зачем.
- Inline-ссылки на соседние/связанные файлы — задают быструю навигацию.

**Лимит:** до 25-30 слов.

**Пример:**
```markdown
Спецификация папки scaffold: типы, имя, миссия, универсальная форма узла, служебные подпапки и правила жизни. Конкретные контуры — в [03_contours.md](03_contours.md).
```

### Обзор (опц.)

Расширенная сводка для длинных справочных файлов, где аннотации мало. **Заменяет аннотацию** — одно из двух.

**Когда нужно:**
- Файл большой (>300 строк), через 25-30 слов суть не передать.
- Полезна и человеку, и LLM (исследование, длинная методология, аналитика).

**Когда не нужно:** операционные файлы (`02_active`, `02_backlog`), короткие справочные файлы, всё, что покрывается аннотацией.

**Формат:** секция `## Обзор`, 2-4 предложения, ≤100 слов.

## 🔵 Тело файла: H2-секции

H2 задают основную структуру файла: человеку дают навигацию, агенту — карту файла без чтения тела (`rg '^## '`) и семантические границы для выборочного чтения.

H2 также являются основной единицей чанкинга для RAG. Ориентир размера секции — 250-500 слов.

**Требования к H2:**
- **Самоописывающие.** `Бюджет проекта Alpha Q1 2026`, не `Бюджет`.
- **С контекстом.** Субъект, период или проект указываются в заголовке, если не очевидны из пути и H1.
- **Уникальные внутри файла.** Повторяющиеся H2 ломают навигацию.
- **Автономные.** Секция должна быть понятна без чтения соседних секций.

**Автономность секции означает:**
- повторять субъект, а не писать «он», «это», «там» без контекста;
- давать короткий контекст в начале секции, если он нужен;
- использовать конкретные термины, а не «сделали это».

**H3** — внутренняя детализация H2, не главный навигационный якорь.

## 🔵 Текст внутри секций

Текст внутри секции должен одинаково хорошо читаться человеком и агентом: без лишних токенов, скрытых смыслов и сложной вложенности.

### Когда параграф, когда список, когда таблица

- **Параграф** — для прозы, рассуждения, объяснения. Один абзац — одна мысль.
- **Маркированный список** — для дискретных элементов (правила, шаги, перечисление). Самая надёжная форма для LLM. Вложенность ≤2 уровня — глубже связи теряются.
- **Key-value список** (`**поле** — описание`) — для набора однотипных параметров с пояснением: поля YAML, атрибуты сущности, параметры. Лучший формат для извлечения LLM.
- **Простая таблица** — когда сравниваешь несколько сущностей по 2-4 одинаковым осям и все ячейки заполнены. Лимит — ≤20 строк; больше — заголовки «забываются».
- **Сложная таблица** (>20 строк, длинные пояснения, пустые ячейки) — нет. Перевести в key-value список или разбить.

### Выделения

- **Жирный** — повышает вес термина для LLM и человека. Для ключевых слов, имён сущностей, важных ярлыков.
- *Курсив* — лёгкий акцент, цитата, термин в первом упоминании.
- Если жирно всё — не выделено ничего. Норма — несколько ключевых слов на абзац.

### Эмодзи-маркеры

Для человека — реальное UX-преимущество, LLM это не путает.

- **Только частотные эмодзи** — кружки/квадраты/галочки/стрелки (🔵 ✅ → 📁), не экзотика.
- **Последовательно** — один эмодзи = один тип секции по всему scaffold. Тогда это мини-онтология, а не украшение.

**Канон в проработке** — см. [open-questions.md § Канонический набор эмодзи-маркеров](open-questions.md). Эмодзи в подзаголовках H2 — на усмотрение автора, жёсткого запрета нет.

#### Текущий рабочий набор

| Маркер | Значение                             |
| ------ | ------------------------------------ |
| 🔵     | Выверено человеком, можно положиться |
| 🤖     | Правил агент, не выверено вручную    |
| ⁉️     | Открытый вопрос                      |
| ✅      | Закрыто                              |

#### Светофор G/Y/R

Частный случай эмодзи-маркера для статуса. Один символ + одна строка комментария. Аббревиатура `G/Y/R` понятна LLM; для человека всегда подписывается «(светофор)».

| Символ | Значение |
|---|---|
| 🟢 | Идёт по плану |
| 🟡 | Риск / задержка / требует внимания |
| 🔴 | Блокер / срыв / эскалация |

**Где появляется (по контексту, опционально):** `01_overview`, `02_active`, `02_backlog`, метрики, KR, проекты.


## 🔵 Правило работы с файлом и секцией (опц.)

Локальная инструкция агенту, которую нельзя вывести из имени, README или canonical templates.

**Когда нужно:**
- локальная специфика этого файла;
- локальное исключение из общего канона;
- предупреждение о типичной ошибке.

**Когда не нужно:** в каноническом файле узла, если правило работы понятно из имени.

**Синтаксис:**

```markdown
> **Правило файла:** {одно конкретное проверяемое правило, действующее на весь файл}

H2-секция

> **Правило секции:** {одно правило, действующее только на эту секцию}
```

**Требования к правилу:**
- конкретное и проверяемое: «новые записи в начало; старые не редактировать», не «будь аккуратен»;
- одно-два правила. Если больше — выносить в README, canonical templates, procedure или skill.

**Enforcement:** хук Rule Injection (тип B) перед Edit подтягивает правило файла + ближайшее правило секции. Спецификация — [memory/01_context_memory.md § 5.5](../../memory/01_context_memory.md). Жёсткий запрет нарушения — отдельный validator, не Rule Injection.

## 🔵 Связи между файлами

В обычных файлах scaffold связи живут inline в тексте через markdown-ссылки с контекстом перехода: `подробнее — [02_folder-spec.md](02_folder-spec.md)`.

Голая ссылка без объяснения, зачем переходить, бесполезна: агент не знает приоритета и причины перехода.

Навигационный граф папки живёт в README, не в обычном файле. Канон README — в [02_readme-spec.md](02_readme-spec.md).

Если возможно, ссылку лучше давать на папку или README: они стабильнее при переименовании файлов внутри папки.
