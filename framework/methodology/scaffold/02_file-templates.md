# Scaffold file templates

## Миссия файла

Канон ключевых файлов scaffold: `01_overview.md`, `profile.md`, `02_active.md`, `02_backlog.md`, `03_progress.md`, `04_decisions.md`, `glossary.md`, `meetings/`.

Для каждого файла: миссия, что сюда, что не сюда, базовая структура и правила применения.

5 блоков `01_overview.md`

Один паттерн для подразделения, проекта, продукта, клиента, направления. Семантика наполнения подстраивается под природу.

1. **Суть** — чем является объект, какую роль играет, в чём специфика.
2. **Цели и КР** — куда идём, как поймём что дошли (метрики).
3. **Команда** — owner / РП + ключевые участники.
4. **Фокус** — узкое место сейчас + куда прикладываем силы.
5. **Дорожная карта (Roadmap)** — ключевые вехи к цели.

Сигнал состояния (🟢🟡🔴) — опционально в любом файле.

#### `glossary.md`

Единый словарь клиентского пространства. Содержит:

- **Клиентские синонимы подразделений/направлений** — как клиент называет свои единицы и группировки («у нас это служба», «у нас дирекция»). Агент маппит на онтологию единица/группировка.
- **Бизнес-жаргон клиента** — специфическая терминология отрасли и компании.
- **Ошибки транскрибации** — типичные ошибки транскрибатора на именах сотрудников, клиентов, терминах. Правильные написания.

Обязательно читается L2-пайплайном meeting-analysis перед обработкой встреч.


### Формат секций в подразделениях/направлениях

`04_company/{node}/02_active.md` использует секции по исполнителям:

```markdown
## Задачи

### {CEO}

- [ ] {Задача (срок | из встречи YYYY-MM-DD)}

### {Лидер}

- [ ] {Задача (срок | из встречи YYYY-MM-DD)}
- [x] {Выполненная — перенести в повестку встречи}

### Не назначен

- [ ] {Задача (YYYY-MM-DD)}
```

Секция = исполнитель. Явный маркер «кто делает», без тегов вроде `@ceo`.


### 1. `01_overview.md`

**Миссия файла:** дать человеку и AI-агенту смысловой вход в сущность объекта папки (подразделение / продукт / проект и т.д.) — ответить на «что это, в каком состоянии, куда дальше». Сводный стратегический документ объекта папки. Стабильный: не задачи, не хроника, не лог решений, не паспорт-доктрина, не навигация.

**Универсальность.** Структура едина для всех типов объектов (подразделение / направление / продукт / проект). Семантика наполнения может меняться (например, у подразделения нет roadmap, но может быть список ключевых проектов). Если блок неестественен для типа сущности — лучше сделать его опциональным, чем «переосмыслять».

**5 блоков:**

1. **Суть** — краткое объяснение того, чем является объект в контексте компании: какую роль он играет, через какую логику работает и в чём его специфика. Главный контекст, который надо передать. Что это вообще?
2. **Цели и ключевые результаты** — вектор движения объекта: конечный результат и критерий успеха (метрики). Отвечает на «куда идём и как поймём, что дошли». Может содержать несколько горизонтов (год / квартал) — сколько необходимо для управления направлением.
3. **Команда** — кто отвечает за объект и кто с ним работает. Owner/РП + ключевые участники.
4. **Фокус** — узкое место объекта на данный момент и точка приложения сил, которая его расшивает.

   - **Узкое место (проблема) на данный момент** — что ограничивает движение к цели сильнее всего.
   - **На чём сейчас сосредоточены** — куда прикладываем силы, чтобы это расшить.

   Не операционная сводка, не статус-репорт, не список текущих задач. Меняется только при расшитии или переоценке главного напряжения; обычно держится месяцами.
1. **Дорожная карта (Roadmap)** — ключевые вехи, инициативы или проекты, через которые объект движется к цели. Соединяет текущий фокус с целью через крупные шаги — без roadmap движение выглядит хаотичным. Не операционка, подробный список задач → в бэклоге.

**Логика родительского узла.** Если узел является родительским (внутри есть дочерние узлы), в блок «Суть» вкладывается логика, по которой объединены дочерние. Типовые:

- **Функциональная** — коммерция = продажи + маркетинг.
- **По OKR** — снижение churn = product + support + implementation + account management.
- **Операционная** — бэкофис = финансы + HR + legal.
- **Продуктовая** — линейка A = разработка A + маркетинг A + поддержка A.
- **Географическая** — хаб Алматы = продажи + операции + склад.

### 2. profile — паспорт сущности

**`profile.md` — опциональный детальный паспорт-файл рядом с overview.** Разворачивается у любой единицы, когда стабильное идеологическое ядро сущности не помещается в overview без потери его компактности.

Ключевое:

- Overview остаётся **первой точкой входа** во всех случаях. Profile читается **по ссылке** из overview, не до него.
- Triggered, не обязателен. Разворачивается, когда в overview хочется писать про принципы / границы / бизнес-модель / ICP / доктрину, и оно начинает мешать «60-секундному входу».

Связь: overview ссылается на profile в смысловых ссылках своего текста. Не наоборот.

**Имя файла — `profile.md`.** Универсальное имя без числового префикса для всех типов единиц.

Обоснование:

- Естественный английский, понятен и человеку, и LLM-агенту без объяснения
- Соответствует русскому «паспорт сущности»
- **Усиливает существующий паттерн** profile в scaffold: `01_ceo/01_my_profile.md` (личность CEO), `03_team/{person}.md` (профайлы людей). Расширяется на сущности — продукт, проект, подразделение
- Без префикса `00_`: префикс ложно сигналил бы, что profile первичен, а у нас первая точка входа — overview
