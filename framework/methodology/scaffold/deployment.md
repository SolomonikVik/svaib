# Scaffold deployment

## Миссия файла

Правила первичного развёртывания scaffold у клиента: что создаётся при старте, что появляется по триггеру живой работы, как запустить новый узел.

Здесь — только установка и расширение под появление новой работы. Миграции существующих данных, перенос/слияние/удаление узлов, превращение файла в папку — за рамками этого файла. См. [open-questions.md](open-questions.md), потенциально отдельный `lifecycle.md`.

## Кратко

При старте клиента создаются **5 обязательных контуров** (`01_ceo`, `02_strategy`, `03_team`, `04_company`, `05_metrics`) с минимальной раскладкой каждого; пустые файлы и пустые секции допустимы. Опциональные контуры (`clients`, `knowledge`, `processes`, `product`, `projects`) разворачиваются **только по живому триггеру**, не «на всякий случай». Дерево подразделений в `04_company/` появляется по триггеру реальной работы клиента с подразделением. Iron Law: разворачивай только живое.

## 1. Iron Law

> **Разворачивай только живое. Пустые ветки не создавай.**
>
> «Живое» = появилась реальная работа (встреча, задача, человек, метрика), а не «заполнены поля».

Текущий этап — развёртывание ручное по этим правилам. Автоматизация в виде скилла — в `framework/skills/scaffold/_draft_spec.md` (откладывается до 2-3 успешных ручных развёртываний).

## 2. Что создаётся при старте клиента

Все 5 обязательных контуров создаются сразу. Пустые файлы и пустые секции допустимы — каркас ставится с первого дня.

### 2.1. `01_ceo/`

```
01_ceo/
├── 01_my_profile.md
├── 02_active.md
├── 03_thoughts.md
└── 04_decisions.md
```

Все 4 файла создаются сразу. Низкий порог записи, нужны с первого дня.

### 2.2. `02_strategy/`

```
02_strategy/
├── 01_vision.md
└── 02_goal.md
```

На старте создаются только `01_vision.md` и `02_goal.md` (кто мы + куда идём). Остальные файлы (`03_plan`, `04_weekly_progress`, `05_timeline`, `06_decisions`) разворачиваются по триггеру живой работы. Полная анатомия контура — в [03_contours.md](03_contours.md).

### 2.3. `03_team/`

```
03_team/
├── README.md
├── org_structure.md
└── {person}.md             ← по одному на каждого C-1
```

При онбординге клиент перечисляет C-1 (прямых подчинённых). Профайлы C-1 нужны с первого дня для обработки встреч и коучинга. Профайлы остальной команды — по триггеру (см. § 4).

### 2.4. `04_company/`

```
04_company/
├── README.md
├── meetings/               ← пустая, для стратсессий и кросс-встреч верхнего уровня
└── glossary.md
```

Дерево подразделений/направлений **не разворачивается сразу** — только по триггеру (см. § 3).

### 2.5. `05_metrics/`

```
05_metrics/
├── README.md
├── 01_metrics.md           ← витрина target metrics (пустая на старте)
├── template-domain.md      ← шаблон domain-файла
└── source/
    └── README.md           ← объяснение, какие xlsx/Sheets сюда класть
```

Базовый каркас контура создаётся сразу — `05_metrics/` обязательный контур по архитектуре. Domain-файлы (`finance.md`, `sales.md`, `operations.md`, `people.md`) — **по триггеру живой работы в домене**, не все сразу. `extractors/` — по триггеру (когда клиент даёт первый источник). Числа не дублируются в md — живут в `source/*.xlsx`. Расчёты — Python через xlsx-skill.

### 2.6. Опциональные контуры

**НЕ создаются при старте.** Разворачиваются по триггеру (см. § 5).

## 3. Триггер разворачивания подразделения/направления `04_company/{node}/`

Подразделение/направление разворачивается, когда клиент **явно работает с ним как существующим**: «у меня есть отдел маркетинга, мы встречаемся, ставим цели, я контролирую».

**Подразделение/направление НЕ разворачивается**, если упомянуто в разговоре без артефактов работы — держим в `02_active.md` как контекст.

При развёртывании подразделения/направления создаются:

```
04_company/{node}/
├── README.md
├── 01_overview.md
├── 02_active.md
├── 02_backlog.md           ← опц.
├── 03_progress.md
├── 04_decisions.md
├── meetings/
└── _inbox/                 ← опц., если есть поток сырья
```

Пустые файлы и пустые секции допустимы.

## 4. Триггер разворачивания `03_team/{person}.md`

Новый файл сотрудника создаётся при:

- Явном перечислении C-1 при онбординге.
- Первой встрече с участием этого человека.
- Первом упоминании с явной ролью.

## 5. Триггеры опциональных папок

Разворачиваются, когда у клиента появляется реальная работа в домене.

| Контур | Триггер |
|---|---|
| **`clients/`** | CEO лично работает с конкретными именованными клиентами, это важная часть управления (аккаунт-менеджмент, B2B-продажи, консалтинг, агентский бизнес). Раскладка узла — универсальная, см. [03_contours.md](03_contours.md). |
| **`knowledge/`** | У CEO есть домен знаний, который он хочет структурировать — живая работа с материалом, а не свалка ссылок. Минимальная раскладка: `README.md` + один `{domain}/`. |
| **`processes/`** | Open question. Не разворачивается до практического запроса клиента и отдельного решения по канону контура. Текущая граница: процесс — постоянный, проект — временный. |
| **`product/`** | У компании есть продукт (то что поставляется клиентам — SaaS, услуги, консалтинг, B2B-сервис, физический товар). В подавляющем большинстве случаев разворачивается. **Раскладка — универсальная форма узла:** `README.md`, `01_overview.md`, `02_active.md`, `02_backlog.md`, `03_progress.md`, `04_decisions.md`, опц. `profile.md`, опц. `architecture.md`. `architecture.md` разворачивается для SaaS, tech-продукта, физического производства; для услуг/консалтинга не разворачивается. Подпапки (`docs/`, `specs/`, `pricing/`) — только по живому триггеру. **Отличие от `projects/`:** продукт — постоянный контур, проект — временная активность с завершением. |
| **`projects/`** | У CEO есть именованные проекты с горизонтом и завершением (миграция CRM, запуск продукта, стройка склада). Раскладка узла — универсальная: `README.md`, `01_overview.md`, `02_active.md`, `02_backlog.md`, `03_progress.md`, `04_decisions.md`, `meetings/`, `_inbox/`. |

Когда опциональный контур с зафиксированным каноном разворачивается — его базовый каркас создаётся сразу внутри (по тому же правилу, что для обязательных): пустые файлы допустимы, плейсхолдеры не оставляем.

## 6. Природа темы выбирает контур

Одна и та же тема может иметь разную природу у разных клиентов. Например, «маркетинг» может быть:

- подразделением в `04_company/`;
- процессом в `processes/`;
- проектом в `projects/`;
- частью продукта в `product/`.

Scaffold не создаёт универсальную папку для слова. Сначала определяется **природа работы CEO с темой**, потом выбирается контур и глубина развёртывания.

## 7. Чего не создаёт ни один шаг развёртывания

- Пустые ветки «на всякий случай».
- Файлы под плейсхолдеры, если подразделения/направления ещё нет.
- Копии файлов для «будущих подразделений».
- Дерево всех опциональных контуров при онбординге, если триггеров ещё нет.
