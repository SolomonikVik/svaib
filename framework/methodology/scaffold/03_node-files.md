# Scaffold node files — миссии универсальных файлов узла

Канон миссий и структур **универсальных** канонических md-файлов узла scaffold (встречаются в нескольких контурах): `01_overview`, `02_active`, `02_backlog`, `03_progress`, `04_decisions`, `profile`, `person`. Контурно-специфичные файлы (`01_my-profile`, `01_vision`, `org-structure`, `glossary` и т.д.) и служебные подпапки (`meetings/`, `_inbox/`, `docs/`) — в [03_contours.md](03_contours.md). Универсальная грамматика md-файла — в [02_file-spec.md](02_file-spec.md), карта папки — в [02_readme-spec.md](02_readme-spec.md).

> **Этот файл не хранит копируемые болванки.** Болванки — в [../../scaffold/](../../scaffold/). Здесь фиксируется методологический канон: миссия, канонические разделы, пример аннотации каждого универсального типа файла.

| [02_readme-spec.md](02_readme-spec.md) | Канон `README.md` как карты папки |

## Связь с README

`03_node-files.md` задаёт **универсальную миссию** каждого типа файла узла. `README.md` конкретной папки **адаптирует** эту миссию под свой контур/узел в колонке «Миссия» таблицы `## Содержимое папки`. Миссии контурно-специфичных файлов (которые живут только в одном контуре) — в [03_contours.md](03_contours.md).

Пример: канон говорит «`02_active.md` — что в работе сейчас, операционный хаб». README маркетингового подразделения уточняет: «активные задачи маркетинга и повестка ближайшей встречи с лидером маркетинга».

## Соответствие file-spec

Все канонические файлы соответствуют [02_file-spec.md](02_file-spec.md):

- YAML по минимуму контентного файла (`title`, `created`, `updated`).
- **Обязательная аннотация** под H1 (без заголовка секции, до 25-30 слов, с inline-ссылками на ключевых соседей).
- Опц. `## Содержание` — только если файл >300 строк.
- Тело — H2-секции, самоописывающие, автономные.
- Связи между файлами — inline в тексте, без отдельной секции сверху.

В описаниях ниже — **специфика конкретного типа файла** (миссия, канонические разделы, пример аннотации).

**Опциональность разделов.** Любой H2-раздел описанных ниже шаблонов опционален. Если у типа сущности или конкретного экземпляра раздел естественно пустой — не держать пустой заголовок, скрыть его. Лучше отсутствие, чем placeholder.

## `01_overview.md` — смысловой вход в сущность

**Миссия:** дать человеку и AI-агенту смысловой вход в сущность папки (подразделение / продукт / проект / клиент / процесс) — ответить на «что это, в каком состоянии, куда дальше». Сводный стратегический документ объекта папки. Стабильный: не задачи, не хроника, не лог решений, не паспорт-доктрина, не навигация.

**Универсальность.** Структура едина для всех типов узлов. Семантика наполнения может меняться (у подразделения нет roadmap, но может быть список ключевых проектов). Если раздел неестественен для типа сущности — лучше сделать его опциональным, чем переосмыслять.

### 5 разделов

1. **Суть** — чем является объект, какую роль играет в контексте компании, через какую логику работает, в чём специфика.
2. **Цели и ключевые результаты** — куда идём и как поймём, что дошли (метрики). Может содержать несколько горизонтов (год / квартал) — сколько необходимо для управления.
3. **Команда** — owner / РП + ключевые участники.
4. **Фокус** — узкое место сейчас + точка приложения сил.
   - **Узкое место** — что ограничивает движение к цели сильнее всего.
   - **На чём сосредоточены** — куда прикладываем силы, чтобы расшить.

   Не операционная сводка, не статус-репорт, не список задач. Меняется только при расшитии или переоценке главного напряжения; обычно держится месяцами.
5. **Дорожная карта (Roadmap)** — ключевые вехи, инициативы или проекты, через которые объект движется к цели. Не операционка; подробный список задач — в `02_backlog.md`.

Сигнал состояния (🟢🟡🔴) — опционально в любом из разделов.

### Логика родительского узла

Если узел является родительским (внутри есть дочерние узлы), в раздел «Суть» вкладывается логика, по которой объединены дочерние:

- **Функциональная** — коммерция = продажи + маркетинг.
- **По OKR** — снижение churn = product + support + implementation + account management.
- **Операционная** — бэкофис = финансы + HR + legal.
- **Продуктовая** — линейка A = разработка A + маркетинг A + поддержка A.
- **Географическая** — хаб Алматы = продажи + операции + склад.

### Пример аннотации

> Маркетинговое направление — состояние, фокус, roadmap. Операционка в [02_active.md](02_active.md), хроника — в [03_progress.md](03_progress.md), решения — в [04_decisions.md](04_decisions.md).

## `profile.md` — паспорт сущности (опц.)

**Миссия:** стабильное идеологическое ядро сущности — продукта, проекта, клиента-как-компании, подразделения. Содержательное наполнение: проблема, ICP, решение, принципы, границы, бизнес-модель, доктрина — то, что не помещается в `01_overview.md` без потери компактности.

**Принципы:**

- **Overview — первая точка входа** во всех случаях. Profile читается **по ссылке** из overview, не до него.
- **Без обязательных разделов.** Triggered-файл для стабильного контекста, который перегружает `01_overview.md`. Набор H2-разделов формируется под тип сущности и реальное содержание (у продукта — ICP/решение/бизнес-модель, у проекта — контекст/границы, у клиента-как-компании — история и стейкхолдеры). Автор создаёт только живые H2-разделы; пустые, симметричные или «на всякий случай» — не делаем.
- **Имя файла — `profile.md`.** Универсальное имя для паспорта сущности.

### Пример аннотации

> Паспорт продукта Second AI Brain: позиционирование, ICP, границы, бизнес-модель. Состояние стройки — в [01_overview.md](01_overview.md), решения — в [04_decisions.md](04_decisions.md).

## `person.md` — профайл человека (опц.)

**Миссия:** профиль человека — идентичность, ценности/принципы, стиль работы, сильные стороны и зоны развития, цели и приоритеты. Блоки разворачиваются только при наличии живого знания; пустые психологические анкеты не создаются. История взаимодействий и коучинг-лог не живут в профиле, если начинают превращаться в хронику.

**Имя и контекст применения** — контурно-специфичны (см. [03_contours.md](03_contours.md)): `01_ceo/01_my-profile.md`, `03_team/{person}.md`, `clients/{client}/{person}.md`. В этом файле — единая рамка, специфика — в секциях контуров.

**Канонические разделы** (все опциональны — разворачиваются под живое знание).

Общая рамка:

1. **Идентичность** — кто, роль, контекст.
2. **Зона ответственности** — за что отвечает, какие проекты ведёт, какие цели/метрики стоят.

Человеческая часть:

3. **Цели и приоритеты** — куда движется (для CEO и контрагента; опц. для сотрудника).
4. **Ценности, принципы, мотивация** — что считает важным, красные линии.
5. **Стиль работы и мышления** — как коммуницирует, как принимает решения.
6. **Сильные стороны / зоны развития** — для коучинга и работы.

### Пример аннотации

> Профайл руководителя продаж: что ведёт, как принимает решения, сильные стороны. Контекст подразделения — в [../04_company/sales/01_overview.md](../04_company/sales/01_overview.md).

## `02_active.md` — операционный хаб

**Миссия:** что в работе **сейчас** — задачи и открытые вопросы. State-файл: перезаписывается, не растёт.

**Канонические разделы** (все опциональны):

1. **Повестка ближайшей встречи** — дата + пункты для обсуждения. Для узлов с регулярной встречей (направление, продукт, проект). У CEO в `01_ceo/` — не используется.
2. **Задачи** — H3 по исполнителям (см. ниже).
3. **Открытые вопросы** — что зреет, требует решения. Сюда же мелкие проблемы и риски, пока их не настолько много, чтобы выделять отдельный раздел.

### Формат раздела «Задачи»

Универсальный паттерн: H3 = исполнитель. Явный маркер «кто делает», без тегов вроде `@ceo`.

```markdown
## Задачи

### {CEO}
- [ ] {Задача (срок | из встречи YYYY-MM-DD)}

### {Name}
- [ ] {Задача (срок | из встречи YYYY-MM-DD)}
- [x] {Выполненная — перенести в повестку встречи}

### Не назначен
- [ ] {Задача (YYYY-MM-DD)}
```

В `01_ceo/02_active.md` H3 не используются — исполнитель один (CEO).

> **Sidenote — svaib-специфика.** В svaib-направлениях (`framework/02_active.md`, `lab/02_active.md`, …) `02_active.md` дополнительно содержит раздел «Session Handoff» — контекст передачи между чатами AI-помощника. У клиента в `02_active.md` Session Handoff не разворачивается — у клиента нет паттерна работы через серию чатов с одним агентом.

### Пример аннотации

> Что в работе сейчас: активные задачи маркетинга, открытые вопросы, повестка встречи. Бэклог — в [02_backlog.md](02_backlog.md), хроника — в [03_progress.md](03_progress.md).

## `02_backlog.md` — будущая работа

**Миссия:** будущая работа узла — инициативы, идеи, отложенные задачи. Что сделать когда-нибудь, но не сейчас.

**Канонические разделы** (оба опциональны):

1. **План** — разобранные задачи и инициативы, упорядоченные. Группировка на выбор: по фокусам roadmap, релизам, вехам, темам или плоский список. Опц. G/Y/R-светофор у горящих позиций.
2. **Идеи и гипотезы** — сырая зона: идеи и гипотезы, требующие проработки прежде чем брать в работу. Разбирается периодически.

Ротация — реже, чем у `02_active.md`.

### Пример аннотации

> Будущая работа маркетинга: отложенные инициативы, идеи на проработку, задачи под условия. Текущая работа — в [02_active.md](02_active.md).

## `03_progress.md` — хроника

**Миссия:** хроника движений узла по времени — встречи, запуски, пивоты, замены людей, внешние события, итоги периода. Сами протоколы встреч — в `meetings/`.

**Без обязательных разделов.** Это последовательность датированных записей. Каждая запись — H2 с датой и заголовком события.

**Формат записи:** новые записи **в начало** файла (свежее сверху). Каждая запись — дата + короткая суть.

```markdown
## {YYYY-MM-DD}: {заголовок события}

{2-5 строк: что произошло, ключевые моменты, что меняется для узла. Для встреч — Участники.}
```

### Пример аннотации

> Хроника маркетинга по неделям: что закрыли, ключевые движения, итоги встреч. Текущая работа — в [02_active.md](02_active.md), протоколы встреч — в [meetings/](meetings/).

## `04_decisions.md` — журнал решений

**Миссия:** реестр решений уровня узла. Что решили, почему, альтернативы, следствия. Новые решения дописываются, принятые записи не переписываются.

**Формат записи:**

```markdown
## {YYYY-MM-DD}: {Краткая формулировка решения}

**Статус:** Accepted / Superseded by → ссылка / Proposed.

**Решение** — что выбрали.
**Почему** — контекст и причина.
**Альтернативы** (опц.) — что рассматривали и почему не взяли.
**Следствия** (опц.) — что меняется для узла, для соседей.
```

**Апгрейд до папки `04_decisions/`** — при >500 строк или одно решение >100 строк: каждое решение в отдельный файл, этот файл становится индексом.

### Пример аннотации

> Реестр решений маркетинга: что решили, почему, последствия. Новые решения дописываются, принятые записи не переписываются. Контекст решений — в [01_overview.md](01_overview.md) и [profile.md](profile.md).
