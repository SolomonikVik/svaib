# Анализ транскриптов встреч

> **DRAFT v2** — черновик методологии. Требует валидации на реальных транскриптах. Промптовые детали (формат кандидата, делай/не делай) вынесены в скилл, здесь — только принципы и модель.

## Кратко

Методология извлечения ценности из транскриптов деловых встреч. Два слоя: **выжимка** (универсальный артефакт встречи) и **надстройки** (скиллы поверх выжимки). Выжимка — в два шага: извлечение (щедрое) → классификация + очистка (точная). Маппинг на сущности из entities.md, не своя таксономия.

## Связанные файлы

- ../ontology/entities.md — каталог сущностей (что извлекаем, речевые маркеры, правила разграничения)
- ../ontology/ontology.md — файловая структура (куда раскладываются извлечённые сущности)
- methodology.md — ритуалы (когда запускаются встречи и что порождают)
- ../plugin/skills/ — скиллы на основе этой методологии

## Модель

### Два слоя

```
СЛОЙ 1: ВЫЖИМКА
  Универсальный артефакт встречи. Живёт как файл.
  Не зависит от канала, получателя, контекста проекта.
  "15 марта была встреча. Кратко: ... Ниже — что извлечено."

СЛОЙ 2: НАДСТРОЙКИ (скиллы поверх выжимки)
  • Протокол в Telegram / для клиента
  • Анализатор (подтягивает контексты → дельты по файлам)
  • Сравнение с прошлыми встречами (contrastive)
  • Рекомендации по проведению (что не спросил, соответствие целям)
  • Обновление лога встреч
```

Выжимка = база. Надстройки = конфигурируемые скиллы, каждый берёт выжимку и делает своё. Эта методология описывает **слой 1**. Надстройки — отдельные скиллы.

### Выжимка: два шага

```
Шаг А: ИЗВЛЕЧЕНИЕ                    Шаг Б: КЛАССИФИКАЦИЯ + ОЧИСТКА
─────────────────────                 ──────────────────────────────
Задача: вытащить всё важное           Задача: разложить по сущностям
Принцип: не потерять                  Принцип: не ошибиться

Вход:                                 Вход:
  транскрипт                            кандидаты с шага А
  + глоссарий (ASR-коррекция)           + (опционально) контексты:
  + участники (имя → роль)                профайл, история встреч, ...

Выход:                                Выход:
  кандидаты                             чистая выжимка
  + мягкие метки типа                   с точной классификацией
  + цитаты-опоры                        по entities.md
```

**Почему два шага, а не один:**
- LLM работает лучше с одной понятной задачей
- Извлечение и классификация — разные когнитивные задачи
- На 600 строках транскрипта классификация страдает ("это инсайт или факт?" — сложно решать, одновременно ковыряя поток речи)
- На шаге Б работаем с компактным набором кандидатов (~100-150 строк) — можно подключить контексты без contamination

---

## Вход

### Обязательное

- **Транскрипт** — текст встречи (после ASR)

### Входной контекст (шаг А)

- **Глоссарий** — коррекция ASR-ошибок + жаргон проекта/компании ("океан" → OKR, "дешборд" → dashboard). Применяется как find-replace до извлечения
- **Участники** — имя → роль. Помогает атрибутировать высказывания, если транскрипт содержит "Speaker 1"

### Дополнительный контекст (шаг Б, опционально)

- **Профайл** — клиента, участника, компании
- **История встреч** — лог предыдущих встреч (краткие выжимки)
- **Данные проекта** — цели, план, метрики
- **Любые релевантные данные** — подтягиваются по потребности (RAG, файлы)

Контекст подключается **только на шаге Б**. Экстрактор (шаг А) работает чисто от транскрипта — это исключает contamination (подмешивание известного в "новое").

---

## Шаг А: Извлечение

**Задача:** вытащить из транскрипта всё важное. Пометить мягкими метками. К каждому — цитату-опору.

**Принципы:**
- Полнота важнее точности классификации — лучше вытащить лишнее, чем потерять
- Одна задача — только извлечение, не классификация
- Мягкая типизация — "похоже на решение", не "это решение"
- Маркеры для извлечения → entities.md (секция "Речевые маркеры")

**Что извлекать:**
1. Метаданные (дата, участники, тема)
2. Темы встречи (3-5)
3. Краткое содержание (2-3 предложения)
4. Кандидаты — всё, что похоже на сущности из entities.md, с мягкой меткой типа и цитатой-опорой

---

## Шаг Б: Классификация + очистка

**Задача:** разложить кандидатов с шага А по сущностям из entities.md. Убрать мусор. Получить чистую выжимку.

**Принципы:**
- Точная классификация по правилам разграничения из entities.md
- Лучше сохранить, чем потерять — неясное в "Требует уточнения", не удалять
- Один факт = одно место (дедупликация)

**Что делает:**
1. Классифицирует каждого кандидата → конкретная сущность из entities.md
2. Удаляет мусор, дедуплицирует
3. Неясное → "Требует уточнения"

**С дополнительным контекстом (если подключен):**
4. Отмечает "новое" vs "уже известное"
5. Уточняет атрибуцию (кто имелся в виду)
6. Корректирует классификацию (с контекстом проще отличить новое от повтора)

## Выход: структура выжимки

Выжимка — артефакт встречи. Живёт как файл. Формат — Markdown.

```
## Метаданные
Дата: ...
Участники: ...
Длительность: ...
Тип встречи: ...

## Краткое содержание
[2-3 предложения: о чём встреча, к чему пришли]

## Извлечённые сущности

### Решения
- [Решение] — опора: "цитата"

### Задачи
[Owner]:
- [ ] [Задача] — срок (если озвучен)
  опора: "цитата"

### Запросы / Потребности
- [Кто]: [что хочет] — опора: "цитата"

### Идеи
- [Идея] — опора: "цитата"

### Инсайты
- [Инсайт] — опора: "цитата"

### Факты
- [Факт] — опора: "цитата"

### Открытые вопросы
- [Вопрос] — опора: "цитата"

### Риски
- [Риск] — опора: "цитата"

## Ключевые цитаты
[5-7 дословных цитат на языке участников]

## Требует уточнения
[Не сущность, а буфер: то, что важно, но не классифицируется однозначно.
Лучше сохранить здесь, чем потерять или приписать неверный тип.]
- [Пункт] — [что неясно]
```

**Правила:**
- Пустые секции не пишем
- Порядок секций в выжимке — фиксированный (по entities.md). Приоритизация и переупорядочивание — задача надстройки (протокол, дельта), которая знает тип встречи и контекст
- Цитаты-опоры остаются — для верификации и надстроек

## Типы встреч

Тип влияет на **приоритеты извлечения** (что важнее), не на алгоритм. Определяется автоматически или задаётся явно.

| Тип | Фокус извлечения |
|-----|-----------------|
| **Вводная (discovery)** | Запросы, Факты, Инсайты |
| **Регулярная (check-in)** | Задачи, Решения, Открытые вопросы |
| **Стратегическая (strategy)** | Решения, Цели, Инсайты, Риски |
| **Рабочая (working)** | Задачи, Решения, Факты |
| **Планёрка (standup/review)** | Задачи, Открытые вопросы, Риски |

Список открытый — конкретные типы настраиваются в скилле.

## Валидация

**Автоматическая (встроена в шаг Б):**
- Каждый пункт имеет цитату-опору
- Участники в выводе = участники во входе
- Задачи имеют owner (или "не назначен")
- Сроки — только если озвучены
- Один факт = одно место
- Решение ≠ задача

**Содержательная (при наличии контекста):**
- Contamination check — не подмешано ли из контекста
- Полнота — не потеряно ли из кандидатов шага А
- Правильность классификации — верно ли разнесено между типами сущностей (решение ≠ задача, инсайт ≠ факт)

---

## Принципы

1. **Сначала извлеки, потом классифицируй** — две разные когнитивные задачи, не смешивать
2. **Лучше вытащить лишнее, чем потерять** — щедрость на шаге А, точность на шаге Б
3. **Одна задача на шаг** — LLM работает лучше с одной понятной задачей
4. **Маппинг на entities.md** — не изобретаем свою таксономию
5. **Контекст — на шаге классификации** — экстрактор работает чисто от транскрипта
6. **Выжимка = универсальная база** — не зависит от канала и получателя
7. **Выжимка ≠ протокол** — протокол, дельты, рекомендации — это надстройки
8. **Живой язык** — не выглаживать в канцелярит, сохранять тон встречи
9. **Идеальная классификация невозможна** — граница между сущностями размыта даже для людей. Поэтому цитаты-опоры и секция "Требует уточнения" — не костыли, а необходимые элементы модели

## Решения и trade-offs

| # | Решение | Почему | Альтернатива |
|---|---------|--------|-------------|
| 1 | Два шага (извлечение → классификация) | LLM лучше с одной задачей. На 600 строках классификация страдает | Один проход |
| 2 | Контекст только на шаге Б | Исключает contamination | Контекст с самого начала |
| 3 | Мягкая типизация на шаге А | Не теряем из-за неправильной классификации | Жёсткая типизация сразу |
| 4 | Выжимка ≠ протокол | Разные задачи. Выжимка — база, протокол — надстройка | Всё в одном |
| 5 | entities.md как источник правды | Единый словарь для всего фреймворка | Своя таксономия для встреч |
| 6 | Приоритеты, не фильтры | Не фильтруем сущности — расставляем приоритеты | Жёсткий фильтр |
| 7 | Типы встреч — открытый список | Методология = механизм, конкретные типы = скилл | Фиксированный набор |
| 8 | Глоссарий + участники на входе | Минимальный контекст без contamination | Без контекста |

## Открытые вопросы

1. **Поведенческие паттерны** — "Коля сопротивляется", "собеседник подавлен", "ушёл от вопроса". Частично ловятся как Инсайт/Факт, но экстрактору сложно одновременно вычленять сущности И анализировать поведение. Возможное решение: отдельный проход/агент по сырому транскрипту, заточенный на поведенческие сигналы (косвенные индикаторы: короткие ответы, уклонение, смена тона, "ну посмотрим"). **Важно:** это и другие аналитические надстройки могут требовать доступа к чистому транскрипту, а не только к выжимке — выжимка теряет тон, паузы, структуру диалога
2. **Рекомендации по проведению** — "ты не спросил про контроль задач", "это не по целям". Задача надстройки-аналитика с контекстом (цели, прошлые задачи), не выжимки. Зафиксировать как тип надстройки
3. **Длинные транскрипты (>600 строк)** — нужен ли чанкинг? Какой? Дополнительная проблема: position bias — LLM склонны переоценивать начало и конец транскрипта, пропуская середину. Для длинных текстов это усиливается
4. **Contrastive summarization** — для регулярных встреч: "что изменилось с прошлой?"
5. **Скрытые барьеры по косвенным индикаторам** — техника из практики Опилкина, возможна как надстройка
6. **Автоопределение типа встречи** — алгоритм по лексике + участникам + длительности
7. **Приоритет источников при конфликте данных** — когда на шаге Б подключается контекст (профайл, история, карточка), данные могут конфликтовать с транскриптом. Нужно правило приоритета: что важнее при расхождении? Требует проработки с примерами
