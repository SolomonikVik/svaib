# Анализ транскриптов встреч

## Кратко

Методология извлечения ценности из транскриптов деловых встреч. Два слоя: **выжимка** (универсальный артефакт встречи) и **надстройки** (скиллы поверх выжимки). Пайплайн: два параллельных экстрактора (бизнес-сущности + поведенческие паттерны) → синтезатор выжимки → выжимка. Чанкинг — опционально, при длинных транскриптах. Маппинг на entities.md и behavioral_patterns.md, не своя таксономия.

## Связанные файлы

- ../ontology/entities.md — каталог бизнес-сущностей (что извлекаем, речевые маркеры, правила разграничения)
- ../ontology/behavioral_patterns.md — каталог поведенческих паттернов (что наблюдаем в динамике встречи)
- ../ontology/ontology.md — файловая структура (куда раскладываются извлечённые сущности)
- ../ontology/rituals.md — каталог ритуалов (встреча как ритуал, связь с другими процессами)
- ../architecture.md — архитектура продукта (выжимка = Данные, процесс = Помощники, контексты = Память)
- meeting_overlays.md — надстройки над выжимкой (Слой 2: протокол, дельта, longitudinal)
- ../skills/meeting-analysis/ — промпты и скиллы на основе этой методологии

## Принципы

1. **Сохранить всё важное** — задача выжимки: собрать из встречи всё, что имеет ценность для бизнеса. Если важное потеряли — система не справилась.
2. **Онтология как компас** — что считать важным определяют entities.md и behavioral_patterns.md. Не изобретаем свою таксономию.
3. **Лучше лишнее, чем потерянное** — при сомнениях в фактах и решениях — в пользу сохранения. Щедрость на входе, точность на выходе.
4. **Сложные когнитивные задачи не смешиваем** — извлечение сущностей, поведенческий анализ, сборка выжимки — разные задачи. Их лучше разделять. Качество через фокус.
5. **Живой язык встречи — часть смысла** — не выглаживать в канцелярит. Сохранять тон и формулировки встречи — в них важная информация.
6. **Идеальная классификация невозможна** — границы между сущностями размыты даже для людей. Поэтому цитаты-опоры и секция "Требует уточнения" — не костыли, а часть модели.
7. **Бережно к сигналам, осторожно в выводах** — лучше сохранить неоднозначное, чем сделать ложный вывод. Гипотеза, не диагноз.

## Модель

### Два слоя

```
СЛОЙ 1: ВЫЖИМКА
  Самодостаточный артефакт встречи — выжимка должна быть достаточно
  полной, чтобы транскрипт стал опциональным. Не зависит от канала,
  получателя, контекста проекта.
  Содержит бизнес-сущности (по entities.md) и поведенческие паттерны
  (по behavioral_patterns.md).

СЛОЙ 2: НАДСТРОЙКИ (скиллы поверх выжимки)
  • Протокол (форматированная версия для Telegram / клиента)
  • Дельта-анализатор (подтягивает контексты → обновления по файлам)
  • Анализ паттернов (longitudinal: повторяющиеся сигналы по серии встреч)
  • Рекомендации по проведению (что не спросил, соответствие целям)
  • И другие при необходимости
```

### Пайплайн

```
транскрипт
    ↓
ЧАНКИНГ (при необходимости, если большая транскрибация — в разработке)
    ↓
транскрипт + контекст (глоссарий, участники, цель — если доступны)
    ↓
┌───────────────────────┬───────────────────────┐
│                       │                       │
│  ЭКСТРАКТОР           │  ПОВЕДЕНЧЕСКИЙ        │
│  СУЩНОСТЕЙ            │  ЭКСТРАКТОР           │
│                       │                       │
│  Задача: вытащить     │  Задача: распознать   │
│  бизнес-сущности      │  поведенческие        │
│                       │  сигналы              │
│  Принцип:             │  Принцип:             │
│  лучше извлечь        │  лучше отметить       │
│  больше, чем          │  меньше, чем          │
│  потерять важное      │  приписать лишнее     │
│  Выход:               │  Выход:               │
│  кандидаты            │  наблюдения           │
│  + мягкие метки       │  (гипотезы)           │
│  + цитаты-опоры       │  + цитаты-опоры       │
│                       │                       │
└───────────┬───────────┴───────────┬───────────┘
            │                       │
            └───────────┬───────────┘
                        ↓
              СИНТЕЗАТОР ВЫЖИМКИ
              + контексты проекта (профайл, история, цели)
                        ↓
              ВЫЖИМКА (единый артефакт встречи)
              — бизнес-сущности (по entities.md)
              — поведенческие паттерны (по behavioral_patterns.md)
                        ↓
              надстройки/скиллы (протокол, дельта, рекомендации)
```

### Почему такая архитектура

- Извлечение бизнес-сущностей и распознавание поведенческих сигналов — разные когнитивные задачи, поэтому два отдельных экстрактора
- Для руководителя встреча — одно событие. Выход — один артефакт (выжимка), не два отчёта
- Синтезатор выжимки собирает оба результата, классифицирует сущности, убирает шум. Контекст проекта помогает отличить новое от известного и уточнить атрибуцию

**Два типа контекста:**
- **Контекст транскрипта** (глоссарий, участники, цель встречи) → подаётся экстракторам. Помогает модели правильно прочитать текст
- **Контекст проекта** (профайл, история встреч, данные проекта) → только синтезатору выжимки. Чтобы известное не подменяло новое из транскрипта

---

## Чанкинг (опционально)

Разбивка длинных транскриптов (>600 строк) на части для обработки. Применяется при необходимости — короткие транскрипты идут напрямую в экстракторы. Алгоритм — в разработке.

## Контекст транскрипта

Подаётся экстракторам вместе с транскриптом — помогает модели правильно прочитать текст. Не является отдельным шагом пайплайна.

- **Глоссарий** — коррекция ASR-ошибок + жаргон проекта/компании ("океан" → OKR, "дешборд" → dashboard)
- **Участники** — имя → роль. Атрибуция высказываний, если транскрипт содержит "Speaker 1"
- **Цель/контекст встречи** (опционально) — о чём встреча, зачем собрались. Без этого LLM выдаёт общие результаты

---

## Экстрактор сущностей

Первый из двух параллельных потоков.

**Вход:** транскрипт + контекст транскрипта (глоссарий, участники, цель — если доступны)

**Задача:** вытащить из транскрипта всё, что похоже на бизнес-сущности. Пометить мягкими метками. К каждому — цитату-опору.

**Принципы:**

- Полнота важнее точности классификации — лучше вытащить лишнее, чем потерять
- Одна задача — только извлечение, не классификация
- Мягкая типизация — "похоже на решение", не "это решение"
- Маркеры для извлечения → entities.md (секция "Речевые маркеры")

**Выход:**

1. Метаданные встречи (дата, участники, длительность)
2. Краткое содержание — о чём встреча, к чему пришли. Служебный выход: используется синтезатором для подбора релевантного контекста проекта
3. Кандидаты в сущности — всё, что похоже на сущности из entities.md, с мягкой меткой типа и цитатой-опорой

---

## Поведенческий экстрактор

Второй параллельный поток.

**Вход:** транскрипт + контекст транскрипта (глоссарий, участники, цель — если доступны)

**Задача:** распознать поведенческие сигналы в динамике встречи. Не "что решили", а "как решали".

**Принципы:**

- Гипотезы, не диагнозы — "похоже на уход от ответственности", не "Петр саботирует"
- Нейтральность — сигнал для размышления, не обвинение
- Точность важнее полноты — лучше отметить меньше, чем приписать лишнее
- Маркеры и причины → behavioral_patterns.md (закрытый каталог)

**Выход:** список наблюдений, каждое содержит:

1. Наблюдение — что произошло, дословно (с цитатой-опорой)
2. Паттерн — по какой оси и полюсу (ответственность +/-, ясность +/-)
3. Возможная причина — из закрытого списка в behavioral_patterns.md, явно помечена как гипотеза
4. Участник — к кому относится

---

## Синтезатор выжимки

**Вход:**

- Выход экстрактора сущностей (метаданные, краткое содержание, кандидаты)
- Выход поведенческого экстрактора (наблюдения с паттернами и гипотезами)
- Контекст проекта (подобранный по краткому содержанию): профайл, история встреч, данные проекта — если доступны

**Задача:** классифицировать кандидатов по entities.md, свести поведенческие наблюдения, убрать шум, собрать единую выжимку одной встречи.

**Принципы:**

- Точная классификация по правилам разграничения из entities.md
- Лучше сохранить, чем потерять — неясное в "Требует уточнения", не удалять
- Один факт = одно место (дедупликация)
- Поведенческие наблюдения переносятся как есть — синтезатор не судья по паттернам
- Не выдавать обсуждение за консенсус — если нет явной фиксации ("решили", "договорились"), это Идея или Открытый вопрос, не Решение

**Что делает:**

1. Классифицирует каждого кандидата → конкретная сущность из entities.md
2. Удаляет мусор, дедуплицирует
3. Неясное → "Требует уточнения"
4. Сводит поведенческие наблюдения в секцию паттернов
5. Отмечает "новое" vs "уже известное" (при наличии контекста)
6. Уточняет атрибуцию (кто имелся в виду)
7. При расхождении транскрипта и контекста — не решает кто прав, а выносит в "Требует уточнения"

**Выход:** выжимка (структура → следующая секция)

> Longitudinal анализ (повторяющиеся паттерны по серии встреч, тренды) — не задача синтезатора. Это отдельная надстройка поверх выжимок.

---

## Выход: структура выжимки

Выжимка — единый артефакт встречи. Живёт в `management/meetings/`. Формат — Markdown.

**Секции:**

| Секция | Назначение |
|--------|-----------|
| Метаданные | Дата, участники, длительность, тип встречи |
| Краткое содержание | О чём встреча, к чему пришли |
| Извлечённые сущности | По типам из entities.md: решения, задачи, цели, запросы, идеи, инсайты, факты, принципы, метрики, открытые вопросы, риски. Каждый пункт с цитатой-опорой |
| Поведенческие паттерны | Наблюдения из поведенческого экстрактора: паттерн (ось + полюс), участник, цитата-опора. Переносятся как есть |
| Ключевые цитаты | Дословные цитаты, значимые целиком (тон, формулировка). Не дублируют цитаты-опоры. Для надстроек и для человека |
| Требует уточнения | Буфер: важное, но неоднозначное. Лучше сохранить, чем потерять |

**Правила:**

- Пустые секции не пишем
- Порядок секций фиксированный. Приоритизация — задача надстройки
- Цитаты-опоры остаются — для верификации и надстроек
