# Анализ транскриптов встреч

## Кратко

Методология извлечения ценности из транскриптов деловых встреч. Два слоя: **выжимка** (универсальный артефакт встречи) и **надстройки** (скиллы поверх выжимки). Маппинг на entities.md и behavioral_patterns.md, не своя таксономия.

**Текущая рабочая версия:** entity extractor → выжимка (бизнес-сущности). Behavioral extractor существует в двух вариантах, но пока не подключён к рабочему пайплайну — открытый вопрос (формат и применение не определены). Оркестратор работает на entity-only версии.

**Целевая архитектура:** два параллельных экстрактора (бизнес-сущности + поведенческие паттерны) → конкатенация → полная выжимка. Будет реализована после стабилизации behavioral extractor.

## Связанные файлы

- ../ontology/entities.md — каталог бизнес-сущностей (что извлекаем, речевые маркеры, правила разграничения)
- ../ontology/behavioral_patterns.md — каталог поведенческих паттернов (что наблюдаем в динамике встречи)
- ../ontology/ontology.md — файловая структура (куда раскладываются извлечённые сущности)
- ../ontology/rituals.md — каталог ритуалов (встреча как ритуал, связь с другими процессами)
- ../architecture.md — архитектура продукта (выжимка = Данные, процесс = Помощники, контексты = Память)
- meeting_overlays.md — надстройки над выжимкой (Слой 2: протокол, дельта, longitudinal)
- ../skills/meeting-analysis/ — промпты и скиллы на основе этой методологии

## Принципы

1. **Сохранить всё важное** — задача выжимки: собрать из встречи всё, что имеет ценность для бизнеса. Если важное потеряли — система не справилась.
2. **Онтология как компас** — что считать важным определяют entities.md и behavioral_patterns.md. Не изобретаем свою таксономию.
3. **Лучше лишнее, чем потерянное** — при сомнениях в фактах и решениях — в пользу сохранения. Щедрость на входе, точность на выходе.
4. **Разные когнитивные задачи разделяем, но не дробим одну** — извлечение бизнес-сущностей и поведенческий анализ — разные задачи, их разделяем. Но внутри одной задачи (extraction + classification сущностей) искусственное дробление на шаги теряет контекст: модель, видящая полный нарратив, классифицирует точнее, чем цепочка с передачей.
5. **Живой язык встречи — часть смысла** — не выглаживать в канцелярит. Сохранять тон и формулировки встречи — в них важная информация.
6. **Идеальная классификация невозможна** — границы между сущностями размыты даже для людей. Поэтому цитаты-опоры и секция "Требует уточнения" — не костыли, а часть модели.
7. **Бережно к сигналам, осторожно в выводах** — лучше сохранить неоднозначное, чем сделать ложный вывод. Гипотеза, не диагноз.

## Модель

### Два слоя

```
СЛОЙ 1: ВЫЖИМКА
  Самодостаточный артефакт встречи — выжимка должна быть достаточно
  полной, чтобы транскрипт стал опциональным. Не зависит от канала,
  получателя, контекста проекта.
  Содержит бизнес-сущности (по entities.md) и поведенческие паттерны
  (по behavioral_patterns.md).

СЛОЙ 2: НАДСТРОЙКИ (скиллы поверх выжимки)
  • Протокол (форматированная версия для Telegram / клиента)
  • Дельта-анализатор (подтягивает контексты → обновления по файлам)
  • Анализ паттернов (longitudinal: повторяющиеся сигналы по серии встреч)
  • Рекомендации по проведению (что не спросил, соответствие целям)
  • И другие при необходимости
```

### Пайплайн

**Текущая рабочая версия** (entity-only):

```
транскрипт + контекст
    ↓
ЭКСТРАКТОР СУЩНОСТЕЙ
(extraction + classification в один проход)
    ↓
ВЫЖИМКА (бизнес-сущности по entities.md)
    ↓
надстройки L2 (протокол, Telegram, дельта-анализатор)
```

Оркестратор: [orchestrator-meeting.md](../skills/meeting-analysis/orchestrator-meeting.md)

**Целевая архитектура** (entity + behavioral):

```
транскрипт + контекст
    ↓
┌───────────────────────┬───────────────────────┐
│  ЭКСТРАКТОР           │  ПОВЕДЕНЧЕСКИЙ        │
│  СУЩНОСТЕЙ            │  ЭКСТРАКТОР           │
│                       │                       │
│  Извлечение +         │  монолит              │
│  классификация        │  (precision важнее)   │
│  в один проход        │                       │
│                       │                       │
│  Принцип:             │  Принцип:             │
│  лучше извлечь        │  лучше отметить       │
│  больше, чем          │  меньше, чем          │
│  потерять важное      │  приписать лишнее     │
│                       │                       │
│  Выход: сущности      │  Выход: наблюдения    │
│  по entities.md       │  (гипотезы)           │
│  + цитаты-опоры       │  + цитаты-опоры       │
└───────────┬───────────┴───────────┬───────────┘
            └───────────┬───────────┘
                        ↓
              СБОРКА ВЫЖИМКИ (конкатенация)
                        ↓
              ВЫЖИМКА (бизнес-сущности + поведение)
                        ↓
              надстройки L2
```

### Почему такая архитектура

- **Два экстрактора, не один** — извлечение бизнес-сущностей и распознавание поведенческих сигналов — разные когнитивные задачи, поэтому два отдельных экстрактора
- **Один проход на задачу** — экстрактор сущностей извлекает И классифицирует в одном проходе. Дробление на extraction → classification (через отдельный синтезатор) теряет контекст при передаче — модель, видящая полный нарратив, делает лучше. Проверено: один промпт с методологией (19/20) побеждает 4-агентный пайплайн (16/20)
- **Выжимка = конкатенация** — для руководителя встреча — одно событие. Два экстрактора работают параллельно, координатор соединяет выходы в один артефакт. Отдельный промпт для сборки не нужен — каждый экстрактор выдаёт готовый результат

**Два типа контекста:**
- **Контекст транскрипта** (глоссарий, участники, цель встречи) → подаётся экстракторам. Помогает модели правильно прочитать текст
- **Контекст проекта** (профайл, история встреч, данные проекта) → подаётся экстрактору сущностей как фон для классификации. Запрещено добавлять информацию из контекста в выжимку

## Чанкинг (для экстрактора сущностей)

> **Статус: в разработке.** Порог и потери тестировались на старом экстракторе (entity extractor + synthesizer). С новым экстрактором (один проход) порог не верифицирован.

Opus 1M контекст тянет ~180KB монолитно (≈2-часовая встреча). Для сверхдлинных встреч (5-10 часов) чанкинг вероятно потребуется — транскрипт разбивается на чанки по границам реплик для параллельной экстракции.

Асимметрия: чанкинг применяется только к экстрактору сущностей (recall критичен). Поведенческий экстрактор работает монолитно — precision важнее, паттерны устойчивы к потере отдельных сигналов.

Процедура, алгоритм нарезки, примеры → [L1-procedure-chunking.md](../skills/meeting-analysis/L1-procedure-chunking.md)

## Контекст транскрипта

Подаётся экстракторам вместе с транскриптом — помогает модели правильно прочитать текст. Не является отдельным шагом пайплайна.

- **Глоссарий** — коррекция ASR-ошибок + жаргон проекта/компании ("океан" → OKR, "дешборд" → dashboard)
- **Участники** — имя → роль. Атрибуция высказываний, если транскрипт содержит "Speaker 1"
- **Цель/контекст встречи** (опционально) — о чём встреча, зачем собрались. Без этого LLM выдаёт общие результаты

## Экстрактор сущностей

Первый из двух параллельных потоков.

**Вход:** транскрипт + контекст транскрипта (глоссарий, участники, цель — если доступны) + контекст проекта как фон (опционально)

**Задача:** вытащить из транскрипта все бизнес-сущности, классифицировать по entities.md, к каждой — цитату-опору. Один проход: extraction + classification.

**Принципы:**

- Полнота важнее точности — лучше вытащить лишнее, чем потерять важное (Iron Law: после экстрактора транскрипт не нужен)
- Quote-first — цитата-опора первична, классификация вторична
- Классификация по 11 типам из entities.md + 18 правил разграничения
- Phantom consensus — нет "решили/договорились" = не решение
- Контекст проекта = фон для понимания, не источник содержания. Запрещено добавлять в выжимку информацию из контекста
- Неясное → "Требует уточнения", не удалять

**Выход:**

1. Метаданные встречи (дата, участники, длительность)
2. Краткое содержание — о чём встреча, к чему пришли
3. Классифицированные сущности по типам из entities.md, с цитатами-опорами
4. Ключевые цитаты — значимые целиком (тон, формулировка)
5. Требует уточнения — важное, но неоднозначное

Промпт → [L1-prompt-entity-extractor.md](../skills/meeting-analysis/L1-prompt-entity-extractor.md)

## Поведенческий экстрактор

> **Статус: открытый вопрос.** Два варианта промпта существуют, но ни один не верифицирован до конца. Формат и применение не определены. В рабочий пайплайн (оркестратор) пока не подключён.

Второй параллельный поток (в целевой архитектуре).

**Вход:** транскрипт + контекст транскрипта (глоссарий, участники, цель — если доступны)

**Задача:** распознать поведенческие сигналы в динамике встречи. Не "что решили", а "как решали".

**Принципы:**

- Гипотезы, не диагнозы — "в трёх высказываниях о задержке названы внешние причины", не "Петр саботирует"
- Нейтральность — сигнал для размышления, не обвинение
- Точность важнее полноты — лучше отметить меньше, чем приписать лишнее
- Каталог осей и ракурсов → behavioral_patterns.md

### Два варианта (оба в тестировании)

**Вариант 1 (текущий):** аналитический формат. Две оси (ответственность, ясность), четыре ракурса, цитаты-опоры, полярность (+/-), техническая разметка. Точный, но навороченный — выход для аналитика, не для клиента.
→ [L1-prompt-behavior-extractor.md](../skills/meeting-analysis/L1-prompt-behavior-extractor.md)

**Вариант 2 (experimental):** клиентский формат. Три оси (ответственность, ясность, вовлечённость), без цитат, группировка по участникам + встреча в целом, дисклеймер. Выход читаем клиентом напрямую. Первый тест на реальном транскрипте — оценка 4/5.
→ [L1-prompt-behavior-extractor-v2.md](../skills/meeting-analysis/L1-prompt-behavior-extractor-v2.md)

Оба варианта работают по сырому транскрипту (не по выжимке L1). Решение о том, какой станет основным — после дополнительных тестов.

**Выход (вариант 1):** список наблюдений, каждое содержит:

1. Цитата — дословно из транскрипта (одна или несколько, если сигнал виден на последовательности реплик)
2. Ось и ракурс — ответственность (причина / энергия) или ясность (конкретность / связность)
3. Полярность — (+) или (-)
4. Наблюдение — что видно в речи (процессными глаголами)
5. Участник — к кому относится

**Выход (вариант 2):** клиентский документ: дисклеймер → инсайты по участникам (три оси, 1-2 предложения каждый) → встреча в целом

## Сборка выжимки

**Текущая версия:** выжимка = выход entity extractor (behavioral не подключён).

**Целевая версия:** координатор соединяет выходы двух экстракторов в единый артефакт. Отдельного промпта нет — каждый экстрактор выдаёт готовый результат, сборка = конкатенация.

**Выход:** выжимка (структура → следующая секция)

> Longitudinal анализ (повторяющиеся паттерны по серии встреч, тренды) — отдельная надстройка поверх выжимок.

## Выход: структура выжимки

Выжимка — единый артефакт встречи. Живёт в `meetings/` на уровне наименьшего общего контейнера: встреча внутри под-направления → `direction/sub_direction/meetings/`, кросс-встреча под-направлений → `direction/meetings/`, кросс-направлений → `management/meetings/` (→ scaffold/direction/README.md). Формат — Markdown.

**Секции:**

| Секция | Назначение |
|--------|-----------|
| Метаданные | Дата, участники, длительность, тип встречи |
| Краткое содержание | О чём встреча, к чему пришли |
| Извлечённые сущности | По типам из entities.md: решения, задачи, цели, запросы, идеи, инсайты, факты, принципы, метрики, открытые вопросы, риски. Каждый пункт с цитатой-опорой |
| Поведенческие паттерны | Наблюдения из поведенческого экстрактора: паттерн (ось + полюс), участник, цитата-опора. Переносятся как есть |
| Ключевые цитаты | Дословные цитаты, значимые целиком (тон, формулировка). Не дублируют цитаты-опоры. Для надстроек и для человека |
| Требует уточнения | Буфер: важное, но неоднозначное. Лучше сохранить, чем потерять |

**Правила:**

- Пустые секции не пишем
- Порядок секций фиксированный. Приоритизация — задача надстройки
- Цитаты-опоры остаются — для верификации и надстроек
