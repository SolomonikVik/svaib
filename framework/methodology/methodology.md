# Методология

## Кратко

Принципы работы с системой Second AI Brain, модели мышления, decision frames. То, что зашивается в skills и agents, но существует как отдельный документ — источник правды. Отвечает на вопросы: как принимать решения о структуре, как работает протокол чтения для LLM, как система масштабируется. Каталог ритуалов: [ontology/rituals.md](../ontology/rituals.md).

- ../ontology/ontology.md — какие сущности существуют и как связаны

- ../scaffold/ — готовый каркас с шаблонами

## Протокол чтения для LLM

### По умолчанию (каждый сеанс работы)

```
README → 01_ceo/01_my_profile.md → 02_strategy/01_vision.md → 02_strategy/02_goal.md → 02_strategy/03_plan.md
```

Эти 5 файлов дают полный контекст: что за проект, кто этот человек, куда идёт компания, что делает сейчас.

### По триггеру (только когда нужно)

| Триггер | Файл |
|---|---|
| Стратегическое обсуждение, пересмотр целей | `02_strategy/05_timeline.md`, `02_strategy/06_decisions.md` |
| Подведение итогов, ретроспектива | `02_strategy/04_weekly_progress.md` |
| Планирование периода | `02_strategy/03_plan.md`, `02_strategy/02_goal.md` |
| Обсуждение задач с участием людей | `03_team/`, `04_management/` |
| Анализ состояния, энергии, паттернов | `01_ceo/03_thoughts.md` |

## Decision Frames

### Когда создавать подпроект?

**Создавать подпроект** когда один plan перестаёт вмещать оперативку — слишком много задач по разным направлениям в одном файле.

**Подпроект → полный проект** когда появилась собственная цель, отличная от родительской.

### Когда выносить секцию в отдельный файл?

**Выносить** когда секция растёт независимо от основного файла и обновляется с другой частотой.

**Известные случаи:**
- Стратегия в goal разрослась → отдельный `strategy.md`
- Team → папка `03_team/` с профайлами участников

**Не выносить** если секция маленькая и обновляется вместе с основным файлом — не плодить файлы заранее.

### Что является «решением» (для decisions.md)?

**Да:** выбор, закрывающий альтернативы — «Не идём в B2C», «Выбрали Dify вместо Langflow».

**Нет:** операционный выбор внутри недели — «Сегодня делаю A, не B».

### Правило каноничности метрик

- В goal метрики существуют только как **план** (целевые значения)
- **Факт** фиксируется только в timeline по итогам периода
- Goal всегда показывает "куда идём", не "где были"

### Правило дословного копирования

При переносе, структурировании, группировке задач и заметок — **КОПИРОВАТЬ текст дословно. НИКОГДА не перефразировать, не сокращать, не "причёсывать".**

Допустимо: добавить заголовок секции, переместить запись в другую группу.
Недопустимо: менять формулировки, убирать детали, обобщать подзадачи в одну строку.

Причина: Человек фиксирует пометки и контекст специально. Каждое слово — напоминалка для будущего действия. "Причёсанная" версия теряет смысл. Это не код и не документация — это рабочие заметки.

## Масштабирование

### Соло-предприниматель / малая команда

- Развёртывается ядро (8 файлов) + мета (2 файла)
- Опциональные файлы — по мере роста
- Один человек ведёт всё, LLM помогает

### CEO с командой (до 100-200 человек)

- Та же структура, но для личного пространства CEO (не для всей компании)
- Team разворачивается в папку с профайлами
- Metrics расширяется (CEO отслеживает больше метрик)
- Strategy может выноситься в отдельный файл
- Подпроекты для ключевых направлений
- Выжимки встреч — в meetings/ наименьшей общей родительской папки (внутри подразделения → 04_management/{org_node}/meetings/, между дочерними одной группировки → 04_management/{parent}/meetings/, между верхнеуровневыми → 04_management/meetings/ — → scaffold/04_management/README.md)

**Принцип:** система не усложняется для клиента. CEO не должен вести больше файлов, чем соло-предприниматель. Разница — в глубине наполнения, не в количестве.
