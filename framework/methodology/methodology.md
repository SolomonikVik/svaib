# Методология

## Кратко

Принципы работы с системой Second AI Brain, модели мышления, decision frames. То, что зашивается в skills и agents, но существует как отдельный документ — источник правды. Отвечает на вопросы: какие ритуалы и с какой периодичностью, как принимать решения о структуре, как работает протокол чтения для LLM.

## Ритуалы

### Сводная таблица

| Когда | Что делаем | Файлы |
|---|---|---|
| Каждое утро | План дня из плана недели | 03_plan |
| Каждый понедельник | Итоги недели → weekly progress. Новый план недели | 03_plan → 04_weekly_progress |
| Начало квартала | Цель квартала + KR. План квартала. Просмотр ideas | 02_goal, 03_plan, ideas |
| Конец квартала | Итоги квартала (KR план vs факт) → timeline | 04_weekly_progress → 05_timeline |
| Конец года | Итоги года → timeline. Пересмотр vision/strategy. Новые цели | 05_timeline, 02_goal, 01_vision |
| По мере | Решения → decisions. Мысли → reflections. Идеи → ideas | decisions, reflections, ideas |

### Ежедневный ритуал

**Утро:** Открыть 03_plan. Составить/обновить план дня из плана недели.

### Еженедельный ритуал (понедельник)

1. Подвести итоги прошлой недели → запись в 04_weekly_progress
2. Просмотреть план недели → отметить выполненное, перенести незавершённое
3. Составить новый план недели

### Квартальный ритуал

**Начало квартала:**
1. Из 02_goal взять цель года → сформулировать цель квартала + KR (3-5 штук)
2. Просмотреть ideas → вытащить релевантное
3. Составить план квартала в 03_plan

**Конец квартала:**
1. Из недельных записей в 04_weekly_progress собрать итоги квартала
2. Зафиксировать KR план vs факт → запись в 05_timeline
3. Почистить ideas — удалить неактуальное

### Годовой ритуал

**Конец года:**
1. Годовая сводка → 05_timeline
2. Перечитать 01_vision — проверить актуальность
3. Пересмотреть стратегию в 02_goal
4. Сформулировать новые цели на год

## Ритуалы по сущностям

### Vision (01_vision)
- Раз в 6-12 месяцев — перечитать и проверить актуальность
- При пивоте или существенном изменении видения
- Конец года — в рамках годовой ретроспективы

### Goal (02_goal)
- **Начало квартала:** сформировать цель квартала + KR
- **Начало года:** пересмотреть цель года, обновить стратегию
- **Конец года:** пересмотреть долгосрочную цель

### Plan (03_plan)
- **Каждое утро:** план дня из плана недели
- **Каждый понедельник:** итоги прошлой недели → weekly progress. Новый план недели
- **Начало квартала:** из goal взять цель и KR → план квартала. Просмотреть ideas

### Weekly Progress (04_weekly_progress)
- **Каждый понедельник:** итоги прошлой недели → новая запись
- **Конец квартала:** из недельных записей → квартальные итоги в timeline

### Timeline (05_timeline)
- **Конец квартала:** из weekly progress собрать итоги → запись в timeline
- **Конец года:** годовая сводка в timeline

### Ideas
- **Планирование квартала:** просмотреть ideas → вытащить релевантное в plan
- **Раз в квартал:** почистить — удалить неактуальное
- **В любой момент:** набросать запись, не думая о структуре

### Decisions
- При накоплении >30-40 решений — архивные (со статусом «Пересмотрено» старше года) переносятся в `archive/decisions_YYYY.md`

### Reflections
- Раз в год старые записи переносятся в `archive/reflections_YYYY.md`

## Decision Frames

### Когда создавать подпроект?

**Создавать подпроект** когда один plan перестаёт вмещать оперативку — слишком много задач по разным направлениям в одном файле.

**Подпроект → полный проект** когда появилась собственная цель, отличная от родительской.

### Что является «решением» (для decisions.md)?

**Да:** выбор, закрывающий альтернативы — «Не идём в B2C», «Выбрали Dify вместо Langflow».

**Нет:** операционный выбор внутри недели — «Сегодня делаю A, не B».

### Правило каноничности KR

- В goal KR существуют только как **план** (целевые значения)
- **Факт** фиксируется только в timeline по итогам квартала
- Goal всегда показывает "куда идём", не "где были"

### Правило дословного копирования

При переносе, структурировании, группировке задач и заметок — **КОПИРОВАТЬ текст дословно. НИКОГДА не перефразировать, не сокращать, не "причёсывать".**

Допустимо: добавить заголовок секции, переместить запись в другую группу.
Недопустимо: менять формулировки, убирать детали, обобщать подзадачи в одну строку.

Причина: Человек фиксирует пометки и контекст специально. Каждое слово — напоминалка для будущего действия. "Причёсанная" версия теряет смысл. Это не код и не документация — это рабочие заметки.

## Масштабирование

### Соло-предприниматель / малая команда

- Развёртывается ядро (8 файлов) + мета (2 файла)
- Опциональные файлы — по мере роста
- Один человек ведёт всё, LLM помогает

### CEO с командой (до 100-200 человек)

- Та же структура, но для личного пространства CEO (не для всей компании)
- Team разворачивается в папку с профайлами
- Metrics расширяется (CEO отслеживает больше метрик)
- Strategy может выноситься в отдельный файл
- Подпроекты для ключевых направлений
- Протоколы встреч — в большой рамке (Процессы), не в management/

**Принцип:** система не усложняется для клиента. CEO не должен вести больше файлов, чем соло-предприниматель. Разница — в глубине наполнения, не в количестве.
