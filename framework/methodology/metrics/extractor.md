# Extractor — как строится per-client извлечение данных

Детальный разбор компонента **extractor**: как его построить под конкретный источник клиента, какие патологии встречаются в реальных xlsx, какие конвенции реализации обязательны. Карта-уровень — что такое extractor, его роль, контракт вход/выход, schema-hash, место в линии данных — в [`architecture.md`](architecture.md), раздел «Extractor». Здесь — уровень сборки.

**Статус — черновик.** Свод материала из старого playbook; probe-процедура, формат JSON-выхода и schema-hash доведены до контракта, остался шаблон реализации (см. «Что доделать»).

## 🔵 Как строится extractor

Extractor создаётся под конкретный источник скиллом-писателем (см. «Скилл-писатель extractor'а») — при онбординге клиента либо когда оркестратор в работе обнаружил непокрытый источник. Последовательность:

1. **Probe источника.** Разведка структуры: листы, размеры, объединённые ячейки, формулы, кэшированные ошибки — только форма, не значения. Детально — раздел «Probe-процедура». Работать на временной локальной копии xlsx — не в репозитории framework, не коммитить: клиентские данные остаются у клиента. Живой источник клиента не править.
2. **Зафиксировать раскладку.** По результату probe — реестр координат «каноническое имя метрики → где её брать». Что входит в раскладку — раздел «Probe-процедура».
3. **Написать реализацию.** Python / SQL / API-клиент под этот адрес. Координаты захардкожены в раскладке, не вычисляются на лету.

## 🔵 Скилл-писатель extractor'а

Три шага выше исполняет не оркестратор, а отдельный **скилл-писатель extractor'а**. Разведение ролей: оркестратор отвечает на вопросы CEO и физику источника не трогает; писатель — собирает extractor и держит всю «грязь» xlsx внутри себя.

**Триггер.** Онбординг клиента — либо runtime: оркестратор в работе натыкается на источник, под который extractor'а ещё нет, информирует CEO и делегирует сборку писателю.

**Что делает.** Probe источника → фиксация раскладки → реализация extractor'а → snapshot эталонного schema-hash. Внутренняя кухня (pandas, `document-skills:xlsx`, [`probe_xlsx.py`](../../skills/metrics-analysis/probe_xlsx.py)) — его дело; наружу он отдаёт готовый extractor в `metrics/extractors/` клиента и возвращает управление оркестратору.

**Статус.** Сам скилл-файл пока не написан — его контракт здесь, реализация отдельным заходом.

## 🔵 Probe-процедура

Probe — разовая разведка источника при онбординге (и повторно — когда schema-hash сигналит об изменении). Цель: понять форму, не значения.

**Что собрать** (helper — [`probe_xlsx.py`](../../skills/metrics-analysis/probe_xlsx.py)):
- список листов, их размеры (строки × колонки);
- заполненность колонок — где данные, где пусто;
- объединённые ячейки и их координаты — они ломают `read_excel(header=N)`;
- формулы и кэшированные ошибки (`#DIV/0!`, `#VALUE!`, `#REF!`);
- несколько таблиц на одном листе — границы каждой, свой header-row.

**Что фиксирует раскладка.** По результату probe раскладка пишется в extractor явной структурой — реестр координат на каждый источник:
- адрес: путь к файлу / URL, имя листа;
- служебная колонка с каноническими именами метрик — её координата;
- ось периодов: где, формат периода (`YYYY-MM` / `YYYY-Qn` / …), гранулярность;
- где план, где факт — колонки или маркер;
- где прошлые периоды;
- единица — общая на источник или координата на метрику.

Раскладка захардкожена, не вычисляется на лету (см. «Контракт и конвенции реализации»).

**Probe-артефакты временные.** Отчёты probe могут содержать фрагменты значений из клиентского источника — это рабочая диагностика на время сборки extractor'а. Их нельзя коммитить, переносить в framework или оставлять как постоянную документацию. После фиксации раскладки и написания extractor'а отчёты удаляются. В постоянных metrics-файлах остаётся только семантика и адрес источника — значения не дублируются. Когда полный дамп не нужен — запускать probe с ограничением строк.

## 🔵 Формат JSON-выхода

Extractor пишет результат в JSON-файл; путь задаётся CLI-параметром `--out`. Оркестратор читает файл через Read. Структура:

```json
{
  "source": "sales.xlsx",
  "schema_hash": "sha256:1f3a…",
  "extracted_at": "2026-05-14T18:30:00+03:00",
  "metrics": [
    {
      "name": "Активные клиенты",
      "period": "2026-04",
      "mode": "факт",
      "value": 142,
      "unit": "шт",
      "status": "ok"
    },
    {
      "name": "Новый MRR",
      "period": "2026-05",
      "mode": "план",
      "value": null,
      "unit": "млн руб",
      "status": "div0"
    }
  ]
}
```

**Поля верхнего уровня:**
- `source` — имя/адрес источника, из раскладки.
- `schema_hash` — отпечаток структуры источника на момент чтения (см. «Schema-hash check»).
- `extracted_at` — момент чтения, ISO 8601 с таймзоной.

**Поля метрики:**
- `name` — каноническое имя метрики, оно же ID (заголовок `### {имя}` из metrics-файла).
- `period` — период значения в формате источника.
- `mode` — `план` / `факт`.
- `value` — число или `null`. `null` всегда, когда `status` ≠ `ok`.
- `unit` — единица из metrics-файла.
- `status` — `ok` | `div0` | `missing` | `error`. Тихий `0`/`null` без статуса запрещён — он убивает классификацию «норма / красная зона / нет данных».

## 🔵 Schema-hash check

Дешёвая защита от дорогой тихой ошибки: клиент поменял источник → extractor останавливается, а не возвращает число из не той ячейки.

**Что входит в отпечаток** — структурный скелет источника, без значений:
- имена листов и их порядок;
- на каждом листе — заголовки колонок и их порядок.

Скелет сериализуется в нормализованный JSON, берётся `sha256`. Эталонный хэш снимается при настройке extractor'а и хранится рядом с ним — константой в коде или файлом `<source>.schema.json`.

**Как сверяется.** При каждом запуске (а если есть слой подготовки — при заборе snapshot) extractor заново снимает структурный скелет, считает хэш, сравнивает с эталоном:
- совпало → продолжает извлечение;
- не совпало → останавливается, в stdout `FAIL: schema changed`, JSON не пишет.

После легитимного изменения источника эталонный хэш пересчитывается вручную — осознанное действие, не автоподстройка.

## 🔵 Патологии реальных источников

Что встречается в живых клиентских xlsx — на опыте первого клиента. Это причина, по которой probe обязателен, а раскладка фиксируется явно:

- **Несколько таблиц на одном листе.** «Шапка-витрина» сверху, «годовая раскладка по месяцам» в середине, «недельная сводка» снизу — у каждой своя система координат, свой header-row, свой формат.
- **Объединённые ячейки в заголовках периодов.** `pandas.read_excel(header=N)` на них уплывает — колонки сдвигаются.
- **Формат меняется по периоду.** Например, до апреля «План / Факт / %», с апреля «План / потенциальный / %». Семантика разная — фиксировать как известную особенность источника.
- **Свободные строки-комментарии между метриками.** Рабочие заметки CEO («триггеры у клиентов», «не отвечают») — не метрики, раскладка их пропускает.
- **`#DIV/0!` в формулах.** Обычно план = 0 для будущих периодов. Решить заранее: пропускать или маркировать статусом.
- **Опечатки и врущие метки.** У первого клиента строка называлась «Новый MRR/доля черна» — опечатка («черна» = churn), и реально это **доля** младенческого churn в новых продажах, а не новый MRR. Имя в источнике может врать — сверять с CEO.

## 🔵 Контракт и конвенции реализации

**Вход / выход** — по контракту из [`architecture.md`](architecture.md), раздел «Extractor»: вход — список канонических имён метрик + параметры (период, фильтры); выход — значения с метаданными (имя, период, режим план/факт, единица, timestamp). Точный формат выхода — раздел «Формат JSON-выхода».

**Один extractor на источник.** У каждого клиента источник свой → раскладка своя → extractor свой. Универсального extractor'а не бывает — он теряет детерминированность.

**Координаты захардкожены, код не генерируется на лету.** Меняется структура источника — правится раскладка, не алгоритм. Генерация кода в момент вызова = недетерминированность + шум в контексте агента.

**Статусы вместо тихих нулей.** Каждой метрике в выходе — `status`: `ok | div0 | missing | error`. Не подставлять `0` или `null` без статуса — это убивает дальнейшую классификацию (норма / красная зона / нет данных).

**Stdout молчит.** Оркестратор дёргает extractor, результат забирает из JSON-файла. В stdout — только короткий статус (`OK: wrote N metrics → <path>` / `FAIL: <reason>`). Дамп результата в stdout = шум в контексте AI-агента.

**CLI явный.** Параметры через argparse: какие метрики, период, путь выхода. Без скрытой магии.

**Schema-hash check** — обязанность extractor'а, его единственного владельца. Механизм — раздел «Schema-hash check».

## 🔵 Как обновлять

- Изменилась структура источника (новая колонка, переименован лист) → правится раскладка + тест-прогон.
- Появилась новая метрика → её паспорт в `{domain}-metrics.md` + строка в раскладке.
- Extractor версионируется через git вместе с остальными артефактами клиента.
