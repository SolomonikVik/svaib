# Intake — внутренний чек-лист координатора

Минимум, который координатор svaib должен собрать у клиента до Шага 3 [`rollout.md`](rollout.md) (черновик `canonical_metrics.md`). Это **внутренний** чек-лист, не анкета на заполнение клиентом. CEO ничего не заполняет — координатор задаёт нужные вопросы голосом / в чате и записывает к себе.

Чек-лист покрывает только первый слой вертикали (см. [`first-layer.md`](first-layer.md)). Полная анкета второго слоя (синонимы, антисинонимы, типичные вопросы, runtime-окружение) — отдельный артефакт, появляется только если в живой работе с клиентом возник запрос за пределы первого слоя.

---

## Что должно быть на руках до старта Шага 3 rollout

### 1. Источник

- [ ] Главный xlsx — имя файла, путь, лист, кто редактирует, как часто обновляется.
- [ ] Структура целеполагания клиента — OKR / KPI / неформальная. **Без расшифровки целей** на первом слое; только тип, чтобы понимать контекст target metrics.

### 2. Базовые target metrics

- [ ] 5–10 метрик, которые CEO держит в голове регулярно (на что смотрит каждую неделю / каждый месяц).
- [ ] На каждую — короткая фраза, как CEO её называет в речи. Это будущее **каноническое имя**; если CEO готов вписать его колонкой в xlsx, оно подходит.
- [ ] На каждую — короткое объяснение бизнес-смысла (как сам CEO формулирует) и алгоритма расчёта.

### 3. Типовые вопросы

- [ ] 3–5 вопросов CEO к этим цифрам в его обычных формулировках. Нужны для smoke-test на Шаге 6 rollout. Регулярные / ad-hoc — фоном, не критично на первом слое.

### 4. Технические условия

- [ ] Где живёт xlsx — локально (через Drive for Desktop), в Google Drive, иное. На первом слое важно только то, что `document-skills:xlsx` может его открыть.
- [ ] Кто на стороне клиента может технически внести колонку с каноническими названиями в главный лист.

---

## Артефакты чек-листа

Записи координатора живут у клиента, не во framework:

- `clients/private/<client>/intake/00_metrics_intake.md` — короткая записка по пунктам 1–4. Создаётся один раз перед Шагом 3 rollout, обновляется по мере уточнений.
- Когда нужно поднять урок в канон (например, обнаружена общая для нескольких клиентов техника подключения колонки) — поднимается в [`first-layer.md`](first-layer.md) или в шаблон `framework/scaffold/05_metrics/canonical_metrics.md`.

---

## Что НЕ собираем на первом слое

Сознательно вне границ. Появляется, только если в живой работе возник соответствующий запрос:

- Синонимы CEO для каждой метрики (нужно для паспорта во втором слое).
- Антонимы / омонимы из речи клиента (защита от спутывания, второй слой).
- Направление метрики (`increase` / `decrease`, нужно для классификации «горит / не горит»).
- Наличие плана для каждой метрики (`plan_required`, второй слой).
- Известные косяки источника (DIV/0, объединённые ячейки, врущие метки) — вводятся через `known_issues` во втором слое, когда первая дыра поймана smoke-тестом.
- Runtime-окружение помощника (Cowork / Claude Code / Web) — конкретные триггеры по runtime обсуждаются при переходе ко второму слою.

Все вопросы выше — нормальные для второго слоя; для первого слоя они избыточны и тормозят запуск.
