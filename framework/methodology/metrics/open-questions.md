# Открытые вопросы — metrics-вертикаль

Архитектурные и методологические вопросы, которые накопились по ходу пилота. Здесь не обсуждаем, а фиксируем — чтобы вернуться в новом чате на свежую голову, не теряя контекст.

Каждый вопрос: что зафиксировано сейчас + что под вопросом + почему ждёт обсуждения.

---

## 1. narrative.py vs LLM-сборка ⚠️ архитектурный

**Где сейчас.** Ветка В оркестратора (compound route) использует Python-скрипт `narrative.py` для **сборки структурированного ответа CEO** из JSON extractor'а. Скрипт классифицирует метрики (red / win / ok / blocked) с фиксированным порогом 10% и собирает markdown с разделами «В красной зоне / Лучше плана / В норме / Не считается». См. [`_inbox/metrics-scaffold/sandbox/extractors/narrative.py`](../../_inbox/metrics-scaffold/sandbox/extractors/narrative.py).

**Что под вопросом** (поднял Виктор 2026-04-30): «откуда взялся этот блок? Я думал, что это как раз решает LLM, а не Python. Python вытаскивает числа, LLM понимает контекст и собирает narrative. У LLM CLAUDE.md подгружен, она знает что обсуждали раньше, что важно подсветить в моменте, 6 метрик показать или 3».

Этот вопрос **не был в исходном ТЗ** — narrative.py добавлен задним числом, чтобы зафиксировать каркас детерминированно и упростить sub-agent test.

**Почему всё-таки сделал так** (мои аргументы):
- Хотел чтобы каркас «red/win/ok» был воспроизводимым — на одних и тех же данных одинаковый ответ.
- Sub-agent test проще: subagent вызывает скрипт → читает md → отдаёт CEO.
- Снижает шанс LLM-галлюцинаций при сборке (например, перепутать direction).

**Аргументы Виктора (контр):**
- Сборка narrative — содержательная, не механическая. Контекст разговора, приоритеты CEO в моменте, сколько подсветить — это работа LLM, а не правило.
- Python создаёт жёсткий каркас и теряет гибкость. Threshold 10% — арбитрарное число.
- LLM с подгруженным CLAUDE.md знает больше, чем может знать Python (контекст истории чата, важность для CEO именно сейчас).

**Возможный компромисс.** Разнести роли:
- `classify_direction(fact, plan, direction)` → Python (чисто механическая логика «факт ниже плана при direction=increase = плохо»).
- Python возвращает JSON с metric + direction + delta + classification.
- **Сборку финального ответа делает LLM** — берёт JSON, добавляет контекст разговора, формирует narrative с учётом того, что важно CEO в моменте. Никакого markdown-каркаса в Python.

**Последствие при принятии компромисса:** `narrative.py` упрощается до «классификатора» (или вообще исчезает, classify уходит в extractor). Каркас красная-зона/лучше-плана/в-норме формирует LLM по системному промпту в `orchestrator-metrics.md`.

**Решение откладывается.** Обсудить в новом чате до старта Б.2 (или после Б.2 — если sub-agent на боевом покажет, что текущая схема работает). Сейчас не ломаем.

## 2. Стоимость sub-agent end-to-end test ⚠️ операционный

**Где сейчас.** Делаем sub-agent test (general-purpose агент эмулирует Cowork-агента CEO на тест-вопросе). Один прогон занимает **~10% контекстного бюджета** основного чата (60-70К токенов на 15 tool_uses).

**Что под вопросом.** Если регулярно тестировать pipeline (после каждого фикса, каждой правки канона) — это съедает контекст. Для разработки метода — ок, но для регулярного eval — слишком дорого.

**Альтернативы для обсуждения:**
- Запускать sub-agent в отдельной сессии (не в основном чате), читать только финальный артефакт через path. Тогда основной чат не платит за tokens агента.
- Использовать более узкий тип субагента (не general-purpose, а Explore — read-only). Но тогда он не может запустить extractor.
- Делать sub-agent test реже — только на ключевых вехах (закрытие этапа), не после каждой правки.

**Решение откладывается.** Обсудить в новом чате при планировании operational tempo eval-набора.

## 3. Дубль метрик KR1 ❓ семантический

**Где сейчас.** В sandbox есть две метрики: `metric_churn_90d_mrr_monthly` (kr1_yearly r31) и `metric_churn_ltv_90d_mrr` (churn_extended r45). За Q1 они дают **идентичные числа** байт-в-байт (факт 5 793 264, план 2 711 000).

**Что под вопросом.** Это одна метрика, записанная дважды для удобства витрины? Или две семантически разные метрики, которые в Q1 случайно совпали?

**Моя ставка.** Одна. Аргумент: если бы LTV было реально другой расчёт (с фильтром по LTV<90d или ещё чем-то), числа бы хоть на копейку отличались. Точное совпадение → одна формула в двух местах.

**Что делать.** Уточнить у клиента. Если одна — schлопнуть в один паспорт `metric_churn_90d_mrr` (без `_ltv`-суффикса), оставить strategy=`kr1_row` r31 (KR1 = semantic source of truth).

**Решение откладывается.** Спросить первого клиента при заполнении intake-form (часть 5.1).

## 4. Threshold vs ranking в narrative ⚙️ улучшение

**Где сейчас.** `narrative.py` использует фиксированный 10% порог для классификации red/win/ok. Для запроса «топ-5 худших» порог формально не нужен — нужен ranking без cutoff.

**Что под вопросом.** На текущих данных первого клиента работает (в красной зоне 4-6 метрик, топ-5 = первые пять из bucket). Реальная разница появится у клиента с 30+ метрик и 15 в красной зоне.

**Решение.** Добавить ranking mode в narrative.py как опциональный флаг `--mode ranking --top 5` в Этап С. Не блокирует Б.2.

Связано с вопросом #1 — если narrative переходит в LLM, ranking тоже становится работой LLM.

## 5. Топ-N + сводка ⚙️ UX

**Где сейчас.** Если CEO просит «топ-5», а в красной зоне ровно 6 — отсечение шестой условно. Если в красной зоне 3 — отдаём топ-3.

**Что предлагается.** Гибрид: давать запрошенный топ-N **плюс сводка** «всего в красной зоне 6 метрик / 3 / 0 — отсёк по запросу top-5 / показал все 3».

**Решение.** В Этап С, не блокирует Б.2.

Также связано с #1 (если narrative в LLM, сводка тоже становится её работой).

## 6. OKR-проекция: правильный паттерн или «костыль»? ❓ архитектурный

**Где сейчас.** Решение №5 в `04_decisions.md`: OKR-навигация — **через секцию «По OKR» в `01_metrics.md`**, не через переименование domain-файлов в `okr1.md`/`okr2.md`. Аргументация: OKR — цель, не бизнес-домен; одна OKR агрегирует метрики из нескольких доменов.

**Что под вопросом** (родился из обсуждения с Виктором 2026-04-29): паттерн «OKR-проекция в витрине + canonical domain-файлы» работает на одном пилоте (первом клиенте). Если у следующих клиентов OKR-структура будет принципиально другой — может оказаться, что мы зафиксировали неоптимальный паттерн.

**Решение.** Проверить на 2-3 клиентах. Если паттерн выдержит — фиксируем в каноне жёстко. Если нет — пересмотр в Этап С.

## 7. Универсальный xlsx-read skill 🔮 будущее

**Где сейчас.** Per-client extractor (`ssp_okr1.py` для первого клиента). Хардкод координат под конкретный лист.

**Что под вопросом.** Если у нескольких клиентов одинаковая структура xlsx (например, все используют один и тот же шаблон от индустриального консультанта) — можно ли вынести extractor в общий `framework/skills/xlsx-read/` с конфигом-маппингом, а не писать с нуля каждому?

**Решение откладывается.** До второго клиента — увидим, насколько похожа структура. Если разная — оставляем per-client. Если одинаковая — выносим.

## 8. Связка metrics ↔ meeting / metrics ↔ initiatives 🔮 кросс-вертикальное

**Где сейчас.** Тест-вопросы первого клиента №2 («бриф к встрече с куратором customer success») и №3 («связка метрика↔инициатива») требуют выхода за рамки metrics — связи с meeting cycle и contracts (BuildIn).

**Что под вопросом.** Архитектурно это — связка вертикалей через горизонтальные слои (по решению №4 в `04_decisions.md`: «зависимости через стабильные ID, не прямые вызовы»). Но конкретных контрактов между metrics и meeting **не написано**.

**Решение.** Этап С, после успешного Б.2. Сначала — что экспортирует metrics (стабильные `metric_<id>` через слой памяти), потом — как meeting cycle их импортирует.

## 9. L3-нормализация на нашей стороне vs L1 (xlsx клиента как источник правды) ⚠️ продуктовый

**Где сейчас.** Архитектура L1: клиентский xlsx — источник правды. На каждого клиента пишем per-client extractor с `known_issues`, паспорта метрик с `regions`/`antonyms`/`direction` под кривости источника. Работает, но требует ручной инженерной работы на каждого нового клиента.

**Что под вопросом** (поднял Виктор 2026-04-30, в конце вечерней+ночной сессии): не корректнее ли создать **наш канонический формат** (нормализованный xlsx, CSV, или БД), куда инженер интегрирует каждый новый клиентский xlsx через ETL? Тогда оркестратор работает с нашим каноном, не с клиентским трешем. Воспроизводимо, тестируемо, паспорта и расчёты простые.

**Уровни зрелости вертикали (мой набросок):**

| L | Источник правды | Работа на клиента | Когда оптимально |
|---|---|---|---|
| **L1 (текущий)** | xlsx клиента как есть | per-client extractor + `known_issues` + паспорта под кривость источника | 1-3 клиента, собираем паттерны |
| **L2 (промежуточный)** | xlsx клиента как есть | typed templates (типовые формы клиентских xlsx) + параметризованный extractor + минимальный per-client config | 5-10 клиентов, паттерны накопились |
| **L3 (вариант Виктора)** | наш канонический формат | инженер пишет ETL «xlsx клиента → наш канон» | 20+ клиентов или продуктовое решение |

**Аргументы за L3:**
- Один формат, простой оркестратор, паспорта без `regions`/`known_issues`.
- Можно строить аналитику, тренды, кросс-domain срезы на чистой базе.
- Изоляция от клиентского хаоса (изменился xlsx → правится только ETL).

**Аргументы против L3 в текущей продуктовой модели:**
- **Нарушает принцип «данные у клиента, методология у нас»** ([00_product.md](../../00_product.md)). Это продуктовое решение, не инженерное.
- Расхождение источников: CEO смотрит в xlsx, AI — в нашу копию. Если ETL отстал — доверие падает мгновенно.
- ETL не дешевле extractor — та же инженерная работа, просто на другом уровне абстракции (mapping в наш канон вместо координат в клиентском xlsx). На малом объёме клиентов экономически не выгодно.
- Семантика теряется: «отток до вывода / 90d / младенческий» у первого клиента — это разные углы взгляда; в каноне они станут одной метрикой с тегами или разными строками — выбор за нас, без знания CEO. Канонизация нивелирует разнообразие.

**Промежуточный вариант — мысленный эксперимент перед вторым клиентом.** Взять копию SSP первого клиента и сделать в sandbox «идеальный нормализованный domain-файл»: что бы заполнил, если бы данные были чистые. Сравнить с текущим — увидеть, что именно обходим в `known_issues`. Не переносить в продакшн, использовать как материал для будущего канона.

**Что делать сейчас.** Идём по траектории L1 → L2. Не прыгаем сразу в L3.

**Триггеры пересмотра:**
1. Достижение 5+ клиентов metrics с разнообразными xlsx — увидим долю общих паттернов.
2. Или раньше — если продуктово решим пересмотреть принцип «данные у клиента» (под подмножество клиентов, которые сами просят и не видят в этом проблему).

**Связь с другими вопросами:**
- #1 (narrative.py vs LLM) — частично снимается в L3, потому что на чистом каноне narrative проще собирать LLM-ом.
- #7 (универсальный xlsx-read skill) — это шаг от L1 к L2.

**Решение откладывается.** Возвращаемся к этому вопросу через 2-3 клиента или при явном продуктовом запросе.

## Что спросить у первого клиента до Б.2

Список **в дополнение** к [`intake-form.md`](intake-form.md). Используется при живой встрече с клиентом / в чате, когда у клиента уже есть заполненная intake-form, и мы добиваем последние неясности.

1. **Подтвердить OKR-структуру.** В intake-form (Часть 1) клиент написал OKR1/2/3. Уточнить вживую: какой OKR берём в Б.2, какие метрики из этого OKR — target metrics CEO (на которые смотрит регулярно), какие — служебные/контекстные.

2. **Дубль метрик KR1 (вопрос #3 выше).** Если в источнике клиента есть две метрики, дающие одинаковые числа за период (как у первого клиента: `metric_churn_90d_mrr_monthly` ≡ `metric_churn_ltv_90d_mrr`) — это одна метрика, записанная дважды для удобства витрины, или две семантически разные? Проверять на этапе аудита синонимов.

3. **Аудит синонимов вживую.** В sandbox несколько метрик помечены «требует подтверждения» (см. `customer.md`, `sales.md`, `operations.md`). Зачитать клиенту его собственные обороты из transcripts: «ты так это называешь?». Где не подтвердилось — пометка `synonyms: ["в речи не зафиксировано"]`.

4. **Антисинонимы (омонимы).** В intake-form (Часть 5.2) клиент перечислил омонимы — слова, звучащие похоже, но означающие разное. Уточнить вживую полноту списка: какие ещё пары встречаются в его речи на встречах за последние 3-6 месяцев?

5. **`plan_required` по каждой target metric.** В intake-form (Часть 2.3) клиент пометил, у каких метрик есть план. Уточнить: при вопросе «топ-N разрывов план/факт» — какие из метрик-без-плана он точно хочет видеть в ответе как «не считается», какие — игнорировать?

6. **Runtime-окружение.** В intake-form (Часть 8) клиент пометил Cowork / Claude Code / иное. Уточнить: где именно лежит xlsx (Drive for Desktop путь или локальная копия), есть ли права на запуск Python-скриптов.

После ответов — финализировать domain-файлы, extractor-координаты и запускать Cowork-чек-лист (см. [`../../skills/metrics-analysis/orchestrator-metrics.md`](../../skills/metrics-analysis/orchestrator-metrics.md)).
