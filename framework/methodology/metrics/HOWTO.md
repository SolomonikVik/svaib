# Как работает metrics-вертикаль: первый слой

Документ объясняет человеческим языком, что происходит, когда CEO задаёт помощнику вопрос про метрику бизнеса **в первом слое** вертикали. Без YAML-портянок и Python-кода. Цель — чтобы любой человек, включая клиента, понял сценарий за 10 минут.

Что такое «первый слой» и где его границы — [`first-layer.md`](first-layer.md). Полная архитектура (база, не переписывается под первый слой) — [`architecture.md`](architecture.md). Развёртывание у нового клиента — [`rollout.md`](rollout.md).

---

## Зачем эта вертикаль

CEO живёт в xlsx и Google Sheets. Его метрики — большая таблица: лист с несколькими блоками, своими названиями, иногда с `#DIV/0!` и пустыми периодами. Когда CEO хочет «посмотреть число» — он либо открывает таблицу руками, либо просит финдира собрать сводку.

Цель metrics-вертикали: **CEO задаёт помощнику вопрос своими словами и получает воспроизводимый ответ.** Не «вот цифра, прими на веру», а «вот цифра, вот источник, вот когда читалось».

Принцип всей вертикали: **помощник не считает в голове.** Любое число приходит из источника, который можно открыть и перепроверить.

Первый слой — самая простая реализация этого принципа: помощник не считает вообще. Он **ищет в xlsx число по каноническому имени колонки**.

## Что у клиента появляется в первом слое

При подключении первого слоя у CEO появляется минимум:

```
metrics/
├── canonical_metrics.md     ← список базовых target metrics:
│                              каноническое название + бизнес-смысл + как считается
└── source/
    └── <боевой xlsx>        ← главный xlsx CEO, в нём добавлена
                              отдельная колонка с каноническими названиями
```

Никаких `01_metrics.md` витрины, никаких domain-файлов, никаких extractor-скриптов, никаких narrative — это всё второй слой, появляется потом по живому триггеру.

### `canonical_metrics.md`

Список базовых target metrics CEO. На каждую:

- **Каноническое название** — стабильная фраза, которую CEO готов произносить вслух и которую он вписывает колонкой в xlsx.
- **Бизнес-смысл** — что метрика измеряет, языком CEO.
- **Как считается** — правило получения числа из реальности бизнеса (а не координата ячейки).

Чего там **нет**: ID метрики, синонимы, антонимы, направление (хорошо когда вверх или вниз), единицы, владелец, версия, формула через датасеты. Всё это — поля паспорта; паспорт появляется во втором слое, когда возникает живой запрос на «разрыв план/факт», классификацию «горит/не горит» и тому подобное.

Полный формат и правила — [`first-layer.md`](first-layer.md), раздел «Формат `canonical_metrics.md`».

### Колонка в xlsx

В главном листе xlsx CEO появляется отдельная колонка с каноническими названиями метрик. Это единственная твёрдая привязка помощника к таблице. Привязка не по тексту описания (он меняется), не по номеру строки (строки сдвигаются), а по содержимому канонической колонки.

Правило: если поле в колонке пустое — **метрика для помощника не существует**. Это сознательно: не описано каноническим именем → не наш слой.

Колонка может быть скрытой. Место выбирается так, чтобы добавление/удаление обычных строк её не ломало.

## Что происходит, когда CEO задаёт вопрос

Пример (на абстракциях, без приватной фактуры):

> «Сколько у нас вышло {каноническое имя метрики} за март?»

Помощник действует так:

### Шаг 1 — сверить, что метрика есть в нашем списке

Помощник читает `canonical_metrics.md`. Находит метрику с этим каноническим именем. Если не находит — отвечает CEO «такой метрики у нас в каноне нет», не выдумывает.

### Шаг 2 — открыть xlsx

Помощник открывает главный xlsx через `document-skills:xlsx` (или эквивалентный read-only инструмент). Не считает в голове — даже если таблица маленькая.

### Шаг 3 — найти каноническую колонку и нужную строку

В главном листе находит колонку с каноническими названиями метрик. В этой колонке ищет строку, где значение совпадает с искомым именем посимвольно. Это и есть строка нужной метрики.

### Шаг 4 — достать число за период

В заголовках xlsx находит колонку запрошенного периода («март» → колонка «март», «март 2026», «03.2026» — по факту, как ведёт клиент). На пересечении строки и колонки лежит число.

### Шаг 5 — вернуть ответ

Помощник возвращает CEO число + короткую метку: «{каноническое имя}, {период}, по данным {имя xlsx}, прочитано {timestamp}».

Всё. Без сравнения с планом, без классификации «хорошо/плохо», без сравнения с прошлым месяцем, без сборки narrative, без выхода в несколько метрик сразу.

## Зачем такое упрощение

Почему не сразу полную архитектуру с паспортами, direction, маршрутами, narrative — а ровно эту узкую плёнку?

- **Стартовая стоимость поднимается до минимума.** Для подключения нужны: список из 5–10 метрик с человеческим описанием + одна колонка в xlsx. Это переносимо на любого первого клиента без специальной техники.
- **Каноническое имя — это первый артефакт, который должен договорить с клиентом любой rollout.** Без него вся последующая работа второго слоя (паспорта, маршруты, синонимы) — переписывание поверх случайных названий.
- **На самом простом потоке (вернуть число) не нужен ни extractor, ни narrative.** Они нужны там, где есть классификация, ранжирование, сравнение — то есть в реальном бизнес-вопросе «что у меня горит». До такого вопроса CEO часто живёт месяцами на запросах «дай число».
- **Воспроизводимость держится на двух вещах:** канон не врёт о метрике, источник видно глазами. Этого хватает, чтобы доверие к помощнику росло — а не падало от каждого выдуманного числа.

## Что будет дальше — второй слой

Первый слой не пытается заменить полную архитектуру; он её **подготавливает**. Когда у CEO появится живой запрос за пределы «достать число», в дело вступает второй слой:

| Запрос CEO | Что добавляется |
|---|---|
| «А план как? Где разрыв?» | паспорт метрики с `plan_required` и `direction` |
| «Что у меня горит?» | compound route + классификация + ранжирование |
| «Как идём по OKR3?» | OKR-проекция в витрине |
| Помощник тупит на оборотах CEO | синонимы и антонимы в паспортах |
| В xlsx начало ломать (DIV/0, объединённые ячейки, врущие метки) | `known_issues` + per-client extractor |

Каждый триггер двигает второй слой по одному элементу, а не «всё разом». Полный набор — [`architecture.md`](architecture.md), пометки `[СЕЙЧАС]`/`[ПОЗЖЕ]`; границы перехода — [`first-layer.md`](first-layer.md), раздел «Триггеры перехода ко второму слою».

## Где жить уроки команды

Опыт развёртывания первого слоя у конкретного клиента (что нашли в его речи, какую формулировку он принял для канонического имени, какие косяки источника всплыли в smoke-test) — остаётся у клиента в его частной папке. Если урок переносимый на следующих клиентов — он поднимается в канон:

- В правила формата → [`first-layer.md`](first-layer.md).
- В шаги rollout → [`rollout.md`](rollout.md).
- В шаблон → `framework/scaffold/05_metrics/canonical_metrics.md`.

Принцип тот же, что в остальных вертикалях: у клиента — конкретика, у нас — методология и шаблоны.
