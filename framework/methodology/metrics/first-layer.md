# Первый слой metrics-вертикали

## Кратко

Минимум, который надо поднять у первого клиента, чтобы Second AI Brain научился отвечать на простой вопрос про метрику числом из xlsx. Без паспортов, без direction, без extractor-скрипта, без narrative. Только три артефакта: каноническое имя метрики у клиента, колонка с этим именем в xlsx, помощник, который умеет по колонке достать число.

Этот слой — точка входа в [`architecture.md`](architecture.md). Базовая 6-слойная архитектура сохраняется как ориентир для второго слоя; здесь — что из неё реально включается у клиента в день 1.

**Работа с файлом:** procedure — обновлять при изменении DoD первого слоя или техники подключения. Меняется редко. Что НЕ сюда: расширенная архитектура (→ [`architecture.md`](architecture.md)), playbook первого слоя пошагово (→ [`rollout.md`](rollout.md)), пользовательский сценарий (→ [`HOWTO.md`](HOWTO.md)).

## Рамка

Metrics-вертикаль — управленческий цикл #4 (см. [`../../ontology/management_cycles.md`](../../ontology/management_cycles.md)). У клиента она разворачивается **слоями**, не сразу полным комплектом.

| Слой | Что входит | Когда включаем |
|---|---|---|
| **Первый слой** | каноническое имя метрики + колонка в xlsx + помощник по колонке | у каждого первого клиента, с него стартуем |
| Второй слой | паспорт метрики (id, version, direction, synonyms, antonyms, plan_required), domain-файлы, per-client extractor, narrative | по живому триггеру: «разрыв план/факт», речевые обороты, OKR-срез |
| Третий слой | регулярный loop, история по месяцам, недельный ритуал, eval, snapshot | когда первые два слоя стабильно работают |

Первый слой — это **продуктовая граница, не пилотная**. Любой новый клиент стартует отсюда; полный комплект из [`architecture.md`](architecture.md) разворачивается над ним по триггерам.

## Definition of Done «первый слой подключён»

Слой считается включённым, когда выполнены все три условия:

1. **Каноническое описание метрик у клиента.** В scaffold клиента лежит файл `canonical_metrics.md` со списком базовых target metrics: каноническое название + бизнес-смысл + как считается. **Без ID, синонимов, направлений, версий, владельцев, единиц.** Канонические названия — единые во всех файлах компании.
2. **Базовый помощник умеет вытащить число.** На вопрос про метрику из этого набора помощник идёт в xlsx клиента через колонку с каноническим названием (правило из канона: «Канонический ID метрики через отдельную колонку SSP» — у первого живого клиента это решение от 2026-05-01) и возвращает число за запрошенный период. **Без narrative, без compound route, без direction-классификации, без extractor-скрипта.**
3. **Методология описывает это как воспроизводимый процесс** для любого первого клиента. Не один разовый кейс; шаг-в-шаг — в [`rollout.md`](rollout.md).

DoD выполнен ⇒ слой работает. Расширение — только по живому триггеру, не «на всякий случай».

## Что не входит в первый слой

Сознательно не пытаемся охватить, даже если методология умеет это во втором слое:

- **Полный паспорт метрики** (`metric_<id>`, `version`, `direction`, `synonyms`, `antonyms`, `plan_required`, `owner`, `formula_human`, `expression`) — второй слой.
- **Domain-файлы** (`finance.md`, `sales.md`, `operations.md`, `people.md`) с разделами `datasets / relationships / metrics / routes` — второй слой.
- **Per-client extractor** с захардкоженными координатами и `regions` — второй слой. На первом слое extractor не пишется: достаточно поиска по колонке.
- **Narrative composer** (red/win/ok классификация, |Δ%| ранжирование) — второй слой; вопрос Python vs LLM см. [`open-questions.md`](open-questions.md) №1.
- **Compound route**, OKR-проекция в витрине, «что у меня горит» — второй слой.
- **Snapshot, eval-набор, observability** — третий слой ([`architecture.md`](architecture.md), пометки `[ПОЗЖЕ]`).
- **Регулярный loop** (история по месяцам, weekly ритуал, недельный бриф) — третий слой.

## Формат `canonical_metrics.md`

Продуктовый шаблон файла, который ложится в `metrics/canonical_metrics.md` у клиента.

### Назначение

Базовый слой target metrics CEO. Каноническое имя — единая привязка между речью CEO, xlsx, помощником и любыми другими файлами компании. Файл — первое и пока единственное описание метрик в scaffold клиента; над ним позже надстраивается витрина + domain-файлы.

### Структура

```markdown
# Канонические метрики {компания}

{Одна-две строки контекста: где живут эти метрики, кто их ведёт, на чём считается C-level.}

Канонические названия — единые во всех файлах компании. В xlsx — колонка с этим названием; во всех других местах — выбор из dropdown, ссылающегося на этот список.

Не используется здесь: синонимы, направления, единицы, версии, владельцы — навешиваются помощником отдельно.

## Метрики

### {Каноническое название метрики 1}
**Бизнес-смысл:** {одно-два предложения, как объясняет CEO без жаргона}.
**Как считается:** {одно-два предложения, какое правило/алгоритм даёт число; ссылается на сущности бизнеса, не на координаты xlsx}.

### {Каноническое название метрики 2}
...

---

## Что НЕ вошло в этот слой

- {подразбиения второго порядка — операционная аналитика}
- {нарастающие итоги — производные, помощник посчитает сам}
- {нестабильные/предварительные метрики}
- {история — отдельный шаг при подключении}
```

### Правила формата

- **Каноническое имя — стабильная фраза**, в которую CEO готов записать колонку xlsx и которую он готов произносить вслух. Не сокращение, не код. Если в xlsx колонка пуста — метрика для помощника **не существует** (см. решение «Канонический ID метрики через отдельную колонку SSP»).
- **Бизнес-смысл** — на языке CEO, без терминов методологии. «Клиенты, которые после заключения контракта ушли до того, как у них реально начал тикать биллинг» — норм. «Churn rate первой когорты» — нет.
- **Как считается** — правило получения числа из реальности бизнеса. Алгоритм («сумма MRR контрактов, расторгнутых до постановки на MRR, за отчётный месяц»), не координата ячейки.
- **Что НЕ кладём:** ID метрики (`metric_<id>`), синонимы из речи, антонимы, `direction`, единицы, владельца, версию, формулу через датасеты. Всё это — поля паспорта; паспорт появляется во втором слое.
- **Размер:** 5–10 базовых target metrics на старте. Если получается больше 12 — у клиента в голове два слоя, отрежьте к target.
- **Раздел «Что НЕ вошло»** — обязательный. Защищает от вопроса «а почему этого нет» через 3 месяца.

## Помощник в первом слое

Что делает помощник, когда CEO спрашивает про метрику из канонического набора (полный сценарий — [`HOWTO.md`](HOWTO.md)):

1. Прочитал `canonical_metrics.md` — сверил, что метрика есть в списке.
2. Открыл xlsx клиента через `document-skills:xlsx` (или эквивалентный read-only доступ).
3. Нашёл **колонку** с каноническими названиями метрик (одна, заданная клиентом).
4. Нашёл **строку**, где значение этой колонки совпадает с каноническим именем из вопроса CEO.
5. Достал число за запрошенный период (колонка периода известна из xlsx — заголовок).
6. Вернул CEO: число + краткая метка «{каноническое имя}, {период}, по данным {имя xlsx}, прочитано {timestamp}».

Что помощник **не делает** в первом слое: не классифицирует «хорошо/плохо», не сравнивает с планом, не сравнивает с прошлым периодом, не пишет narrative, не ходит в несколько метрик сразу, не строит compound route. Любой шаг сверх «вернул число» — выход за границы первого слоя.

Технически:
- LLM ходит в xlsx через инструмент. **Не считает в голове** — даже если таблица маленькая, см. главный принцип [`architecture.md`](architecture.md).
- На первом слое — нет per-client extractor. Поиск по канонической колонке — универсальный паттерн, реализуется через `document-skills:xlsx` без хардкода координат.

## Триггеры перехода ко второму слою

Слой расширяется, когда **появилась живая работа**, а не «потому что у нас есть методология».

| Запрос CEO | Что добавляем |
|---|---|
| «А план как? Где разрыв?» | `plan_required` в паспорте + `direction` |
| «Что у меня горит?», топ-N | compound route + narrative + ранжирование |
| «По OKR3 как идём?» | OKR-проекция в витрине |
| Помощник тупит на речевых оборотах CEO | аудит синонимов/антисинонимов в паспортах |
| Появились косяки источника (DIV/0, объединённые ячейки, врущие метки) | `known_issues` + per-client extractor |
| Метрика стала зависеть от другой вертикали (meeting/contracts) | стабильные ID + горизонтальный контракт |

Каждый триггер двигает один-два элемента второго слоя, не «всё разом». Дальше — по канону [`architecture.md`](architecture.md) и [`rollout.md`](rollout.md), второй и третий слои.

## Связь с архитектурой

Первый слой реализует только часть слоёв из [`architecture.md`](architecture.md):

- **Слой 1 (Клиентские таблицы)** — read-only, как описано в каноне.
- **Слой 4 (Семантический слой)** — включён в минимальной форме: `canonical_metrics.md` вместо `01_metrics.md` + domain-файлов. Это сознательное упрощение «семантического слоя» до плоского списка target metrics; нарезка по доменам, паспорта, маршруты вводятся вторым слоем.
- **Слой 6 (Обработка запроса)** — только Ветка А в редуцированной форме: «найти метрику по имени → достать число». Без классификатора маршрутов, без планирования с валидацией, без compound.

Snapshot, canonical layer, ритуалы как формат, observability, eval — не входят в первый слой.
