# Metrics: спецификация формата файлов

Source of truth для формата metrics-файлов клиента (`business-metrics.md`, `{domain}-metrics.md`) — общие правила формата + раскладка базового и функциональных domain-файлов. Шаблоны в `scaffold/05_metrics/` собираются по этой спеке.

Metrics-файлы — это семантический слой вертикали: описывают бизнес-смысл метрик (что метрика значит, как считается, в каких единицах, куда растёт). Без него LLM гадает — у одного клиента `revenue` это отгрузка, у другого оплата, у третьего признание выручки.

Связанные артефакты:
- [`../../scaffold/05_metrics/business-metrics.md`](../../scaffold/05_metrics/business-metrics.md) — шаблон базового файла метрик бизнеса
- [`../../skills/metrics-analysis/business-metrics-intake.md`](../../skills/metrics-analysis/business-metrics-intake.md) — промпт-помощник заполнения `business-metrics.md` с CEO
- **Ориентир:** [Open Semantic Interchange](https://open-semantic-interchange.org/) — внешний reference по semantic models, не источник контракта svaib.

## 🔵 Общие правила для всех metrics-файлов

### Имя метрики = её ID

Каноническое имя метрики, записанное заголовком `### {имя}` в metrics-файле, одновременно является её идентификатором.

Требования к имени:
- **Нормализованность** — Имя метрики = как её реально называют внутри компании (речь, встречи). Индустриальный канон (`MRR`, `Валовая прибыль`, `Конверсия`) — рекомендация; при расхождении побеждает внутренняя норма, чтобы CEO и команда узнавали свою метрику. 
- **Краткость** — имя называет метрику, а не описывает её.
- **Однозначность** — по имени должно быть ясно, о какой именно метрике идёт речь в контексте клиента.
- **Стабильность** — меняется формула, имя не меняется. Меняется сама управленческая сущность — это новая метрика.
- **Уникальность** — имя уникально в пределах всех metrics-файлов одного клиента.

### Базовые поля метрики

В любом metrics-файле каждая метрика описывается пятью полями; при нескольких источниках добавляется шестое — `**Источник:**`.

> ### {Каноническое имя}
> **Бизнес-смысл:** {что метрика значит для бизнеса}
> **Как считается:** {правило расчёта, в котором явно указаны временная логика метрики и способ её вычисления; например: «количество клиентов, у которых на конец отчётного месяца статус активный»}
> **Единица:** {млн руб / % / шт}
> **Направление:** {рост — хорошо / рост — плохо / нейтрально}
> **Источник:** `{source_id}` — только если в файле несколько источников (см. «Контракт привязки к источнику»).

**Пример:**
> ### Активные клиенты
> **Бизнес-смысл:** Клиентская база, которая реально приносит деньги в текущем периоде. Показывает, сколько клиентов «живых» — на ком держится выручка прямо сейчас.
> **Как считается:** Число уникальных клиентов, по которым за календарный месяц прошла хотя бы одна оплаченная реализация. Срез фиксируется на последний день месяца.
> **Единица:** шт
> **Направление:** рост — хорошо

### Контракт привязки к источнику

Metrics-файлы делятся по бизнес-смыслу, не по источнику. Один файл может покрывать метрики из разных xlsx / Google Sheets / листов / API / SQL-источников. Привязка живёт в самом metrics-файле — не в routes.

**Один источник** — в начале файла секция `## Источник`:

```md
## Источник

**Источник:** `{file}.xlsx`, лист «{sheet}».
```

Поле `**Источник:**` у метрик в этом режиме не указывается — все они наследуют общий источник.

**Несколько источников** — в начале файла секция `## Источники`, внутри каждый источник = подраздел `### {source_id}` (snake_case, ASCII, уникальный в пределах файла):

```md
## Источники

### source_finance

**Источник:** `finance.xlsx`, лист «P&L».

### source_sales

**Источник:** `sales.xlsx`, лист «Pipeline».
```

В этом режиме каждая метрика обязана указать свой источник через `**Источник:** {source_id}`:

> ### Маржа
> **Бизнес-смысл:** ...
> **Как считается:** ...
> **Единица:** %
> **Направление:** рост — хорошо
> **Источник:** `source_finance`

### Рекомендация по обустройству источника (Google Sheets)

Когда источник — Google Sheets, рабочий паттерн:

- Отдельный служебный лист со списком канонических имён метрик.
- В рабочих листах колонка с именем метрики — **dropdown через data validation**, ссылается на служебный лист. Клиент не может вписать произвольное имя — только выбрать из списка.
- Через `ImportRange` список канон-имён распространяется на связанные файлы компании.

Один канонический список имён на всю компанию. Опционально, но сильно усиливает контракт: исключает опечатки, гарантирует совпадение имени в metrics-файле и в источнике.

## 🔵 Базовый файл `business-metrics.md`

Базовый metrics-файл компании. Один на компанию — ключевые бизнес-метрики, по которым отслеживается движение бизнеса в целом. 

### Структура секций

1. **Источник** — секция `## Источник` (один источник) или `## Источники` с подразделами `### {source_id}` (несколько источников); см. «Контракт привязки к источнику» выше.
2. **Метрики** — каждая на `### {каноническое имя}` (заголовок третьего уровня — он же ID). Два варианта раскладки, **взаимоисключающие**:
   - **Без групп.** Один контейнер `## Метрики`, внутри подряд `### {имя}` по каждой метрике. Используется, когда у CEO нет естественных блоков target metrics.
   - **С группами.** Каждая группа — отдельная секция `## {имя группы}` (Продажи / Клиенты / Финэффективность и т.п.), внутри `### {имя метрики}`.

   В обоих вариантах метрика всегда оформляется заголовком третьего уровня: `### {имя}`.
### Группы как точка стыковки с функциональными domain-файлами

Группы в базовом файле могут опционально ссылаться на функциональные domain-файлы, когда те появляются:

> ## Финансы
>
> Развёрнутая раскладка финансовых метрик — [finance-metrics.md](finance-metrics.md).
>
> ### {метрика 1}
> ...

Сама группа в `business-metrics.md` остаётся краткой верхнеуровневой навигацией.

### OKR

OKR живут в `goal.md` как цели. Метрики, которыми измеряется выполнение OKR, лежат в `business-metrics.md` как обычные метрики; одна OKR может агрегировать метрики из разных доменов.

## 🔵 Функциональные файлы метрик `{domain}-metrics.md`

То же, что `business-metrics.md`, но по одной функции и с более подробной семантикой. Нарезка — по бизнес-смыслу (продажи / удержание / операции / финансы), не по источнику (см. «Контракт привязки к источнику»).

### Канонические имена domain-файлов

Каноническое имя файла — `{domain}-metrics.md`. Допустимы только эти имена (SOT):

| Файл | Что покрывает |
| --- | --- |
| `business-metrics.md` | базовый файл, ключевые метрики бизнеса. Один на компанию, точка входа в metrics-вертикаль |
| `finance-metrics.md` | прибыль, маржа, расходы, cash, рентабельность, P&L |
| `sales-metrics.md` | выручка, pipeline, конверсии, средний чек, LTV* |
| `operations-metrics.md` | delivery, качество, сроки, capacity, SLA |
| `people-metrics.md` | команда, найм, удержание, производительность |
| `customer-metrics.md` | retention, churn, LTV, когортный анализ, customer success |
| `marketing-metrics.md` | каналы, CAC по каналам, контент-воронка, атрибуция |
| `product-metrics.md` | DAU/MAU, adoption, feature usage, метрики использования цифрового продукта |

\* LTV пока в `sales-metrics.md`, переезжает в `customer-metrics.md`, когда customer success становится отдельным контуром.
