# Metrics: спецификация формата файлов

## Кратко

Source of truth для формата metrics-файлов клиента (`business-metrics.md`, `{domain}-metrics.md`). Описывает общие правила формата + раскладку L1 (базовый файл) + раскладку L2 (функциональные domain-файлы). Если scaffold удалят, формат восстанавливается отсюда.

**Работа с файлом:** specification — обновлять при изменении формата или контракта привязки к источнику. Что сюда: правила формата, поля метрики, контракт привязки, рекомендации по обустройству источника. Что НЕ сюда: архитектура вертикали и канон имён → [architecture.md](architecture.md); продуктовое решение о раскладке metrics → [../../04_decisions.md](../../04_decisions.md) №5.

## Общие правила для всех metrics-файлов

### Имя метрики = её ID

Каноническое имя метрики, записанное заголовком `### {имя}` в metrics-файле — её ID. Никаких отдельных `metric_<id>` полей.

Свойства имени-как-ID:

- **Произносимая фраза CEO** — то, что он готов сказать вслух и вписать в xlsx.
- **Стабильность** — меняется формула, имя не меняется. Меняется суть — это новая метрика, не та же.
- **Уникальность** в пределах всех metrics-файлов одного клиента.

Имя меняется только при явном переосмыслении метрики самим CEO (фиксируется в `decisions.md` клиента), не для косметики.

### Базовые поля метрики

В любом metrics-файле каждая метрика описывается пятью полями:

> ### {Каноническое имя}
> **Бизнес-смысл:** {речью CEO, что метрика значит для бизнеса}
> **Как считается:** {правило на языке бизнеса, не на координатах xlsx}
> **Единица / период:** {млн руб MRR / месяц}
> **Направление:** {рост — хорошо / рост — плохо / нейтрально}

Без направления и единицы помощник не может думающе подать число: «−0.5» не значит «просело на 0.5 млн MRR». Это минимум на L1; на L2 поля паспорта расширяются (см. ниже).

**Чего НЕ добавляем в базовые поля.** Синонимы (CEO говорит каноном — имя метрики и есть язык речи), версия (есть git), владелец (на старте один CEO = единый владелец), формула в SQL/expression (это L2).

### Контракт привязки к источнику

В шапке metrics-файла фиксируется явный блок «Источник»:

> **Источник:** `{file}.xlsx`, лист «{sheet}», колонка с каноническим именем метрики: «{имя как клиент назвал}».

Имя колонки выбирает клиент — `Метрика`, `KPI`, `Название метрики`, любое. В каждой строке-метрике этой колонки помещено каноническое имя из metrics-файла.

**Правило существования метрики.** Если в строке источника поле канон-имени пусто — для помощника метрики не существует. Имя в колонке — единственная твёрдая привязка между metrics-файлом и xlsx.

**Fallback на поиск по содержимому.** Если помощник по имени колонки не нашёл (клиент переименовал/сдвинул), он проходит первые N колонок листа и ищет ту, чьи значения пересекаются с каноническими именами из metrics-файла. Нашёл — использует, в trace помечает «контракт устарел: имя колонки сейчас = X, в metrics-файле = Y», подсказывает обновить шапку. Не нашёл — отказ с явным сообщением.

Это правило поведения оркестратора (системный промпт), не правило формата файла; здесь описано, потому что без него контракт хрупкий.

### Рекомендация по обустройству источника (Google Sheets)

Когда источник — Google Sheets, рабочий паттерн:

- Отдельный служебный лист со списком канонических имён метрик.
- В рабочих листах колонка с именем метрики — **dropdown через data validation**, ссылается на служебный лист. Клиент не может вписать произвольное имя — только выбрать из списка.
- Через `ImportRange` список канон-имён распространяется на связанные файлы компании.

Один канонический список имён на всю компанию. Опционально, но сильно усиливает контракт: исключает опечатки, гарантирует совпадение имени в metrics-файле и в источнике.

### Что НЕ кладём в metrics-файл

- **Текущие значения метрик** — только в источнике (`source/*.xlsx`). LLM не дублирует цифры в md (тихий рассинхрон).
- **OKR** — это цели, не измерение. Живут в стратегическом контуре клиента (`02_strategy/`).
- **На L1** — маршруты-запросы (они в L2, секция `routes`).
- **На L1** — поля паспорта L2: версия формулы, владелец, синонимы, `datasets`, `relationships`.

## L1 — базовый файл `business-metrics.md`

Базовый metrics-файл вертикали. У каждого клиента ровно один — точка входа CEO в собственные target metrics. На нём работает первый слой metrics-помощника: AI-аналитик базовых target metrics CEO («что у CEO в голове как весь бизнес одной страницей»).

### Структура секций

1. **Контекст бизнеса** — 1–2 абзаца на речи CEO: на чём считается C-level, что значит «весь бизнес» для этого CEO, какие у бизнеса есть особенности (например, считаем кэшфло, не реализацию).
2. **Источник + контракт привязки** — один блок-цитата (см. «Контракт привязки к источнику» выше). В L1 — один xlsx-лист. Если у CEO target metrics на нескольких листах, это сигнал разворачивать функциональные domain-файлы L2.
3. **Метрики** — список с подзаголовками `### {имя}`. Опционально сгруппирован по `## Блок «{имя}»`, если у CEO есть естественные блоки (Продажи / Отток / Финэффективность и т.п.).
4. **Что НЕ вошло** — явный список: подразбиения второго порядка, нарастающие итоги, история, нестабильные метрики. Чтобы помощник не пытался отвечать про не-target.

### Группы как точка стыковки с L2

Группы в L1 могут опционально ссылаться на функциональные domain-файлы, когда те появляются:

> ## Блок «Финансы»
>
> Развёрнутая раскладка финансовых метрик — [finance-metrics.md](finance-metrics.md).
>
> ### {метрика 1}
> ...

Сама группа в `business-metrics.md` остаётся как точка входа CEO «верхним взглядом»; детальный паспорт метрики живёт в L2.

## L2 — функциональный domain-файл `{domain}-metrics.md`

Канон 9 имён закреплён в [architecture.md § «Канонические имена domain-файлов»](architecture.md):

- **Базовый:** `business-metrics.md` (см. L1 выше).
- **Минимальный функциональный канон:** `finance-metrics.md` / `sales-metrics.md` / `operations-metrics.md` / `people-metrics.md`.
- **Расширение по триггеру:** `customer-metrics.md` / `marketing-metrics.md` / `product-metrics.md` / `strategy-metrics.md`.

Структура секций функционального файла (по решению №5 в [04_decisions.md](../../04_decisions.md), адаптация Open Semantic Interchange):

- `datasets` — какие источники (xlsx/sheet) и поля
- `relationships` — связи между источниками
- `metrics` — паспорта метрик (базовые пять полей + поля паспорта: версия формулы, синонимы, expression и др.)
- `routes` — маршруты-запросы под типовые вопросы CEO

**Полная спека L2 — отдельной задачей** (Шаг 2 пересмотра методологии metrics). Сейчас здесь — точка стыковки: канон имён + ссылка на решение №5 + правило «при разворачивании метрика из L1 переезжает в L2 с обогащением, имя не меняется».

## Эволюция L1 → L2

Когда у CEO появляется функциональный домен как отдельный фокус («теперь подключай финансы», «теперь маркетинг»):

1. Создаётся `{domain}-metrics.md` рядом с `business-metrics.md`.
2. Метрики, относящиеся к этому домену, переезжают из `business-metrics.md` в `{domain}-metrics.md`, обогащаясь полями паспорта L2.
3. В `business-metrics.md` блок группы остаётся как верхний взгляд; на месте метрик — ссылка `→ паспорт в [{domain}-metrics.md]({domain}-metrics.md)`.

**Имя метрики при переезде не меняется.** Source of truth для метрики после разворачивания — L2. В L1 на эту метрику — ссылка, не определение. Один паспорт — один файл.
