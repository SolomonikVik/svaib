# Надстройки над выжимкой (Слой 2)

## Кратко

Скиллы, которые строятся поверх выжимки встречи (Слой 1). Выжимка — самодостаточный артефакт, хранит ВСЁ. Надстройки извлекают из неё конкретную ценность под задачу: протокол для клиента, обновления по файлам проекта, анализ паттернов по серии встреч.

## Связанные файлы

- meeting_analysis.md — методология Слоя 1 (пайплайн: транскрипт → выжимка)
- ../ontology/entities.md — каталог сущностей (что лежит в выжимке)
- ../ontology/behavioral_patterns.md — каталог паттернов (что лежит в выжимке)
- ../skills/meeting-analysis/ — реализация (промпты и скиллы)
- ../architecture.md — архитектура продукта (надстройки = Помощники)

---

## Принцип

Выжимка хранит всё. Надстройка — фильтр и трансформация под конкретную задачу.

Вход всегда — выжимка (+ контекст проекта при необходимости). Выход — зависит от надстройки. Надстройки независимы друг от друга и комбинируются.

## Каталог надстроек

### Протокол (семейство форматов)

Упаковка выжимки для отправки участникам. Не один формат, а семейство: полный протокол, Telegram-сводка, и другие по мере необходимости.

**Вход:** выжимка + досье (опционально)
**Выход:** структурированный документ в нужном формате
**Когда:** после каждой клиентской встречи

**Форматы:**

| Формат | Что делает | Промпт |
|--------|-----------|--------|
| Полный | Все секции, без ограничений длины | `skills/meeting-analysis/L2-prompt-protocol.md` |
| Telegram | Ядро (решения, задачи, проблемы), лимит ~4000 символов, эмодзи | `skills/meeting-analysis/L2-prompt-telegram.md` |

**Архитектурное решение:** каждый формат — отдельный самостоятельный промпт, не один промпт с параметром. Причины:
- Фокусированный промпт работает лучше, чем промпт с условными ветками (подтверждено на L1: один промпт 19/20 vs пайплайн 16/20)
- Форматы имеют разную жёсткость фильтрации и форматирование — один промпт с ветвлением усложняет отладку
- Дублирование маппинга (11 типов → секции) — осознанная цена, маппинг привязан к entities.md и меняется редко
- Альтернатива (один промпт с параметром `формат: полный | telegram | email`) может стать актуальной когда форматов станет 4+

**Особенность:** выжимка может содержать 30-50 сущностей. Полный протокол берёт почти всё. Telegram — только ядро. Механизм: приоритетная фильтрация в промпте (ядро → средний → низкий). FRAME-скоринг (отдельный LLM-вызов для оценки значимости) — отложен, см. ниже.

### Дельта-анализатор

Подтягивает контексты проекта, сопоставляет с выжимкой → генерирует обновления по файлам.

**Вход:** выжимка + файлы проекта (direction/, team/, management/)
**Выход:** конкретные дельты — что добавить/обновить в каком файле
**Когда:** после встречи, если нужно обновить проект
**Особенность:** самая ценная надстройка для системы — замыкает цикл "встреча → действие"
**Реализация:** `skills/meeting-analysis/L2-procedure-scaffold-update.md` (DRAFT v1)

### Анализ паттернов (longitudinal)

Повторяющиеся сигналы по серии встреч: тренды, динамика, паттерны поведения участников.

**Вход:** несколько выжимок (серия встреч с одним клиентом / командой)
**Выход:** тренды, повторяющиеся паттерны, динамика по осям behavioral_patterns.md
**Когда:** по запросу, при подготовке к стратегической встрече
**Особенность:** требует доступа к истории выжимок. Не задача синтезатора — отдельная надстройка

### Мета-анализатор (проверки целостности)

Проверка извлечённых сущностей на логическую согласованность между собой и с онтологией.

**Вход:** выжимка + entities.md (правила целостности)
**Выход:** список несогласованностей — задача без цели, решение без основания, решение противоречит принципу, метрика без привязки к цели
**Когда:** после синтезатора, перед отправкой протокола
**Особенность:** не ловит ошибки извлечения (это задача синтезатора), а проверяет бизнес-логику: имеет ли картина встречи смысл целиком. Правила — в entities.md (секция "Проверки целостности")

### Рекомендации по проведению

Анализ встречи с точки зрения качества: что не спросил, соответствие заявленным целям, баланс участия.

**Вход:** выжимка + цель встречи (если была)
**Выход:** рекомендации для следующей встречи
**Когда:** по запросу, для развития навыков проведения встреч

## Механики и инсайты

### FRAME-скоринг (для протокола)

Паттерн: атомарные факты → скоринг значимости → саммари. Промежуточный шаг между выжимкой и протоколом.

**Логика:** выжимка хранит ВСЁ (принцип Слоя 1). Протокол — только важное. Скоринг значимости = фильтр: какие сущности из выжимки попадут в протокол.

**Trade-off:** 3-4x по стоимости API (дополнительный LLM-вызов). Оправдано для high-stakes встреч, избыточно для рутинных.

### Factual spot-check

Выборочная верификация 3-5 утверждений из выжимки отдельным LLM-вызовом. Дешёвый smoke test качества.

**Логика:** не полная верификация, а быстрая проверка — не галлюцинирует ли модель. Берём случайные утверждения из выжимки, проверяем по транскрипту.

**Когда полезно:** при новых клиентах, при длинных встречах, как часть QA-пайплайна.

### Бизнес-фреймворки как шаблоны

Разные типы встреч → разные шаблоны протокола поверх одной и той же выжимки.

**Примеры:**
- BANT / MEDDIC — для sales calls (квалификация лида)
- ADR (Architecture Decision Record) — для стратегических встреч
- MoSCoW — для встреч по планированию

**Принцип:** выжимка универсальна. Бизнес-фреймворк — это линза, через которую надстройка смотрит на выжимку.
