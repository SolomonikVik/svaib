# deploy-scaffold — драфт спецификации

Техзадание на будущий скилл автоматизации развёртывания scaffold у клиента. Это не `SKILL.md` в Anthropic-формате — описательный драфт. Когда дойдём до разработки (после 2-3 успешных ручных развёртываний), содержимое переводится в реальный `SKILL.md` + `references/` + `assets/templates/` в этой же папке.

**Связь с MODEL.md:** правила развёртывания — источник истины [../../scaffold/MODEL.md, раздел 8](../../scaffold/MODEL.md). Скилл реализует эти правила автоматически. При расхождении — правда в `MODEL.md`, скилл догоняет.

**Статус:** отложено. До готовности — развёртывание ручное по `MODEL.md`.

---

## 1. Назначение

**Одна задача скилла:** расширить scaffold клиента новой сущностью по запросу CEO/консультанта.

Примеры запросов:

- «разверни scaffold у нового клиента»
- «добавь направление commercial»
- «добавь подразделение marketing внутри commercial»
- «добавь человека Иванова в team»
- «перенеси sales из корня внутрь commercial» (миграция)
- «разверни опциональную папку processes»

Скилл не плодит отдельные команды под каждый тип сущности (`deploy-direction`, `deploy-person`, `deploy-subdivision`...) — это одна задача с разными техническими вариациями.

## 2. Iron Law

> **Разворачивай только живое. Пустые ветки не создавай. «Живое» = появилась реальная работа (встреча, задача, человек), а не «заполнены поля».**

**Уточнение:** «реальная работа» = CEO регулярно взаимодействует с этим подразделением/направлением (встречи, задачи, цели, контроль). Факт существования отдела в компании — **не** триггер. У клиента может быть 50 отделов, но управленческий контур CEO — только те, с которыми он работает напрямую.

Этот закон — фильтр для каждого действия скилла. Если запрос пытается развернуть подразделение/направление без реальной работы за ним — скилл останавливается и спрашивает: «работает ли CEO с этим подразделением/направлением напрямую, или только знает о его существовании?»

## 3. Один скилл, а не набор — обоснование

Anthropic рекомендует «один скилл = одна задача». Наша задача одна: расширить scaffold клиента. Разные типы сущностей (подразделение/направление, человек, опциональная папка) — технические варианты одной операции, не разные задачи.

**Почему не набор скиллов (`deploy-direction`, `deploy-person`, `deploy-subdivision`):**

- Source of truth один, задача одна по сути.
- Iron Law общий для всех типов сущностей.
- Правила взаимосвязаны — создание дочернего подразделения/направления может требовать родительского. Разбивая на скиллы, заставляем их вызывать друг друга.
- Шаблоны должны жить рядом с правилами — иначе source of truth рассыпается между скиллами.

Аналогия: `skill-creator` Anthropic создаёт скиллы любого типа одним скиллом, а не набором `create-python-skill`, `create-doc-skill`, `create-api-skill`.

## 4. Что делает скилл (шаги)

На каждый запрос:

1. **Прочитать запрос** — определить, какая сущность разворачивается (подразделение/направление, человек, опциональная папка, миграция).
2. **Проверить Iron Law** — есть ли за запросом живой артефакт (встреча, задача, человек, решение). Нет — остановиться, спросить.
3. **Прочитать `MODEL.md` (раздел 8)** — какие правила применимы к этому типу сущности.
4. **Определить место в дереве** — для подразделения/направления: родитель или корень `04_management/`. Для человека: `03_team/`. Для опциональной папки: корень scaffold.
5. **Создать папку и все её обязательные файлы сразу** — пустые файлы допустимы. Плейсхолдеры не оставляем: либо реальный контент, либо пустая секция с подсказкой как HTML-комментарий.
6. **Обновить связанные файлы** — при создании подразделения/направления: обновить `03_team/org_structure.md` (если у подразделения/направления есть лидер); при создании человека: обновить `org_structure.md`; при переносе подразделения/направления: пофиксить ссылки в связанных файлах.
7. **Проверить консистентность** — нет ли битых ссылок, нет ли файлов-сирот.
8. **Сообщить результат** — что создано, что обновлено, какие связанные правки ждут внимания CEO.

## 5. Что скилл не делает

- **Не удаляет и не сливает подразделения/направления автоматически.** Запрос на удаление/слияние → остановка, подтверждение CEO/Виктора. Риск потерять историю работы.
- **Не создаёт пустые ветки «на всякий случай».** Если подразделение/направление упомянуто без живого артефакта — скилл не разворачивает, а записывает в `02_active.md` как контекст.
- **Не правит модель scaffold.** Скилл реализует правила из `MODEL.md`, не модифицирует их.

## 6. Целевая структура скилла (Anthropic-формат)

Когда скилл будет разрабатываться — переводится в стандартный формат skills:

```
framework/skills/scaffold/
├── SKILL.md                      ← главный файл (≤500 строк)
├── _draft_spec.md                ← этот файл (удаляется после переноса в SKILL.md)
├── references/
│   └── rules.md                  ← правила развёртывания (копия или выжимка из MODEL.md, раздел 8)
└── assets/
    └── templates/
        ├── org_node/             ← шаблон подразделения/направления: 01_overview, 02_active, 03_progress, 04_decisions, meetings/
        ├── person.md             ← шаблон профайла сотрудника
        ├── 01_ceo/               ← шаблоны 4 файлов 01_ceo
        ├── 02_strategy/          ← шаблоны 6 файлов 02_strategy
        ├── 03_team/              ← README + org_structure
        ├── clients/              ← опциональная папка
        ├── knowledge/            ← опциональная папка
        ├── processes/            ← опциональная папка
        └── product/              ← опциональная папка
```

**`SKILL.md`** — триггеры скилла, Iron Law, описание шагов, ссылки на references и assets.

**`references/rules.md`** — правила развёртывания (раздел 8 из `MODEL.md`). Живёт отдельно, чтобы SKILL.md оставался компактным. Когда обновляется `MODEL.md` — обновляется `rules.md`.

**`assets/templates/`** — готовые шаблоны файлов. Скилл копирует их при разворачивании, подставляет клиентские данные.

## 7. Когда разрабатываем скилл

**Не раньше, чем после 2-3 успешных ручных развёртываний по правилам из `MODEL.md`.** Цель — зафиксировать рабочий паттерн на реальных клиентах, не кодировать гипотезу.

Критерий готовности к разработке скилла:

- Ручное развёртывание прошло минимум на 2-3 клиентах (включая миграцию первого клиента).
- Правила развёртывания не требуют правок после каждого клиента.
- Типовые ситуации (новое подразделение/направление, миграция, добавление человека, разворачивание опциональной папки) прошли хотя бы один раз.

До этого момента — развёртывание ручное, правила живут в `MODEL.md`, скилл не существует как исполняемый артефакт.
