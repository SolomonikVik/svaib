# Оркестратор: вопрос CEO про метрики

Процедура для координатора (основной чат). CEO задаёт вопрос про метрики бизнеса — координатор ведёт цикл: определить домен → найти маршрут → исполнить через Python → собрать ответ + trace.

## Scope

Любой вопрос CEO про метрики (состояние, отклонения, подготовка к встрече, прогноз). Опирается на `metrics/` в пространстве клиента (семантика) и `metrics/source/*.xlsx` (числа).

Вне scope: вопросы вне домена бизнеса, вопросы без данных (помечать как «нет данных», не угадывать).

## Когда использовать

CEO спрашивает: «сколько», «какая выручка», «что с маржой», «почему упало», «готов меня к встрече с X», «покажи отклонения». Любой количественный вопрос про бизнес.

## Критические правила

1. **LLM не считает в голове.** Любой расчёт — через Python. Даже на простых вопросах. Соблазн «прикинуть на маленькой таблице» — главный источник тихих ошибок.
2. **Два разных xlsx-инструмента, не путать:**
   - `document-skills:xlsx` (Anthropic) — пишет pandas-код **на лету**. Для **разведки** структуры (один раз при онбординге клиента и при изменении источника). Не для регулярных запросов CEO.
   - **Per-client extractor** — заранее написанный Python-скрипт со списком метрик и **точными координатами** (sheet, row, col). Стабильный, детерминированный, не генерирует код заново. Для **регулярных запросов CEO** — только этот путь.

   Правило: «один раз через `document-skills:xlsx` понял структуру → написал extractor → дальше регулярно через extractor». См. Шаг 2.
3. **Stdout vs файл.** Python-экстракторы пишут результат **в файл** (JSON/markdown), агент читает через Read. В stdout — только короткий статус (`OK: wrote N metrics to ...`). Дамп ~7KB в stdout = шум в контексте + потеря детерминированности.
4. **Перед запуском extractor — самопроверка.** Все ли фильтры из вопроса попали в запрос. Не выдуманы ли имена колонок. Соответствует ли scope агрегации фильтрам.
5. **Trace доступен по запросу.** В ответе CEO — цифра + одна фраза «как считал». На «покажи trace» — маршрут, файл, код, timestamp данных.
6. **Если данных нет или маршрут не найден — сказать прямо.** Не угадывать. Спросить уточнение или предложить добавить маршрут.
7. **Синонимы — только из речи CEO.** Аудит синонимов по transcripts и decisions клиента — обязателен **до** первого тест-промпта. Где синонима в речи нет — пометка «в речи не зафиксировано», не выдумка. **Антисинонимы** — омонимы, которые звучат похоже, но означают другое (например, «контракт с сотрудником» ≠ «контракт с клиентом») — фиксируются явной секцией `antonyms` в `datasets`/`metrics` или в `01_metrics.md` глоссарий.
8. **План у клиента — первичный, не факт.** Если CEO спрашивает «топ-N разрывов план/факт», а у некоторых метрик план = 0 (или DIV/0 в источнике) — извлекатель должен пропускать или маркировать, не считать «разрывом 100%».

## Шаг 0: Понять вопрос и собрать контекст

1. Прочитай вопрос CEO.
2. Прочитай `metrics/01_metrics.md` (витрина) — определи **домен по канону**.

   **Canonical domain names** (зафиксированы в `framework/04_decisions.md` №5):

   | Минимальный канон | Расширение по триггеру |
   |---|---|
   | `finance.md` — прибыль, маржа, расходы, cash, P&L | `customer.md` — retention/churn/LTV |
   | `sales.md` — выручка, pipeline, конверсии, средний чек | `marketing.md` — каналы, CAC |
   | `operations.md` — delivery, качество, capacity, SLA, onboarding | `product.md` — метрики продукта |
   | `people.md` — найм, удержание, производительность | `strategy.md` — long-term цели |

   **Правило «не выдумывать имя».** Допустимы только эти 8 имён. Если вопрос CEO ложится в канон — используй каноническое имя; если не ложится (например, CEO мыслит через OKR / cross-functional срез) — **не изобретай новый domain-файл**. Иди в Ветку В (compound route, Шаг 1) или используй секцию «По OKR» в `01_metrics.md`. Если домен принципиально не покрыт каноном — остановись и подними вопрос (не разворачивай новый domain без решения в `04_decisions.md`).

3. Прочитай domain-файл (`metrics/{domain}.md`) — секции `datasets`, `metrics`, `routes`. Для compound-вопросов — все затронутые domain-файлы.
4. Если вопрос про подготовку к встрече — прочитай контекст встречи (`04_management/meetings/...` или `clients/{client}/`).

Если домен непонятен → переспроси CEO одной фразой («это про продажи или про операции?»).

## Шаг 1: Выбор пути

Определи тип вопроса и выбери ветку:

- **Один маршрут на одну метрику** («какая выручка за март?») → **Ветка А**
- **Cross-domain срез / OKR / бриф** («что с OKR3?», «подготовь к встрече с X», «топ-5 разрывов план/факт») → **Ветка В** (compound route)
- **Готового маршрута нет, строим на лету** → **Ветка Б** (думающая)

Cross-domain признак: вопрос требует агрегацию по нескольким метрикам, потенциально из разных domain-файлов, с финальным narrative-ответом, а не одним числом.

### Ветка А — готовый маршрут (single-metric)

В секции `routes` domain-файла найди маршрут, где `trigger` совпадает (по смыслу или по синонимам) с вопросом CEO.

Если маршрут найден:
1. Подставь параметры (даты, фильтры) в `query`.
2. Покажи CEO маршрут одной фразой («использую `route_<id>` из `{domain}.md`») — для прозрачности.
3. Переходи к Шагу 2 (исполнение).

### Ветка В — compound route (OKR / cross-domain)

Триггер: вопрос про OKR, cross-functional разрез, бриф к встрече, топ-N по срезу. Метрики могут быть из 1+ доменов.

1. **Найди набор метрик.** В `01_metrics.md`:
   - Для OKR-вопроса — секция «По OKR» → список метрик с ссылками на паспорта в domain-файлах.
   - Для cross-domain среза — секция «Target metrics CEO» с фильтром по контексту (встреча с X, период, тип разрыва).
   - Если набор не найден явно → переходи в Ветку Б, формулируй план fan-out на лету.

2. **Fan-out по маршрутам.** Для каждой метрики в наборе:
   - Открой паспорт в `{domain}.md` → найди маршрут (как в Ветке А).
   - Если маршрута нет → локально применяй Ветку Б для этой метрики.
   - Покажи CEO одной фразой: «по {OKR/срезу} — {N} метрик из {список доменов}».

3. **Исполнение.** Шаг 2. Если все метрики в одном xlsx — один read + N агрегаций. Если в разных — N чтений. Pandas умеет читать несколько листов одним проходом.

4. **Сборка narrative.** Не конкатенация цифр, а структурированный ответ. Каркас:
   - **Что в норме** (метрики без значимого разрыва план/факт).
   - **Что в красной зоне** (значимый разрыв, упорядочено по величине).
   - **Что не считается** (дыры в данных, отсутствующие маршруты, устаревшие источники).
   - **Trace по запросу** — список paths по каждой метрике (`{domain}.md#metric_<id>` + источник + timestamp).

Если хоть один маршрут провалился (ошибка xlsx, отсутствует поле) — не блокируй сборку: фиксируй метрику в «не считается» с причиной, остальные собирай.

### Ветка Б — думающая (маршрут не найден)

Сообщи CEO: «готового маршрута нет, строю на лету». Дальше — 4 фазы:

**Фаза 1. Планирование.** Прочитай `datasets` и `metrics` домена. Сформулируй план: какую метрику считаем, на каких полях, какими шагами. План — 2–4 строки текста.

**Фаза 2. Валидация плана.** Проверь:
- Все ли фильтры из вопроса CEO попали в план?
- Имена колонок берутся из `datasets.fields[].expression`, не выдуманы?
- Scope агрегации (по дню / месяцу / всему периоду) соответствует вопросу?
- Не подставляются ли «выдуманные значения» вместо обращения к xlsx?

Если хоть одна проверка не прошла — переделать план до исполнения.

**Фаза 3. Исполнение.** Сгенерируй Python-код для xlsx-skill. Код:
- Открывает `source/<file>.xlsx`
- Читает Pandas-ом нужный лист
- Применяет фильтры и агрегацию
- Возвращает результат

Передай код в xlsx-skill. При ошибке — retry с правкой.

**Фаза 4. Сборка ответа.** Цифра + 1–2 фразы narrative + timestamp данных.

## Шаг 2: Исполнение

Два режима — выбираются по типу запроса.

### Режим А — extractor (для регулярных запросов и Веток А/В)

Per-client extractor — Python-скрипт, живущий **рядом с клиентским пространством** (`metrics/_extractors/<source>_<sheet>.py` или аналогичный путь). Скрипт:

- Знает точные координаты метрик (sheet, row, col) — **захардкожены** или подгружены из конфига рядом.
- Принимает на вход: список `metric_id` (из паспорта), параметры (период, фильтры).
- Открывает xlsx, читает Pandas-ом, применяет агрегацию.
- **Пишет результат в файл** — `metrics/_runs/<timestamp>_<question_id>.json`. Формат: `{metric_id: {value, unit, period, source_path, source_timestamp, status: "ok"|"div0"|"missing"|"error"}}`.
- В stdout — только `OK: wrote N metrics to <path>` или `FAIL: <reason>`.

Координатор:
1. Запускает extractor через Bash (`python3 metrics/_extractors/<name>.py --metrics m1,m2,m3 --period ...`).
2. Читает результат через Read из файла.
3. Передаёт в Шаг 3 (сборка ответа).

Преимущества: детерминированность (один и тот же ответ на тот же вопрос), нет генерации кода каждый раз, не засоряет контекст stdout-дампами, легко версионировать.

### Режим Б — `document-skills:xlsx` (для разведки и think-on-the-fly)

Используется только в:
- Онбординг клиента (один раз, для понимания структуры xlsx).
- Изменилась структура источника (CEO добавил колонку, переименовал лист).
- Ветка Б (думающая), когда extractor ещё не написан и вопрос разовый.

После того как через `document-skills:xlsx` понял структуру и собрал ответ — **зафиксировать паттерн в extractor**, чтобы следующий тот же вопрос шёл через Режим А.

### Обработка ошибок

- Файл/лист не найден → проверь `datasets[].source` в domain-файле, попроси CEO уточнить.
- Колонка не найдена → проверь `fields[].expression`. Возможно, клиент изменил структуру источника → обнови domain-файл и extractor.
- `#DIV/0!` или `#VALUE!` в источнике → extractor возвращает `status: "div0"` для этой метрики. Координатор фиксирует в «не считается» (Ветка В) с причиной «деление на ноль в источнике».
- Stdout > 1KB → ошибка extractor'а (он не должен дампить в stdout). Поправить extractor, не пытаться парсить stdout.

## Шаг 3: Ответ CEO

Формат:

```
<цифра + единица>. <1–2 фразы narrative — что это значит>
Данные на: <timestamp xlsx>.
```

Для маршрута: формат из `routes[].response_format`.

На «покажи trace»:

```
Маршрут: <route_id или "думающая ветка">
Файл данных: source/<file>.xlsx (timestamp: ...)
Метрика: <metric_id>, version <N>
Код:
<python>
```

## Шаг 4: Кандидат на маршрут (если ветка Б)

После успешного ответа из думающей ветки — спроси CEO: «такой вопрос будет повторяться? добавить как маршрут?». Если да — записать в `routes` domain-файла кандидатом (status: candidate). После 2–3 повторений и подтверждения — promote в готовый маршрут.

## Cowork-чек-лист (перенос на runtime клиента)

При переносе пайплайна в runtime клиента (например, Cowork-агент Клиента) проверить **до** первого CEO-запроса:

1. **`document-skills:xlsx` доступен?** Триггерится ли скилл в этом runtime? Если нет — Ветка Б (думающая) с генерацией кода на лету не работает; fallback только через extractor.
2. **Extractor подхватывается?** Положен ли `metrics/_extractors/*.py` в путь, видимый агенту? Bash-вызов из агента работает? `python3` доступен?
3. **Allowlist для extractor.** В Cowork есть sandbox/прокси (см. `knowledge/tools/cowork.md`). Запуск Python-скрипта на локальный xlsx-файл — нужен ли в allowlist? Тестовый прогон `--dry-run` обязательно перед боевым CEO-вопросом.
4. **Stdout/stderr ограничения.** Если runtime обрезает stdout — extractor должен молчать (только `OK: ...` коротко).
5. **Запись в файл.** Доступ на запись в `metrics/_runs/` — есть? Альтернатива: `/tmp/` если sandbox не пускает в рабочую папку.

Без прохождения этих 5 пунктов — Б1 (боевой пилот) не стартует.
