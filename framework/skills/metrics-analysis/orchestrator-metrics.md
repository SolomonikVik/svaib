# Оркестратор metrics — операционный пайплайн

Как оркестратор исполняет поток обработки запроса в проде. Концептуальная модель — поток, режимы, инструменты — в [`architecture.md`](../../methodology/metrics/architecture.md) (разделы «Линия анализа», «Оркестратор»). Здесь — операционный слой: чем исполняется каждый шаг, какие правила нельзя нарушать, как обрабатываются ошибки.

**Статус — черновик.** Вычищен из старой версии под отменённую модель (ветки А/В/Б, `01_metrics.md`, секции `datasets`/`routes`). Приведён к `architecture.md` v2, нуждается в обкатке на практике.

## Когда активируется

Базовая беседа определила, что вопрос CEO про метрики бизнеса, и подгрузила этот skill. Граница домена решена до активации — оркестратор внутри домена метрик не переспрашивает «а это вообще про метрики».

Опирается на пространство клиента `metrics/`: `*-metrics.md` (семантика), `extractors/` (извлечение), `source/` (числа). Устройство — [`architecture.md`](../../methodology/metrics/architecture.md), раздел «Линия данных».

## Критические правила

1. **LLM не считает в голове.** Любое число — либо извлечено extractor'ом, либо посчитано calculator'ом. Даже на простом вопросе. «Прикинуть на маленькой таблице» — главный источник тихих ошибок.
2. **Оркестратор не трогает источник сам.** Его инструменты данных — только extractor и calculator. Ни `document-skills:xlsx`, ни pandas на лету. Нет extractor'а под источник — оркестратор не лезет в xlsx руками, а делегирует сборку скиллу-писателю extractor'а (см. шаг 5). Разведка источника и написание extractor'а — работа писателя, не оркестратора.
3. **Stdout молчит.** Extractor пишет результат в JSON-файл, оркестратор читает его через Read. В stdout extractor'а — только короткий статус. Дамп в stdout = шум в контексте + потеря детерминированности.
4. **Перед запросом данных — самопроверка.** Все ли фильтры из вопроса попали в запрос. Не выдуманы ли имена метрик. Соответствует ли scope агрегации вопросу.
5. **Нет данных / не уверен — сказать прямо.** Метрики нет — вернуть это явно, предложить близкие. Не уверен, какая метрика — переспросить CEO. Не угадывать.
6. **Trace по запросу.** В ответе — число + одна фраза «как считал». На «покажи trace» — план, источник, расчёт, timestamp данных.
7. **План у клиента первичный, не факт.** Если у метрики план = 0 или `#DIV/0!` в источнике — extractor маркирует статусом, оркестратор не считает это «разрывом 100%».

## Пайплайн

Семь шагов потока обработки запроса из [`architecture.md`](../../methodology/metrics/architecture.md). Здесь — чем исполняется каждый.

### 1. Понять вопрос

Разобрать формулировку CEO: о чём, за какой период, в каком разрезе. Разрешить анафоры и склеить с контекстом разговора — «а почему упало?» осмысленно только вместе с предыдущим ходом чата.

### 2. Опознать метрики

Какие метрики стоят за вопросом. Точка входа — `metrics/business-metrics.md`: прочитать целиком — бизнес-смысл, как считается, единицу, направление. Без направления нельзя подать число как «просело/выросло», без единицы — как «−0.5».

Если вопрос уходит в домен — открыть соответствующий `{domain}-metrics.md`. Канонические имена domain-файлов — [`metrics-spec.md`](../../methodology/metrics/metrics-spec.md), новых не изобретать.

Две развилки:
- **Не уверен, какая метрика** — переспросить CEO.
- **Метрики нет** — вернуть явно, предложить близкие по смыслу.

### 3. Маршрут или размышление

Есть ли готовый маршрут под вопрос. **На текущем уровне готовых маршрутов нет** — оркестратор всегда идёт в режим размышления и строит план под каждый вопрос с нуля.

Маршруты и ритуалы (снимок проработанной линии «вопрос → план → ответ»; расписание поверх маршрутов) заложены в [`architecture.md`](../../methodology/metrics/architecture.md), но механизм их хранения ещё не зафиксирован — появятся позже как оптимизация поверх размышления.

### 4. Построить план

Определить: какие метрики нужны, за какой период, какие сравнения (план/факт, динамика, бенчмарк), какие данные запросить у extractor'а, что посчитать calculator'ом, нужно ли уточнение у CEO. План — 2–4 строки.

Самопроверка перед запросом (правило 4): все ли условия вопроса в плане, имена метрик каноничные, scope агрегации верный. Не прошла проверка — переделать план до исполнения.

### 5. Запрос данных и расчёта

**Извлечение — extractor.** Per-client extractor живёт в `metrics/extractors/`, по одному на источник. Оркестратор:
1. Запускает через Bash: `python3 metrics/extractors/<name>.py --metrics "имя1,имя2" --period 2026-04 --out metrics/_runs/<run>.json`.
2. Читает результат из JSON-файла через Read. Формат выхода и статусы метрик — [`extractor.md`](../../methodology/metrics/extractor.md), раздел «Формат JSON-выхода».
3. В stdout extractor'а — только `OK: ...` / `FAIL: ...`. Дамп в stdout — баг extractor'а: поправить extractor, не парсить stdout.

**Нет extractor'а под источник.** Оркестратор в xlsx сам не лезет. Он сообщает CEO, что по этому источнику готового extractor'а ещё нет и он сейчас его соберёт — чтобы было понятно, почему дольше обычного, — и делегирует сборку скиллу-писателю extractor'а (контракт — [`extractor.md`](../../methodology/metrics/extractor.md), раздел «Скилл-писатель extractor'а»). Писатель разведывает источник, пишет extractor, кладёт в `metrics/extractors/`, возвращает управление — оркестратор продолжает с готовым extractor'ом.

**Расчёт — calculator.** Производные (рост, доля, отклонение план/факт, агрегаты, ряды для графика) — через calculator, не в голове. Вход: сырые значения от extractor + что посчитать.

### 6. Анализ

Работа с сырыми и вычисленными значениями: что метрика значит, почему двинулась, что в норме, что в красной зоне. Сверка с направлением и единицей из metrics-файла. При необходимости — контекст соседних вертикалей (`decisions`, `weekly_progress`, `meeting`). Не хватает данных — возврат к шагу 4 или 5.

### 7. Ответ CEO

Не сырое число, а управленческий вывод. Формат по вопросу — число, таблица, график, narrative. Адресат: ad-hoc вопрос → чат; ритуал → канал.

Базовый формат:
```
<число + единица>. <1–2 фразы — что это значит>
Данные на: <timestamp из JSON extractor'а>.
```

На «покажи trace»:
```
План: <2–4 строки плана из шага 4>
Источник: <source из JSON> (timestamp: ...)
Метрики: <канонические имена>
Расчёт: <что считал calculator>
```

## Обработка ошибок

- **Файл/лист не найден** — проверить адрес источника в metrics-файле, попросить CEO уточнить.
- **Schema-hash не сошёлся** — extractor остановился: структура источника изменилась. Оркестратор делегирует пересборку скиллу-писателю extractor'а — тот переразведывает источник, правит раскладку, пересчитывает эталонный хэш. Механизм — [`extractor.md`](../../methodology/metrics/extractor.md), раздел «Schema-hash check».
- **`#DIV/0!` / план = 0** — extractor вернул `status: div0`. Зафиксировать метрику как «не считается» с причиной, остальные собрать.
- **Метрика не найдена в источнике** — `status: missing`. То же: «не считается», не подставлять `0`.
- Не блокировать сборку ответа из-за одной упавшей метрики — собрать остальные, упавшие вынести явным блоком.

## Cowork-чек-лист (перенос на runtime клиента)

До первого CEO-запроса в runtime клиента проверить:

1. **Разведка источника доступна?** Скиллу-писателю extractor'а нужны `document-skills:xlsx` / `probe_xlsx.py` для probe. Недоступны — новые источники в runtime не подключить, работают только заранее собранные extractor'ы.
2. **Extractor подхватывается?** `metrics/extractors/*.py` в видимом агенту пути, Bash-вызов работает, `python3` доступен.
3. **Allowlist для extractor.** В Cowork есть sandbox (см. [`cowork.md`](../../../knowledge/tools/cowork.md)). Запуск Python на локальный xlsx — нужен ли в allowlist. Тестовый `--dry-run` обязателен до боевого вопроса.
4. **Stdout-ограничения.** Если runtime обрезает stdout — extractor должен молчать (только короткий статус).
5. **Запись результата.** Доступ на запись в рабочую папку прогонов (`metrics/_runs/`) есть? Альтернатива — `/tmp/`, если sandbox не пускает в рабочую папку.

Без этих пяти пунктов регулярный metrics-runtime у клиента не стартует.
