# Rollout playbook: metrics для нового клиента

Пошаговая процедура развёртывания вертикали `metrics` у нового клиента. Превращает опыт первого пилота в воспроизводимый процесс.

**Когда это нужно.** CEO-клиент:
1. Ведёт метрики бизнеса в xlsx или Google Sheets (не в BI-системе).
2. Хочет, чтобы AI отвечал на вопросы про эти метрики (состояние, отклонения, бриф к встрече).
3. Готов открыть нам свой источник для одноразовой разведки.

**Когда это НЕ нужно.** У клиента уже выстроенная BI-система с API; или метрики не существуют как отдельный слой (живут в нарративе). Тогда — отдельное решение, не этот playbook.

## Шаг 1. Понять CEO

Прежде чем трогать xlsx — собрать контекст клиента. Вопросы CEO (или субагенту по его пространству):

1. **Какая структура целеполагания?** OKR? KPI? Цели + ключевые результаты по-другому? Где они формально записаны (`02_strategy/02_goal.md` или аналог)?
2. **Какой xlsx считаешь главным?** Имя файла, путь, как часто обновляется, кто редактирует.
3. **5-10 метрик, которые держишь в голове регулярно** — на что смотришь каждый понедельник.
4. **Какой типовой вопрос задаёшь к этим цифрам?** (Это будут наши тест-вопросы.)
5. **Где transcripts твоих встреч и `decisions.md`?** Нам нужно для аудита синонимов — какими словами ты сам называешь метрики.

> **Если CEO мыслит через OKR-структуру** — фиксируем это: в витрине будет секция «По OKR». Не переименовываем domain-файлы под OKR (см. `04_decisions.md` №5).

**Артефакт шага:** короткая записка `_inbox/metrics-rollout/{client}/00_context.md` — кто CEO, какая OKR-структура, какие тест-вопросы планируются, ссылки на источники.

## Шаг 2. Разведка xlsx (одноразово, через `document-skills:xlsx`)

Цель: понять структуру источника. **Это единственный шаг, где AI пишет pandas-код на лету.** Дальше — только готовый extractor.

1. Положить копию xlsx клиента в **рабочее sandbox-пространство под git** (например, `framework/_inbox/metrics-rollout/{client}/sandbox/source/`). Не работать сразу в `clients/private/<client>/drive/` — там нет git, ошибки не откатить.

2. Через `document-skills:xlsx` или Python-скрипт `probe.py` (см. шаблон в `_inbox/metrics-scaffold/sandbox/_b0_findings/probe.py`) собрать отчёт о структуре:
   - Имена всех листов.
   - Для каждого ключевого листа: размер (rows × cols), первые 10-20 строк целиком, заполненность колонок.
   - **Через openpyxl** — формулы, объединённые ячейки, кэшированные ошибки (`#DIV/0!`, `#REF!`).
   - **Без агрегаций.** Только описание формы и содержания.

3. Записать результат разведки в `sandbox/_findings/01_xlsx_structure.md`. **Не дамп stdout** — markdown в файл.

**Что искать.** На опыте Клиента:
- **Несколько таблиц на одном листе.** «Шапка-витрина» сверху, «годовая раскладка по месяцам» в середине, «недельная сводка» снизу. У них разные системы координат, header-row, форматы. → нужны `regions`.
- **Объединённые ячейки** в заголовках периодов. Pandas `read_excel(header=N)` уплывает.
- **Формат меняется по периоду** — например, до апреля «План/Факт/%», с апреля «План/потенциальный/%». Семантика разная — фиксировать в `known_issues`.
- **Свободные строки-комментарии** между метриками («триггеры у клиентов», «не отвечают» — рабочие заметки CEO). Не метрики, парсер пропускает.
- **`#DIV/0!` в формулах** — план = 0 для будущих периодов. Решить: пропускать (`status: "div0"`) или маркировать.
- **Опечатки и врущие метки** — у первого клиента row 2 называлась «Новый MRR/доля черна» (опечатка «черна» = «churn»; реально это **доля** младенческого churn в новых продажах, не сам новый MRR).

**Артефакт шага:** `_findings/01_xlsx_structure.md` (структура) + `_findings/02_data_map.md` (маппинг строк xlsx → metric_id → canonical domain).

## Шаг 3. Аудит синонимов и антисинонимов

Цель: знать, как CEO **сам называет** свои метрики, и какие омонимы есть в его речи.

Запускается параллельно с шагами 4-6, но обязан быть готов **до тест-промпта**.

1. Запустить субагент Explore в пространство клиента (`clients/private/<client>/`):
   - Прочитать transcripts всех встреч за последние 3-6 месяцев.
   - Прочитать `decisions.md` всех уровней.
   - Извлечь все упоминания метрик и обороты, по которым CEO их называет.
   - Найти антисинонимы: «контракт» в речи клиента = договор с сотрудником, не с клиентом — спутать критично.

2. Записать результат в `_findings/03_synonyms_audit.md`.

**Жёсткие правила** (см. [`orchestrator-metrics.md`](orchestrator-metrics.md), Критическое правило 7):
- Синонимы **только из речи CEO**. Не выдумывать.
- Где синонима в речи нет — `synonyms: ["в речи не зафиксировано"]`.
- Антисинонимы — отдельный список в паспорте (`antonyms`).

**Артефакт шага:** `_findings/03_synonyms_audit.md` с цитатами «дословно (источник, дата)».

## Шаг 4. Спроектировать domain-файлы

По data-map (шаг 2) распределить метрики по каноническим доменам ([`04_decisions.md`](../../04_decisions.md) №5):

- Минимальный канон: `finance.md`, `sales.md`, `operations.md`, `people.md`.
- Расширение по триггеру: `customer.md`, `marketing.md`, `product.md`, `strategy.md`.

**Iron Law.** Domain-файл создаётся, когда у CEO живая работа в этом домене. Не «на всякий случай».

**На каждый создаваемый domain-файл:**

1. Скопировать `framework/scaffold/metrics/template-domain.md` как отправную точку.
2. Заполнить:
   - `datasets` — источники + `regions` (если на листе несколько таблиц) + `fields` со синонимами/антисинонимами/`is_time`.
   - `metrics` — паспорта со всеми обязательными полями: `direction`, `formula_human`, `synonyms` из речи CEO, `antonyms`, `plan_required`.
   - `routes` — готовые маршруты для типовых вопросов CEO (мин. 1-2 на старте). В `routes[].extractor` — путь к per-client extractor с параметрами вызова.
   - `known_issues` — все косяки источника, обнаруженные на шаге 2. Без этой секции агент тихо отвечает неточно.

**Артефакт шага:** заполненные `customer.md`, `sales.md`, `operations.md` (или какие нужны) в `sandbox/`.

## Шаг 5. Спроектировать витрину

`01_metrics.md` — точка входа агента в metrics. Заполнить:

1. **Target metrics CEO** — 5-10 ключевых метрик, на которые CEO смотрит регулярно. Карточки-ссылки на паспорта (без дублирования значений и формул).
2. **Карта доменов** — какой domain покрывает какую категорию.
3. **Источники** — таблица с описанием xlsx (что внутри, обновляется как).
4. **Секция «По OKR»** (опционально) — если у CEO OKR-структура. Для каждой OKR: название, KR, владелец, **список метрик с ссылками на паспорта**. Сами OKR живут не здесь, а в `02_strategy/02_goal.md` — здесь только навигация.
5. **Глоссарий CEO** — обороты CEO («что у меня горит», «бриф к встрече с X», «как у нас с …») и на какую процедуру они маппятся.
6. **Антисинонимы / защита от омонимов** — если в его речи есть слова, которые легко спутать.

**Артефакт шага:** заполненный `sandbox/01_metrics.md`.

## Шаг 6. Написать per-client extractor

`{client}_{sheet}.py` — Python-скрипт со списком метрик и **захардкоженными координатами**.

Структура (см. шаблон `framework/_inbox/metrics-scaffold/sandbox/_extractors/ssp_okr1.py`):

1. **`METRIC_REGISTRY`** — словарь `metric_id → {strategy, row(s), label, unit, direction, plan_required, ...}`. Координаты захардкожены, не вычисляются.
2. **Region column maps** — для каждого региона на листе (если их несколько) — словарь «период → колонка».
3. **`OKR_GROUPS`** (опционально) — алиасы вида `okr1 → [список метрик]` для compound route.
4. **`extract_metric(df, metric_id, period)`** — диспетчер по `strategy`, возвращает `{value, status, ...}`. Status ∈ `ok | missing | div0 | error`.
5. **CLI** — argparse `--metrics` или `--okr`, `--period`, `--out`. Пишет JSON в `--out`. **В stdout — только короткий статус** (`OK: wrote N metrics ...`).

**Жёсткие правила:**
- Скрипт **не генерирует код** во время вызова. Логика — в `METRIC_REGISTRY`.
- Любой `#DIV/0!` или пустая ячейка → `status: "div0"` или `"missing"`. Не подставлять `0` или `null` без статуса.
- Stdout > 1 KB = ошибка скрипта.

**Артефакт шага:** `sandbox/_extractors/{client}_{sheet}.py`.

## Шаг 7. Перенести narrative composer

`narrative.py` — **универсальный**, копируется как есть из `framework/_inbox/metrics-scaffold/sandbox/_extractors/narrative.py`.

Что он делает: читает JSON от extractor → классифицирует метрики (`red | win | ok | blocked`) с учётом `direction` → пишет structured markdown с разделами «В красной зоне / Лучше плана / В норме / Не считается».

**Если у клиента понадобится правка** (например, изменить порог 10% или добавить новый bucket) — править копию у клиента, **синхронизировать обратно в канон по триггеру** (если правка пригождается следующим клиентам).

**Артефакт шага:** `sandbox/_extractors/narrative.py` (копия).

## Шаг 8. Smoke-test в sandbox

В sandbox у себя (под git) прогнать pipeline на реальном тест-вопросе клиента:

1. `python3 _extractors/{client}_{sheet}.py --okr okr1 --period <реальный период> --out _runs/test_v1.json`
2. `python3 _extractors/narrative.py --in _runs/test_v1.json --out _runs/test_v1.narrative.md`
3. Проверить визуально: цифры сходятся со скриншотом xlsx, классификация по `direction` корректна, нет «выдуманных» значений.

**Если что-то не так** — править в sandbox, не в боевом scaffold клиента. Под git, всё откатывается.

**Артефакт шага:** прогон в `sandbox/_runs/`, отчёт в `sandbox/_findings/04_smoke_test.md`.

## Шаг 9. Sub-agent end-to-end test

Прежде чем нести клиенту — эмулировать его агента у себя.

1. Создать папку `framework/_inbox/subagents/metrics-{client}-test/`.
2. Запустить subagent в **foreground** с задачей «ты — AI-помощник CEO, ответь на вопрос: …», дать доступ только к `sandbox/` (без `_findings/`, `_runs/` — не подсказывать). Бриф включает явные ограничения:
   - `Use only simple, single bash commands - no &&, no ||, no pipes, no ;, no redirects.`
   - Не открывай xlsx сам через `document-skills:xlsx` — найди extractor через `routes[].extractor`.
   - Не считай в голове.
   - Запиши финальный ответ в указанный путь.
3. Прочитать результат, сверить с ожидаемым.

**Что проверяешь:**
- Subagent сам нашёл ветку (А/В/Б) — должен правильно по триггеру.
- Subagent сам нашёл extractor через routes — без подсказки пути.
- Direction учтён в классификации.
- DIV/0 и missing корректно ушли в «не считается».
- Финальный ответ структурирован, не каша из чисел.

**Если subagent ошибся** — это сигнал об дыре в pipeline. Зафиксировать как открытый вопрос, починить, перезапустить.

**Артефакт шага:** `_inbox/subagents/metrics-{client}-test/answer.md`.

## Шаг 10. Cowork-чек-лист (если runtime клиента — Cowork)

До переноса в боевой scaffold клиента — пройти 5 проверок (см. [`orchestrator-metrics.md`](orchestrator-metrics.md), секция «Cowork-чек-лист»):

1. `document-skills:xlsx` доступен в Cowork-агенте клиента?
2. Per-client extractor подхватывается? `python3` доступен? Extractor читает локальный xlsx (через Drive for Desktop)?
3. Allowlist для запуска Python-скрипта — нужен или нет?
4. stdout/stderr ограничения runtime?
5. Запись в `metrics/_runs/` доступна или fallback в `/tmp/`?

**Без прохождения этих 5 пунктов — боевой пилот не стартует.** Ошибка тут даст «вроде работает локально, у клиента не работает».

## Шаг 11. Перенос в боевой scaffold клиента + rclone push

1. Скопировать из sandbox в `clients/private/<client>/drive/<scaffold-path>/metrics/`:
   - `01_metrics.md`
   - все domain-файлы
   - `source/README.md` (xlsx уже там, обычно)
   - `_extractors/{client}_{sheet}.py` + `narrative.py`
2. **НЕ копировать** `_findings/`, `_runs/` — это рабочие материалы команды.
3. Скорректировать пути если нужно (extractor использует `Path(__file__).resolve().parent.parent` — должно работать без правок).
4. `rclone push` (или аналог синхронизации) в Google Drive клиента.
5. Сообщить клиенту: «metrics развёрнут, можно задавать вопросы».

## Шаг 12. Eval-набор первый

Прогнать у клиента в его runtime 3-5 живых вопросов CEO **без подсказок про `metrics/`** (минимальный агент, только системный промпт оркестратора). Записать дыры:

- Какие маршруты не сработали.
- Какие синонимы оказались пропущены.
- Какие косяки источника не были предусмотрены в `known_issues`.
- Есть ли расхождение между sandbox-результатом и Cowork-результатом.

**Артефакт шага:** `clients/private/<client>/eval/01_first_run.md`.

Дыры — двигают канон (Этап С): обновить `framework/scaffold/metrics/template-domain.md`, `orchestrator-metrics.md`, `HOWTO.md` если урок общий для всех клиентов.

## Когда вертикаль готова к расширению

После успешного шага 12 вертикаль `metrics` у клиента — **на уровне early-adopter beta**. Дальше расширения по триггеру:

- **Новые domain-файлы** — когда у CEO появляется живая работа в новом домене.
- **OKR2, OKR3, …** — расширить секцию «По OKR» в витрине + extractor.
- **Top-N с временным анализом** — обогащение narrative.
- **Связки с другими вертикалями** (meeting, contracts) — через горизонтальные слои по стабильным ID, не прямые вызовы.

## Что из этого нужно сохранить у нас (svaib)

После успешного развёртывания у клиента — что-то остаётся **только** у клиента, что-то возвращается в канон.

| Артефакт | Где живёт у клиента | Источник правды у svaib |
|---|---|---|
| `01_metrics.md` витрина | `clients/<client>/drive/.../metrics/` | образец витрины — `framework/scaffold/metrics/01_metrics.md` |
| Domain-файлы | `clients/<client>/drive/.../metrics/` | шаблон — `framework/scaffold/metrics/template-domain.md` |
| Per-client extractor | `clients/<client>/drive/.../metrics/_extractors/` | шаблон + примеры в `framework/_inbox/metrics-scaffold/sandbox/_extractors/` |
| Narrative composer | `clients/<client>/drive/.../metrics/_extractors/narrative.py` | **универсальный канон** — `framework/skills/metrics-analysis/narrative.py` (вынести как общий после 2-3 клиентов) |
| Системный промпт оркестратора | `clients/<client>/drive/.../CLAUDE.md` или плагинный slot | `framework/skills/metrics-analysis/orchestrator-metrics.md` |
| Уроки клиента (что нашли в его речи / xlsx-косяки) | `clients/private/<client>/drive/.../metrics/` (там) | если урок общий — обогатить `template-domain.md`, `HOWTO.md`, `orchestrator-metrics.md` |
| Eval-набор клиента | `clients/private/<client>/eval/` | первые 2-3 клиента — собрать в `framework/_inbox/metrics-evals/` для будущего регресс-теста |

**Принцип:** у клиента — конкретика (его метрики, его координаты, его синонимы). У нас — методология и шаблоны (что универсально и переносимо). Между ними — explicit перенос: уроки одного клиента не остаются у него, поднимаются в канон, помогают следующему.
