# Обновление файлов проекта по выжимке встречи

L2-надстройка над выжимкой встречи. Получает готовую выжимку, делает файл-центричный проход по файлам проекта, предлагает обновления руководителю.

**Слой:** L2 (→ methodology/meeting_overlays.md, секция "Дельта-анализатор").

## Вход

1. **Выжимка** — готовый файл `_summary.md` (результат L1-пайплайна)
2. **Контекст от руководителя** — ответ на вопрос: "Что за встреча? По какому направлению? Кто участвовал?"

## Выход

Список обновлений по файлам. Каждое обновление: какой файл → что добавить/изменить → почему (ссылка на выжимку). Применяется ТОЛЬКО после ОК руководителя.

## Scope

Встречи по одному направлению или кросс-функциональные (2+ направления). Предполагает развёрнутый team/ (team/{person}.md, не management/team.md).

Вне scope этой версии: стратегические встречи уровня компании (management-слой), свёрнутая форма team/, контейнерная логика вложенных направлений.

---

## Шаг 1: Определи scope обновления

По ответу руководителя определи затронутые направления.

**Встреча по одному направлению** (1-on-1, планёрка отдела):
- Целевая папка: `{direction}/` (или `{direction}/{sub_direction}/`)
- Читать: файлы проекта целевой папки + team/ участников

**Кросс-функциональная встреча** (2+ направления):
- Целевые папки: ВСЕ затронутые направления
- Читать: файлы проекта КАЖДОГО направления + team/ участников + management/decisions.md

---

## Шаг 2: Собери контекст

1. Прочитай выжимку
2. Прочитай файлы проекта целевой папки (папок): 01_overview.md, 02_active.md, 03_progress.md, [04_decisions.md если есть]
3. Прочитай team/{person}.md для каждого участника встречи
4. При кросс-функциональной встрече — прочитай management/decisions.md

---

## Шаг 3: Файл-центричный проход

### Iron Law обновления

После обновления — состояние проекта в файлах актуально. Выжимка остаётся как полная запись встречи. Не всё из выжимки должно попасть в файлы — это нормально.

### Принцип: выжимка щедрая, файлы проекта строгие

Выжимка сохраняет всё (Iron Law L1). Файлы проекта — только то, что имеет устойчивую ценность: стратегические решения, факты о бизнесе, задачи с owner, изменения в целях и структуре. Операционная мелочь, разовые реплики, бытовые просьбы — остаются в выжимке, не попадают в файлы.

---

Пройди по каждому прочитанному файлу и спроси по его миссии: "нужно ли обновить этот файл с учётом новой информации из выжимки?"

**НЕ раскидывай сущности из выжимки по файлам.** Проверяй файлы — это принципиально другой алгоритм (→ scaffold/principles.md, Принцип 1).

### Чеклист

| Файл | Миссия | Тест-вопрос |
|------|--------|-------------|
| 01_overview.md | ЧТО это направление и КУДА идёт | Новое о целях, KPI, scope, команде, ключевых проектах? Тест: будет актуально через квартал? |
| 02_active.md | ЧТО АКТУАЛЬНО сейчас | Сначала сопоставь новое с existing — связанные объединяй, не дублируй. Новые задачи (по owner)? Проблемы? Открытые вопросы? Риски? Идеи? Повестка следующей встречи? Выполненные задачи → отметить `[x]`, перенести в "Подтвердить" |
| 03_progress.md | КАК ДВИГАЛИСЬ | Дописать запись о встрече (формат ниже) |
| 04_decisions.md | ПОЧЕМУ так решили (уровень направления) | Стратегические решения? Тест: через 3 мес спросят "почему так?" |
| management/decisions.md | ПОЧЕМУ так решили (company-wide) | Решения уровня всей компании? Только при кросс-функциональных встречах |
| team/{person}.md | КТО этот человек | Наблюдения о стиле, сильных сторонах, зонах развития? Коучинг? Тест: про ЧЕЛОВЕКА или про НАПРАВЛЕНИЕ? |

При кросс-функциональных встречах — проход по чеклисту для КАЖДОГО затронутого направления. Одна сущность из выжимки может породить обновления в файлах разных направлений.

---

## Шаг 4: Покажи обновления руководителю

Формат предложения:

```
### {имя файла} ({путь})

**Обновление:**
- {конкретная правка}

**Основание:** {ссылка на пункт выжимки}
```

Применяй ТОЛЬКО после ОК руководителя. Если руководитель отклоняет обновление — не спорь, оставь в выжимке (catch-all).

---

## Правила маршрутизации

**Один аспект — одно место.** Одно событие может обновить несколько файлов, если затрагивает разные аспекты. "Решили сменить лидера направления" → 01_overview (структура) + team/{person}.md (роль) + 02_active (задачи по передаче). Разные файлы отвечают на разные вопросы — это не дублирование.

**Catch-all.** Не можешь однозначно маршрутизировать — оставь в выжимке. Лучше не разложить, чем разложить неправильно.

**Антипаттерны:**
- Проблемы/задачи в overview → active
- Хроника в overview → progress
- Решения без контекста → decisions (нужны why + альтернативы)
- Коучинг в direction/ → team/{person}.md
- Операционные выборы в decisions → остаются в выжимке
- Company-wide решения в direction/04_decisions → management/decisions.md

**Даты-источники.** Volatile-данные (цифры, статусы, решения) — inline-дата `(YYYY-MM-DD)`. Стабильные факты — без дат.

---

## Форматы записей в log-файлах

**03_progress.md:**
```
## YYYY-MM-DD: {тема встречи}

**Участники:** {кто был}
**Ключевое:** {2-4 строки — что обсудили, что решили, что сдвинулось}
**Следующий шаг:** {главное действие}
```

**04_decisions.md / management/decisions.md:**
```
## YYYY-MM-DD: Название решения
**Статус:** Актуально
**Решение:** Что решили
**Почему:** Контекст
**Альтернатива:** Что рассматривали и отвергли
```

**team/{person}.md → Коучинг-лог:**
```
### YYYY-MM-DD
{Наблюдение: что заметили, как проявилось, контекст}
```

## Ограничения DRAFT v4

1. **Предполагает развёрнутый team/.** Если team/ не развёрнут (management/team.md) — коучинг-наблюдения остаются в выжимке.
2. **Company-wide встречи вне scope.** Стратегические встречи уровня компании (management-слой целиком) требуют отдельного чеклиста по management-файлам.
3. **Контейнерная логика вне scope.** Встречи внутри направления-контейнера (commercial/ → sales + marketing) обрабатываются как кросс-функциональные (по обоим направлениям), без специфики контейнерных файлов.
4. **Нет автоопределения направления.** Руководитель указывает направление и участников.
