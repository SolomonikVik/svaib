# Чанкинг транскриптов

Процедура разбиения длинных транскриптов на части для параллельной экстракции сущностей (entity extractor). Координатор (главный агент) управляет процессом, не загружая результаты в свой контекст.

## Зачем

Монолитная экстракция теряет кандидатов на длинных транскриптах. Тестирование на двух транскриптах показало:

| Транскрипт | Размер | Потери монолита | Подтверждено верификацией |
|---|---|---|---|
| Е. (менторская, ~1:46) | 182KB | 12-14 важных кандидатов | Да, 0 выдумок |
| С. (клиентская, ~1:01) | 105KB | 14 важных кандидатов | Да, 0 выдумок |

Чанкинг добавляет ~15% к расходу токенов (overlap), но возвращает 30-40% потерянного содержания.

Тестировано на модели Opus 4.6 (субагенты — та же модель). При смене модели субагентов результаты и порог могут измениться.

## Когда применять

- Файл > 60KB → чанкинг
- Файл ≤ 60KB → монолитная экстракция

Порог 60KB — рабочая гипотеза, не проверена на границе. Проверено: на 105KB и 182KB монолит теряет 30-40% кандидатов.

## Процедура

### 1. Определить количество чанков

```
wc -c файл → размер
ceil(размер / 60000) = количество чанков
```

### 2. Нарезать по границам реплик

Атомарная единица — реплика (от `Имя • таймкод` до следующего `Имя • таймкод`). Нарезка по репликам, а не по строкам — потому что длина строк в транскриптах сильно варьируется (от 5 слов до 2000 слов в одной строке).

Алгоритм:
1. Найти номера строк всех реплик: `grep -n " • " файл`
2. Для каждой реплики посчитать байтовый offset
3. Найти реплику ближайшую к N-й доле файла по байтам (52KB для 2 чанков, 35KB/70KB для 3 чанков)
4. Overlap = 3 реплики назад от точки разреза

Пример для 105KB, 2 чанка (С.):
- Реплика на строке 256 (`Viktor • 32:07`) ≈ 52KB — середина файла
- 3 реплики назад = строка 247
- Чанк 1: строки 1-258
- Чанк 2: строки 247-577

Агенту передаётся через Read tool: offset и limit в строках.

Позже: алгоритм нарезки свернуть в Python-скрипт (одна команда на вход файл + кол-во чанков → на выход диапазоны строк).

### 3. Запустить экстракторов параллельно

Каждый агент получает:
- Промпт entity extractor (без секций метаданных и краткого содержания)
- Указание: "читай строки X-Y файла"
- Указание: "ЗАПИШИ кандидатов в файл chunk_N.md"
- Указание: "верни мне ТОЛЬКО несколько предложений краткого содержания твоего куска"

Каждый агент независим — не знает о других чанках. Дубли из overlap — нормально, синтезатор уберёт.

### 4. Собрать результат

```bash
cat chunk_1.md chunk_2.md [chunk_3.md ...] > extract.md
```

Координатор интегрирует краткое от каждого агента в единое краткое содержание и вставляет header в начало файла через Edit (БЕЗ чтения файла — первая строка известна из инструкции агенту).

### 5. Передать синтезатору

Синтезатор получает один файл с header (метаданные + краткое) и кандидатами. Дубли из overlap — нормально: синтезатор дедуплицирует по совпадению цитаты-опоры и содержания (не схлопывает кандидатов с одной цитатой, если они относятся к разным типам сущностей).

## Что координатор НЕ делает

- Не читает файлы с кандидатами (экономия контекста)
- Не читает файлы перед Edit, когда первая строка известна из инструкции агенту
- Не делает merge-шаг моделью (cat достаточно)

## Шум и дедупликация

Чанкинг добавляет ~10-15% шумовых/дублирующих кандидатов (из overlap и из-за независимости агентов). Это нормально — дедупликация на этапе синтеза. Синтезатор и так убирает дубли и шум.

## Асимметрия: entity vs behavior

Чанкинг применяется только к entity extractor. Behavior extractor работает монолитно.

Причина: разные приоритеты качества.
- Entity extractor: recall критичен — потерянный факт, цифра, задача не восстановится. Чанкинг возвращает 30-40% потерянного
- Behavior extractor: precision критичнее — лучше отметить меньше, чем приписать лишнее. Чанкинг может раздуть шум и ложные сигналы. Паттерны — тренды, не дискретные факты; потеря отдельного сигнала не критична, если тренд виден

Behavior extractor через чанкинг не тестировался. Fallback на чанкинг — только при явном провале качества на очень длинных транскриптах.
