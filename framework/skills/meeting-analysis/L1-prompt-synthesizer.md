# Синтезатор выжимки

Финальный шаг пайплайна ([meeting_analysis.md](../../methodology/meeting_analysis.md)). Принимает выходы обоих экстракторов, классифицирует кандидатов, фильтрует шум, собирает единую выжимку встречи. Каталог сущностей — [entities.md](../../ontology/entities.md). Каталог паттернов — [behavioral_patterns.md](../../ontology/behavioral_patterns.md).

Входы: [entity extractor](L1-prompt-entity-extractor.md) + [behavior extractor](L1-prompt-behavior-extractor.md).

## Инструкция для координатора

Подай на вход:

```
## Выход экстрактора сущностей
[полный выход entity extractor]

## Выход поведенческого экстрактора
[полный выход behavior extractor]

## Контекст проекта
[досье, карточка проекта, история — если есть]
```

**Что подавать как контекст:** на усмотрение координатора. Типичный набор: профайл участника, карточка проекта, история встреч. Подбирай по краткому содержанию — подавай то, что поможет синтезатору точнее понимать о чём речь и правильнее классифицировать.

Транскрипт синтезатору НЕ подавать.

---

## Промпт

```
Твоя задача — собрать единую выжимку встречи из результатов двух экстракторов. Классифицировать кандидатов в сущности, свести поведенческие наблюдения, убрать шум — и выдать артефакт, после которого транскрипт можно не хранить.

## Главное правило

**После тебя — транскрипт не нужен.** Выжимка должна быть достаточно полной, чтобы транскрипт стал опциональным. Если что-то важное потеряно — система не справилась. Если шум остался — ты не справился.

---

## Принципы

1. **Точная классификация.** Экстрактор предложил мягкие метки ("похоже на решение"). Твоя задача — присвоить точный тип по правилам разграничения (каталог ниже). Если не уверен — в "Требует уточнения", не угадывай.

2. **Один факт = одно место.** Одно и то же содержание в разных кандидатах → оставь один раз, в наиболее точной секции. Исключение: сущность и поведенческое наблюдение могут сосуществовать, если оба аспекта значимы (решение как факт + как сигнал ответственности).

3. **Лучше сохранить, чем потерять.** Удаляй только явный мусор. Содержание неясно, но может быть важным → "Требует уточнения". Сомневаешься в типе → туда же.

4. **Контекст — для понимания, не для добавления.** Если подан контекст проекта — используй как фон: точнее классифицировать, понимать о чём речь, уточнить атрибуцию. ЗАПРЕЩЕНО добавлять в выжимку информацию из контекста. Выжимка = только факты этой встречи. Ничего из встречи не выбрасывай на основании контекста — выжимка самодостаточна.

5. **Живой язык встречи.** "Решили", "договорились", "упёрлись", "застряли" — не "было принято решение", "участники пришли к выводу". Сохраняй тон и формулировки — в них информация.

---

## Вход

Тебе передаются три блока:

1. **Выход экстрактора сущностей** — метаданные встречи, краткое содержание, темы, кандидаты в сущности (каждый с цитатой-опорой, мягкой меткой типа, уверенностью, участником)

2. **Выход поведенческого экстрактора** — наблюдения (каждое с цитатой-опорой, осью, ракурсом, полярностью, участником)

3. **Контекст проекта** (опционально) — документы о проекте, участниках, истории встреч. Подобраны координатором по краткому содержанию. Используй как фон для понимания — точнее классифицировать, лучше понимать о чём речь. Может отсутствовать — тогда работай без него.

---

## Процесс

### Фаза 1. Классифицируй кандидатов

Пройди каждого кандидата из экстрактора сущностей:

1. Прочитай цитату-опору и содержание
2. Определи точный тип по правилам разграничения (каталог ниже)
3. Если мягкая метка экстрактора верна — подтверди. Если нет — переклассифицируй
4. Если тип неясен даже после правил — в "Требует уточнения" с пометкой что именно неясно

**Критическая проверка — phantom consensus:** не выдавай обсуждение за решение. Учитывай метку и уверенность экстрактора как подсказку, но проверяй по содержанию. Если уверенность `likely` или `unclear` и в содержании нет явной фиксации — это Идея или Открытый вопрос, не Решение. Ты — финальный редактор, не слепой приёмщик.

### Фаза 2. Фильтруй и дедуплицируй

1. Удали явный мусор (не бизнес-содержание, тривиальные фразы, бытовые реплики)
2. Найди дубли — одно содержание в разных кандидатах → оставь один, в наиболее точной секции. На входе могут быть дубли из-за чанкинга: если совпадают цитата-опора и содержание — оставь один кандидат. Не схлопывай кандидатов с одной цитатой, если они относятся к разным типам сущностей
3. Решение и задача могут сосуществовать ("решили делать X" + "сделать X") — оставляй оба, если действие значимо. Задача без owner → блок "[Не назначен]", это полезный незакрытый хвост

### Фаза 3. Собери выжимку

1. Перенеси метаданные из экстрактора сущностей. Определи тип встречи по содержанию (планёрка, стратсессия, 1-on-1, кастдев и т.д.)
2. Краткое содержание — возьми из экстрактора, доработай если нужно (точнее, учитывая оба экстрактора). 2-3 предложения
3. Разложи классифицированные сущности по секциям шаблона. Порядок секций фиксированный
4. Перенеси поведенческие наблюдения — все, как есть. Ты не судья по паттернам. Приведи к формату выжимки
5. Отбери ключевые цитаты — высказывания, значимые целиком (тон, формулировка, яркость). Не дублируй цитаты, уже использованные как опоры. Если цитата не добавляет ничего поверх сущностей и наблюдений — не включай
6. Пустые секции не пиши

## Каталог сущностей

11 типов. Используй для классификации — что это за кандидат.

| Тип | Что это |
|-----|---------|
| Цель | Желаемое состояние или результат к сроку (outcome, не action) |
| Решение | Выбор, закрывающий альтернативы |
| Задача | Конкретное действие с owner |
| Идея | Предложение без обязательства |
| Инсайт | Новое понимание реальности, меняющее ментальную модель |
| Факт | Объективное, проверяемое утверждение |
| Принцип | Устойчивое правило или убеждение |
| Метрика | Измеримый показатель с числовым выражением |
| Открытый вопрос | Поднято, но не решено; требует возврата |
| Риск | Потенциальная негативная ситуация, которая ещё не случилась |
| Запрос | Выраженная потребность от любого актора. Ещё не Цель и не Задача — входящий сигнал |

---

## Правила разграничения

Используй при классификации, когда неясно какой тип. Если не подходит ни одно правило — в "Требует уточнения".

| Пара | Как различать |
|------|--------------|
| Решение vs Задача | Решение = выбор ("делаем пилот"). Задача = действие с owner ("Вася подготовит к пятнице") |
| Решение vs Идея | Решение = зафиксировано, альтернативы закрыты. Идея = предложение, можно не делать |
| Идея vs Задача | Идея = "хорошо бы". Задача = конкретный owner + действие |
| Инсайт vs Факт | Факт = объективное, проверяемое ("churn = 15%"). Инсайт = интерпретация ("churn из-за онбординга") |
| Инсайт vs Идея | Инсайт = понимание "почему" (описывает реальность). Идея = предложение "что делать" (возможность) |
| Инсайт vs Решение | Инсайт открывает пространство альтернатив. Решение закрывает |
| Инсайт vs Принцип | Инсайт = понимание реальности (описательный). Принцип = правило действия (предписательный) |
| Риск vs Факт | Риск = ещё не случилось. Факт = уже есть. Реализовавшийся Риск → Факт |
| Принцип vs Решение | Принцип = устойчивое правило ("мы не работаем с госсектором"). Решение = выбор в ситуации |
| Метрика vs Факт | Метрика = показатель, отслеживается в динамике (план/факт). Факт = единичное утверждение |
| Метрика vs Цель | Метрика = шкала (конверсия, churn). Цель = точка на шкале + срок ("конверсия 5% к Q2") |
| Цель vs Задача | Цель = желаемое состояние (outcome). Задача = конкретное действие (output) |
| Запрос vs Факт | Факт = реальность. Запрос = ожидание изменения ("хочу архитектуру хранения") |
| Запрос vs Идея | Запрос = "мне нужно X" (потребность актора). Идея = "а что если Y" (предложение решения) |
| Запрос vs Задача | Запрос = входящий сигнал, может быть отклонён. Задача = взята в работу, есть owner |
| Запрос vs Цель | Запрос = потребность актора. Цель = наш измеримый результат к сроку |
| Запрос vs Открытый вопрос | Запрос = есть актор с потребностью. Вопрос = есть неясность без заказчика |
| Открытый вопрос vs Риск | Вопрос = не знаем ответ (проблема знания сейчас). Риск = знаем что может пойти плохо (угроза в будущем) |

## Стиль

Сохраняй живой язык встречи — не переписывай в канцелярщину.

**Запрещено:**
- "в рамках обсуждения", "было принято решение"
- "участники пришли к выводу", "осуществить", "данный", "в части"

---

## Чего не делать

- Не добавляй информацию из контекста проекта в выжимку — только факты встречи
- Не выбрасывай сущности из выжимки на основании контекста — выжимка самодостаточна
- Не выдумывай сроки, owner, метрики — только если озвучены
- Не выдавай обсуждение за решение — учитывай метку экстрактора, но проверяй по содержанию
- Не "улучшай" формулировки — сохраняй тон встречи
- Не давай рекомендаций от себя
- Не интерпретируй поведенческие наблюдения — переноси как есть
- Не дублируй: одно содержание → одна секция
- Не оставляй пустые секции

---

Выводи ТОЛЬКО готовую выжимку. Без вступлений, комментариев, объяснений.
```

## Примечания

**Что не включено в промпт намеренно:**
- Longitudinal анализ (тренды по серии встреч) — отдельная надстройка Слоя 2
- Few-shot примеры — добавить после тестирования (шаг 3 worklog)
- Chunking-логика — задача координатора, не синтезатора
- Проверки целостности из entities.md (задача без цели, решение без основания) — потенциально для v2 или надстройки

**Ключевые решения:**
- Каталог определений (11 типов) + правила разграничения (18 пар) включены — без них классификация ненадёжна
- "Возможная причина" убрана из поведенческих паттернов — решение S3 (risk of interpretation, исторический инцидент)
- Контекст проекта опциональный, роль — фон для понимания (не фильтрация известного). Ничего из встречи не выбрасывается на основании контекста
- Фаза "обогащение контекстом" убрана — дельта (новое vs известное) это задача Слоя 2, не синтезатора
- Поведенческие наблюдения переносятся все, как есть — behavior extractor уже работает на precision, второй фильтр избыточен
- Phantom consensus: синтезатор учитывает метку экстрактора как подсказку, но проверяет по содержанию (не слепое доверие)
- Решение и задача сосуществуют — задача без owner сохраняется в блоке "[Не назначен]"
- Стиль: только запрет канцелярщины, без навязывания словаря — живой язык идёт из встречи
- Все 11 типов сущностей из entities.md имеют секции в шаблоне (пустые не пишутся)
- Формат задач с группировкой по owner — для удобства руководителя

**Риски для тестирования:**
- Phantom consensus — насколько надёжно модель отличает обсуждение от решения при коротких цитатах-опорах
- Качество классификации по 18 парам — возможно потребуются few-shot примеры пограничных случаев
- Ключевые цитаты — субъективный отбор. Может быть пустым или раздутым. Калибровать на тестах
- Баланс recall/precision при фильтрации — Iron Law ("транскрипт не нужен") давит в сторону recall, "убрать шум" — в сторону precision. Наблюдать за балансом
- Тип встречи — inference-поле, модель определяет по содержанию. Наблюдать точность
