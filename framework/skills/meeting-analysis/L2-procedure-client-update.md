# Обновление досье контрагента по выжимке встречи

L2-надстройка над выжимкой встречи. Получает готовую выжимку, делает файл-центричный проход по досье контрагента, предлагает обновления исполнителю.

## Вход

1. **Выжимка** -- готовый файл `_summary.md` (результат L1-пайплайна)
2. **Контекст от исполнителя** -- ответ на вопрос: "Что за встреча? Кто контрагент? Кто участвовал?"

## Выход

Список обновлений по файлам. Каждое обновление: какой файл -> что добавить/изменить -> почему (ссылка на выжимку). Применяется ТОЛЬКО после ОК исполнителя.

## Scope

Встречи с одним контрагентом (клиент, партнёр, подрядчик). Предполагает досье по стандартной структуре клиентского проекта (profile.md, company.md, project.md и т.д.).

Вне scope этой версии: встречи с несколькими контрагентами одновременно, внутренние встречи (-> L2-procedure-scaffold-update.md).

---

## Шаг 1: Определи scope обновления

По ответу исполнителя определи:
- `{project_dir}` -- папка проекта контрагента (содержит profile.md, meetings/ и т.д.)
- Тип встречи (кикофф, регулярная, стратсессия и т.д.)

---

## Шаг 2: Собери контекст

1. Прочитай выжимку
2. Прочитай файлы досье **которые существуют**: profile.md, company.md, project.md, active.md, setup.md, decisions.md, progress.md. 

---

## Шаг 3: Файл-центричный проход

### Iron Law обновления

После обновления -- состояние проекта в файлах актуально. Выжимка остаётся как полная запись встречи. Не всё из выжимки должно попасть в файлы -- это нормально.

### Принцип: выжимка щедрая, файлы досье строгие

Выжимка сохраняет всё (Iron Law L1). Файлы досье -- только то, что имеет устойчивую ценность: стратегические решения, факты о бизнесе, задачи с owner, изменения в целях и структуре. Операционная мелочь, разовые реплики, бытовые просьбы -- остаются в выжимке, не попадают в файлы.

---

Пройди по каждому прочитанному файлу и спроси по его миссии: "нужно ли обновить этот файл с учётом новой информации из выжимки?"

**НЕ раскидывай сущности из выжимки по файлам.** Проверяй файлы -- это принципиально другой алгоритм.

### Чеклист

| Файл | Миссия | Тест-вопрос |
|------|--------|-------------|
| profile.md | КТО этот человек | Узнали новое о человеке (стратегия, предпочтения, боли, принципы, язык)? |
| company.md | ГДЕ он работает | Новые факты о бизнесе (цифры, люди, оргструктура, стратегия компании)? Тест: останется при смене руководителя? |
| project.md | ЧТО мы делаем | Новые запросы на решения? Изменилась модель работы, метрики успеха, программа? |
| decisions.md | ПОЧЕМУ так решили | Были стратегические решения? Тест: через 3 мес спросят "почему так?" -- это ответ? |
| active.md | ЧТО АКТУАЛЬНО сейчас | Сначала сопоставь новое с existing -- связанные объединяй, не дублируй. Новые задачи (по owner)? Открытые вопросы? Идеи? Риски? Повестка следующей встречи? Выполненные задачи -> отметить `[x]`, перенести в "Подтвердить" |
| setup.md | КАК настроено | Изменилась конфигурация (инструмент, структура, настройки)? |
| progress.md | КАК двигались | Дописать 2-4 строки о встрече (что сделано, ключевой сдвиг, обратная связь) |

---

## Шаг 4: Покажи обновления исполнителю

Формат предложения:

```
### {имя файла} ({путь})

**Обновление:**
- {конкретная правка}

**Основание:** {ссылка на пункт выжимки}
```

Применяй ТОЛЬКО после ОК исполнителя. Если исполнитель отклоняет обновление -- не спорь, оставь в выжимке (catch-all).

---

## Правила маршрутизации

**Один аспект -- одно место.** Одно событие может обновить несколько файлов, если затрагивает разные аспекты. "Перешёл на новый инструмент" -> setup (конфигурация) + profile (опыт человека). Разные файлы отвечают на разные вопросы -- это не дублирование.

**Catch-all.** Не можешь однозначно маршрутизировать -- оставь в выжимке. Лучше не разложить, чем разложить неправильно.

**Антипаттерны:**
- Конфигурацию (инструмент, настройка) в profile -> setup
- Прогресс ("разобрал 50 файлов") в profile/company -> progress или протокол
- Задачи в project -> active
- Решения без контекста в project -> decisions (нужны why + альтернативы)
- Конкретные события/даты в "Кратко" -> progress
- Нет данных -- не додумывать. Если факт не подтверждён источником (выжимка, транскрипт) -- спрашивать исполнителя
- Операционный шум в reference-файлы. company.md = ключевые люди и их роли, НЕ "Коля попросил фидбэк". Тест: поможет ли это при подготовке к следующей встрече через месяц? Нет -> не записывать

**Даты-источники.** Volatile-данные (цифры, статусы, решения, люди) -- inline-дата `(YYYY-MM-DD)`. Стабильные факты -- без дат.

---

## Форматы записей в log-файлах

**progress.md:**
```
## Встреча N (YYYY-MM-DD) -- тема
{2-4 строки: что сделано, ключевой сдвиг, обратная связь}
```

**decisions.md:**
```
## YYYY-MM-DD: Название решения
**Статус:** Актуально
**Решение:** Что решили
**Почему:** Контекст
**Альтернатива:** Что рассматривали и отвергли
```

---

## Ограничения DRAFT v1

1. **Предполагает стандартную структуру клиентского проекта.** Файлы без миссии в frontmatter -- маршрутизировать по чеклисту выше.
2. **Встречи с несколькими контрагентами вне scope.** Обрабатывать как отдельные проходы по каждому досье.
3. **Нет автоопределения проекта.** Исполнитель указывает контрагента и project_dir.
