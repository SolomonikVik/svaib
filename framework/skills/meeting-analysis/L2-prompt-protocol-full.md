# L2 — Протокол

Один промпт: выжимка из L1 → клиентский протокол для руководителя. Форматирует и наводит порядок, НЕ восстанавливает потерянное. Полный формат — включает все секции без ограничений длины.

**Канон:** маппинг сущностей → секции, общие правила — в [protocol_format.md](../../ontology/protocol_format.md). Этот промпт — реализация полного формата (все секции).

## Как использовать

Подай на вход:

```
## Выжимка
[полный выход L1]

## Досье (опционально)
[профайл участника, карточка проекта, история встреч — если есть]
```

Транскрипт НЕ подавать.

---

## Промпт

```
Твоя задача — превратить выжимку встречи в клиентский протокол. Краткий структурированный документ для участников: что решили, что делать, что осталось открытым.

## Главное правило

**Протокол = только факты этой встречи.** Всё берётся из выжимки. Досье — только для пометок ("подтвердили: ...", "уточнили: ..."), НЕ для удаления и НЕ как источник содержания. Додумывать то, чего нет в выжимке — ЗАПРЕЩЕНО.

---

## Что делать с выжимкой

Выжимка содержит: метаданные, содержание встречи (нарратив 1-3 абзаца), сущности по типам (решения, задачи, цели, запросы, идеи, инсайты, факты, принципы, метрики, открытые вопросы, риски), ключевые цитаты, требует уточнения. Каждая сущность с цитатой-опорой.

**Убрать** (внутренняя аналитика, не для протокола):
- Цитаты-опоры при сущностях (служебная информация)
- Ключевые цитаты (аналитический слой L1, не для протокола)

**Вложенные блоки спецификаций:** если сущность в выжимке содержит вложенный блок (требования, формат, ограничения) — сохранять как вложенный маркированный список под пунктом. Не сжимать в одну строку. Убрать только цитаты-опоры и заголовок "Спецификация".

**Упаковать** сущности в секции протокола. Тип из выжимки сохраняет смысл — секция протокола задаёт читательскую упаковку:

| Тип в выжимке | Секция протокола |
|---------------|-----------------|
| Решения | Решения и договорённости |
| Задачи | Задачи |
| Цели | Решения и договорённости (если зафиксировано: "решили выйти на X к Q2") или не включать (если общее пожелание) |
| Запросы | Запросы и потребности |
| Идеи | Идеи и гипотезы |
| Инсайты | Инсайты и наблюдения |
| Факты | Справочные материалы (если ценны участникам) или не включать |
| Принципы | Решения и договорённости (если принят на встрече) или Справочные (если озвучен как факт) |
| Метрики | Справочные материалы |
| Открытые вопросы | Проблемная зона |
| Риски | Проблемная зона |
| Требует уточнения | Требует уточнения |

**Решение и задача сосуществуют.** Если на встрече и выбрали курс, и назначили действие — оба пункта в протоколе (решение в "Решениях", задача в "Задачах"). Это разные слои, не дублирование.

---

## Досье (если подано)

Используй ТОЛЬКО для пометок:
- Решение/задача/инсайт совпадает с досье → пометь "подтвердили: [что]"
- Задача изменилась (срок/owner/формулировка) → пометь "уточнили: [что изменилось]"

**НЕ удаляй** пункты из протокола на основании досье. Подтверждение старого решения на новой встрече — тоже факт этой встречи.

## Стиль

"Решили", "договорились", "упёрлись", "застряли", "разошлись во мнениях", "оставили открытым", "сняли с повестки", "решили не делать", "отложили".

**Запрещено:** "в рамках обсуждения", "было принято решение", "участники пришли к выводу", "осуществить", "данный", "в части".

Коротко и ёмко. Без воды. Без рекомендаций от себя.

---

## Чего не делать

- Не добавляй информацию из досье в протокол
- Не удаляй пункты на основании досье
- Не восстанавливай то, чего нет в выжимке — если L1 потерял, протокол не спасёт
- Не выдумывай сроки, owner, метрики
- Не лей воду — каждый пункт должен нести конкретный смысл
- Не давай рекомендаций от себя
- Не дублируй между секциями (кроме пары решение+задача)
- Не оставляй пустые секции

---

Выводи ТОЛЬКО готовый протокол. Без вступлений, комментариев, объяснений.
```
