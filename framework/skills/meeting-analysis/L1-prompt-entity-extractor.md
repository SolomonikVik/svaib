# Экстрактор сущностей

Первый параллельный поток пайплайна ([meeting_analysis.md](../../methodology/meeting_analysis.md)). Каталог сущностей — [entities.md](../../ontology/entities.md). Параллельно: [behavior extractor](L1-prompt-behavior-extractor.md). Следом: [synthesizer](L1-prompt-synthesizer.md).

## Инструкция для координатора

Подай на вход промпт + транскрипт. Если есть — добавь перед транскриптом блок контекста (глоссарий, маппинг Speaker N → имя/роль, цель встречи). Досье проекта экстрактору не давать — только синтезатору.

При файле >60KB — применить чанкинг: [L1-procedure-chunking.md](L1-procedure-chunking.md).

```
## Контекст
Глоссарий: ...
Участники: Speaker 1 → Виктор (основатель), Speaker 2 → Алексей (CTO)
Цель встречи: ...

## Транскрипт
[текст встречи]
```

---

## Промпт

```
Твоя задача — извлечь из транскрипта встречи всё, что похоже на бизнес-сущности: решения, задачи, цели, идеи, факты, риски и другое.

## Главное правило

**Не бойся ошибиться — бойся упустить важное.** После тебя будет редакция: синтезатор проверит, уточнит, уберёт лишнее. Лучше извлечь лишнее, чем потерять.

---

## Принципы

1. **Текст первичен.** Извлекай только то, что сказано в транскрипте. Не додумывай, не интерпретируй, не добавляй из контекста. Если факт не прозвучал — его нет.

2. **Quote-first.** Работай от текста, не от схемы. Найди значимый фрагмент → определи что это. Нет цитаты — нет кандидата.

3. **Мягкие метки.** Ты предлагаешь тип ("похоже на решение"), не утверждаешь. Точная классификация — задача синтезатора, у которого будет больше контекста.

4. **Живой язык.** Сохраняй тон встречи. "Решили", "упёрлись", "забили" — не "было принято решение", "участники столкнулись с затруднением".

---

## Вход

Тебе передаётся транскрипт встречи. Перед транскриптом может быть блок контекста — глоссарий, участники, цель встречи. Контекст помогает правильно прочитать текст, не более. Не извлекай сущности из контекста.

---

## Процесс

### Шаг 1. Определи темы

Прочитай транскрипт целиком. Выдели основные темы, которые обсуждались.

### Шаг 2. Извлеки кандидатов

По каждой теме пройди транскрипт и найди фрагменты, которые содержат бизнес-сущности. Один фрагмент может содержать несколько сущностей — извлекай каждую отдельным кандидатом. Для каждого кандидата:

1. Выпиши цитату (5-15 слов, дословно)
2. Определи тип — на что похоже? Используй каталог ниже
3. Сформулируй содержание своими словами (кратко, 1-2 предложения)
4. Отметь уверенность:
   - `certain` — явный маркер в речи ("решили", "сделаю к пятнице", число + источник)
   - `likely` — похоже, но нет явного маркера ("надо бы", "хорошо бы", "может стоит")
   - `unclear` — важное, но неясно что это
5. Укажи участника (кто сказал / к кому относится)

### Шаг 3. Собери метаданные и краткое содержание

- Метаданные: дата, участники, длительность (из транскрипта или контекста; если нет — unknown)
- Краткое содержание: о чём встреча, к чему пришли (коротко)

---

## Каталог сущностей

11 типов. Используй как ориентир — что искать и по каким маркерам в речи.

| Тип | Что это | Маркеры в речи |
|-----|---------|---------------|
| Цель | Желаемое состояние к сроку (outcome, не action) | "хотим достичь", "к концу квартала", число + срок |
| Решение | Выбор, закрывающий альтернативы | "решили", "договорились", "делаем так" |
| Задача | Конкретное действие с owner | имя + действие + срок: "Вася подготовит к пятнице" |
| Идея | Предложение без обязательства | "а что если", "можно попробовать", "хорошо бы" |
| Инсайт | Новое понимание реальности | "оказывается", "я понял что", "проблема не в X, а в Y" |
| Факт | Объективное, проверяемое | число, ссылка на источник: "churn = 15%" |
| Принцип | Устойчивое правило / убеждение | "мы всегда", "мы никогда", "для нас важно" |
| Метрика | Измеримый показатель | "будем мерить", "KPI", "план X — факт Y" |
| Открытый вопрос | Поднято, но не решено | "нужно разобраться", "непонятно", "вернёмся" |
| Риск | Потенциальная угроза | "если не..., то...", "есть опасность" |
| Запрос | Выраженная потребность | "хочу", "мне нужно", "не хватает", "можно ли" |

## Чего не делать

- Не додумывай то, чего не было ("вероятно имелось в виду")
- Не давай рекомендаций от себя
- Не угадывай Speaker N ↔ имена — используй метки как в транскрипте
- Не угадывай исполнителя задачи — если не назван, не указывай
- Не выдумывай сроки — только если озвучен
- Не дублируй одно и то же содержание в разных кандидатах (но одна цитата может породить несколько кандидатов разных типов)
- Не выдавай обсуждение за решение — нет "решили/договорились" = не решение

---

Выводи ТОЛЬКО структурированный анализ. Без вступлений, комментариев, объяснений.
```

## Примечания

**Что не включено в промпт намеренно:**
- Правила разграничения из entities.md — задача синтезатора
- Контекст проекта (досье, история) — только синтезатору
- Проверки целостности — синтезатор или надстройка
- Few-shot примеры — добавить после тестирования (шаг 3 worklog)

**Риск для тестирования:** поле `участник` не различает "кто сказал" и "кого касается" (owner задачи, источник запроса). При короткой цитате синтезатор может не восстановить атрибуцию. Триггер на возврат: если на lebedev/smolianinov начнут путаться owner у задач.
