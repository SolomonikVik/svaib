# Протокол встречи

Надстройка Слоя 2 ([meeting_overlays.md](../../methodology/meeting_overlays.md)). Принимает выжимку ([L1-prompt-synthesizer.md](L1-prompt-synthesizer.md)) и превращает в клиентский протокол — краткий, структурированный, читаемый без контекста проекта.

## Инструкция для координатора

Подай на вход:

```
## Выжимка
[полный выход синтезатора]

## Досье
[опционально: профайл участника, карточка проекта, история встреч — если есть]
```

**Досье** — на усмотрение координатора. Если есть — промпт использует для пометок ("подтвердили", "уточнили"), НЕ для удаления содержания. Если нет — работает без него.

Транскрипт протоколу НЕ подавать.

---

## Промпт

```
Твоя задача — превратить выжимку встречи в клиентский протокол. Краткий структурированный документ для участников: что решили, что делать, что осталось открытым.

## Главное правило

**Протокол = только факты этой встречи.** Всё берётся из выжимки. Досье — только для пометок ("подтвердили: ...", "уточнили: ..."), НЕ для удаления и НЕ как источник содержания. Додумывать то, чего нет в выжимке — ЗАПРЕЩЕНО.

---

## Что делать с выжимкой

Выжимка содержит: метаданные, краткое содержание, сущности по типам (решения, задачи, цели, запросы, идеи, инсайты, факты, принципы, метрики, открытые вопросы, риски), поведенческие паттерны, ключевые цитаты, требует уточнения. Каждая сущность с цитатой-опорой.

**Убрать** (внутренняя аналитика, не для клиента):
- Поведенческие паттерны (правило v1 для внешнего протокола)
- Цитаты-опоры (служебная информация)
- Ключевые цитаты (по умолчанию убрать; оставить только если цитата — лучший носитель смысла, чем пересказ)

**Упаковать** сущности в секции протокола (структура ниже). Тип из выжимки сохраняет смысл — секция протокола задаёт читательскую упаковку:

| Тип в выжимке | Секция протокола |
|---------------|-----------------|
| Решения | Решения и договорённости |
| Задачи | Задачи |
| Цели | Решения (если зафиксировано: "решили выйти на X к Q2") или не включать (если общее пожелание) |
| Запросы | Решения (если оформлено как договорённость) или Идеи (если без обязательства) |
| Идеи | Идеи и гипотезы |
| Инсайты | Инсайты и наблюдения |
| Факты | Справочные материалы (если ценны участникам) или не включать |
| Принципы | Решения (если принят на встрече) или Справочные (если озвучен как факт) |
| Метрики | Справочные материалы |
| Открытые вопросы | Проблемная зона |
| Риски | Проблемная зона |
| Требует уточнения | Требует уточнения |

**Решение и задача сосуществуют.** Если на встрече и выбрали курс, и назначили действие — оба пункта в протоколе (решение в "Решениях", задача в "Задачах"). Решение = что выбрали, задача = кто делает. Это разные слои, не дублирование.

---

## Досье (если подано)

Используй ТОЛЬКО для пометок:
- Решение/задача/инсайт совпадает с досье → пометь "подтвердили: [что]"
- Задача изменилась (срок/owner/формулировка) → пометь "уточнили: [что изменилось]"

**НЕ удаляй** пункты из протокола на основании досье. Подтверждение старого решения на новой встрече — тоже факт этой встречи.

## Структура протокола

КОНТЕКСТ ВСТРЕЧИ
├── Тема / проект
├── Дата
├── Участники
└── Цель встречи (если озвучена)

ИТОГ ВСТРЕЧИ
└── Саммари в 2-3 предложениях: что произошло, что прояснилось, что изменилось.
    Не список тем. Не копия решений/задач.

РЕШЕНИЯ И ДОГОВОРЁННОСТИ
└── Что выбрали и готовы так действовать

ЗАДАЧИ
├── По исполнителям:
│   ├── Кто делает
│   ├── Что делает
│   └── К какому сроку (только если озвучен)
└── Исполнитель не назначен (если есть такие задачи)

ВАЖНАЯ ИНФОРМАЦИЯ (если есть)
├── Проблемная зона — риски, проблемы, открытые вопросы
├── Идеи и гипотезы — предложения без обязательства
├── Инсайты и наблюдения — что поняли/заметили
└── Справочные материалы — ссылки, цифры, документы

ТРЕБУЕТ УТОЧНЕНИЯ (если есть)
└── [Пункт] — [что неясно]

**Правила:**
- Нет информации — нет блока. Пустые секции не пишем
- Один факт = одно место. Не дублируй между секциями
- Подсекции "Важной информации" тоже пустые → не пиши

Дата: [дата]
Участники: [список]

---

ИТОГ ВСТРЕЧИ

[2-3 предложения]

## Стиль

"Решили", "договорились", "упёрлись", "застряли", "разошлись во мнениях", "оставили открытым", "сняли с повестки", "решили не делать", "отложили".

**Запрещено:** "в рамках обсуждения", "было принято решение", "участники пришли к выводу", "осуществить", "данный", "в части".

Коротко и ёмко. Без воды. Без рекомендаций от себя.

---

## Чего не делать

- Не добавляй информацию из досье в протокол
- Не удаляй пункты на основании досье
- Не включай поведенческие паттерны и цитаты-опоры
- Не выдумывай сроки, owner, метрики
- Не лей воду — каждый пункт должен нести конкретный смысл
- Не давай рекомендаций от себя
- Не дублируй между секциями (кроме пары решение+задача)

---

Выводи ТОЛЬКО готовый протокол. Без вступлений, комментариев, объяснений.
```

## Примечания

**Что переиспользовано из наработок:**
- Структура протокола — из meeting_protocol_ontology.md (meta/_inbox/)
- Контракт "факт встречи vs контекст" — из meeting_protocol_finalizer.md (meta/_inbox/)
- Стиль и запрещённые обороты — из meeting_protocol_finalizer.md
- Правила досье (пометки, не удаление) — уточнено по ревью Codex

**Что нового vs v1:**
- Структура из онтологии протокола (ВАЖНАЯ ИНФОРМАЦИЯ как зонтик для подсекций)
- Принципы Слоя 1 убраны ("лучше сохранить" → не для протокола)
- Маппинг: упаковка, не переклассификация ("тип сохраняет смысл, секция задаёт упаковку")
- Решение + задача сосуществуют (исправлено по ревью Codex)
- Открытые вопросы → Проблемная зона (исправлено по ревью Codex)
- Досье: только пометки ("подтвердили/уточнили"), НЕ удаление (исправлено по ревью Codex)
- Промпт компактнее: ~100 строк vs ~200

**Решения v1:**
- Поведенческие паттерны не включаются — правило v1 для внешнего протокола, не абсолют
- Ключевые цитаты — по умолчанию не включаются, исключение если цитата = лучший носитель смысла
- FRAME-скоринг — отложен
- Досье опционально, роль: пометки, не фильтрация

**Риски для тестирования:**
- **Агрессивная фильтрация без FRAME** — главный риск. Модель может слишком агрессивно выкидывать Цели / Запросы / Принципы при упаковке. Без FRAME-скоринга нет формального порога значимости — модель решает сама. Наблюдать: что теряется, что остаётся
- Упаковка Целей/Запросов/Принципов — насколько стабильно модель распределяет по секциям
- Итог в 2-3 предложения — качество саммари
- Досье-пометки — не раздувают ли протокол ("подтвердили: ...")
