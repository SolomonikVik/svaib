# email-assistant

Скилл триажа входящей почты с сигналом в Telegram.

## Что делает

3 раза в день (или по ручному вызову) читает новые письма в Gmail с момента последнего запуска, классифицирует по 3-tier (Reply Needed / Review / Noise), отправляет сводку важного в Telegram. Не действует на ящике (не отвечает, не помечает, не двигает).

Полная логика: [SKILL.md](SKILL.md).

## Файлы

| Файл | Что |
|------|-----|
| `SKILL.md` | Runtime-промпт для Claude. Минимальный, без воды |
| `README.md` | Этот файл — для разработчика/девопса при переносе |
| `rules.md` | Правила классификации. Приходит с минимальными дефолтами (noreply/newsletter/digest в «Шум»). Клиент дополняет под себя по мере появления шума |
| `state.json` | Чекпойнт `last_processed_at`. Создаётся скиллом автоматически при первом запуске |

## Требования к runtime

Скилл runtime-agnostic, но нуждается в двух доступных инструментах:

### 1. Gmail-tool

Скилл сам выберет доступный из tools контекста. Подключается по-разному:

| Runtime | Как подключить |
|---------|---------------|
| **Cowork / Claude.ai** | В чате: «подключи Gmail-коннектор» → Connect → OAuth Google. Включить Settings → Capabilities → Allow network egress: ON, после чего открыть **новую сессию** |
| **Claude Code локально** | Установить gmail-cli (например, `brew install gmail` или [google-api-python-client](https://github.com/googleapis/google-api-python-client)) и пройти OAuth. Альтернатива — MCP Gmail-сервер, подключённый к проекту |
| **Другой host** | Любой MCP-сервер с Gmail-инструментами или CLI-обёртка над Gmail API |

Если Gmail-tool недоступен — скилл сломается на Шаге 1. Решение текстом в промпте невозможно: нужно физически подключить инструмент.

### 2. Telegram-доставка

Скрипт `framework/skills/meeting-analysis/send_telegram.sh` (после рефакторинга channels — `framework/skills/channels/telegram/`). Скрипт читает `.env` в корне рабочей папки клиента:

```
TELEGRAM_BOT_TOKEN=...
TELEGRAM_CHAT_ID=...
```

`.env` должен быть в `.gitignore`.

## Развёртывание у клиента

1. Скопировать папку (`SKILL.md` + `rules.md`) в `claude/skills/email-assistant/` рабочей папки клиента (Drive). `rules.md` приходит с дефолтами — заполнять руками не надо
2. Подтвердить что Gmail-tool подключён к runtime клиента (см. таблицу выше)
3. Подтвердить что `.env` с TELEGRAM_BOT_TOKEN/CHAT_ID на месте, send_telegram.sh работает (тест: `bash claude/skills/send_telegram.sh "тест"` → `"ok":true`)
4. Передать клиенту короткую инструкцию: подключить Gmail-коннектор, создать 3 scheduled tasks (9:00/13:00/18:00 локального времени), запустить вручную «проверь почту» для проверки
5. Если клиент захочет фильтровать почту точнее — пишет в Cowork «помоги дополнить rules для email-assistant», Claude пройдётся по `rules.md` и допишет под него

## Конфигурация под клиента

`rules.md` — единственная точка кастомизации. Скилл его перечитывает при каждом запуске, поэтому правки можно вносить на лету без перезапуска чего-либо.

Структура `rules.md`: см. [rules.md](rules.md).

## Известные ограничения

- Только входящие, не работает с отправленными / черновиками / архивом
- Только чтение, никаких действий на ящике (Iron Law)
- Дедупликация через `state.json` — если файл удалить, скилл прочитает последние 24 часа как «новые» (может быть дубль)
- Лимиты Gmail API/коннектора зависят от runtime — конкретные числа смотреть в доках runtime'а
