# Контекстная память Second AI Brain

## Цель

Построить методологию контекстной памяти svaib — систему организации, поддержания и сборки информации, в которой: 

* информация хранится так, что агент легко и быстро может найти необходимое
* агент собирает минимально необходимый пакет контекста и достаточный для качественного решения пользовательской задачи
* система остается в актуальном состоянии — при добавлении, изменении, удалении материалов

## Связанные файлы

- [./file_spec.md](./file_spec.md) — формат файла (YAML, секции, H2) — требует обновления под конвенции из блока 7
- [../architecture.md](../architecture.md) — архитектура продукта, куда эта методология встроится
- [../../knowledge/context/](../../knowledge/context/) — база знаний по контекстной памяти (ICM, RAG, arscontexta, llm-wiki)
- [../../.claude/hooks/](../../.claude/hooks/) — реализованные хуки svaib, на которые ссылаются примеры в блоке 5

## Базовая архитектура управления контекстом

1. Работа с информацией, организация и поиск по ней, выполняется двумя разными способами:

   * структурированно-детерминированно → вот задача, вот конкретный маршрут её решения, вот так организовано хранение — папки, файлы
   * неструктурированно → вопрос открытый, маршрут неизвестен, хранение как в вики через связи всё со всем
1. Как организована информация:

   * через организацию и устройство папок как инфо-группы: название — вложенность — readme
   * через файлы-сборники/методички: readme, обзорный файл (как !context в knowledge)
   * через устройство файла как единицы информации: коротко — структура — требования к тексту
   * через связи между файлами: ссылки
1. Как агент собирает необходимый контекст:

   * базовый контекст на входе
   * протоколы навигации (и для структурированной, и для неструктурированной информации)
   * послойное углубление — как агент добирает контекст
       - поиск по базе через агентный grep
       - поиск через RAG
   * критерий достаточности — по какому признаку должен остановиться
1. Как документация поддерживается:

   * что происходит после появления документа — обновление связей, файлов-сборников: методология и работа агента
   * что происходит после редактирования документа: методология и работа агента
   * регулярный анализ и рефакторинг всей базы (ночной прогон и т.д.)
1. Уровни детерминированности агента:

   * работа через текст — инструкция для агента встроена в текст файла, слабая управляемость
   * работа через скилл — чёткая последовательность выполнения шагов со свободой агента
   * работа через хук/скрипт — жёстко заданный программный алгоритм

## 1. Базовая развилка: структурированная vs неструктурированная модель

### Структурированная модель

**Повторяемые операции**: задача понятна, маршрут известен, последовательность действий предопределена. Примеры: протокол встречи, обновление цели, weekly ritual, delivery pipeline.

**Модель управления:** папки = группы информации, предопределенные миссии файлов, чёткие маршруты сбора информации и контракты. Агент идёт по рельсам.

→ Референс:  [icm.md (knowledge)](../../knowledge/context/icm.md)

### Неструктурированная модель

**Произвольные задачи**: вопрос открытый, маршрут неизвестен, нужно исследовать и находить связи. Примеры: "что у нас есть по этой теме", "давай найдем общие паттерны / исследуем тему"

**Модель управления:** плоская структура + гибкий граф связей (wikilinks, MOC). Агент сводбодно навигирует по смыслу, не по папкам, останавливается на слое, который отвечает на вопрос

→ Референс: 

* **Arscontexta:**  [architecture.md (arscontexta)](../../knowledge/context/skill-graphs/architecture.md)
* **LLM Wiki (Karpathy):** [llm-wiki.md](../../knowledge/context/llm-wiki.md)

### Следствие для протокола агента

Первый шаг агента после определения задачи — понять: **я иду по намеченному маршруту или навигирую свободно?**

|                       | Структурированная                               | Неструктурированная                 |
| --------------------- | ----------------------------------------------- | ----------------------------------- |
| Маршрут               | Предопределён (контракт/skill)                  | Агент строит сам (граф/поиск)       |
| Контекст              | Фиксированный                                   | Адаптивный                          |
| Остановка             | По контракту (всё из Inputs загружено)          | По достаточности (ответ найден)     |
| Обновление информации | Агент пишет обновление по контрактам обновления | Wiki: ищет связи смыслы и обновляет |

**В svaib оба режима сосуществуют**. Есть предопределенные операции для помощников и ритуалов. Но в целом агент может переключаться между режимами внутри одной сессии.

## 2. Как организована информация

### 2.1. Папки

1. Папка — сгруппированная в одном месте информация
2. Вложенность папок — способ постепенного погружения в контекст (папка всех процессов → процессы по клиентам → процессы предпродаж)
3. Имя папки имеет значение — агент видит и читает полный путь как предложение. `clients/playbook/delivery/01_delivery_plan.md` — четыре уровня, каждый добавляет смысл. Имя папки = фрейминг, так же как имя файла
4. У папки есть README.md — обязательный файл для понимания структуры папки: что за информация, как устроена, куда идти дальше

   * также есть файлы CLAUDE.md, AGENTS.md — перебрасывают агентов на README (ссылкой)

**Ключевое ограничение папки:** невозможно разделить все сущности ровно по папкам. Некоторые файлы логически принадлежат нескольким папкам. Для этого существуют домены (см. п.2.2)

**Открытый вопрос:**

* как правильно называть README.md (некоторые называют !context.md) и как перебрасывать агентов?
* создавать ли пачку claude/agents/gemini в каждой папке?
* нужны ли README в маленьких папках где несколько документов?

### 2.2. Домены (области знаний)

Область связанных знаний, которые невозможно уложить в одну папку. Например, "управление целями" затрагивает management/, clients/, framework/. Домен управляется через файл-навигатор (`!domain_цели.md`) — описание, ландшафт, какая информация в каких файлах и зачем.

**Статус:** проектируемый элемент, в проекте пока не реализован.

### 2.3. Файлы-навигаторы (сборники, методички)

1. **Корневой README** — точка входа в воркспейс. Агент читает его первым. Не перечисляет все файлы — показывает ландшафт и направляет внимание. Описывает домены/папки верхнего уровня, их миссии, как они связаны. Агент решает: идти в файл домена или в конкретную папку
2. **README.md папки** — объясняет устройство папки (см. 2.1)
3. **!domain_sth.md** — объясняет устройство домена (см. 2.2)

### 2.4. Файлы

→ Детали: [file_spec.md](./file_spec.md) — файл требует обновления под конвенции этого блока

Главная задача: агент должен быстро найти файл, взять из него что нужно и не засорить контекст лишним.

1. **Название файла** = фрейминг для агента. "Обновление файлов" → агент думает "что в файле нужно обновить". Префикс `01_` означает "главный файл папки" (`01_presale_plan.md`, `01_delivery_plan.md`) — в обсуждении.

2. **YAML** — стандартизированные метаданные для поиска: description, status, type, tags. Агент не парсит YAML как базу данных — ищет текстовым поиском (`rg '^status: raw$'`). YAML работает потому что это конвенция: одинаковый формат, предсказуемое место, индустриальный стандарт.

3. **YAML `description`** — обязательное поле, одна строка = миссия файла — для чего он предназначен _(пример: "Операционный хаб проекта: всё, что требует внимания сейчас, в одном месте")_. Ключевая конвенция — поддерживает паттерн "карточки файла без чтения" (см. 3.0.1).

4. **Секция "Кратко"** — не резюме, а **поисковая точка входа**. Формулируется с ключевыми словами, по которым файл должны находить. Одно-два предложения, содержащие термины предметной области, имена систем, названия процессов _(пример: "Архитектура MVP svaib: трёхслойная система Google Workspace → Supabase → n8n для транскрибации встреч и извлечения задач" — в grep находится по запросам "архитектура", "MVP", "транскрибация", "встречи", "задачи")_. Без ключевых слов секция "Кратко" становится обычным резюме и перестаёт работать как входная точка поиска.

5. **Правила файла** — инструкция агенту в виде inline-правила: что можно добавлять, что нет, можно ли редактировать _(пример: "это лог — новая запись в начало файла")_. Детали механизма — см. 5.5.

6. **Текст** структурирован и размечен: заголовки, подзаголовки.

7. **Заголовки H2** — обязательно говорящие, самодостаточные. Агент получает структуру через команду (`Grep '^## '`), не читая содержимое, и может прочитать только нужную секцию. H2 внутри файла уникальны; самоописывающие; являются основной единицей чанкинга. H3 не используются как главный навигационный якорь, только как внутренняя детализация.
 "всё что касается блока — в блоке, он должен быть самодостаточным" - т.к. агент может взять только этот блок в память, а не весь файл (например, открытый вопрос связанный с темой блока - должен быть здесь же)

8. **Правила блока** (опционально) — локальная инструкция агенту, действует только внутри одного блока _(пример: "блок Открытые вопросы — если решено, удалить; если важное решение, перенести в decisions")_. Детали механизма — см. 5.5.

Совокупность этих конвенций (путь + имя + YAML description + карта заголовков) позволяет агенту быстро получить "карточку файла" и решить, читать его или нет. Механика карточки (способы A и B) описана в 3.0.1.

### 2.5. Связи между файлами

1. **Блок "Связанные файлы"** — файл + что там и зачем смотреть
2. **Ссылки внутри текста** — каждая ссылка внутри предложения даёт агенту контекст перехода: когда и зачем идти в другой файл (см. arscontexta)
3. **Ссылайся на папки и сводки, не на конкретные файлы.** При переименовании/переносе файлов перекрёстные ссылки ломаются. По умолчанию ссылка идёт на папку или её README, а не на отдельный файл. Прямые ссылки — только на стабильные файлы с явными маркерами стабильности (префикс `01_` и подобные).
4. **Один источник правды.** По умолчанию факт хранится в одном файле, остальные ссылаются обычной markdown-ссылкой ("метрики — см. `goal.md`"). Маркеры `[SOURCE] / [REF: file.md]` — крайний случай, когда факт обязан дублироваться в нескольких местах: единственное место обновления помечено `[SOURCE]`, копии — `[REF:]`; при расхождении верь `[SOURCE]`. Протокол проверки: при правке факта помеченного `[SOURCE]` прогнать `rg 'REF: filename.md'` по проекту, чтобы найти все зависимые места и либо обновить их, либо пометить как устаревшие.

### 2.6. Температура файла (hot → cold)

У каждого места хранения есть **температура** — как часто информация занимает контекст агента. Это ортогонально релевантности.

| Уровень | Где живёт | Когда загружено |
|---------|-----------|-----------------|
| **Hot** | `MEMORY.md`, session-start injection | Весь сеанс |
| **Warm** | `CLAUDE.md`, корневой README | С момента старта сессии |
| **On-demand warm** | README папки, `!domain_X.md` | При заходе в папку/домен |
| **Cold** | Обычные файлы | Только при целевом чтении |

**Принцип.** Место хранения выбирается по частоте использования, не только по релевантности. Hot-слой оплачивается токенами в каждом диалоге — туда попадает только то, что реально нужно всегда.

Симптом нарушения — когда `MEMORY.md` или `CLAUDE.md` растут до сотен строк: туда просочилась информация, которая нужна раз в месяц. Её место — cold или on-demand warm.

## 3. Как агент собирает контекст

### 3.0. Как агент видит информацию

Агент не видит репозиторий. Он видит текстовые строки: путь файла, результат поиска, фрагмент содержимого. Каждое из этих — отдельный явный запрос. Между запросами агент принимает решение: копать глубже или хватит.

#### 3.0.1 Текстовый поиск и навигация

**Операции:**

- **Поиск по пути / имени файла** — найти файлы по паттерну имени или расположения _(например: Claude Code `Glob`; Codex `rg --files`, shell search)_
- **Поиск по содержимому / метаданным** — найти текст внутри файлов, включая YAML-поля и заголовки _(например: Claude Code `Grep`; Codex `rg`, shell search)_
- **Построение карты заголовков** — получить структуру файла по заголовкам без чтения содержимого целиком _(например: Claude Code `Grep '^## '`; Codex `rg '^## '`)_
- **Диапазонное чтение файла** — прочитать начало файла или конкретный диапазон строк/блок _(например: Claude Code `Read(offset, limit)`; Codex — чтение диапазона через shell/local commands)_

**Три фильтра — цикл принятия решений (от дешёвого к дорогому)**

**Фильтр 1 — путь и имя**\
Самый дешёвый сигнал. Агент получает список путей (из поиска по именам/путям, поиска по содержимому, README, ссылок в тексте) и читает их как предложения. `framework/scaffold/principles.md` — три слова, каждое добавляет смысл. Имя папки = фрейминг, имя файла = суть, вложенность = иерархия контекста. На этом этапе агент отбирает кандидатов для чтения; сколько - зависит от задачи.

**Фильтр 2 — карточка файла.** 

Цель — понять релевантность файла быстрее, чем прочитать его целиком. Есть два способа собрать карточку; выбор между ними — вопрос реализации в конкретной среде, а не принципиальной развилки.

**Способ A — single-Read.** Один вызов `Read(limit=30)` — первые 30 строк содержат YAML, заголовок, "Кратко", "Связанные файлы". Решение "нужен ли остальной файл" принимается после этого единственного чтения. Подходит там, где диапазонное чтение дёшево и хорошо поддерживается.

**Способ B — metadata scan без Read.** Карточка собирается через два дешёвых поиска, не читая содержимое файла:

1. **Путь** — контекст (из поиска по пути/имени или ссылок в README)
2. **Имя файла** — суть (уже в пути)
3. **YAML `description`** — миссия файла (`Grep '^description:'` / `rg '^description:'` → одна строка)
4. **Карта заголовков** — структура (`Grep '^## '` / `rg '^## '` → все H2 с номерами строк)

Пример карточки, которую агент получает без единого Read:

```
framework/memory/file_spec.md

description: Стандарт файла Second AI Brain — формат, YAML, секции

## Кратко (12)

## Принципы (29)

## Именование файлов (41)

## Анатомия файла (76)

## YAML-заголовок (108)
```

Подходит там, где поиск по паттерну дешевле чтения — либо когда файлов-кандидатов много и single-Read каждого бы раздул контекст.

**Выбор способа — вопрос реализации среды**, не методологии. Конкретная политика (какой способ по умолчанию, когда fallback) фиксируется при разворачивании на конкретном runtime — см. блок 6.

**Требования к модели (работают для обоих способов):**

- `description` — обязательное поле YAML, одна строка, формат "миссия файла"
- Заголовки H2 — говорящие, самодостаточные (не "Раздел 1", а "Именование файлов")

Оба требования поддерживаются хуками — см. 5.4 (Description Updater, Heading Validator).

**Фильтр 3 — целевое чтение.** 

По карточке агент решает: читать файл или нет. Если нужен — делает точечное чтение нужной секции по найденной границе блока (например: `Read(offset=N, limit=M)` или shell-based range read) либо читает файл полностью при явной необходимости.

**Навигация между файлами**

Агент не обходит папки. Он двигается по следам релевантности:

- README → увидел ссылку → перешёл
- Поиск по содержимому нашёл совпадение → открыл файл → в связях увидел другой файл → перешёл
- YAML-поле tags или type → поиск по нему → нашёл соседей

Это не дерево (сверху вниз по папкам), а граф (от точки входа по связям). Папки задают грубую территорию, а внутри — свободная навигация по ссылкам и поиску.

#### 3.0.2 Семантический поиск (RAG)

В разработке. Текстовый поиск (grep) ищет точное совпадение слов. Семантический поиск (RAG) ищет по смыслу — находит релевантное даже если слова другие. Дополняет текстовый поиск, может обнаружить неожиданные находки.

→ Референсы: [rag.md](../../knowledge/context/rag.md), [agent-memory.md](../../knowledge/context/agent-memory.md)

### 3.1. Кто собирает контекст

Четыре варианта — от максимальной детерминированности к максимальной свободе:

1. **Код собирает** — для повторяемых задач с известной структурой (обработка встречи, план недели). Скрипт в хуке/скилле читает нужные файлы, агент получает готовый контекст. Не может пропустить, не может забыть.
2. **Координатор собирает сам** — координатор знает где искать, файлов немного. Читает напрямую, формирует полное понимание. Минус: если файлов много, забивает контекст поисковым мусором.
3. **Субагент-скаут** — территория незнакомая, файлов много. Субагент делает разведку: находит нужные файлы, возвращает координатору карту ("вот 4 файла, в каком что искать"). Координатор сам читает и формирует понимание. По сути субагент выполняет фильтр 1 (отбор кандидатов), координатор — фильтры 2–3 (чтение и осмысление).

   **Открытый вопрос:** где живут предуготовленные промпты/инструкции для таких разведывательных запросов — как часть соответствующего skill'а, как отдельная библиотека агентных шаблонов, или формируются координатором на лету? Без проработанного промпта субагент возвращает поверхностный мусор, и координатор всё равно вынужден читать сам.
4. **Субагент делает выжимку** — когда нужна справка, а не глубокое понимание (факт-чекинг, "есть ли у нас что-то по теме X"). Субагент читает, возвращает сжатое резюме. Минус: координатор оперирует чужими выводами — видит фильтрованное, не чувствует нюансов. Для принятия решений на основе материала — ненадёжно.

### 3.2. Базовый контекст

Что агент обязан знать до первого шага. Для повторяемых задач: собирается кодом (хук/скрипт), не агентом.

#### 3.2.1 Идентичность агента

- идентичность — задаёт профессиональную рамку
- методологические принципы
    - в т.ч. принципы работы с контекстом
- долгосрочные цели

#### 3.2.2 Базовая информация о проекте

- описание пользователя
- миссия и краткое описание проекта
- ключевые папки и файлы проекта
- типовые сценарии и ключевые маршруты

### 3.3. Сбор контекста под задачу

#### 3.3.1 Понимание задачи

Поток мысли клиента → перевести в структурированный запрос/задачу, синхронизироваться и только потом двигаться.

До чтения файлов:

- **Task** — какая задача и что должно быть в результате её решения
- **Scope** — какие домены затронуты, какие маршруты могут быть задействованы
- **Route** — куда идти за контекстом

#### 3.3.2 Выбор режима работы (предустановленные роли / команды)

Фрейм роли. Примеры: стратег, аналитик встреч, маркетолог, CTO.
Определяет профессиональную рамку и тип мышления. Опционально подтягивает обязательные файлы (стратег → цели, CTO → архитектуру). Включается пользователем через команды/роли, но агент может подтянуть исходя из задачи.

#### 3.3.3 Готовые маршруты (протоколы чтения)

Под определённые задачи есть готовые протоколы чтения — что именно читать и в каком порядке.

Примеры:

- Стратегия: README → my_profile → vision → goal → plan
- Встреча: participant_profile → previous_meetings → active → transcript

Если есть готовый протокол — используем его.

#### 3.3.4 Открытые маршруты (свободная навигация)

Когда готового маршрута нет — агент навигирует сам (см. 3.0).

### 3.4. Критерий достаточности

Контекст достаточен, если агент уже может дать надёжный ответ или выполнить следующий шаг без существенного риска ошибиться из-за непрочитанного файла.

**Проверка достаточности:**

* Задача понята — определены Task / Scope / Route
* Ключевые источники прочитаны — для важных фактов открыт канонический файл, а не только соседние упоминания
* Нет критической неопределённости — не осталось вопроса, который может существенно изменить ответ, решение или действие
* Следующий шаг ясен — агент уже может отвечать или работать, а не читать дальше “на всякий случай”

**Правило остановки:** 
агент прекращает чтение, когда следующий файл, скорее всего, добавит детали, но не изменит вывод, решение, план действия или формат ответа.

Читать дальше всё же обязательно, если файл входит в жёсткий протокол чтения

**Открытый вопрос:** 
как настраивать порог достаточности под разные типы задач. Для факт-чекинга, стратегии, редактирования файлов и записи в память этот порог может быть разным.

### 3.5. Связка с блоком 5

Всё описанное в блоке 3 — инструкции, которые агент *читает*. Memory, text, README помогают навигации, но не enforce'ят поведение. Критичные требования (например, "перед Read использовать limit", "перед запуском субагента определить scope", "при создании файла обновить связи") должны уходить в хуки. См. блок 5 — детерминированность.

## 4. Поддержка документации

Как система остаётся актуальной: что происходит при создании, изменении и накоплении документов.

### 4.1. Запись и обновление документа

При создании или изменении файла — два разных действия:

**Валидация** (безопасно, можно автоматизировать):

- Проверка соответствия архитектуре (формат, YAML, секции)
- Проверка: не дублируется ли информация с существующими файлами

**Обновление связей** (изменяет другие файлы, требует осторожности):

- Обновление README, файлов-сборников, обратных ссылок
- Важно разделять: валидация и автоизменение — разные классы действий

**Переименование, перенос, удаление файлов** — отдельный случай, где чаще всего ломаются связи. При rename/move нужно обновить все ссылки на файл по проекту. При удалении — проверить, не осиротели ли ссылки в других файлах.

**Механизм:** хук Write Validate — срабатывает перед/после сохранения. См. п.5 (детерминированность).

### 4.2. Закрытие сессии

Что из разговора достойно стать частью постоянной памяти системы, а что должно остаться просто следом рабочего процесса и не засорять будущее.

**Что попадает в память:**

- Решения, выводы, устойчивые факты
- Новые маршруты, улучшения методологии
- Развёрнутые ответы-анализы, которые стоит сохранить (query-as-accumulation — знание компаундится через вопросы, не только через обработку источников)

**Что не попадает:**

- Черновые рассуждения, сомнения без вывода
- Одноразовые промежуточные ответы
- Сырой материал без обработки

→ Референс: [llm-wiki.md](../../knowledge/context/llm-wiki.md)

**Механизм:** хук/скилл закрытия сессии. См. п.5 (детерминированность).

### 4.3. Обработка сырых данных

Raw-материалы не должны сразу становиться памятью. Транскрипт встречи, черновик идеи, набросок в _inbox — это сырьё. Если агент читает сырой транскрипт как контекст, он получает шум: повторы, отступления, незавершённые мысли.

**Правило:** raw-файлы не участвуют в маршрутах чтения. Сырьё должно пройти обработку (extraction → structured file), и только структурированный файл становится частью контекста, на который агент опирается.

→ Референс: [langextract](../../knowledge/context/!context.md) (Google) — extraction-слой для массовой обработки документов по онтологии.

### 4.4. Периодический аудит и рефакторинг

Регулярная проверка целостности базы — "ночной прогон":

**Чеклист аудита:**

- Противоречия между файлами
- Устаревшие даты и факты
- Битые ссылки
- Пропущенные кросс-ссылки
- Ослабление веса неактуальных документов
- "Дыры" в покрытии тем

→ Референсы:

- **Lint** ([llm-wiki.md](../../knowledge/context/llm-wiki.md)) — паттерн периодического аудита вики
- **6Rs Pipeline** ([architecture.md (arscontexta)](../../knowledge/context/skill-graphs/architecture.md)) — конвейер обработки знаний: Reflect (поиск неочевидных связей, обновление MOC) → Verify (аудит ссылок, соответствие форматам) → Rethink (рефакторинг базы)

**Статус:** в разработке.

## 5. Детерминированность: код vs текст

### 5.1. Принцип

То, что должно произойти гарантированно — делает код. То, что требует мышления — делает LLM.

Текст в файле — это контекст. Агент его читает, но может проигнорировать. Код — это исполнение. Агент не может его обойти, потому что код срабатывает до или после его хода.
Если написать в файле "всегда проверяй связи после правки" — агент это прочитает, но может не выполнить. 

**Доказательство:** rescue-log (9 из 18 кейсов, [rescue-log.yml](../../.claude/rescue-log.yml)). Текстовая инструкция "прочитай досье перед работой" не работает как enforcement — агент регулярно пропускает. Хук на тот же паттерн срабатывает всегда (кейс 2026-03-17, строка 196).

### 5.2. Уровни управления — от жёсткого к мягкому

| Уровень           | Что это                                                                    | Управляемость                                                | Пример                                      |
| ----------------- | -------------------------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------- |
| **Хук**           | Код, который срабатывает на событие жизненного цикла, перед/после действия или в validation pipeline | Жёсткий — агент не может обойти                              | before tool call / session start / session close |
| **Скилл**         | Предустановленный маршрут: последовательность шагов + опциональные скрипты | Средний — агент идёт по шагам, но интерпретирует внутри шага | `/weekly-planning`, `/close-session`        |
| **Текст в файле** | Инструкция агенту: "делай так", "всегда проверяй X"                        | Мягкий — агент читает, но может проигнорировать              | README, CLAUDE.md, Правила файла (5.5) |

**Правило выбора:** чем критичнее требование, тем жёстче уровень. Критичное (нельзя забыть) → хук. Повторяемое (одна и та же последовательность) → скилл. Рекомендация (полезно, но не критично) → текст.

### 5.3. Три типа хуков

"Хук" — это код, который гарантированно срабатывает на событие (старт сессии, перед записью файла, на новый промпт). Гарантирована **доставка** (хук точно отработает), но не результат — он зависит от того, что внутри хука. Код сам думать не умеет, поэтому есть три разных класса хуков.

**Тип A — детерминированные (чистый код).** Механические операции, где правила полностью формализуемы. Код проверяет, блокирует, переписывает. Надёжно на 100%. Внутри типа A два практических подкласса:

- **Content / behavior guard** — проверяет содержимое или структуру действия агента (текст ответа, параметры tool call, структура хода). Блокирует нежелательное. Работает на событиях `Stop`, `PreToolUse`, `PostToolUse`. Примеры: фильтр запрещённых фраз, валидация структуры хода, проверка YAML при Write.
- **Dependency / phase gate** — проверяет наличие обязательного артефакта или состояния, блокирует продолжение если условие не выполнено. Работает на границе между фазами pipeline или между шагами skill. Пример: Phase Gate не пропускает в фазу Design без артефакта Research.


**Тип B — впрыски контекста или инструкции.** Код гарантированно подаёт агенту что-то (файл, набор файлов, текст-напоминание) перед тем, как агент начинает работу. Решает, что делать с впрыснутым — LLM. Код не делает работу за агента, но гарантирует, что нужная информация окажется у агента на глазах в нужный момент.

**Тип C — гибриды код + LLM.** Хук вызывает отдельный маленький LLM (дешёвый, например Haiku), получает от него ответ и возвращает результат в основной контекст. Используется там, где нужно понимание (классификация задачи, отбор "что сохранить в память"), но это понимание должно произойти гарантированно — поэтому вынесено в отдельный вызов, которого основной агент пропустить не может.

**Важно:** тип B — не enforcement поведения, а enforcement доставки. Агент получит инструкцию, но может не выполнить. Тип C сильнее: классификация произойдёт даже если агент её пропустил бы в свободном режиме.

Ниже перечислены **архитектурные паттерны enforcement**, а не универсальные API конкретной среды. Один и тот же паттерн может реализовываться через tool hooks, lifecycle hooks, rules, skill scripts или post-validation. Примеры реализации для Claude Code, Codex и других сред вынесены в раздел 6.

### 5.4. Примеры хуков для контекстной архитектуры

**Тип A — детерминированные (content/behavior guard):**

1. **Write Validate** (проверка до/после изменения файла) — проверка формата, YAML, секций перед сохранением. Блокирует или помечает невалидный файл. Реализует 4.1.
2. **Rename Guard** — при переименовании/переносе файла автоматически обновляет все ссылки по проекту. Find/replace по файлам. Реализует 4.1.
3. **Raw Gate** — блокирует чтение raw-файлов из маршрутов чтения, если нет явного разрешения. Проверка по путям. Реализует 4.3.
4. **Nightly Audit** (по расписанию) — периодический прогон чеклиста из 4.4 (битые ссылки, противоречия, устаревшие даты). Скрипт обходит файлы и выдаёт отчёт.
5. **Content Guard** (Stop hook на ответ ассистента) — проверяет текст перед отправкой регуляркой, блокирует если найден запрещённый паттерн. _Реализовано в svaib:_ [`.claude/hooks/banned_phrases.py`](../../.claude/hooks/banned_phrases.py) (блокирует предложения закрыть сессию), [`.claude/hooks/check_deferred_actions.py`](../../.claude/hooks/check_deferred_actions.py) (блокирует "в следующий раз сделаю X" — заставляет сформулировать правку сейчас).
6. **Turn Structure Validator** (Stop hook на структуру хода) — анализирует последний ход: если substantive text AND action tool_use в одном ходе — блокирует и требует пересобрать. _Реализовано в svaib:_ [`.claude/hooks/enforce_rule10.py`](../../.claude/hooks/enforce_rule10.py) — enforce правила "текст отдельно от действий".

**Тип A — детерминированные (dependency / phase gate):**

7. **Read Limit Guard** (проверка параметров Read) — переписывает вызов Read: если нет ограничения диапазона, автоматически ставит `limit=30`. Заставляет агента сначала пройти дешёвую навигацию, потом решить, что читать дальше. Реализует активное правило 3.0.1.
   
   _Кейс-обоснование (2026-04, session-log):_ файл 21k токенов, 527 строк. Агент попробовал прочитать целиком — спасся только потому, что не влез в технический лимит. Если бы файл был 9k токенов (под лимит), все 9k попали бы в контекст впустую, хотя нужно было 15 строк. Хук предотвращает эту ситуацию детерминированно.
8. **Phase Gate** (проверка артефакта предыдущей фазы pipeline) — на старте новой фазы код проверяет наличие и валидность обязательного артефакта (файла) предыдущей. Если артефакт отсутствует или не соответствует контракту — следующая фаза не стартует. Пропустить шаг pipeline физически невозможно. → Референс: [knowledge/skills/last30days-skill.md](../../knowledge/skills/last30days-skill.md) — раздел "Code-Enforced Phases".

**Тип B — впрыски:**

9. **Session Orient** (на старте сессии) — при старте сессии код читает базовый контекст (идентичность агента, инфо о проекте, методологию) и инжектит в начало сессии. Реализует 3.2. _Реализовано в svaib:_ [`.claude/hooks/inject_session_context.py`](../../.claude/hooks/inject_session_context.py) — впрыскивает в субагенты последние N ходов родительской сессии + правила субагента.
10. **Context Injection** (на старте подзадачи или через skill-команду) — под конкретную задачу или субагента код собирает набор файлов и подаёт готовым. Референс: [OpenAI Symphony](https://github.com/openai/symphony). Общий паттерн для всех повторяемых операций.
11. **Task Classifier Prompt** (перед обработкой нового пользовательского запроса) — хук впрыскивает в начало каждого промпта напоминание: "Прежде чем читать файлы, определи task/scope/route по таблице папок". Гарантирует, что агент увидит инструкцию на каждом новом шаге, а не только при старте сессии.
12. **Pre-Action Reminder** (PreToolUse hook на конкретный инструмент) — перед вызовом инструмента код впрыскивает координатору чеклист или напоминание, релевантное этому действию. Отличается от Task Classifier Prompt тем, что срабатывает не на новый user prompt, а на намерение агента вызвать конкретный tool. _Реализовано в svaib:_ [`.claude/hooks/remind_subagent_rules.py`](../../.claude/hooks/remind_subagent_rules.py) — перед запуском субагента впрыскивает координатору 4-пунктовый чеклист (границы задачи, bash-правила, путь к артефакту, foreground).

**Тип C — гибриды:**

13. **Task Classifier LLM** (перед обработкой нового пользовательского запроса) — хук вызывает малую LLM с промптом + таблицей доменов, получает `{task, scope, route}`, инжектит в контекст основного агента. Реализует 3.3.1.

    _Кейс-обоснование (2026-04, telegram-delivery):_ задача "создать Telegram-бота для клиента" пересекает `clients/` + `framework/skills/`. Агент увидел только `clients/`, пошёл в Grep по содержимому (230 файлов мусора), канонический `telegram-delivery.md` в `framework/skills/` не нашёл. Если бы task/scope/route был первым обязательным шагом, оба домена попали бы в scope, и протокол чтения повёл бы через `framework/skills/`.

14. **Session Close** (при завершении сессии или по явной команде) — код собирает транскрипт сессии, LLM решает что попадает в память по критериям 4.2, код записывает результат в нужные файлы.

15. **Description Updater** (после создания или существенного изменения файла) — хук вызывает малую LLM: "прочитай файл, обнови YAML `description` так, чтобы одна строка точно отражала миссию файла". Обеспечивает, что description всегда актуален. Реализует конвенцию `description` из 2.4 и паттерн "навигация через метаданные" из 3.0.1. Без этого хука description обречён устаревать — человек его ручками не поддерживает.

16. **Heading Validator** (проверка при сохранении файла) — хук проверяет заголовки H2: соответствуют ли конвенции (говорящие, самодостаточные, не дублирующие друг друга). Вызов малой LLM с промптом "вот список заголовков этого файла, вот правила — всё ок или есть проблемы?". Если есть — возвращает предупреждение. Поддерживает "карту заголовков" в рабочем состоянии для паттерна из 3.0.1.

Примеры реализации по конкретным средам см. в разделе 6.

### 5.5. Rule Injection — правила внутри markdown

**Проблема.** Самый естественный способ зафиксировать правило работы с файлом — написать его прямо в файле, рядом с контентом. Человек открывает файл и видит: "открытый вопрос решён → удалить, не зачёркивать". Агент тоже видит — но при большом контексте (150k+ токенов) он инструкцию читает, помнит, но в момент действия игнорирует. Инструкция "размыта" в общем контексте.

**Идея.** Правила остаются внутри markdown как обычный текст с префиксом, но специальный хук инжектит их в контекст **непосредственно перед моментом действия** и **только релевантные** к изменяемому блоку.

**Формат в файле:**

```markdown
# Заголовок

> **Правило файла:** каждое изменение фиксируй в changelog внизу

## Открытые вопросы

> **Правило блока:** решён → удалить строку, не зачёркивать

- вопрос 1
- вопрос 2
```

**Два уровня:**

- **Правило файла** — глобальное, применяется при любом Edit в этом файле
- **Правило блока** — локальное, применяется только если Edit меняет контент этого блока

**Механизм (hook/tool interceptor перед изменением файла, тип B):**

1. Получает путь файла и payload изменения (например: `file_path`, `old_string`, `new_string`)
2. Читает файл, находит позицию `old_string`
3. Собирает **правила файла** (весь файл) + **правила блока** (ближайший `##` вверх от позиции)
4. Инжектит в контекст агента как отдельный блок: `"Перед применением Edit проверь правила: [...]"`

Реализация — bash-скрипт ~30 строк (grep + awk) или аналогичный обработчик в зависимости от среды. Правила пишутся человеком в свободном тексте, парсер не интерпретирует смысл — только доставляет в нужный момент.

**Почему это работает:**

- Правила живут рядом с контентом, а не в параллельной структуре (скилл/хук для каждого файла)
- Человек видит правило там, где оно применяется
- Агент получает не все 300 строк файла с размытой инструкцией, а сфокусированный блок "вот правила для этого конкретного изменения"
- Внимание актуализируется в момент действия, а не при старте сессии

**Ограничение.** Это тип B (впрыск), не enforcement — агент всё ещё может проигнорировать. Но вероятность резко падает, потому что правило попадает прямо перед действием, в жёсткой форме. Если нарушения продолжаются на конкретном правиле — апгрейд до типа C (LLM-валидатор вызывается только для этого правила).

**Кейс-обоснование (2026-04, active.md):** агент предложил зачеркнуть открытый вопрос, хотя в начале блока написано "решён → удалить строку, не зачёркивать". Правило было в файле, агент его прочитал, но при действии проигнорировал. Rule Injection Hook решил бы эту проблему: прямо перед Edit правило было бы инжектировано как отдельный блок, а не прочитано раз в середине 300-строчного файла.

**Развитие паттерна.** Тот же механизм (маркер в markdown + хук, читающий его в момент действия) применим к смежной задаче — отделению зон автоматической правки от авторского контента. Маркеры `<!-- AGENT_EDITABLE_START:name -->` / `<!-- END -->` задают границы, внутри которых хуки-автоматизаторы (Description Updater, Rename Guard и т.п.) могут менять текст; всё снаружи — только по явному указанию пользователя.

## 6. Реализация в разных средах (Claude Code, Codex и т.д.)

Архитектура выше описана в vendor-neutral терминах. Ниже — примеры её реализации в конкретных средах на момент обновления этого файла. Инструментарий сред развивается; при расхождении между архитектурным смыслом и текущим toolset приоритет у архитектурного смысла.

| Архитектурная операция | Claude Code | Codex | Комментарий / ограничение |
| ---------------------- | ----------- | ----- | -------------------------- |
| Поиск по пути / имени файла | `Glob` | `rg --files`, shell search | Эквивалент по смыслу, разный интерфейс вызова |
| Поиск по содержимому / метаданным | `Grep` | `rg`, shell search | Подходит для YAML-полей, заголовков, ключевых паттернов |
| Карта заголовков | `Grep '^## '` | `rg '^## '` | Позволяет собрать структуру файла без полного чтения |
| Диапазонное чтение файла | `Read(offset, limit)` | shell-based range read / локальные команды | В Codex это чаще реализуется менее нативно, чем в Claude Code |
| Context injection | `SessionStart`, `SubagentStart`, `!command`, additional context | `SessionStart`, `UserPromptSubmit`, rules, shell-mediated workflows | Оба рантайма поддерживают доставку контекста, но разными механизмами |
| Task classification before reading | `UserPromptSubmit`, малая LLM в hook | `UserPromptSubmit`, rules, малая LLM во внешнем workflow | Скорее lifecycle-level orchestration, чем file-tool hook |
| Validation before/after edit | `PreToolUse` / `PostToolUse` на `Write/Edit` | lifecycle hooks, rules, bash validators, post-validation | Claude Code сильнее в tool-level interception |
| Metadata update (`description`) | `PostToolUse` + малая LLM | `Stop`, `UserPromptSubmit`, shell-mediated workflow, batch audit/fixer | В Codex удобнее опираться на lifecycle hooks и внешние пайплайны |
| Session close / запись в память | `SessionEnd` / команда | `Stop` / команда / внешний workflow | В обоих случаях реализуемо, но триггеры и степень нативности различаются |

**Наблюдение по средам:**
- **Claude Code** сильнее там, где нужно жёсткое перехватывание конкретных tool calls.
- **Codex** сильнее опирается на lifecycle hooks, rules и shell-mediated workflows.
- **Другие среды / API-агенты** могут не иметь встроенных hooks вообще; тогда архитектура реализуется через skills, оркестратор и внешние валидаторы.

Референсы:
- [claude-code.md](../../knowledge/coding/claude-code.md) — текущие заметки по Claude Code
- Codex docs — hooks / rules / config (внешняя документация продукта)

## 7. Конвенции на выходе

Этот документ описывает **почему и как работает** контекстная память. Из него в svaib рождается набор **конвенций** — договорённостей о формате, которые потом оформляются как спецификации в `framework/memory/`.


### Список конвенций, которые нужно оформить
1. **Пути = название папок + вложенность папок.** Имя папки как фрейминг, иерархия как контекст. Что именно кладём на каждом уровне. (основа в п.2.1)
2. **Имя файла.** Префикс `01_` для главных файлов папки; имя файла = поисковый запрос, раскрывающий суть без чтения содержимого. Конкретные паттерны (case-стайл, датированные файлы) — на уровне [file_spec.md](./file_spec.md), а не методологии.
3. **YAML `description`.** Обязательное поле, одна строка, формат "миссия файла". Поддерживается хуком Description Updater. (новая, из п.2.4 и 3.0.1)
4. **Заголовки H2.** Говорящие, самодостаточные, работают как оглавление. Поддерживается хуком Heading Validator. (новая, из п.2.4 и 3.0.1)
5. **Inline-правила.** Префикс `> **Правило файла:**` / `> **Правило блока:**`. Обрабатывается хуком Rule Injection. (из п.5.5)
6. **"Работа с файлом".** Как задаются режим обновления, ограничения и допустимые действия над файлом; что остаётся текстовой инструкцией, а что мигрирует в inline-правила. (связка п.2.4 и 5.5)
7. **Обязательные маршруты чтения.** Какие файлы входят в протокол чтения для каких типов задач (стратегия, встреча, бот-клиент). (основа в п.3.3.3)
8. **`SOURCE / REF`.** Как обозначается канонический источник факта и как на него ссылаются производные файлы. (основа в п.2.5)
