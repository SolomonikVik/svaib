# Memory update protocol

## Миссия файла

Общий протокол обновления памяти Second AI Brain: как агент раскладывает новую информацию, обновляет существующие файлы, работает с сомнениями, сохраняет источники и не ломает связи.

## Граница

Этот файл отвечает за **как обновлять память безопасно**.

Не сюда:
- модель папок scaffold → `../methodology/scaffold/01_architecture.md`
- спецификация файла → `file_spec.md`
- канон файлов scaffold → `../methodology/scaffold/02_file-templates.md`
- исполняемые шаги конкретного помощника → `../skills/`

## Черновик

Заполнить из:

- `file_spec.md` (части про `[SOURCE]` / `[REF:]`, связи и актуальность)
- старых scaffold-принципов про маршрутизацию, сомнение, движение задач и сборки


### Сомнение остаётся в источнике

Если агент не может уверенно выбрать место, информация не раскладывается насильно. Она остаётся в исходном протоколе, `_inbox` или другом источнике и помечается как требующая разбора.

### AI пишет по контракту

Агент не имеет безусловного права переписывать память. Для каждого обновления он должен понимать режим записи:

- **обновить сам** — рабочие state/log-секции, где протокол разрешает автоматическое обновление;
- **предложить** — смысловые формулировки, где нужна проверка человека;
- **остановиться и спросить** — изменение стабильного ядра, удаление, слияние, перенос или риск потери контекста.

Если режим неясен, агент не переписывает файл молча, а оставляет предложение или просит подтверждение.


## 6. Маршрутизация решений (проверить)

В модели есть три уровня файлов с решениями — личный, компанейский, по подразделению/направлению. Граница:

| Критерий                                                                                                                                     | Куда                                                             |
| -------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Решение влияет на курс компании, цели, стратегию, vision. Через 3 месяца спросят «почему мы двигаемся туда?».                                | `02_strategy/06_decisions.md`                                    |
| Решение касается только одного подразделения/направления и не меняет курс. Через 3 месяца спросят «почему у commercial такой подход?».       | `04_company/{node}/04_decisions.md`                              |
| Решение охватывает несколько подразделений/направлений, но не меняет курс (например, как перераспределить нагрузку между sales и marketing). | `04_company/{parent}/04_decisions.md` (общая родительская папка) |
| Личное решение CEO — ещё не вынесено наружу, или принципиально не пойдёт наружу.                                                             | `01_ceo/04_decisions.md`                                         |
| Сомнение «курс или одно подразделение/направление?»                                                                                          | **Выше**, в `02_strategy/`. Компания важнее.                     |

Мелкие операционные выборы («не делать сегодня X», «перенести звонок») — не решения, не маршрутизируются, остаются в протоколах встреч или `02_active.md`
