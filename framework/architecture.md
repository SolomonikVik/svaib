# Архитектура продукта

## Кратко

Как устроен продукт Second AI Brain изнутри: слои, компоненты, связи. Три слоя (данные → память → помощники) внутри мета-рамки (методология + онтология). Разработка в skills/, реализация через plugin. Интерфейс — не наше.

## Связанные файлы

- 00_product.md — видение продукта, принципы, границы, бизнес-модель
- 01_overview.md — состояние реализации шести частей
- 04_decisions.md — продуктовые решения (runtime, границы, путь skills)
- ontology/ontology.md — сущности, связи, правила размещения
- methodology/methodology.md — протоколы, decision frames, масштабирование
- ontology/rituals.md — каталог ритуалов
- memory/file_spec.md — формат файлов: YAML, секции, ограничения
- memory/01_context_memory.md — протокол работы агента с информацией (операционализация слоя "Память")
- scaffold/ — готовый каркас (воплощение онтологии)
- skills/ — мастерская промптов и навыков (разработка помощников)
- plugin/README.md — деплоимый пакет (реализация помощников)

## Мета-рамка: Методология + Онтология

Не компонент продукта, а то, что удерживает всё вместе. Источник правды.

**Онтология** ([ontology.md](ontology/ontology.md)) — какие сущности существуют, как связаны, что куда кладётся. Справочник.

**Методология** ([methodology.md](methodology/methodology.md)) — как с этими сущностями работать: протоколы, decision frames, масштабирование. Каталог ритуалов — в [ontology/rituals.md](ontology/rituals.md). То, что зашивается в skills и agents.

Клиент может не читать онтологию и методологию напрямую — они зашиты в scaffold (структуру) и plugin (автоматизацию). Но они существуют как отдельные документы для развития продукта и обучения.

### Каноническая цепочка изменений

Любое изменение в продукте идёт сверху вниз:

```
методология / онтология   →  source of truth: что и как устроено
        ↓
schema (entities, protocol_format, file_spec)  →  контракты: формат данных
        ↓
промпты и процедуры (skills/)  →  реализация контрактов
        ↓
оркестраторы  →  связывание промптов в пайплайн
        ↓
README, навигация  →  описание того что есть
        ↓
производная документация  →  playbook, overlays, _inbox
```

Каждый нижний слой реализует контракт верхнего, не придумывает свою логику. Если промпт противоречит методологии — правим промпт. Если оркестратор передаёт не то, что требует промпт — правим оркестратор.

## Вертикали управленческих циклов

Третий разрез поверх двух уже описанных — продуктовых слоёв и частей framework. **Вертикаль — это не новая папка во framework и не часть продукта рядом со слоями.** Это управленческий цикл (встречи, метрики, стратегия, команда), который проходит **сквозь обе уже существующие карты**.

### Три карты архитектуры

**1. Три слоя продукта** — то, что работает у клиента:

- **Данные** — что лежит в клиентском пространстве
- **Память** — как AI понимает что читать, что актуально, как связаны сущности
- **Помощники** — что превращает данные и память в действие

**2. Шесть частей framework** — наши производственные модули, через которые мы продукт проектируем, собираем и доставляем:

| Часть framework | Роль в трёх слоях                           |
| --------------- | ------------------------------------------- |
| `ontology/`     | Мета-рамка: какие сущности существуют       |
| `methodology/`  | Мета-рамка: как с этими сущностями работать |
| `scaffold/`     | Реализация слоя **Данные** у клиента        |
| `memory/`       | Контракты слоя **Память**                   |
| `skills/`       | Разработка слоя **Помощники**               |
| `plugin/`       | Доставка помощников клиенту                 |

**3. Вертикали управленческих циклов** — третий разрез. Каждая вертикаль (`meeting`, `metrics`, `strategy`, `team`) проходит сквозь обе карты:

```text
Вертикаль meeting:
ontology/      → какие сущности встречи существуют
methodology/   → как анализировать встречу
scaffold/      → где у клиента живут протоколы и транскрипты
memory/        → как искать историю и контекст встречи
skills/        → промпты и процедуры meeting-analysis
plugin/        → что устанавливается клиенту
```

Никаких «3 vs 6 vs N» альтернатив. Это три ортогональных разреза одной системы.

Онтологический источник понятия — [ontology/management_cycles.md](ontology/management_cycles.md).

### Тест границы: что в горизонтали, что в вертикали

| Признак                                                                                              | Куда                                 |
| ---------------------------------------------------------------------------------------------------- | ------------------------------------ |
| Работает для любого CEO и любой темы (контекстная память, snapshot, версионирование, trace, semantic layer как паттерн) | **Горизонталь** (общая инфра слоёв)  |
| Привязано к конкретной теме (паспорта метрик, оркестратор встреч, шаблон стратсессии)                | **Вертикаль**                        |
| Конкретные значения у конкретного CEO (его таблицы, его метрики, его участники)                      | **Клиентская настройка** в скаффолде |

### Вертикаль ≠ скилл

- **Вертикаль** — управленческий цикл с собственными сущностями в слоях. Тяжёлая, долгоживущая, развивается уровнями зрелости.
- **Скилл** — атомарный помощник без своей инфраструктуры в слоях. Легковесный, переиспользуется.

Email, Telegram, работа с документами — **скиллы**, не вертикали (нет своих сущностей в слоях памяти и данных). Если у нового помощника нет своих сущностей в слоях — это скилл. Не размножать вертикали под каждый помощник.

### Зависимости вертикалей

**Только через горизонтальные слои, по стабильным ID.** Прямые вызовы внутренностей чужой вертикали запрещены.

- Каждая вертикаль декларирует, что **экспортирует** (например, metrics — `metric_revenue_daily` как стабильный ID).
- Каждая вертикаль декларирует, что **импортирует** (например, meeting может использовать `metric_*`).
- Meeting не вызывает metrics напрямую — запрашивает у слоя памяти «дай метрику X на дату Y», слой памяти знает провайдера.

Без этого правила вертикали срастаются.

### Лимит

Управленческие циклы конечны. **10 вертикалей максимум** в обозримой перспективе. Сейчас на горизонте: strategy, team, metrics, meeting, product, marketing, sales, projects (см. [ontology/management_cycles.md](ontology/management_cycles.md)). Если режется мельче — это скиллы.

## 1. Данные

Бизнес-контекст руководителя: цели, планы, решения, встречи, команда, подразделения/направления. Личные домены вне бизнеса (здоровье, хобби, семья) — вне scope продукта. `01_ceo/01_my_profile.md` содержит личность руководителя (ценности, стиль), но как входные данные для бизнес-решений.

**Принцип: LLM + People centered.** Каждый файл одинаково удобен для человека и для LLM-агента. Читаем глазами, парсится моделью без потери смысла. Это проектное требование к слою данных: структура и обвязка строятся так, чтобы обе стороны получали максимум.

### Структура

Какие файлы, какие папки, что куда кладётся.

- **Scaffold** ([scaffold/](scaffold/)) — готовый каркас папок и документов. Корневой канон: `01_ceo/`, `02_strategy/`, `03_team/`, `04_management/` + опциональные домены
- **Онтология** ([ontology.md](ontology/ontology.md)) — описание сущностей и связей поверх этого каркаса
- **Связи между файлами** — чёткая иерархия: `01_ceo/01_my_profile.md` → `02_strategy/01_vision.md` → `02_strategy/02_goal.md` → `02_strategy/03_plan.md` → `02_strategy/04_weekly_progress.md` → `02_strategy/05_timeline.md`

Наша ценность: клиент получает не пустую папку, а архитектуру — понятно что куда класть, как файлы связаны, когда какой обновлять.

### Обвязка

Как файл устроен внутри. Правила, которые делают данные машиночитаемыми и связными.

- **YAML-заголовок** — title, updated, version, scope, priority ([file_spec.md](memory/file_spec.md))
- **Стандартные секции** — "Кратко", "Связанные файлы", контентные секции
- **Связи** — каждый файл явно указывает связанные файлы → LLM может навигировать
- **Формат** — Markdown с ограничениями: <300 строк, ≤2 уровня вложенности, таблицы <20 строк
- **Маркеры источников** — [SOURCE], [REF:] для отслеживания фактов

Наша ценность: не просто файлы, а файлы с конвенциями. AI может парсить, находить связи, понимать актуальность.

## 2. Память

Свойство правильно устроенных данных — не отдельная технология, а результат структуры + обвязки + помощников.

Детали протокола (как агент находит и читает файлы, как собирает контекст под задачу, как поддерживает документацию, уровни детерминированности через хуки) — в [memory/01_context_memory.md](memory/01_context_memory.md). Спецификация единицы памяти — в [memory/file_spec.md](memory/file_spec.md).

### Протокол чтения

LLM знает в каком порядке читать файлы для получения полного контекста:

**Пример:**

```
README → 01_ceo/01_my_profile.md → 02_strategy/01_vision.md → 02_strategy/02_goal.md → 02_strategy/03_plan.md
```

Эти 5 файлов дают полный контекст за минимум токенов. Остальные файлы — по триггеру (methodology.md, секция "Протокол чтения для LLM").

### Связи и темпоральность

- **Версии и даты** в YAML → LLM понимает что актуально, а что устарело
- **Явные связи** между файлами → LLM видит граф, а не набор документов
- **Статусы решений** в decisions.md → LLM знает что в силе, а что пересмотрено

Это реализация "темпорального графа" без отдельной технологии — через конвенции файлов.

### Сборки (будущее развитие)

Дополнительные файлы под капотом, которые агрегируют данные из файлов клиента для быстрого контекста:

- Индексы (что изменилось за неделю)
- Сводки (ключевые решения за период)
- Профили контактов (собранные из meetings/)

Клиент их не ведёт — они генерируются автоматически помощниками.

### Горизонтальные контракты слоя

Слой памяти владеет общими механизмами для всех вертикалей. Делятся на две группы:

- **Контекстная память** (как агент находит и читает файлы, протоколы навигации, карточка файла, rule injection) — описана в [memory/01_context_memory.md](memory/01_context_memory.md).
- **Воспроизводимость и ответ** (snapshot, версионирование сущностей, trace ответа, semantic layer как паттерн, маршруты как готовые решения, границы ответа) — пока живут в [methodology/metrics.md](methodology/metrics.md) как часть metrics-вертикали; по мере оформления как общих контрактов переедут в `memory/` отдельной задачей.

Любая новая вертикаль опирается на эти механизмы, а не изобретает свои.

## 3. Помощники

Всё, что извлекает ценность из данных. Разделение по триггеру: кто запускает.

### По запросу (клиент вызывает)

Клиент явно запускает: "обработай встречу", "подготовь план", "проанализируй решения".

Примеры:

- Обработка транскрипции встречи → протокол + дельта по файлам
- Подготовка к встрече → сводка контекста по участнику
- Ревью плана недели → проверка на соответствие целям периода
- Заполнение scaffold → интервью по шаблону

Под капотом это может быть команда, скилл или агент — клиенту неважно.

### Автоматизация (работает сама)

Настраивается один раз — работает по расписанию или триггеру.

Примеры:

- Утро понедельника → план недели приходит в Telegram
- Перед встречей → сводка по участнику автоматически
- Конец недели → черновик weekly_progress из событий недели
- При изменении файла → обновление связанных файлов

Разница с "по запросу" — не в технологии, а в триггере: человек запустил / система запустила сама.

### Горизонтальные контракты слоя

Паттерны обработки запроса, общие для всех вертикалей:

- **Маршрут + думающая ветка** — сначала готовое решение из библиотеки, затем построение нового через инструменты.
- **Границы ответа** — когда отказаться, когда уточнить, когда не угадывать.

Сейчас оба паттерна описаны в [methodology/metrics.md](methodology/metrics.md) как часть metrics-вертикали; при оформлении как общих контрактов переедут на этот слой.

## Универсальность

- **Любой интерфейс** — Claude Code, Cowork, VS Code, будущая веб-морда
- **Любая модель** — Claude, GPT, DeepSeek (данные в открытом Markdown, не привязаны к API)
- **Данные у клиента** — не хостим данные, не привязываем к вендору

Дополнение: **любой runtime**, если он способен доставить наш plugin-слой. Runtime не считается частью ядра продукта. Это delivery-инфраструктура, а не наша уникальность.

## Skills и Plugin

**Skills** ([skills/](skills/)) — мастерская: промпты, скиллы, агенты по доменам. Здесь разрабатываем и итерируем.

**Plugin** ([plugin/](plugin/)) — деплоимый пакет: готовые skills + agents + hooks. То, что клиент устанавливает.

```
skills/ (разработка)              plugin/ (деплой)
├── meeting-analysis/             ├── skills/
│   ├── prompt-entity-*           ├── agents/
│   ├── prompt-behavior-*         └── hooks/
│   ├── prompt-protocol/telegram
│   ├── procedure-scaffold-update
│   └── orchestrator-meeting
└── [будущие домены]/
```

- Skills → мастерская, Plugin → витрина
- Плагин не хранит данные клиента — только интеллектуальный слой поверх
- Обновляется через Git
- Работает поверх внешнего runtime / host, который умеет доставить этот слой клиенту

## Runtime и delivery-слой

Runtime — это внешний исполняющий слой, через который клиент получает наш plugin: AI-host, orchestration engine, scheduling, интерфейсные коннекторы. Он не является частью ядра Second AI Brain.

Принцип:

- **plugin** = наш продуктовый слой;
- **runtime** = заменяемая инфраструктура доставки.

Что из этого следует:

- мы не проектируем продукт вокруг одного конкретного runtime;
- `Claude Code`, `Cowork`, `nanobot` или будущий managed runtime — это варианты delivery;
- при проектировании skills, agents и hooks мы должны по возможности сохранять runtime-agnostic контракт.

Текущее решение: свой runtime не пишем; внешний lightweight runtime проверяем как delivery-слой. Детали и статус — в [04_decisions.md](04_decisions.md) (пункт 1 — Выбор runtime).
