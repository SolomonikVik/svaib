# Архитектура продукта

## Кратко

Как устроен продукт Second AI Brain изнутри: слои, компоненты, связи. Три слоя (данные → память → помощники) внутри мета-рамки (методология + онтология). Реализация через plugin. Интерфейс — не наше.

## Связанные файлы

- ontology/ontology.md — сущности, связи, правила размещения
- methodology/methodology.md — ритуалы, протоколы, decision frames
- ontology/file_spec.md — формат файлов: YAML, секции, ограничения
- scaffold/ — готовый каркас (воплощение онтологии)
- plugin/README.md — деплоимый пакет (реализация помощников)
- ../meta/product/product_vision.md — видение продукта, бизнес-модель
- ../meta/product/model.md — модель продукта для клиентов (презентационный слой)

## Мета-рамка: Методология + Онтология

Не компонент продукта, а то, что удерживает всё вместе. Источник правды.

**Онтология** ([ontology.md](ontology/ontology.md)) — какие сущности существуют, как связаны, что куда кладётся. Справочник.

**Методология** ([methodology.md](methodology/methodology.md)) — как с этими сущностями работать: ритуалы, протоколы, decision frames. То, что зашивается в skills и agents.

Клиент может не читать онтологию и методологию напрямую — они зашиты в scaffold (структуру) и plugin (автоматизацию). Но они существуют как отдельные документы для развития продукта и обучения.

## 1. Данные

Фундамент системы. Всё, что клиент хранит и с чем работает AI.

**Принцип: LLM + People centered.** Каждый файл одинаково удобен для человека и для LLM-агента. Читаем глазами, парсится моделью без потери смысла. Это проектное требование к слою данных: структура и обвязка строятся так, чтобы обе стороны получали максимум.

### Структура

Какие файлы, какие папки, что куда кладётся.

- **Scaffold** ([scaffold/](scaffold/)) — готовый каркас папок и документов. Клиент разворачивает и начинает заполнять
- **Онтология** ([ontology.md](ontology/ontology.md)) — описание сущностей: 8 файлов ядра (vision, goal, plan, weekly_progress, timeline, ideas, decisions, reflections) + мета (README, my_profile, SKILL.md) + опциональные блоки (product, processes, knowledge)
- **Связи между файлами** — чёткая иерархия: vision → goal → plan → weekly_progress → timeline

Наша ценность: клиент получает не пустую папку, а архитектуру — понятно что куда класть, как файлы связаны, когда какой обновлять.

### Обвязка

Как файл устроен внутри. Правила, которые делают данные машиночитаемыми и связными.

- **YAML-заголовок** — title, updated, version, scope, priority ([file_spec.md](ontology/file_spec.md))
- **Стандартные секции** — "Кратко", "Связанные файлы", контентные секции
- **Связи** — каждый файл явно указывает связанные файлы → LLM может навигировать
- **Формат** — Markdown с ограничениями: <300 строк, ≤2 уровня вложенности, таблицы <20 строк
- **Маркеры источников** — [SOURCE], [REF:] для отслеживания фактов

Наша ценность: не просто файлы, а файлы с конвенциями. AI может парсить, находить связи, понимать актуальность.

## 2. Память

Свойство правильно устроенных данных — не отдельная технология, а результат структуры + обвязки + помощников.


### Протокол чтения

LLM знает в каком порядке читать файлы для получения полного контекста:

**Пример:**
```
README → my_profile → 01_vision → 02_goal → 03_plan
```
Эти 5 файлов дают полный контекст за минимум токенов. Остальные файлы — по триггеру (methodology.md, секция "Протокол чтения для LLM").

### Связи и темпоральность

- **Версии и даты** в YAML → LLM понимает что актуально, а что устарело
- **Явные связи** между файлами → LLM видит граф, а не набор документов
- **Статусы решений** в decisions.md → LLM знает что в силе, а что пересмотрено

Это реализация "темпорального графа" без отдельной технологии — через конвенции файлов.

### Сборки (будущее развитие)

Дополнительные файлы под капотом, которые агрегируют данные из файлов клиента для быстрого контекста:

- Индексы (что изменилось за неделю)
- Сводки (ключевые решения за период)
- Профили контактов (собранные из meetings/)

Клиент их не ведёт — они генерируются автоматически помощниками.

## 3. Помощники

Всё, что извлекает ценность из данных. Разделение по триггеру: кто запускает.

### По запросу (клиент вызывает)

Клиент явно запускает: "обработай встречу", "подготовь план", "проанализируй решения".

Примеры:

- Обработка транскрипции встречи → протокол + дельта по файлам
- Подготовка к встрече → сводка контекста по участнику
- Ревью плана недели → проверка на соответствие целям периода
- Заполнение scaffold → интервью по шаблону

Под капотом это может быть команда, скилл или агент — клиенту неважно.

### Автоматизация (работает сама)

Настраивается один раз — работает по расписанию или триггеру.

Примеры:

- Утро понедельника → план недели приходит в Telegram
- Перед встречей → сводка по участнику автоматически
- Конец недели → черновик weekly_progress из событий недели
- При изменении файла → обновление связанных файлов

Разница с "по запросу" — не в технологии, а в триггере: человек запустил / система запустила сама.

## Реализация через Plugin

Plugin ([plugin/](plugin/)) — деплоимый пакет: skills + agents + hooks. То, что клиент устанавливает.

```
Plugin
├── skills/    ← методология в исполняемом формате
├── agents/    ← автоматизация
└── hooks/     ← триггеры
```

- Плагин не хранит данные клиента — только интеллектуальный слой поверх
- Обновляется через Git
- Работает с любым AI-интерфейсом, поддерживающим формат
