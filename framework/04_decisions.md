# Product Decisions

## Кратко

Реестр крупных продуктовых решений Second AI Brain. Здесь — выборы, которые меняют траекторию продукта: закрывают альтернативы, задают границы архитектуры, delivery-модели и эволюции. Операционные выборы внутри спринта сюда не выносим.

**Работа с файлом:** decision log — append-only. Новое решение добавляется в конец. Изменение статуса (accepted → superseded) записывается обновлением, оригинал решения не стирается. Формат каждого решения: статус → решение → контекст → альтернативы → следствия.

**Апгрейд до папки `04_decisions/`** — когда файл перестанет умещаться (>500 строк или одно решение >100 строк), выносим каждое решение в отдельный файл, этот файл превращается в индекс.

## Связанные файлы

- [00_product.md](00_product.md) — что за продукт и зачем (принципы, границы)
- [architecture.md](architecture.md) — архитектурные следствия решений
- [01_overview.md](01_overview.md) — состояние реализации

## Формат записи

Каждое решение:

- **Статус** — текущее состояние (`accepted`, `proposed`, `superseded by …`) + дата
- **Решение** — что именно выбрали
- **Альтернативы** — что рассматривали и почему не берём

## 3. Путь Claude skills: канон vs legacy delivery

**Статус:** Accepted (2026-04-14).

### Решение

1. **Канон:** `.claude/skills/` — единственный правильный путь для Claude-specific навыков. Использовать в framework, ontology, architecture, plugin и прочих канонических документах.
2. **Legacy-исключение:** для текущих живых клиентских воркспейсов на Google Drive допускается временное развёртывание в `claude/skills/` без точки.
3. **Явная пометка:** каждый документ, где появляется legacy-путь, должен помечать его как временное client-specific исключение, не как норму продукта.
4. **Не смешивать оба пути в одном клиентском воркспейсе.** Для конкретного клиента выбирается один путь.

### Контекст

7 апреля 2026 разделили два слоя: `CLAUDE.md` — инструкция агенту, `.claude/skills/` — исполняемые навыки Anthropic runtime. После этого часть канона обновлена, но scaffold-шаблон и клиентский delivery остались на старом пути `claude/skills/`.

Почему legacy-путь пока не убираем у клиентов:
- Живые клиентские воркспейсы уже развёрнуты без точки.
- Текущая delivery-операционка завязана на этот путь.
- Для Google Drive / Finder dot-папки менее удобны в ручной работе и поддержке.
- Прямо сейчас важнее не сломать живую поставку, чем дожать миграцию пути.

## 7. Матричная модель scaffold вместо Аспект > Домен

**Статус:** accepted (2026-05-12). Канон методологии и клиентский каркас переписываются отдельными треками.

**Решение.** Главная ось scaffold — **матрица**. На корне выделяется компанийный уровень (`01_company/`) и доменные области (`marketing/`, `sales/`, `finance/`, `operations/`, …). Аспекты (стратегия, команда, метрики, проекты, процессы) разворачиваются на двух уровнях: компанийном (`01_company/01_strategy / 02_team / 03_metrics / …`) и доменном (`{domain}/strategy.md / team / metrics / projects / …`). Распределительный принцип — **владелец первый, наименьший общий предок второй** (приоритет: владелец перевешивает НОП).

**Контекст.** Прежняя модель v1 ставила типы сущностей на корень (`01_ceo / 02_strategy / 03_team / 04_company / 05_metrics`), домены — узлами внутри `04_company/`. На практике это ломалось: «стратегия маркетинга» и «target metrics CEO» не одного уровня, насильно лежали рядом или дублировались.

**Альтернативы.**
- **Вариант 1 — Аспект > Домен** (текущий v1). Один дом по типу. Не различает компанийный и доменный уровень. Отвергнуто.
- **Вариант 2 — Домен > Аспект.** Корень = функции, аспекты внутри. Кросс-функция требует дублей или индексов. Отвергнуто.
- **Вариант 3 — Матрица** (выбрано). Уровень определяет дом по природе факта.

Распределительный принцип опирается на индустриальные паттерны: bounded context в DDD, lift state up в React, нормализация в реляционных БД, каскад в CSS/config-системах.

**Следствия.**
- Канон методологии переписывается: `01_architecture`, `02_folder-spec`, `03_node-files`, `03_contours`, `deployment`, `open-questions`, `README` (Трек 1 плана трансформации).
- Клиентский `framework/scaffold/` мигрирует v1 → v2: `02_strategy → 01_company/01_strategy`, `03_team → 01_company/02_team`, `05_metrics → 01_company/03_metrics`, `04_company` распадается на корневые домены (Трек 3).
- Open questions закрываются: `org-structure` локация, `finance` где живёт, knowledge размещение, место `glossary.md`.
- Распределительный принцип распространяется с встреч/задач/решений на team / metrics / strategy / projects / processes — обобщение «правила общей родительской папки».

**Открытые вопросы.** Терминология (узел / домен / поддомен / контур / аспект), cross-domain navigation механизм, корневой vs доменный `knowledge/`.

**Полный ADR:** [methodology/scaffold/00_dilemma.md](methodology/scaffold/00_dilemma.md).
