# Product Decisions

## Кратко

Реестр крупных продуктовых решений Second AI Brain. Здесь — выборы, которые меняют траекторию продукта: закрывают альтернативы, задают границы архитектуры, delivery-модели и эволюции. Операционные выборы внутри спринта сюда не выносим.

**Работа с файлом:** decision log — append-only. Новое решение добавляется в конец. Изменение статуса (accepted → superseded) записывается обновлением, оригинал решения не стирается. Формат каждого решения: статус → решение → контекст → альтернативы → следствия.

**Апгрейд до папки `04_decisions/`** — когда файл перестанет умещаться (>500 строк или одно решение >100 строк), выносим каждое решение в отдельный файл, этот файл превращается в индекс.

## Связанные файлы

- [00_product.md](00_product.md) — что за продукт и зачем (принципы, границы)
- [architecture.md](architecture.md) — архитектурные следствия решений
- [01_overview.md](01_overview.md) — состояние реализации

---

## Формат записи

Каждое решение:

- **Статус** — текущее состояние (`accepted`, `proposed`, `superseded by …`) + дата
- **Решение** — что именно выбрали
- **Контекст** — почему вопрос возник
- **Альтернативы** — что рассматривали и почему не берём
- **Следствия** — что меняется для продукта, framework, delivery
- (опционально) **Открытые вопросы**, **Критерии проверки**, **Следующий шаг**

## 1. Выбор runtime для Second AI Brain

**Статус:** Accepted (2026-03-28). Решение по направлению принято. Конкретный runtime — в пилотной проверке.

### Решение

На текущем этапе **не пишем свой runtime с нуля**, а используем **готовый lightweight runtime как delivery-слой**.

Граница продукта:

> Second AI Brain = данные клиента + наша методология + plugin-слой.

Runtime, интерфейс, orchestration engine и конкретный AI-host — **заменяемые детали**, не ядро продукта.

**Текущие кандидаты для пилота:**

- **Claude Code Scheduled Tasks** (cloud-режим) — приоритетный, discovery 28.03.2026: нативный Claude без адаптерного слоя, коннекторы Google Drive / Google Calendar / GitHub из коробки, bash (curl) для Telegram проверен, не требует self-hosting. Research preview — возможны изменения лимитов и pricing.
- **`HKUDS/nanobot`** — запасной вариант: зрелый (~34K stars), формат `SKILL.md` совпадает с нашим, `HEARTBEAT.md` под ритуалы, Telegram + MCP из коробки, model-agnostic, lightweight.

### Контекст

Весной 2026 возник вопрос delivery-слоя продукта. Соблазн собрать свой минимальный runtime из зрелых кирпичей, но это уводит внимание в инфраструктурную разработку — не наша core value.

Из [00_product.md](00_product.md), ранних клиентских разговоров и экспериментов видно: клиентам нужен не «ещё один AI-фреймворк», а работающие управленческие циклы. Барьер входа критичен. Выигрыш даёт не код рантайма, а качество ритуалов, контекста, навыков и автоматизации.

Отдельный фактор: недавний опыт с meeting analysis показал, что избыточная самодельная сложность съедает десятки часов без пропорционального прироста качества.

### Альтернативы

- **OpenClaw** — слишком тяжёлый для нашей стадии, хуже security-профиль для CEO-кейса, тащит лишнюю платформенную сложность.
- **Свой runtime** — уводит в инфраструктурное R&D вместо методологии. Есть негативный опыт с meeting analysis.
- **NanoClaw** — сильная привязка к одному вендорному стеку, проигрывает по model-agnostic. Lock-in важнее, чем выигрыш от минимализма.
- **ZeroClaw** — серьёзный по безопасности, потенциально силён как future runtime. В watchlist. Проигрывает по скорости кастомизации и time-to-demo.

### Следствия

- **Продукт:** усиливает позицию «методология важнее инструмента», снижает риск инфраструктурного R&D, ускоряет показ клиенту работающего сценария.
- **Framework:** зафиксировать plugin-контракт независимо от runtime; отделить skill/agent/hook от специфики Claude Code / Cowork; описать минимальный adapter-слой между plugin и runtime.
- **Dev:** не пишем свой runtime. Проверяем Scheduled Tasks и nanobot на реальных клиентских сценариях.

### Что НЕ утверждается

- Что Scheduled Tasks или nanobot — постоянный runtime на годы.
- Что текущий `plugin/` уже полностью runtime-agnostic.
- Что нужно перестраивать продукт под экосистему nanobot.
- Что свой runtime никогда не понадобится.

### Критерии проверки решения

В коротком пилоте внешний runtime должен без существенной самописной инфраструктуры реализовать 3 вещи:

1. Приём запроса через простой интерфейс для клиента.
2. Запуск проактивного сценария по расписанию.
3. Подключение методологического слоя без переписывания логики продукта.

Минимальный сценарий: обработка встречи → подготовка к ближайшей встрече → короткий proactive brief в Telegram.

Если для этого потребуется писать заметный объём glue-кода, обходить ограничения платформы или менять структуру plugin под runtime — решение пересматривается.

### Открытые вопросы

1. В каком формате зафиксировать runtime-agnostic контракт plugin?
2. Что в текущем `framework/plugin/` — общая модель, а что — наследие Claude Code / Cowork?
3. Нужен ли отдельный документ про adapter layer между plugin и runtime?
4. Scheduled Tasks vs nanobot — какой берём для первого клиентского пилота?

### Следующий шаг

1. Отразить решение в [architecture.md](architecture.md) — явно развести продуктовый plugin и заменяемый runtime.
2. Пилот Scheduled Tasks на реальном клиентском сценарии (утренний план дня: scaffold из Google Drive + календарь → Telegram).
3. Если Scheduled Tasks не закроет — вернуться к пилоту nanobot.

## 2. Границы продукта (product-scope)

**Статус:** Accepted (2026-04-07).

### Решение

**Формула:** помощник, который знает тебя как личность, чтобы помогать тебе в бизнесе.

**Входит:**

- Управленческие циклы: цели, встречи, команда, решения, направления
- Личность руководителя в контексте бизнеса: ценности, стиль мышления, компетенции, слабые стороны — всё, что влияет на управленческие решения
- Бизнес-контекст: продукт, процессы, клиенты, метрики
- База знаний по бизнесу и управлению

**Не входит как отдельная инфраструктура:**

- Личные домены: здоровье, хобби, семья, путешествия, творческие проекты
- Бытовые задачи и персональная организация вне бизнеса
- Scaffold для не-бизнес-проектов не разворачивается

**Нюанс:** значимые не-бизнес-проекты (обучение, переезд, важные жизненные изменения) могут упоминаться в MY_PROFILE как контекст, если влияют на бизнес-решения. Отдельная инфраструктура для них не создаётся — только упоминание в секции «Проекты и контекст».

**MY_PROFILE** — мост, а не фундамент. Содержит личность руководителя (ценности, миссию, стиль) как входные данные для бизнес-решений, не как начало развёрнутой персональной инфраструктуры.

**«Second AI Brain»** — маркетинговый бренд (SEO). Внутри продукта и в коммуникации с клиентом scope описывается точно: AI-инфраструктура для управления бизнесом.

### Контекст

Название «Second AI Brain» работает маркетингово, но создаёт ложное ожидание: клиент читает «второй мозг» и воспринимает как систему для всей жизни. Реальный кейс: клиент разместил медкарту и хобби-проекты в рабочем пространстве.

В документации продукта граница не была явно проведена: «всё, что руководитель знает», «система начинается с человека» — формулировки, допускающие расширительное толкование.

### Следствия

- [00_product.md](00_product.md) — секция «Границы» (явно зафиксирована здесь).
- [architecture.md](architecture.md) — уточнён слой «Данные» (бизнес-контекст).
- `scaffold/my_profile.md` — секция 5 уточнена (бизнес-проекты).
- `scaffold/README.md` — knowledge/ уточнена (бизнес и управление).
- [methodology/onboarding.md](methodology/onboarding.md) — добавлена scope-нота.
- При онбординге клиенту явно объяснять scope системы.

## 3. Путь Claude skills: канон vs legacy delivery

**Статус:** Accepted (2026-04-14).

### Решение

1. **Канон:** `.claude/skills/` — единственный правильный путь для Claude-specific навыков. Использовать в framework, ontology, architecture, plugin и прочих канонических документах.
2. **Legacy-исключение:** для текущих живых клиентских воркспейсов на Google Drive допускается временное развёртывание в `claude/skills/` без точки.
3. **Явная пометка:** каждый документ, где появляется legacy-путь, должен помечать его как временное client-specific исключение, не как норму продукта.
4. **Не смешивать оба пути в одном клиентском воркспейсе.** Для конкретного клиента выбирается один путь.

### Контекст

7 апреля 2026 разделили два слоя: `CLAUDE.md` — инструкция агенту, `.claude/skills/` — исполняемые навыки Anthropic runtime. После этого часть канона обновлена, но scaffold-шаблон и клиентский delivery остались на старом пути `claude/skills/`.

Почему legacy-путь пока не убираем у клиентов:

- Живые клиентские воркспейсы уже развёрнуты без точки.
- Текущая delivery-операционка завязана на этот путь.
- Для Google Drive / Finder dot-папки менее удобны в ручной работе и поддержке.
- Прямо сейчас важнее не сломать живую поставку, чем дожать миграцию пути.

### Следствия

- **Framework:** `.claude/skills/` остаётся единственным каноническим путём. Упоминания `claude/skills/` допустимы только как явно помеченный legacy-case.
- **Playbook:** в delivery-операциях можно оставлять `claude/skills/` для текущих клиентов с пояснением «временная совместимость с уже развёрнутыми Drive-воркспейсами». При создании нового delivery-контура путь проверяется явно, а не наследуется молча.
- **Миграция клиентов:** перенос с `claude/skills/` на `.claude/skills/` — отдельный проход, не смешивать с текущим delivery.

### Что НЕ утверждается

- `claude/skills/` равноправен `.claude/skills/`.
- Legacy-путь нужен всем новым клиентам.
- Миграцию нужно делать немедленно посреди delivery.

### Следующий шаг

1. Синхронизировать scaffold-шаблон и playbook с этим решением.
2. В фазе клиентской миграции отдельно решить, когда и как переводить живые воркспейсы на `.claude/skills/`.
