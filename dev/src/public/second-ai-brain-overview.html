// SECOND AI BRAIN

Продукт собирается на пересечении трёх горизонталей и восьми вертикалей.

Горизонтали — общая инфра: данные, память, помощники. Они одинаковы для любого CEO. Вертикали — управленческие циклы: стратегия, команда, метрики, встречи, продукт, маркетинг, продажи, проекты. Каждый цикл проходит сквозь все три горизонтали.

// часть 01

Горизонтали — три слоя продукта

общая инфра для любого CEO
1

Данные

Бизнес-контекст руководителя в одном месте. Не свалка файлов — связанная модель, в которой каждая папка отвечает на один вопрос.

  • scaffold — каркас папок: 01_ceo / 02_strategy / 03_team / 04_management + опциональные
  • обвязка — YAML, говорящие H2, секция «Кратко», связи
  • правило — миссия папки как фильтр размещения информации
  • iron law — разворачивай только живое, пустых веток нет
  • формат — открытый Markdown, не привязан к API
2

Память

Правила, по которым AI находит в данных нужное и понимает что актуально. Загружает контекст под задачу, а не всё подряд.

  • протокол чтения — что и в каком порядке читает агент
  • 3 фильтра сбора — путь → карточка файла → целевое чтение
  • температура — hot / warm / on-demand / cold
  • сборки — агрегации (например «всё на CEO» из секций active.md)
  • детерминированность — критичное закрыто хуками, не текстом
3

Помощники

То, что превращает данные и память в действие. Два режима: по запросу руководителя и автоматически — упакованные ритуалы.

  • по запросу — «обработай встречу», «подготовь план», «разбери метрику»
  • автоматически — бриф перед встречей, weekly progress, метрики к планёрке
  • разработка — в skills/ по доменам (мастерская)
  • доставка — через plugin (наш продуктовый слой)
  • интерфейс — любой: Claude Code, Cowork, IDE, веб-морда
// мета-рамка

Над тремя слоями — методология и онтология

То, что удерживает всё вместе. Онтология — какие сущности существуют и как связаны. Методология — как с ними работать: протоколы, decision frames, ритуалы. Клиент их не читает напрямую — они зашиты в scaffold (структуру) и plugin (автоматизацию). Любое изменение продукта идёт сверху: методология → схемы → промпты → оркестраторы → документация.

// часть 02

Вертикали — управленческие циклы

проходят сквозь все три горизонтали
Управленческий цикл — повторяющийся контур работы CEO: постановка направления → сбор информации → решения → контроль → обратная связь → корректировка. Цикл связывает сущности (что создаётся), scaffold (где живёт), память (как находится), помощников (кто автоматизирует). Не атомарная сущность, а контекст, в котором сущности рождаются и используются.

Циклы связаны в систему: стратегия задаёт направление команде, через команду понятно какими встречами управлять, метрики распределяются на команду и подсвечиваются на встречах, продукт и маркетинг ведут к продажам, проекты пересекают функции. Ни один цикл не существует изолированно.
8
циклов · ориентир 5–10
// 01 strategy

Управление стратегией

Vision, цели периода, страт-сессии, пересмотр курса. Хроника по неделям, timeline план/факт, реестр стратегических решений с обоснованиями.

// как разворачивается
  • vision и цели периода живут в отдельных файлах с YAML
  • weekly progress собирается автоматически из дельт по другим циклам
  • каждое стратегическое решение пишется с обоснованием — через 3 месяца виден контекст
  • timeline сводит план и факт по периодам
// 02 team

Управление командой

Оргструктура, профайлы людей, роли, наблюдения, коучинг-лог. Стиль и сильные стороны как входные данные для встреч и делегирования.

// как разворачивается
  • оргструктура маппит каждого на узел в управленческом дереве
  • профайл сотрудника: стиль работы, сильные стороны, рост, риски
  • перед каждым 1:1 AI собирает контекст — что обещал, что висит, какие тревоги
  • наблюдения и коучинг-лог копятся, видны паттерны на дистанции
// 03 metrics

Управление по метрикам

Хранение метрик с источником и формулой, план/факт, разбор отклонений как процесс. Метрики попадают в контекст встреч автоматически.

// как разворачивается
  • паспорт метрики: что считаем, как считаем, откуда берём, target/baseline
  • числа живут в источнике (xlsx, Google Sheets), расчёты — через скилл
  • отклонение фиксируется в разборе с гипотезой и решением
  • метрики цикла подтягиваются в бриф перед встречей
// 04 meeting

Управление встречами

Подготовка, анализ, протоколирование, дельты по файлам. Транскрипт → выжимка → протокол + обновления в scaffold + повестка следующей встречи.

// как разворачивается
  • транскрипт после встречи разбирается AI: задачи, решения, риски, метрики
  • дельты разносятся по файлам — active, decisions, profile, метрики
  • за час до следующей встречи приходит бриф: что обещал, что сделал, что горит
  • longitudinal-анализ серии встреч — повторяющиеся паттерны и тренды
// 05 product

Управление продуктом

Позиционирование, внутреннее устройство продукта, roadmap, релизы, продуктовые решения. Отдельно от стратегии компании — тут устройство самого продукта.

// как разворачивается
  • 00_product.md — позиционирование, ЦА, принципы, границы
  • architecture.md — внутреннее устройство продукта (если оно есть)
  • roadmap по декадам, релизы в timeline
  • продуктовые решения отделены от стратегических — не смешиваются
// 06 marketing

Управление маркетингом

Бренд, каналы, контент, дистрибуция. Карта каналов с конверсиями, такт публикаций, контент-конвейер, эксперименты с гипотезами.

// как разворачивается
  • карта каналов с фактическими конверсиями — какой работает
  • контент-такт: что выходит, когда, на каком канале
  • контент-конвейер из материалов — переупаковка под форматы
  • гипотезы тестируются с критериями и зафиксированным выводом
// 07 sales

Управление продажами

Воронка, конверсии по этапам, работа с клиентами, продление. Связан с marketing на входе и с metrics через целевые показатели.

// как разворачивается
  • воронка с этапами и конверсиями — видно где течь
  • работа с именованными клиентами, если CEO ведёт сам
  • продление и LTV — через цикл metrics
  • обратная связь от продаж в продукт и маркетинг через файлы
// 08 projects

Управление проектами

Именованные проекты с горизонтом и завершением — миграции, запуски, временные инициативы. Отличие от направления: проект конечен, направление постоянно.

// как разворачивается
  • каждый проект — отдельный узел: overview, active, backlog, progress, decisions
  • конкретные сроки, ресурсы, статус — не размывается в направления
  • встречи проекта собираются в meetings/ внутри узла
  • после завершения проект уходит в архив с timeline

// действие в цикле

Ритуал

Повторяющееся действие внутри цикла. Конкретная форма, в которой цикл реализуется во времени.

«утренний брифинг», «weekly review»
// исполнитель операции

Скилл

Атомарный помощник для одной операции. Не имеет своих сущностей в слоях — переиспользуется внутри разных циклов.

«собрать бриф», «обработать встречу»

Лимит вертикалей — 10. Email, Telegram, документы, календарь — это не циклы, а каналы и скиллы. Если у нового помощника нет своих сущностей в слоях памяти и данных — это скилл, не вертикаль. Не размножать вертикали под каждый помощник.

Скорость операций

Конкретные операции, которые сейчас занимают часы и дни, схлопываются в минуты.

  • Подготовка к встрече — день разбора → 5 минут (бриф собирается сам)
  • Разбор гипотезы — неделя тестов → минуты в диалоге с AI
  • Создание контента/статьи — день работы → минуты через AI-конвейер
  • Weekly progress — час сбора → авто из дельт по циклам
  • «Что мы решали 3 месяца назад» — 30 минут архивов → секунды
// плоскость 02 +уровень

Качество управления

Не «быстрее то же самое» — а вещи, которые без системы CEO просто не делает.

  • Метрики честные — не «где-то 60%», а цифра + формула + источник + тренд
  • Решения видны — через 3 месяца понятно почему повернули туда
  • Команда видна — перед каждым 1:1 контекст, а не «как дела» вслепую
  • Паттерны на дистанции — AI замечает, что подменяешь действие отчётом
  • План/факт замкнут — стратегия → план → задачи → встречи → дельты
// плоскость 03 −нагрузка

Освобождение головы

Самое неочевидное и самое сильное. CEO перестаёт быть носителем оперативной памяти компании.

  • Сейчас — сотни нитей в голове: обещания, метрики, висящие задачи, риски
  • С системой — всё в файлах, не в голове
  • Освобождается для того, для чего CEO нужен: думать о направлении
  • Симптом — после рабочего дня есть силы на стратегическое
  • Это — самый ценный эффект, описать сложнее всего

// главный буст — интегральный

Циклы не дают +10% к каждому процессу. Они меняют то, как руководитель работает целиком.

Вместо «я и мои инструменты» — «я и мой партнёр, который знает мой бизнес и видит то, что я упускаю». Это уровень, на котором AI перестаёт быть инструментом и становится частью контура управления.

Каждый отдельный цикл — это видимая ценность (быстрее, чётче, прозрачнее). Совокупность работающих циклов — это другая модель управления компанией, в которой контекст не теряется, решения замкнуты в файлах, а голова руководителя свободна для стратегического мышления.

// часть 05

Почему так получается

архитектурные причины
01

Каждый цикл — продуманный, не «ещё одна фича»

Стратегия не путается с продуктом, маркетинг не размывается во встречи. У каждого цикла свои файлы, маршруты, помощники.

02

Слои общие — циклы переиспользуют

Память не изобретается под каждый цикл. Контекстный сбор, температура, протоколы чтения — общая инфра. Цикл подключается, не строит с нуля.

03

Зрелость растёт уровнями

Цикл стартует базово (один помощник + сущности в scaffold), потом обрастает ритуалами, автоматизациями, longitudinal-анализом. Не «всё или ничего».
