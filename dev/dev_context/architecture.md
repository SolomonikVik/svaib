## 1. Общая модель

### 1.1 Суть продукта

**svaib** — AI-помощник по командным встречам и проектам.

Svaib записывает встречи (Zoom/Meet/Teams), транскрибирует с разметкой спикеров, извлекает задачи и решения, обновляет презентацию проекта, отправляет протоколы в Telegram. AI-ассистент отвечает на вопросы по контексту проекта.

### 1.2 Трёхслойная архитектура

```
┌─────────────────────────────────────────────────────────────┐
│           СЛОЙ 1: ДАННЫЕ КЛИЕНТА (Google Workspace)         │
├─────────────────────────────────────────────────────────────┤
│  Google Drive     │  Google Sheets   │  Google Slides      │
│  (документы)      │  (задачи,        │  (презентация       │
│                   │   метрики)       │   встречи)          │
│                                                             │
│  → Клиент работает здесь, это его интерфейс                │
│  → Sheets = источник истины по задачам                     │
└─────────────────────────────────────────────────────────────┘
                            ↕
              [Google Sync Layer — синхронизация]
                            ↕
┌─────────────────────────────────────────────────────────────┐
│              СЛОЙ 2: ЯДРО SVAIB (Supabase)                  │
├─────────────────────────────────────────────────────────────┤
│  clients          │  meetings        │  transcripts        │
│  (клиенты)        │  (встречи)       │  (транскрипты)      │
├───────────────────┼──────────────────┼─────────────────────┤
│  tasks            │  documents       │  oauth_tokens       │
│  (snapshot)       │  (карта файлов)  │  (encrypted)        │
├───────────────────┼──────────────────┼─────────────────────┤
│  prompts          │  pipeline_runs   │  [reasoning_bank]   │
│  (AI-промпты)     │  (observability) │  (Phase 2)          │
│                                                             │
│  → Метаданные, связи, статусы, история                     │
│  → RLS: client_id изолирует данные клиентов                │
│  → Контент НЕ дублируется (живёт в Google)                 │
└─────────────────────────────────────────────────────────────┘
                            ↕
                   [Webhooks / API calls]
                            ↕
┌─────────────────────────────────────────────────────────────┐
│         СЛОЙ 3: ОРКЕСТРАЦИЯ И AI (n8n + LLM)               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │ meeting_    │  │ google_     │  │ ai_         │        │
│  │ pipeline    │  │ sync        │  │ assistant   │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                             │
│  + Внешние сервисы:                                        │
│    • Recall.ai (запись встреч)                             │
│    • Soniox (транскрибация)                                │
│    • OpenAI/Claude API (LLM)                               │
│                                                             │
│  → Stateless: не хранит состояние                          │
│  → Промпты читает из Supabase                              │
│  → Результаты пишет в Google + Supabase                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Ключевые принципы

| Принцип                       | Описание                                  | Почему важно                                      |
| ----------------------------- | ----------------------------------------- | ------------------------------------------------- |
| **Google-first**              | Контент живёт в Google, структура у нас   | Привычный интерфейс для клиента, нет дублирования |
| **Sheets \= источник истины** | Статусы задач меняются в Sheets, не у нас | Минимум трения, клиент работает где привык        |
| **Тонкое ядро**               | Supabase хранит метаданные, не контент    | Простота, нет синхронизации контента              |
| **Stateless AI**              | LLM не держит состояние, контекст из БД   | Масштабируемость, предсказуемость                 |
| **Промпты \= данные**         | Промпты в Supabase, не в коде n8n         | Версионирование, A/B тесты, без редеплоя          |
| **Модульность**               | Каждый компонент заменяем                 | Можно сменить Soniox на другой ASR без переделки  |

## 3. Схема данных (Контракт системы)

### 3.1 Сущности ядра

| Сущность           | Назначение                                 | Ключевые поля                                                                        |
| ------------------ | ------------------------------------------ | ------------------------------------------------------------------------------------ |

| Сущность                   | Назначение                 | Ключевые поля                                                   |
| -------------------------- | -------------------------- | --------------------------------------------------------------- |
| **reasoning\_bank\_items** | Паттерны команды           | title, content, type (pattern/insight), embedding, usage\_count |

### 3.3 Принципы схемы

* **RLS везде** — client\_id изолирует данные, политика "Users see own data"
* **documents vs files** — documents = текстовый контент (досье, протоколы), files = карта внешних файлов для RAG
* **tasks в MVP** — хранятся в Supabase, sync в Sheets добавляется в Phase 2
* **svaib\_task\_id** — hash для идемпотентности (meeting\_id + normalized(title) + assignee)
* **Промпты** — системные (client\_id IS NULL) видны всем, кастомные — только владельцу

### 3.5 Идентификация клиента (MVP)

**Принцип:** Telegram чат = Client

| Тип чата | Что создаётся |
|----------|---------------|
| Групповой | client с названием чата |
| Личный | client с именем пользователя |

**Логика:**
1. Получаем `chat_id` из Telegram сообщения
2. Ищем `clients` по `telegram_chat_id`
3. Если нашли → используем существующего
4. Если нет → создаём нового client

**Что это даёт:**
- Изоляция данных между чатами
- Не требует регистрации (добавил бота = стал клиентом)
- Простое тестирование

**Ограничения MVP:**
- Нет объединения чатов в одного клиента
- Нет иерархии компания → проекты

**Масштабирование:** `clients` → `workspaces`, добавить `organizations` над ними.

## 4. MVP: Технические решения

> **Что входит в MVP и детали релиза:** см. `product_release.md`
> **План реализации по неделям:** см. `product_sprint.md`

Этот раздел описывает ТЕХНИЧЕСКИЕ РЕШЕНИЯ для MVP.

### 4.1 Цель фазы

Работающий продукт для 3-5 пилотных клиентов:

* Бот приходит на встречу и записывает
* Транскрипт с разметкой спикеров
* Автоматическое извлечение задач
* Запись в Google Sheets
* Протокол в Telegram

### 4.2 SGR: Schema-Guided Reasoning

**Зачем:** LLM "плывут" на сложных задачах. Разбиваем на простые шаги с JSON Schema.

**Реализация в n8n (6 промптов, разветвление после finalizer):**

```
Сырой транскрипт
        ↓
[meeting_cleaner] Удаление мусора, сжатие 2-3×
→ Чистый транскрипт
        ↓
[meeting_analyst] Анализ + контекст (досье)
→ RAW_ANALYSIS (всё извлечено, с опорами-цитатами)
        ↓
[meeting_protocol_finalizer] Классификация, фильтрация, проверка повторов
→ CANONICAL_PROTOCOL (универсальный markdown)
        ↓
┌───────────────────────────────────────────────────┐
│  [meeting_protocol_telegram] Форматирование       │
│  → Telegram-сообщение с эмодзи                    │
│                                                   │
│  [meeting_docs_editor] Обновление досье           │
│  → Полный текст досье                             │
│                                                   │
│  [meeting_task_extractor] JSON mode               │
│  → JSON: [{title, assignee, deadline, priority}]  │
└───────────────────────────────────────────────────┘
```

> **Примечание:** Модель выбирается на этапе реализации. Оптимизация моделей — отдельная задача.
> **Связанные документы:** `meta/_inbox/meeting_protocol_ontology.md`, `meta/_inbox/dossier_structure.md`

**Контекстная матрица (кто что получает):**

| Промпт | Transcript | Dossier | RAW_ANALYSIS | CANONICAL_PROTOCOL |
|--------|------------|---------|--------------|-------------------|
| meeting_cleaner | ✅ | ❌ | — | — |
| meeting_analyst | ✅ (cleaned) | ✅ | — | — |
| meeting_protocol_finalizer | ❌ | ✅ | ✅ | — |
| meeting_protocol_telegram | ❌ | ❌ | ❌ | ✅ |
| meeting_docs_editor | ❌ | ✅ | ❌ | ✅ |
| meeting_task_extractor | ❌ | ❌ | ❌ | ✅ |

> **Важно:** История протоколов не передаётся. Досье — единственный источник контекста.

**Формат входных данных для meeting_analyst (шаблон):**

```
## МЕТАДАННЫЕ ВСТРЕЧИ
Проект: {{project_name}}         ← clients.name
Дата: {{meeting_date}}           ← meetings.scheduled_at
Участники: {{participants}}      ← meetings.participants (TODO)

## ДОСЬЕ ПРОЕКТА
{{dossier}}                      ← documents WHERE doc_type='dossier'

## ТРАНСКРИПЦИЯ
{{transcript}}                   ← transcripts.cleaned_content (fallback: content)
```

> **TODO:** Поле `meetings.participants` нужно добавить. Данные доступны из Recall API (`meeting_participants`).
> **Структура досье:** См. `meta/_inbox/dossier_structure.md`

**Преимущества:**

* Дебажится (видно где сломалось)
* Предсказуемо (JSON Schema гарантирует формат)
* Экономия токенов (cleaner сжимает вход для всех последующих шагов)

**Structured Outputs (декабрь 2025):**

Claude API теперь поддерживает гарантированный JSON по схеме (`response_format: json_schema`). Это означает:
* 100% валидный JSON — без "лечащего" Code Node
* Меньше retry логики — ответ всегда соответствует схеме
* Проще промпты — не нужны многословные инструкции про формат

При реализации SGR использовать Structured Outputs для каждого шага цепочки.

### 4.3 Google Sync Layer

**Принцип:** Отдельная группа флоу для всех операций с Google.

```
google_sync_*
├── sync_tasks_to_sheets      — Batch write новых задач
├── sync_tasks_from_sheets    — Периодическое чтение статусов
├── sync_docs_metadata        — Индексация документов для RAG
└── update_slides             — Обновление презентации
```

**Критичные правила:**

1. **Batch operations везде** — никаких циклов "по одной строке"
2. **Идемпотентность** — svaib\_task\_id в скрытой колонке Sheets
3. **Exponential backoff** — retry при quota errors

### 4.4 Умный RAG (lite версия для MVP)

**Архитектура:**

```
Вопрос клиента
        ↓
[Embed вопроса]
        ↓
[Semantic search в documents] → Топ-3 файла по релевантности
        ↓
[Google Docs API] → Читаем файлы ЦЕЛИКОМ
        ↓
[LLM] Контекст (3 файла) + Вопрос → Ответ
```

**Для MVP:**

* Индексируем только ключевые файлы (5-10 на клиента)
* Embeddings на уровне ФАЙЛА (summary), не чанков
* Читаем целиком (современные LLM держат 128K+ токенов)

**Phase 2:** Гибридный подход для больших документов (>20 страниц).

### 4.5 Observability: pipeline\_runs

**Зачем:** Когда что-то сломается, нужно быстро понять ГДЕ.

**Что логируем:**

| stage         | Описание                  |
| ------------- | ------------------------- |
| bot\_joined   | Бот подключился к встрече |
| recording     | Идёт запись               |
| transcription | Транскрибация в процессе  |
| extraction    | Извлечение задач          |
| google\_sync  | Запись в Sheets           |
| notification  | Отправка в Telegram       |

**Простой дашборд:** Supabase Studio или Metabase — список встреч и их статус.

## 5. Будущие технические компоненты

### 5.1 ReasoningBank (детали)

**Что это:** Хранилище паттернов командной динамики, извлечённых из встреч.

**Цикл:**

```
Встреча → Транскрипт → Задачи
                    ↓
        [LLM выжимает паттерны]
                    ↓
        "80% решений принимает один человек"
        "Тема X поднималась 4 встречи без closure"
                    ↓
        Сохранение в reasoning_bank_items
                    ↓
        Ежемесячный дайджест / AI-инсайты
```

**Типы items:**

* `pattern` — повторяющееся поведение команды
* `insight` — наблюдение для контекста

**Детали фичи:** См. MVP сценарий, раздел 3.4 "Паттерны команды" и Приложение А.

### 5.2 ReasoningBank: контекст исследования

**Исследование:** [ReasoningBank: Scaling Agent Self-Evolving with Reasoning Memory](https://arxiv.org/abs/2509.25140v1) (Google, 2025)

**Суть:** Агенты учатся на успехах и провалах, выжимая обобщённые стратегии действий (memory items). Применимо к агентам, которые сами выбирают действия.

**Применимость к svaib:**

| Подход                   | Статус             | Комментарий                                                                                   |
| ------------------------ | ------------------ | --------------------------------------------------------------------------------------------- |
| Самообучающийся пайплайн | Пока не релевантно | SGR — детерминированная схема. Обратная связь обрабатывается вручную через таблицу `feedback` |
| Паттерны команды         | Адаптировано       | Извлекаем паттерны командной динамики из контента встреч — фича продукта, не самообучение AI  |

**Сбор обратной связи (MVP):**

* Telegram: реакции на протокол (👍/👎)
* Периодический опрос: "Насколько точно извлечены задачи?"
* Хранение: таблица `feedback` в Supabase
* Анализ: вручную (Виктор + CTO), корректировка промптов/схемы

## 6. Критические предупреждения

### 🔴 OAuth Token Limit (БОМБА)

**Проблема:** Google ограничивает 100 active refresh tokens на OAuth Client ID.

| Клиентов | Статус      | Действие                            |
| -------- | ----------- | ----------------------------------- |
| 10-50    | ✅ ОК        | User OAuth работает                 |
| 50-80    | ⚠️ Опасно   | Готовиться к Domain-Wide Delegation |
| 100+     | 🔴 Критично | ОБЯЗАТЕЛЬНО Domain-Wide Delegation  |

**Что происходит при 101 клиенте:** Самый старый токен аннулируется БЕЗ ПРЕДУПРЕЖДЕНИЯ. Клиент #1 перестаёт работать.

**Action:** Изучить Domain-Wide Delegation ДО достижения 50 клиентов.

### 🔴 OAuth Verification (время!)

**Проблема:** Без верификации Google показывает scary warning и ограничивает до 100 test users.

**Timeline:** 3-5 рабочих дней (может больше).

**Требования:**

* Privacy Policy (URL)
* Terms of Service (URL)
* Demo video приложения
* Объяснение использования каждого scope

**Action:** Начать процесс на Неделе 1 параллельно с разработкой.

### 🟡 Google API Quotas

**Лимиты:**

* Sheets: 300 req/min per project
* Docs: 300 req/min per project
* Slides: 300 req/min per project

**При 50 клиентах:** \~5 calls/min — в лимитах с запасом 60x.

**Правила:**

1. **Batch operations всегда** — 10-100x эффективность
2. **Exponential backoff** — обязательно на все API calls
3. **Rate limiting** — при пиковых нагрузках

### 🟡 Идемпотентность задач

**Проблема:** При retry n8n может записать одну задачу дважды.

**Решение:**

1. Генерируем `svaib_task_id` (hash от meeting\_id + текст задачи + assignee)
2. Скрытая колонка в Sheets с этим ID
3. Перед записью — проверяем существование
4. Если есть — skip, нет — пишем
