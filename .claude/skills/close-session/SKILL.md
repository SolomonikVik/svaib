# Close Session

Структурированное завершение рабочей сессии. Три фазы: лог, проверка несохранённого, маршрутизация вопросов.

## Iron Law

**Не пиши без подтверждения Виктора.** Каждая фаза заканчивается показом предложения и ожиданием реакции. Фаза 2 и 3 — это диалог, не автоматизация.

## Core Process

### Phase 0: Определи тип закрытия

Задача в этой сессии **завершена** или **будет продолжена** в новом чате?

**Признаки продолжения:** контекст на пределе, задача явно не закончена, Виктор говорит "закрываем сессию, задача не закончена" / "продолжим в новом чате".

**Признаки завершения:** задача сделана, Виктор явно завершает ("всё, готово", "закрываем"), или сессия была разовой (консультация, ответ на вопрос).

Если не очевидно — спроси: "Задача завершена или продолжим в новом чате?"

→ **Завершена** — Phase 1A → Phase 2 → Phase 3
→ **Продолжение** — Phase 1B → Phase 2 → Phase 3

### Phase 1A: Запись в session-log (задача завершена)

1. Получи текущее время: выполни `date "+%H:%M"` через Bash — используй результат как HH:MM в записи
2. Прочитай `meta/management/session-log.md`
3. Определи направление сессии по активной команде или контексту разговора:
   - `lab` — лаборатория, помощники, скиллы, агенты
   - `knowledge` — база знаний, исследования
   - `framework` — онтология, методология, scaffold, plugin
   - `dev` — сайт, код, архитектура
   - `clients` — клиенты, встречи, трекинг
   - `meta` — стратегия, маркетинг, управление
4. Сформулируй запись: **HH:MM | направление** — Что сделано. 1-2 предложения. (длительность)
   - HH:MM — время из шага 1, не придумывай
   - Если Виктор указал длительность — добавь в конец записи в скобках: `(55 мин)`, `(1ч 15 мин)`
   - Пиши **достижение + зачем оно нужно**, не детали реализации
   - Этот лог читают через неделю/месяц чтобы понять прогресс проекта. 15-20 сессий в день — каждая запись должна быть понятна без контекста
   - НЕ пиши: какие файлы менялись, какие баги фиксились, технические названия паттернов
   - Инсайт — только если он меняет подход к будущей работе
   - Если задача велась через worklog — добавь "(S{N})" в конец, где N — число сессий из мини-лога worklog
   - Пример хорошей записи: "Создан скилл close-session — формализует закрытие сессии: пишет достижение в лог, проверяет несохранённое, фиксирует незакрытые вопросы."
   - Пример с сессиями: "Система анализа встреч готова — методология + скилл meeting-processor. (S5)"
5. Покажи Виктору запись → дождись подтверждения → запиши под сегодняшнюю дату
6. Если секции с сегодняшней датой нет — создай `## YYYY-MM-DD` после `---`
7. Обнови поле `updated:` в YAML-заголовке
8. Если существует worklog с данными по этой задаче — очисти его до шаблона (по правилам в самом файле)
9. Если направление = `framework` И задача завершена (Phase 1A, не 1B) — добавь строку в мини-лог `framework/status.md` (формат: `- YYYY-MM-DD: Что сделано`). Если задача продолжается — НЕ трогай status.md. Правила файла — в самом status.md.

### Phase 1B: Обновление worklog (продолжение в новом чате)

НЕ пиши в session-log — запись появится когда задача будет завершена (Phase 1A).

1. Прочитай worklog текущего направления (`{направление}/_inbox/_worklog.md`)
2. Обнови worklog по правилам, описанным в самом файле: перезапиши состояние и handoff до актуального, добавь строку в сессии
3. Покажи Виктору что написал → дождись подтверждения → запиши

### Phase 2: Проверка несохранённого

1. Пройдись по разговору: инсайты, решения, находки, новые термины, скорректированные предположения
2. Для каждого: было ли это записано в файл проекта во время сессии?
3. **Проверь `_inbox/` папки.** Если в сессии был создан или изменён файл в `_inbox/` — проверь, есть ли соответствующая строка в `_TODO.md` этого inbox. Нет → предложи добавить.
4. Незаписанные → предложи Виктору конкретный файл + формулировку
4. **СТОП.** Жди ответ по каждому пункту. Подтвердил → запиши. Отклонил → пропусти.
5. Если всё записано: "Всё из сессии зафиксировано, ничего не потерялось."

### Phase 3: Маршрутизация открытых вопросов

1. Найди нерешённые вопросы, отложенные решения, пометки "потом"
2. Предложи маршрут по правилам [inbox-rules](../../../lab/inbox-rules.md)
3. Покажи предложенные записи Виктору
4. **СТОП.** Жди подтверждение по каждому. Подтвердил → добавь строку в `_TODO.md` соответствующего inbox.
5. Если открытых вопросов нет: "Открытых вопросов нет." → завершай.

## Завершающая нота

После всех фаз — заверши сессию коротко и тепло (1-2 предложения). Это момент благодарности: Виктор переходит из режима «делаю» в «сделал». Не шаблон — искренне, по-партнёрски, исходя из того, что реально произошло в сессии.

## Edge Cases

| Ситуация | Действие |
|----------|----------|
| Ничего значимого не произошло | Спроси: "Ничего не записываем?" — если да, пропусти Phase 1 |
| Не удаётся определить направление | Спроси Виктора: "Какое направление у этой сессии?" |
| Несколько направлений в одной сессии | Создай отдельную запись для каждого |
| Виктор говорит "быстро закрываем" | Только Phase 1 (A или B), пропусти 2 и 3 |
| Виктор говорит "закрываем, задача не закончена" | Phase 1B: обнови worklog, не пиши в session-log |
| Первая сессия по задаче, worklog пуст | Заполни worklog с нуля по правилам в файле |
| Задача завершена, worklog заполнен | Phase 1A: запиши в session-log с (S{N}), очисти worklog |
| session-log.md не существует | Создай с YAML-заголовком + первую запись |
