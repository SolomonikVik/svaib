# Ритуал планирования недели

Понедельник утром, один раз в неделю. Семь фаз: закрытие → итоги → анализ → бэклог → план недели → пост → план дня.

Бэклоги — доменные: `{direction}/02_backlog.md` в каждом направлении (framework, lab, dev, clients, marketing). План недели/дня и регулярные — в `meta/management/03_plan.md`. Модель работы и связи — в [lab/work-model.md](../../../lab/work-model.md).

## Iron Law

**Каждая фаза = отдельное сообщение. Не начинай следующую фазу без явного «ОК» от Виктора.**

## Принцип

Сформировать план недели, который ведёт к цели декады.

## EXECUTE NOW

1. Загрузи файлы из секции «Подготовка». Прочитай каждый
2. Возьми номер текущей недели из заголовка «План недели» в `03_plan.md`
3. Скажи Виктору: "Планёрка: закрываем W{N}, планируем W{N+1}. Верно?"
4. После «ОК» — выполняй фазы строго по порядку:

**Фаза 1** → Закрытие прошлой недели
**Фаза 2** → Итоги недели (weekly progress)
**Фаза 3** → Анализ
**Фаза 4** → Чистка бэклогов
**Фаза 5** → План недели
**Фаза 6** → Пост Telegram
**Фаза 7** → План дня на понедельник

**Iron Law: каждая фаза = отдельное сообщение. Не начинай следующую фазу без «ОК» от Виктора.**

**START NOW.**

## Фаза 1: Закрытие прошлой недели

Цель: зафиксировать прогресс в плане периода, очистить план недели от задач, вернуть невыполненное в доменные бэклоги.

### Алгоритм

1. Пройтись по плану недели W(N). Каждую задачу определить как регулярную (есть в шаблоне) или разовую.

2. **Выполненные задачи (любые):**
   - Удалить из плана недели (структуру и заголовки не трогать)

3. **Невыполненные регулярные:**
   - Удалить из плана недели
   - В шаблоне регулярных задач добавить метку пропуска: `[пропущено W21]` или дополнить существующую `[пропущено W20, W21]`

4. **Невыполненные разовые:**
   - Определить исходное направление задачи по смыслу (клиент → `clients`, скилл/помощник → `lab`, scaffold/плагин → `framework`, сайт/инфра → `dev`, пост/семинар → `marketing`)
   - Перенести в `{direction}/02_backlog.md` в подходящую подсекцию (если подходящей нет — спросить Виктора или создать новую по смыслу)
   - Добавить метку возраста: без метки → `[W1]`, с `[WN]` → `[W(N+1)]`. Метка едет вместе с задачей
   - Удалить из плана недели
   - Обновить `updated:` в YAML затронутого `02_backlog.md`

5. **Обновление плана периода:**
   - На основе выполненных задач — предложить обновления прогресса: количественные цели `(сейчас: N)`, завершённые `[x]`
   - НЕ вносить до согласования с Виктором

После шагов 1–5: в плане недели остаётся только структура (заголовки, имена клиентов), задач нет.

**ВАЖНО:** Список клиентов в плане недели — постоянная структура. Удалять можно только задачи под клиентом, не самого клиента. Клиент убирается из списка только по явному решению Виктора.

### Output фазы 1

Показать Виктору одно сообщение:
1. Предлагаю обновить в плане периода: {конкретные строки с новыми значениями}
2. В доменные бэклоги возвращаю: {задачи с метками и целевыми файлами}

**После «ОК» — обновить план периода, перенести в `{direction}/02_backlog.md` (обновив YAML затронутых файлов), проставить метки пропуска в шаблоне регулярных, очистить задачи из плана недели.**

## Фаза 2: Итоги недели

Цель: собрать факты недели, зафиксировать летопись.

**Iron Law фазы:** Имена клиентов в летописи не указывать — только «первый/второй/третий клиент». Номера — по хронологии появления.

### Алгоритм

1. **Clockify за всю неделю:** `lab/tools/clockify-fetch.sh {start} {end+1}` (диапазон, end не включается). Из ответа получить Total и группировку часов по направлениям по префиксам:

   | Clockify префикс | Направление |
   |---|---|
   | svaib клиенты | clients |
   | svaib-frame | framework/product |
   | svaib-pub: пост / svaib-pub: семинар | content |
   | svaib лаб / svaib-lab | lab |
   | svaib разное | meta |
   | svaib dev | dev |
   | обучение AI / svaib-learn | knowledge |
   | план дня / план недели | meta |

2. **Прочитать session-log** за даты недели — источник правды по содержанию (Clockify = правда по времени, session-log = правда по содержанию).

3. **Прочитать `{direction}/03_progress.md`** всех направлений за даты недели — результаты в доменах. Это основной источник «что доделано», дополняет session-log.

4. **Собрать выжимку** по направлениям: часы из Clockify, содержание из session-log + результаты из 03_progress. Подсветить ключевые события, решения, сбои.

5. **Дополнить из контекста фазы 1** — закрытые задачи, прогресс по плану периода, флаги.

6. **Написать драфт летописи** в формате и стиле `04_weekly_progress.md`. Ориентироваться на последние 3-4 недели как эталон тона (введение 2-3 предложения + блоки по направлениям с часами).

7. Показать Виктору драфт.

### Output фазы 2

Распределение часов по направлениям + развёрнутая выжимка + драфт летописи.

**После «ОК» — записать летопись в `04_weekly_progress.md` (сверху, перед предыдущей неделей). Обновить `last_recorded_week` в frontmatter.**

## Фаза 3: Анализ

Цель: понять, правильно ли движемся к цели декады.

### Алгоритм

Используй полную картину из фазы 2 (сырые данные, выжимку, решения, часы)
и обновлённый план периода из фазы 1 (метрики, прогресс):

1. Распределение часов по направлениям vs фокус декады —
   соответствует ли то, на что тратим время, тому, что заявлено как приоритет?

2. Метрики декады: конкретные значения «было → стало»
   (из плана периода, обновлённого в фазе 1). Что двигалось, что стояло.

3. Клиентская работа — вытеснялась ли чем-то?
   (клиенты = приоритет первой декады)

4. Прямой вердикт: «в фокусе» или «ушли не туда, потому что...»

### Output фазы 3

Аналитическая оценка: пункты 1-4 выше. Может быть короткой —
если всё в порядке, достаточно «в фокусе, метрики двигаются».

**После «ОК» — переходим к фазе 4.**

## Фаза 4: Чистка бэклогов

Цель: привести доменные бэклоги в порядок перед формированием плана недели.

### Алгоритм

1. Пройтись по бэклогам направлений в порядке: `clients` → `framework` → `dev` → `lab` → `marketing`.
   В каждом — по подсекциям, по каждой задаче:
   - **Актуальна для декады** → оставить
   - **Не актуальна** → предложить удалить
   - **[W3]+** — красный флаг: обратить внимание Виктора, задача трижды не выполнена

2. **Приоритизация:** внутри каждой подсекции — важное наверх.
   Конвенция: порядок строк = приоритет.

### Output фазы 4

Показать Виктору по каждому направлению:
1. Предлагаю удалить: {задачи с обоснованием}
2. [W3]+: {задачи} — красный флаг
3. Предлагаю порядок приоритетов (если менялся)

**После «ОК» — обновить затронутые `{direction}/02_backlog.md` (в том числе `updated:` в YAML).**

## Фаза 5: План недели

Цель: сформировать план недели W(N+1) от стратегии, а не от инерции.

### Алгоритм

Создать TodoWrite на всю фазу как чеклист.

1. Напомнить: vision → цель экватора → цель декады (2–3 строки, не пересказ)
2. Посмотреть анализ из фазы 3 — куда корректировать
3. **Регулярные задачи** — из шаблона в `03_plan.md`, фильтр по чёт/нечёт W(N+1).
   Встречи → в блок встреч соответствующих дней.
4. **Разовые задачи — обсуждение по направлениям:**
   По каждому доменному бэклогу (clients → framework → dev → lab → marketing):
   - Показать задачи (сверху вниз = по приоритету, по подсекциям)
   - Предложить что взять и ПОЧЕМУ (привязка к цели декады)
   - Если бэклог пуст или подходящих задач нет — посмотреть план периода:
     какую следующую цель превратить в задачу?
   - Виктор корректирует → фиксируем

   Это ОБСУЖДЕНИЕ, не готовый список. У агента нет полного
   стратегического контекста — Виктор дополняет из головы.
5. Собрать единый план W(N+1) в `03_plan.md` по категориям плана недели.
   Маппинг направление → категория — в секции «Подготовка» выше.
   Правило: клиентская работа всегда первая.
   **Правило метки [WN]:** при переносе задачи из `{direction}/02_backlog.md` в план недели — **сохранять метку [WN] в тексте задачи**. Метка — счётчик возраста задачи, а не атрибут позиции в бэклоге. Метка едет вместе с задачей.
6. Обновить заголовок плана недели: номер W(N+1) и даты.

### Output фазы 5

Согласованный план W(N+1) — результат обсуждения по направлениям.

Виктор корректирует — агент вносит правки и показывает обновлённый план.
Итерации до финальной версии.

**После согласования финальной версии — записать план недели в `03_plan.md`.
Взятые в план задачи удалить из соответствующих `{direction}/02_backlog.md`.
Обновить `updated:` в YAML затронутых файлов.**

**Правило удаления из бэклога:** перед удалением сравнить формулировку задачи в доменном бэклоге и в плане недели. Удалять только если план недели покрывает ВЕСЬ скоуп задачи из бэклога. Если в план взята только часть (например, "обсудить" из задачи "расшифровать + обсудить") — задача остаётся в бэклоге.

## Закрытие

Запись в session-log: "Ритуал планирования недели W(N+1)." + длительность. Спросить Виктора сколько заняло.

## Red Flags

Если ты делаешь что-то из этого списка — СТОП, ты нарушаешь скилл:

- Добавил задачу в `03_plan#Бэклог` (такой секции больше нет — бэклоги доменные)
- Читаешь общий бэклог из `03_plan.md` (не существует в этой модели)
- При переносе задачи из доменного бэклога в план недели — теряешь метку `[WN]`
- Удалил задачу из доменного бэклога, но план недели покрывает её не полностью

## Edge Cases

| Триггер | Действие |
|---------|----------|
| В домене нет `02_backlog.md` | Направление не имеет задач на будущее — пропусти. Если Виктор предлагает добавить задачу в это направление, создай файл по `file_spec` |
| Задача бэклога не вписывается в категории плана недели (Клиенты/Продукт/Лаборатория/Публикации/Обучение/Прочие) | Спроси Виктора — возможно, завести новую категорию (например `### Dev` при dev-задачах) |
| Задача из `dev/02_backlog.md` | По умолчанию — в `### Продукт` плана недели. Если dev-задач несколько — спроси Виктора про создание `### Dev` |
| Клиентская папка в `.gitignore`, не видна глобальным инструментам | `clients/02_backlog.md` существует локально — читай, обновляй. Изменения не коммитятся, это норма |
