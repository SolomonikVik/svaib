# Roadmap Джарвиса

## Миссия

Виктор приходит с болью или задачей — Джарвис делает так, чтобы эта боль исчезла навсегда. Результат: помощники в .claude/ (skills, agents, hooks, commands), которые потенциально идут в плагин для клиентов.

Но Джарвис — не только конвейер. Он развивает и себя: знания, процессы, инструменты. Каждая сессия — шаг в развитии, а не изолированный эпизод.

## Оси развития

### 1. Knowledge Integration
Виктор вкладывает 8-10ч/нед в knowledge/. Джарвис должен быть на вершине этих знаний — не обязательно в памяти, но знать куда пойти и что подгрузить.

### 2. Progressive Disclosure
Не грузить всё сразу. Lean core (процесс) + reference files (глубокие знания по типу помощника) + knowledge/ (внешние знания). Подгрузка по потребности.

### 3. Architectural Maturity
Процесс шире чем "построить помощника": разбор inbox, развитие принципов, архитектурные решения, рефлексия. Архитектурная идея: принципы + прецеденты (как прецедентное право) — меняем конституцию или прецеденты, а не вечно правим код.

### 4. Cross-session Memory
Стратегический контекст между сессиями. Roadmap (этот файл), principles.md, rescue-log.yml — след каждой сессии. Не эпизоды, а траектория.

### 5. Cross-team Sync
Knowledge flows between commands. Когда knowledge/ обогатился — framework/ и lab/ узнают об этом. Виктор не должен быть единственным мостом.

## Текущие ограничения

- **Изоляция от knowledge/** — research формальный, не глубокий
- **Всё или ничего** — нет reference-файлов, чеклист грузится всегда
- **Процесс = только конвейер** — нет процесса для inbox, принципов, архитектуры
- **Знания разрозненны** — principles, inbox, MEMORY, rescue-log в разных местах
- **Нет стратегии** — нет карты помощников, нет связи с целями декады

- **Передистилляция knowledge/ → todo-файлы (knowledge distillation pipeline)**

  **Откуда:** Сессия 2026-03-02. При верификации todo-skill.md обнаружили дилемму: шаг 2 требует чтения 4 файлов из knowledge/skills/. Если координатор читает сам — теряет контекст для основной задачи (создание скилла). Если делегирует субагенту — получает выжимку, но может пропустить важное, потому что файлы не дистиллированы под координатора.

  **Проблема:** knowledge/ пишется как справочник (для любого AI), а todo-файлы lab/ нуждаются в выжимке (для координатора с конкретной задачей). Если встроить выжимку в todo — она устаревает при обновлении knowledge/. Нет механизма узнать что устарело.

  **Решение (event-driven):**
  1. knowledge-координатор (svaib-knowledge) обновляет файл в knowledge/skills/
  2. Он же фиксирует дифф в `lab/_inbox/`: "обновился !skills.md — добавлен YAML-атрибут context: fork"
  3. lab-координатор на старте сессии: проверяет inbox → если есть дифф по skills/ → передистиллирует todo-skill.md → потом работает

  **Принцип #8 (контекст — главный ресурс):** todo-файл с дистиллированной выжимкой = один Read вместо четырёх. Inbox с диффами = знаем когда обновлять, а не перечитываем всё каждый раз.

  **Первый шаг:** сделать дистилляцию вручную один раз (встроить выжимку в todo-skill.md). Потом — автоматизировать через inbox.

- **Навигация по связям, не по путям (arscontexta model)**

  **Откуда:** Сессия 2026-03-03. Глубокий разбор arscontexta architecture — верифицировано по коду на GitHub.

  **Проблема:** Джарвис находит файлы по путям (`knowledge/skills/!skills.md`). Реорганизация файлов ломает все ссылки. knowledge/ обновляется — lab/ не узнаёт. Каждая сессия начинается с чистого листа без понимания ландшафта.

  **Что видели в arscontexta (верифицировано по коду):**

  1. **Session-orient hook** — при старте сессии автоматически инжектит: file tree (`tree -L 3`), identity-файлы (кто агент, методология, цели), condition signals (observations ≥10 → предложи reduce, inbox ≥3 → обработай, methodology staleness ≥30d → пересмотри). Джарвис ничего из этого не делает — стартует холодно.

  2. **Wikilinks `[[name]]`** — ссылки по имени файла, не по пути. Файл можно переместить — все ссылки работают. У нас markdown links `[text](path)` — привязаны к путям.

  3. **Progressive Disclosure (5 уровней)** — `tree + self/ → YAML descriptions → MOC hierarchy → wiki-link traversal → full content`. У Джарвиса 2 уровня: lean core (svaib-lab.md) → todo-файлы. Нет промежуточного слоя YAML-описаний и MOC-навигаторов.

  4. **Three-Space Architecture** — self/ (identity, slow), notes/ (knowledge, steady), ops/ (operations, fluctuating). У нас: .claude/ (identity + operations смешаны), knowledge/ (отдельно), lab/ (meta).

  **Что можно взять:**
  - Session-orient hook → hook в `.claude/hooks/`, инжектит tree + текущую версию Джарвиса + inbox count + последний changelog entry
  - YAML descriptions → каждый файл knowledge/ уже имеет description в YAML, нужен навигатор (субагент или скилл), который сканирует descriptions без чтения файлов целиком
  - Condition signals → hook проверяет `lab/_inbox/` (есть ли диффы), последнюю запись changelog (свежесть), `rescue-log.yml` (частые сбои) — и подсказывает координатору что требует внимания
  - Crossfeed (уже в backlog выше) получает конкретную механику: knowledge-координатор пишет дифф → session-orient показывает его lab-координатору

  **Связь с осями развития:** Knowledge Integration (#1), Progressive Disclosure (#2), Cross-session Memory (#4), Cross-team Sync (#5) — arscontexta model даёт конкретную реализацию для всех четырёх.

  **Ирония сессии:** мы обсуждали "в какую папку положить файлы skill-graphs/" — именно ту проблему, которую arscontexta architecture решает. Навигация по связям делает вопрос папки вторичным.
