# Rescue Reference

Справочник для агента rescue. Не загружается в контекст при старте — агент читает нужную секцию когда она ему нужна.

---

## Таксономия сбоев

Используй на шаге CLASSIFY. Если сбой не вписывается — сформулируй новый тип, это ценно.

| Категория | Тип | Описание |
|-----------|-----|----------|
| **Behavioral** | sycophancy | Угодливость вместо экспертизы. Отдаёт решение пользователю, хотя по роли это его зона |
| | premature-agreement | Соглашается с первым предложением не проверив альтернативы |
| | authority-bias | Принимает слова пользователя как факт без верификации |
| | over-hedging | Избыточные оговорки вместо конкретного ответа |
| **Instruction** | token-костыль | Предлагает дописать в инструкцию то, что уже следует из существующего текста |
| | literal-reading | Читает инструкцию буквально, не выводя следствия |
| | priority-inversion | Второстепенное правило перевешивает ключевое |
| | overloaded-prompt | Промпт перегружен правилами, AI перестаёт думать от принципов и механически чекает список. Любой непокрытый кейс проваливается |
| **Context** | post-compaction-amnesia | После компакции контекста забыл ранее данные инструкции |
| | displacement | Новая информация вытеснила ранее загруженную |
| | misreferencing | Ссылается на файл/секцию которые не читал или читал давно |
| | self-reinforcement | Верит собственным предыдущим записям в файлах как фактам, не ставя под сомнение. Отличие от authority-bias: там верит пользователю, тут — самому себе через файлы. Каскад: ошибка в сессии N → файл → "факт" в сессии N+1 |
| **Planning** | premature-termination | Остановился раньше чем задача выполнена |
| | repetition-loop | Повторяет одно и то же действие без прогресса |
| | process-step-skip | Пропустил шаг собственного задокументированного процесса |
| **Meta** | protocol-violation | Нарушил явное правило из CLAUDE.md или команды |
| | scope-creep | Сделал больше чем просили, добавил лишнее |
| | tool-misuse | Использовал инструмент не по назначению или не тот |
| | diagnosis-without-remedy | Нашёл причину но не предложил как предотвратить |

## Чеклист типа фикса

Используй на шаге REMEDIATE, перед тем как предлагать конкретный фикс.

"Добавить правило" — дефолтный инстинкт, но часто неправильный. Чем больше точечных правил в промпте, тем меньше AI думает от принципа и больше механически чекает список.

1. **Принцип уже есть?** Прочитай целевой файл. Если общий принцип покрывает этот кейс → AI не вывел следствие. Фикс: заточить формулировку принципа, не плодить конкретные правила.
2. **Сколько правил уже?** Если в целевом файле >8 специфических правил/red flags по теме → ещё одно ухудшит, не улучшит. Фикс: консолидировать несколько правил в один ясный принцип.
3. **Это уже патчили?** Проверь rescue-log.yml: были ли похожие фиксы к этому файлу. Если да → текстом не чинится. Нужно структурное: разбиение файла, gate в процессе, смена подхода.
4. **Это ограничение модели?** Некоторые паттерны (многословие, стремление показать работу, eager-to-please) — тенденции модели, которые промптом до конца не убираются. Если так — скажи прямо. Не делай вид что ещё одно правило поможет.
5. **Это ограничение платформы (Claude Code)?** Причина может быть не в тексте инструкций, а в механике платформы: tool calls и текст уходят в одном ходе, контекстная компакция теряет данные, субагент не видит историю. Запусти субагента `claude-code-guide` для исследования механики. Если причина платформенная — фикс не в тексте, а в использовании правильного инструмента, хука или изменении процесса.

**Иерархия фиксов:** Принципы > Правила > Конкретные запреты.

## Red Flags

Признаки что диагностика идёт не так. Если заметил у себя — остановись и перепроверь.

1. **Диагноз без цитат** — ни одной цитаты из диалога или файла-источника. Вернись к OBSERVE.
2. **"Проблема в интерпретации" без конкретики** — какой именно текст был неправильно интерпретирован? Какое следствие не выведено?
3. **Фикс = "добавить правило"** — пройди чеклист типа фикса выше. Больше правил ≠ лучше поведение.
4. **Превентивная мера = "быть внимательнее"** — это не мера. Мера = изменение файла, процесса или инструмента.
5. **Один тип без проверки каскада** — trace upstream. Может за sycophancy стоит overloaded-prompt.
6. **Не прочитал rescue-log.yml** — нарушение Iron Law. Может это уже третий раз.
7. **Категория "Meta" при первом анализе** — перепроверь. Meta-сбои обычно маскируют более глубокую причину.

## Common Rationalizations

| Отговорка | Почему неправильно | Что делать вместо |
|-----------|--------------------|-------------------|
| "Проблема понятна и без чтения файлов" | Диагноз из головы, без проверки — так работал v1 | Iron Law: прочитай файл-источник |
| "Тип сбоя не вписывается в таксономию" | Новый тип — это ценность, не проблема | Сформулируй новый тип, добавь в паттерн |
| "Фикс очевиден, превентивная мера не нужна" | Без превентивной меры тот же сбой повторится | Обязательно: что изменить в среде |
| "Это разовый сбой, не стоит записывать" | Без записи нет данных для анализа частотности | Всегда добавляй запись в rescue-log.yml |
| "Виктор сам виноват в нечёткой постановке" | Координатор должен уточнять, а не обвинять | Проверь: была ли возможность уточнить? |
| "Ещё одно правило точно закроет проблему" | Чем больше правил, тем меньше AI думает от принципа | Пройди чеклист типа фикса. Принцип > правило |

## Верификация (перед выводом)

Прежде чем отдать диагноз координатору:

- [ ] Iron Law соблюдён — прочитан файл-источник сбоя
- [ ] Есть конкретные цитаты (из диалога И из файлов)
- [ ] Тип назван точно (не "что-то из behavioral")
- [ ] Trace upstream выполнен (каскад проверен)
- [ ] Чеклист типа фикса пройден
- [ ] Фикс конкретный (файл:строка или действие)
- [ ] Превентивная мера — изменение среды, не "быть внимательнее"
- [ ] Запись для rescue-log.yml сформирована
