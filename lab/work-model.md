# Work Model

## Кратко

Оперативный гайд по модели работы svaib: из каких файлов состоит направление, как задача движется от входящего до результата, как передаётся контекст между чатами, как закрывается сессия. Единый паттерн для всех направлений (`framework/`, `lab/`, `clients/`, `dev/`, `meta/marketing/`) и для клиентских `projects/`. Источник правды — один словарь `_inbox / 01_overview / 02_active / 02_backlog / 03_progress / 04_decisions`.

**Работа с файлом:** procedure — читай перед первой работой в направлении. Меняй при изменении модели (не при ежедневной работе).

## Связанные файлы

- [framework/scaffold/MODEL.md](../framework/scaffold/MODEL.md) — Iron Law, единица развёртывания
- [framework/memory/file_spec.md](../framework/memory/file_spec.md) — канон формата файлов (YAML, секции, размер)
- [.claude/skills/close-session/SKILL.md](../.claude/skills/close-session/SKILL.md) — закрытие сессии (контракт в разделе ниже)
- [.claude/skills/plan/SKILL.md](../.claude/skills/plan/SKILL.md) — быстрые операции над `02_backlog`
- [.claude/skills/weekly-planning/SKILL.md](../.claude/skills/weekly-planning/SKILL.md) — недельный ритуал
- [meta/management/session-log.md](../meta/management/session-log.md) — лог сессий работы с Claude (процесс)

## Модель направления

Каждое направление устроено одинаково:

```
/{direction}/
├── _inbox/
│   ├── 01_inbox.md          — индекс входящего на разбор
│   └── [сырые материалы, подпапки допустимы]
│
├── 01_overview.md           — миссия, стадия, цель декады (reference)
├── 02_active.md             — что в работе сейчас + Session Handoff (plan)
├── 03_progress.md           — хроника завершённого (log)
│
├── 02_backlog.md            — расширение: задачи на будущее
├── 04_decisions.md          — расширение: реестр решений
│
└── [docs/, подпапки контента]
```

### Ядро и расширения — Iron Law

**Ядро (всегда):** `_inbox/01_inbox.md`, `01_overview.md`, `02_active.md`, `03_progress.md`.

**Расширения (по нагрузке):** `02_backlog.md`, `04_decisions.md`. **Пустыми не создаются.** Появляются, когда есть реальный контент:

- `02_backlog.md` — задач больше, чем помещается в active
- `04_decisions.md` — накопились развилки, которые стоит помнить

### Логика нумерации

Номер = этап жизненного цикла: `inbox → backlog (02) → active (02) → progress (03) → decisions (04)`. Одинаковый номер = один семантический слой: `02_active` и `02_backlog` оба про задачи (сейчас vs весь список) — соседние файлы.

---

## Движение задачи

```
_inbox → (разбор) → 02_backlog → 02_active → 03_progress
                                     ↓
                              04_decisions  (поперечно)
```

- **Появилась** — положи во входящий буфер (`_inbox/01_inbox.md` направления).
- **Разобрана** — если это задача на будущее, перенеси в `02_backlog.md`; если срочно — в `02_active.md`.
- **Делается** — живёт в `02_active.md#Активные задачи`.
- **Завершена** — строка в `03_progress.md` (дата, суть), из `02_active.md` убрана.
- **Развилка** — при необходимости зафиксируй в `04_decisions.md` (из любой точки цикла).

## _inbox — входящее на разбор

**Назначение.** Буфер входящих материалов и задач. Стремится к пустоте, не к архиву выполненного.

**Где живут.** В каждом направлении есть свой `_inbox/`. Не знаешь куда — кидай в `meta/_inbox/` (единственный «нейтральный» инбокс, без `01_overview/02_active/03_progress`).

### Как добавить

Открой `_inbox/01_inbox.md` нужного направления, добавь строку. Запись самодостаточна — агент понимает что это, зачем и надо ли действовать, не открывая других файлов.

Простая запись:

```
- [ ] Разобрать: нужен ли отдельный механизм обмена находками между ролями
```

Запись с отдельным файлом:

```
- [ ] В [crossfeed.md](crossfeed.md) — идея механизма обмена находками. Разобрать
```

Сложная запись — подзадачи вложенным списком (≤2 уровня).

**Когда нужен отдельный файл.** Если материал требует развёрнутого описания — создай файл в той же `_inbox/`, сошлись на него из строки в `01_inbox.md`. Подпапки допустимы (`_inbox/subagents/`, `_inbox/context-management/`).

### Как разобрать

Разобрал запись → **удали строку** из `01_inbox.md`. Не ставь `[x]`, не зачёркивай. Список стремится к пустоте. Если был отдельный файл и его содержимое больше не нужно — удали и его.

**Куда уходит содержимое:**
- Задача на будущее → `{direction}/02_backlog.md`
- Срочное → `{direction}/02_active.md`
- Полезное знание → `knowledge/` или в docs направления
- Устарело → удалить

## Session Handoff — передача контекста между чатами

**Проблема.** AI теряет контекст при смене чата. Виктор не должен заново объяснять что делаем.

**Решение.** В `02_active.md` каждого направления — секция **Session Handoff** сверху. Это файл СОСТОЯНИЯ активной задачи для передачи между чатами: на чём остановились, что проверить следующему чату, ключевые ссылки.

### Правила

- **Секция, не файл.** Session Handoff живёт внутри `02_active.md`, не в отдельном файле. Порядок секций: Session Handoff → Активные задачи → Открытые вопросы.
- **Перезаписывается, не растёт.** При обновлении — заменяй секции на актуальные. Это состояние, не хроника.
- **Не каждая задача требует Handoff.** Только если работа идёт >1 сессии или контекст не очевиден из `02_active.md#Активные задачи`.
- **Когда задача завершена** — Session Handoff по ней очищается (см. `/close-session`, Phase 1A).
- **Не ссылайся на эфемерные файлы.** Содержимое `.claude/plans/` и временных артефактов копируй в Session Handoff — следующий чат их не увидит.

### Для клиентских шаблонов

Session Handoff — **только для направлений svaib.** В клиентских шаблонах (`framework/scaffold/04_management/{org_node}/`, `framework/scaffold/projects/`) секции нет — у клиента другой режим работы.

## Жизненный цикл сессии

```
Команда /svaib-{direction} → работа → /close-session
```

### 1. Старт сессии

Вызов `/svaib-{direction}` подгружает: `README.md`, `01_overview.md`, `02_active.md` направления + `product_vision.md` (для `framework`). Команда проверяет Session Handoff — если есть активная задача, предлагает продолжить.

### 2. Работа

- Результаты пишутся в доменные файлы (`02_active`, `02_backlog`, `03_progress`, `04_decisions`).
- Новые материалы на разбор → `_inbox/01_inbox.md` нужного направления.
- Если задача перескакивает на >1 сессии — обнови Session Handoff в `02_active.md` того направления, куда перевешиваешь контекст.

### 3. Закрытие — `/close-session`

Контракт — в следующем разделе.

## Close-session: 3 ситуации

Контракт в трёх ситуациях:

| Ситуация | Куда пишем |
|----------|-----------|
| Значимая сессия (большой кусок работы), независимо от завершения задачи | Строка в `meta/management/session-log.md` (дата, тема, длительность, достижение) |
| **Задача полностью завершена** | Дополнительно: запись в `{direction}/03_progress.md` + очистка Session Handoff в `{direction}/02_active.md` |
| Продолжение задачи (чат закрываем, задача живёт) | Обновляем **только** `{direction}/02_active.md#Session-Handoff`. В session-log не пишем, если сессия короткая/техническая |

### Оси файлов

`03_progress` и `session-log` — разные оси:

- **`03_progress.md` в направлении** — **результат.** Что сделано в домене. Пишется только при полном завершении задачи.
- **`session-log.md` в meta/management/** — **процесс работы с Claude.** Дата, тема, длительность, инсайты. Пишется по факту значимой сессии, независимо от завершения задачи. Используется Clockify-сверкой как источник правды.
- **`04_weekly_progress.md`** агрегирует из доменных `03_progress`, не из `session-log`.

| Скилл / команда | Что читает | Что пишет |
|-----------------|-----------|-----------|
| `/svaib-{direction}` | `{direction}/01_overview.md`, `02_active.md` | — |
| `/plan` | `meta/management/03_plan.md` (day/week/regular), `{direction}/02_backlog.md` | то же |
| `/weekly-planning` | `{direction}/02_active.md`, `{direction}/02_backlog.md`, `meta/management/*` | план недели, `04_weekly_progress.md` |
| `/close-session` | `02_active.md`, `session-log.md`, `03_progress.md` | по 3 ситуациям выше |

## Правила безопасности

- **Удаление источника — только после verification переноса.** Перенёс задачу из `02_active` в `03_progress` → проверил, что запись на месте → удалил из `02_active`. То же для `_inbox` → `02_backlog`.
- **Секции сверху — самое горячее.** Session Handoff — в начале `02_active.md`, потому что читается первым после открытия файла.
- **Фиксируй конкретно.** В `03_progress.md` — что сделано и зачем, не процесс. В Session Handoff — на чём остановились и что проверить, не пересказ сделанного.
