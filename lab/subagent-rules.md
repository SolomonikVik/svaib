# Правила работы с субагентами

Операционный протокол для запуска субагентов через Agent tool в Claude Code.

**Главный адресат:** координатор. Именно он решает, нужен ли субагент, как поставить задачу и как забрать результат.

**Субагент** не читает этот файл целиком. Он получает только секцию **"Правила субагента"** через хук `SubagentStart`.

## Как устроен этот файл

- Здесь только принципы работы, а не история инцидентов.
- Правила координатора живут только здесь.
- Правила субагента помечены `⚡` и дублируются в `.claude/hooks/inject_session_context.py` (`SUBAGENT_RULES`).
- Напоминание координатору перед запуском субагента дублируется в `.claude/hooks/remind_subagent_rules.py`.
- Если правило должно соблюдаться жёстко, одного текста в `CLAUDE.md` недостаточно. Нужен hook, permission rule или другой элемент ландшафта.

## Как обновлять правила

1. Сначала определи адресата: это правило для координатора или для субагента.
2. Если правило для субагента: обнови текст здесь и синхронно обнови `SUBAGENT_RULES` в `.claude/hooks/inject_session_context.py`.
3. Если правило для координатора и оно должно всплывать перед каждым запуском: обнови текст здесь и чеклист в `.claude/hooks/remind_subagent_rules.py`.
4. Пиши правило как принцип. Без дат, баг-репортов, ссылок на issues и случайных деталей из конкретной сессии.

## Enforcement

- `PreToolUse` с matcher `Agent` → напоминает координатору правила перед запуском субагента.
- `SubagentStart` → инжектит `⚡`-правила в контекст субагента.

## Правила координатора

### 1. Сначала определи границы задачи

Субагент получает один понятный кусок работы.

Координатор в промпте явно задаёт:
- что именно нужно сделать;
- какой результат нужен на выходе;
- нужен ли Bash;
- нужен ли файл-артефакт.

Если задача требует координации нескольких кусков, это работа координатора, а не субагента.

### 2. Если ожидается Bash — пропиши ограничение прямо в промпте

В промпте субагенту явно писать:

`Use only simple, single bash commands - no &&, no ||, no pipes, no ;, no redirects.`

Иначе субагент почти наверняка начнёт собирать составные команды, а Claude Code будет дёргать пользователя на permission prompt.

### 3. Если нужен файл-результат — путь задаёт координатор

Для выходов субагентов используется только `_inbox/subagents/` внутри нужного `_inbox/`.

Координатор:
- заранее создаёт папку, если её нет;
- в промпте указывает точный путь записи;
- не оставляет субагенту самому выбирать место для файлов.

### 4. Запускать только foreground

Не использовать `run_in_background: true`.

Subagent workflow у нас рассчитан на foreground-запуск. Background ломает permission flow и делает поведение менее предсказуемым.

### 5. Координация остаётся у координатора

Координатор сам решает:
- сколько нужно субагентов;
- кого и в каком порядке запускать;
- как соединить их результаты.

Нельзя делегировать вниз оркестрацию, план работ или маршрутизацию между субагентами.

### 6. Результат передавать по пути, а не перечитыванием

Если субагент записал файл, координатор по умолчанию работает с путём к артефакту, а не читает его содержимое обратно в свой контекст.

### 7. Архитектурные задачи — не делегировать планирование

Субагенты годятся для разведки (найти файлы, проверить ссылки, собрать данные). Но план, архитектурные решения и верификацию плана координатор делает сам. Нельзя запустить Plan-субагента и взять его output as-is — координатор обязан сам прочитать спецификацию, сам построить маппинг, сам сверить результат с каноном.

**Инцидент:** миграция clients/ 2026-04-06. План был скомпилирован из output субагентов без собственного осмысления координатором. Результат: не воспроизводил целевую структуру, пропускал пути, тихо принимал незакрытые архитектурные решения. Потребовалась полная переделка.

Исключение: аварийный режим, когда субагент не смог записать файл и результат приходится вытаскивать из лога.

## Правила субагента

Эти правила инжектятся автоматически через `SubagentStart`.

### 1. Bash-команды: только простые ⚡

Одна команда за вызов Bash. Без `&&`, `||`, `|`, `;` и редиректов.

### 2. Запись: только в указанный путь внутри _inbox/subagents/ ⚡

Если координатор дал путь записи — писать только туда.
Если путь не дан, не придумывать его самостоятельно.

### 3. Работай только в пределах порученного куска ⚡

Не придумывай свой workflow, дополнительные роли и новые контуры координации.
Если для выполнения не хватает границ задачи или пути записи — вернись к координатору с блокером.
