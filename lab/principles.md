# Принципы

Четыре принципа. Каждый — из конкретного сбоя.

---

## 1. Необходимо и достаточно

Каждый элемент — файл, секция, правило, токен в контексте — должен быть оправдан задачей конкретного потребителя в конкретный момент.  

Это означает: подумай, ГДЕ информация реально нужна, и положи её ТОЛЬКО туда. 
Не дублируй, не клади в место с высокой частотой чтения то, что нужно редко.
Нет обоснования — убери. Нужно в двух местах — ссылка, не дубль.
---

## 2. Ландшафт важнее инструкции

Агент ошибётся везде, где может ошибиться. Не объясняй правильный путь — заблокируй неправильные. Структурой, форматом, ограничениями.

**Тест:** «какие нежелательные траектории существуют и чем заблокированы?»

**Источник:** Красинский, energy landscape метафора. Подтверждено: process-step-skip (2026-02-27, 2026-03-05).

---

## 3. Принципы важнее правил

Один ясный принцип покрывает десять конкретных запретов. После ~8 правил по одной теме AI перестаёт думать и становится чекером списка.

**Тест:** «можно ли заменить эти N правил одним принципом?»

**Источник:** rescue-агент 2026-02-28. Агент с 10 Red Flags — всё равно предложил ещё.

---

## 4. Четыре слоя системы

Каждый элемент живёт в одном слое. Смешение слоёв = bias или потеря enforcement.

| Слой | Что в нём | Где живёт |
|------|----------|-----------|
| **Core** | Нужно при любой задаче | `CLAUDE.md` |
| **Reference** | Нужно по типу задачи | include-файлы, knowledge/, lab/ |
| **Modes** | Task-specific маршруты | команды `/svaib-*`, маршруты внутри команд |
| **Enforcement** | Нельзя оставлять на усмотрение модели | hooks, permissions, scripts |

**Тест:** «в каком слое это живёт? Не смешиваю ли я два слоя в одном месте?»

**Источник:** runtime fix S5-S6 (2026-03-17). Always-on identity в Core → стратегический bias. Memory как enforcement → не работает, hook работает.
