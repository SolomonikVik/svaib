# Миграция контекста SVAIB

## ИНСТРУКЦИЯ (читай первым, выполняй всегда)

Этот файл — единственная память между сессиями миграции. Новый чат НАЧИНАЕТ с его чтения.

1. **Фиксируй ВСЁ по ходу** — каждое решение, каждую деталь обсуждения. Если Виктор что-то объяснил — запиши сюда немедленно. Не держи в контексте — следующий чат этого не увидит
2. **Перерабатывай, не допихивай** — файл должен оставаться чистым и актуальным. Устаревшее — удаляй, дублирующее — объединяй. Не превращай в свалку дописок
3. **Перед закрытием сессии** — обязательно пройдись по файлу: что сделали, что осталось, все ли решения записаны
4. **Структурные изменения → сюда немедленно** — любое изменение структуры (новые папки, перемещение файлов, переименование, новые файлы) → сразу фиксировать в этом файле. Это единственный источник правды о состоянии миграции
5. **Субагенты** — используй субагентов для тяжёлых операций (создание файлов, форматирование, проверка), чтобы сохранять контекст координатора для стратегических решений
6. **context-editor НЕ вызывать до Фазы 3** — во время Фазы 2 мы меняем структуру и наполняем файлы. Вызывать context-editor на каждом файле бессмысленно: структура ещё не стабильна, связи меняются, README устареют через день. Сначала наводим порядок в содержании (Фаза 2), потом один прогон context-editor по всему (Фаза 3). Иначе — бесконечный цикл "поменял файл → вызвал editor → поменял другой файл → editor сломал связи первого"

## Контекст проблемы

Проект прошел 5 пивотов за 5 месяцев (сентябрь 2025 — февраль 2026). Контекстные файлы не обновлялись систематически — context-editor не подключался, потому что приоритетом было встраивание, а не обновление контекста. Результат: meta/ и dev/ содержат устаревшую информацию, которая противоречит текущему фокусу (Second AI Brain).

### Хронология пивотов (из поста "100 недель кринжа")
1. Непонятная "AI-лаборатория"
2. "Продуктовая AI-мастерская"

### Структура meta/ (утверждена, v3)

```
meta/
├── _old/           ← временная свалка (бывший meta_context + pub + management из корня)
├── management/     ← чистая, наполняем из scaffold (vision, goals, plan, progress)
├── marketing/      ← каналы, контент, бренд (подробности ниже)
├── product/        ← продуктовая стратегия: что строим, зачем, куда развиваем
├── z_archive/      ← постоянный архив (research, старые версии)
└── viktor_profile.md  ← профиль основателя
```

**Корень:**
- `PROJECT.md` — полное описание проекта для AI-сессий. Отличие от других корневых файлов:
  - README.md = для людей/GitHub ("что это за проект")
  - CLAUDE.md = техническая инструкция для Claude Code ("как работать с репо"), грузится в контекст на КАЖДОМ ходу автоматически — поэтому должен быть максимально сжатым
  - PROJECT.md = "знание о проекте" ("кто мы, что делаем, куда идём"), подгружается по запросу в конкретные команды, может быть большим. Не нужен для knowledge или dev, нужен для стратегических сессий

### Структура meta/marketing/ (утверждена)

**Почему "marketing", а не "publications":** всё что делает Виктор наружу — это маркетинг (посты, семинары, конференции, YouTube). Нет ничего, что "публикация, но не маркетинг". Название консистентно со scaffold — клиентам даём marketing/, сами используем marketing/.

**Три потока контента:**
1. **Публикации** — Telegram (основной канал, уже очень много постов), дублирование в Twitter/X, Threads. Виктор приучился делиться информацией в Telegram и хочет масштабировать на другие платформы
2. **Knowledge → SEO/GEO** — статьи из knowledge/ с `publish: true` идут на сайт (механизм в dev/). GEO = Generative Engine Optimization, AEO = Answer Engine Optimization — оптимизация для AI-поисковиков
3. **События** — семинары (серия стартует ~10 марта 2026, делится инсайтами с людьми), записи возможно на YouTube. Конференции (Liga и др.). YouTube — маленькие ролики, по аналогии с привычкой делиться в Telegram

**Структура:**
```
meta/marketing/
├── README.md
├── strategy.md          ← позиционирование, каналы, воронка (из старого marketing.md)
├── brand.md             ← цвета, шрифты, tone of voice
├── telegram/            ← посты (основной канал, дублируется в X/Threads)
├── seminars/            ← программы, записи, материалы
├── conferences/         ← Liga и т.д.
└── youtube/             ← сценарии, ролики
```

**SEO/GEO (AI-SEO):** knowledge/ файлы с `publish: true` → парсятся dev/ → страницы на сайте. Это задача разработки, не marketing/.

## Принятые решения

1. **Подход:** "напиши заново, потом подмени" — не обновляем по одному, а создаём новые версии пачками
2. **Migration-файл:** заполняем шаг за шагом по ходу обсуждения
3. **Субагенты:** используем для исполнения, координатор (Claude) бережёт контекст
4. **context-editor:** один прогон в конце, когда всё консистентно
5. **pub/ → meta/marketing/**: публичные материалы, конференции, семинары = маркетинг
6. **research/ → архив**: исследования под старый продукт (транскрибация, МСБ-планёрки)
7. **prompts/ → .claude/prompts/**: все промпты из meta/ и dev/ в одно место. Отдельная сессия: "помедитировать" — какие скиллы для какого домена (knowledge, framework, meta), делить или не делить. Сейчас просто собрали чтобы не потерять
8. **marketing/ консистентна со scaffold**: eat your own dogfood — та же структура, что предлагаем клиентам. meta/ для svaib — живой экземпляр scaffold/management/
9. **knowledge/ publish:true → SEO/GEO на сайт**: механизм публикации через dev/ (задача разработки)
10. **meta/context/ отменён**: дублирует management/, всё что было "стратегия/vision/OKR" — это и есть management
11. **PROJECT.md в корне**: полное описание проекта для AI-сессий (не README, не CLAUDE.md — а "знание о проекте")
12. **viktor_profile.md в meta/**: на верхнем уровне meta/, не в подпапке
13. **Подход "чистый лист" для meta/**: НЕ переименовывать старые папки, а: (1) всё старое → meta/_old/, (2) создать чистую структуру, (3) заносить только то, что Виктор согласует. meta/_old/ — временная свалка, не архив
14. **Безопасность bash**: обновили settings.json — разрешены безопасные команды (git, ls, mkdir, mv и т.д.), деструктивные (rm, git reset, git push, git checkout --) требуют подтверждения
15. **CLAUDE.md грузится на каждом ходу**: не перечитывается с диска, но занимает место в контексте постоянно. При обновлении — максимально сократить, вынести знания в PROJECT.md
16. **.mcp.json в корне**: правильное место, Claude Code ищет его именно там (конвенция, как .gitignore). В .claude/ молча игнорируется
17. **meta/product/ — отдельная папка**: продуктовая стратегия (что строим, зачем, куда развиваем) — не в management/ и не в framework/. Три уровня: meta/product/ (стратегия) → framework/ (артефакт для клиента) → dev/ (реализация). Три параллельных блока в meta/: management (управление), marketing (рынок), product (продукт). Scaffold для клиентов обновляется параллельно
